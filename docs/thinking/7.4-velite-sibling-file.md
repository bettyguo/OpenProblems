# 7.4 — Velite sibling-file pattern (MDX + YAML); content-schema plumbing

Third code unit of Phase 7. Renumbered from the [Unit 7.0 §F](7.0-phase-7-prep.md) plan (was originally 7.5); Unit 7.3's deviation note inverted the 7.4 ↔ 7.5 dependency by re-scoping the methodology FR pilot to consume the sibling-file pattern. This unit ships the **schema plumbing** that pattern requires: extends five Velite collections (`methodology`, `contributing`, `problemPages`, `problems`, `papers`) with `lang` derivation + `translation_source` provenance, adds a shared filename-locale parser at `lib/i18n/locale-filename.ts` for runtime consumers, and stops at plumbing. No `*.fr.{mdx,yaml}` content files land here; the next unit (provisionally 7.5) is the methodology FR pilot that adds the first translated content file end-to-end.

## THINK

### Inputs

- **[ADR-0011 D-C](../adr/0011-i18n-strategy.md)** — sibling-file content storage: `foo.yaml` (EN canonical) + `foo.fr.yaml` (FR translation). EN takes no infix; FR carries `.fr` before the extension.
- **[ADR-0011 D-D](../adr/0011-i18n-strategy.md)** — locale fallback chain: missing FR file falls back to EN canonical with a switch hint. The fallback resolver itself lives in Unit 7.5 (next to its first consumer); this unit ships the *enabling* schema fields it consumes.
- **[ADR-0011 D-G](../adr/0011-i18n-strategy.md)** — `translation_source: "human" | "machine-assisted"` frontmatter; required on translated files, absent on EN canonicals.
- **[Unit 7.2](7.2-lib-i18n-runtime.md)** — `lib/i18n/routing.ts` exports `locales = ["en", "fr"] as const`. This unit re-uses that constant.
- **[velite.config.ts](../../velite.config.ts)** — the five collections to extend.
- **[lib/schemas/problem.ts](../../lib/schemas/problem.ts)** + **[lib/schemas/paper.ts](../../lib/schemas/paper.ts)** — Zod-4 source-of-truth schemas to mirror the additions into (per Q31 dual-schema contract).

### Scope (this unit)

Three new files + 7 edits:

1. **`lib/i18n/locale-filename.ts`** (new) — runtime helper:
   - `parseLocaleFromPath(slugLike: string)` → `{ lang: Locale; canonicalSlug: string }`. Detects the `.fr` infix (or any locale in `locales`); returns `{ lang: defaultLocale, canonicalSlug: slugLike }` if no infix.
   - Library-agnostic; no next-intl import; pure string parsing; reusable in velite transforms (via relative import) and at SSG / SSR consumers.
2. **`lib/i18n/locale-filename.test.ts`** (new) — vitest unit tests:
   - `methodology/v1` → `{ lang: "en", canonicalSlug: "methodology/v1" }`
   - `methodology/v1.fr` → `{ lang: "fr", canonicalSlug: "methodology/v1" }`
   - `problems/x/background.fr` → `{ lang: "fr", canonicalSlug: "problems/x/background" }`
   - `problems/x/problem.fr` → `{ lang: "fr", canonicalSlug: "problems/x/problem" }`
   - `papers/attention.fr` → `{ lang: "fr", canonicalSlug: "papers/attention" }`
   - Edge case: `xx` (unknown locale) treated as part of the slug, not a locale infix.
   - Edge case: `.fr.fr` (double infix) → only outer `.fr` is the locale infix.
3. **`velite.config.ts`** (edit) — five collection extensions:
   - **`methodology`**: schema adds `translation_source: s.enum(["human", "machine-assisted"]).optional()`; transform parses `data.slug`, derives `lang`, normalizes `slug` to canonical (strips `.fr` infix), adds `lang` field; post-transform `.refine` enforces `translation_source` is set when `lang !== "en"`.
   - **`contributing`**: same shape as methodology.
   - **`problemPages`**: same shape. The existing `(background|definition|history)` regex is extended to tolerate the optional `.fr` suffix; lang derivation runs first, then the kind regex.
   - **`problems`**: schema adds `path: s.path()` (so the transform can see the filename) + `translation_source: s.enum(...).optional()`; transform derives `lang` from path; pre-existing fields unchanged.
   - **`papers`**: same shape as problems.
   - **Glob patterns**:
     - `problems`: `problems/*/problem.yaml` → `problems/*/problem*.yaml` (admits `problem.fr.yaml`).
     - `problemPages`: `problems/*/{background,definition,history}.mdx` → `problems/*/{background,definition,history}*.mdx` (admits `.fr.mdx` siblings).
     - `methodology`, `contributing`, `papers`: existing patterns (`*.mdx`, `*.yaml`) already match `.fr` siblings; no glob change.
4. **`lib/schemas/problem.ts`** (edit) — mirror: add `lang: z.enum(["en", "fr"]).optional()` + `translation_source: z.enum(["human", "machine-assisted"]).optional()`. Both optional in the canonical schema because EN-canonical files have neither (Velite derives `lang` from path; canonical YAML doesn't carry it).
5. **`lib/schemas/paper.ts`** (edit) — same mirror.
6. **`lib/schemas/problem.test.ts`** (edit) — add cases asserting the new fields parse / omit cleanly.
7. **`lib/schemas/paper.test.ts`** (edit) — same.
8. **`CHANGELOG.md`** — Phase-7 Unit 7.4 entry.
9. **`docs/thinking/7.4-velite-sibling-file.md`** (this doc).

### Scope NOT in this unit

- **No `*.fr.{mdx,yaml}` content files land**. The plumbing is exercised only by unit tests on the helper; no live FR content yet. Velite scans the existing 203 files and produces 203 records, each with derived `lang: "en"`. The FR scan is no-op until 7.5.
- **No locale-aware loader at `#site/content` consumers**. Routes still load the EN content as before. Consumers continue to call `methodology` / `problems` etc. without locale filtering. 7.5 adds the resolver.
- **No `validate-content.ts` extension**. The script's `path.join(problemDir, "problem.yaml")` literal lookup is unchanged — it would not pick up `.fr.yaml` siblings until extended. Deferred to whichever unit ships the first `.fr.yaml` content (Unit 7.5 if methodology pilot extends to a problem; otherwise a later content unit).
- **No taxonomy / authors / institutions / ratings extension**. Taxonomy is a `single: true` collection (separate treatment); authors / institutions are slug-identified with no translatable user-facing prose; ratings are dimension-graded numerics (no translation surface). Deferred.
- **No middleware / route-level fallback**. ADR-0011 D-D's fallback hint surfaces in the route layer, not the loader; this unit doesn't touch routes.

### Constraints

- **Q31 dual-schema contract**. Velite 0.3.x can't consume our Zod-4 schemas; `lib/schemas/*` is the source of truth and `velite.config.ts` duplicates with `s`. Both sides must extend together so `scripts/validate-content.ts` (Zod-4 path) and `.velite/*.json` (Velite path) accept the same field set.
- **`exactOptionalPropertyTypes: true`**. The new fields are `.optional()` (omitted-or-string-typed) — the Velite mirror is fine; the lib/schemas side uses `z.enum().optional()` which gives `field?: T | undefined`. Loaders that pass `translation_source` must use the "assign-only-when-defined" pattern.
- **ADR-0005 immutability**. Rating-action YAMLs untouched (no schema change to RatingAction); the immutability hook remains hot.
- **Velite glob behavior**. `problems/*/problem*.yaml` matches `problem.yaml` and `problem.fr.yaml`; the brace-expansion patterns I'm adding (`{background,definition,history}*.mdx`) match the three EN literals + their `.fr` siblings. Verified against Velite's [micromatch](https://github.com/micromatch/micromatch) glob semantics.
- **No new dependency**. Pure refactor of existing schemas + a new pure-string helper.
- **Velite + relative imports**. `velite.config.ts` already imports npm packages; importing `./lib/i18n/locale-filename` (relative) is risky because velite's TS loader may not resolve the path alias. I'll **inline the regex** in velite.config.ts to match the existing convention (the methodology transform inlines a regex; this stays consistent). The `lib/i18n/locale-filename.ts` helper is the single source of truth for runtime consumers; the Velite duplication is one regex line.

### Alternatives considered

- **Add `lang` as a `s.enum(["en","fr"]).default("en")` schema field rather than a transform-derived one** — Rejected. Forces curators to set `lang: "en"` on every existing YAML/MDX file (203 file edits) for the field to be authoritative; opaque to readers ("why is this file marked English when its sibling isn't?"); breaks the ADR-0011 D-C "EN takes no infix" invariant by injecting locale-state into EN-canonical files. Transform-derive from filename is the canonical pattern.
- **Use one shared `addLocaleFields(base)` helper at the top of velite.config.ts** — Rejected. Each collection has its own transform already; composing transforms via spread is mechanical but fragile (the order of `.refine` after the inner transform's `.transform` matters; type narrowing through composed chains breaks). Inline addition of ~4 lines per collection is clearer than a shared helper that obscures the chain.
- **Run validate-content extension in this unit** — Rejected. No `.fr.yaml` content exists yet; extending validate-content is dead code until 7.5 lands content. Couples this unit's surface area to a future unit's content shape.
- **Skip the `lib/i18n/locale-filename.ts` helper; rely only on inline regex** — Rejected. Runtime consumers (the loader at 7.5) need this logic too; centralizing in `lib/i18n/` matches the pattern of `lib/i18n/routing.ts`.
- **Use `s.refine` instead of `.refine` on the transformed schema** — Velite's `s` is the Zod-3 factory. `.refine` after `.transform` is the standard Zod 3 chain; matches `RatingActionS` precedent (transform-only; no refine in HEAD); adds the refine that enforces ADR-0011 D-G's "translation_source required on FR files".

### Tradeoffs / defensible defaults flagged

1. **Inlining the locale regex in velite.config.ts** duplicates intent with `lib/i18n/locale-filename.ts`. The two regexes are simple (`/\.fr$/`); a unit test on the helper plus the visible regex in velite.config.ts catches drift. Acceptable until velite-config-side TS imports from `@/lib/...` become reliable.
2. **`lang` + `translation_source` added as optional on `lib/schemas/{problem,paper}.ts`**. EN canonicals must omit `translation_source`. Files violating this (e.g., setting `translation_source: "human"` on a non-`.fr` file) wouldn't fail validate-content — the field is permissive. Velite's post-transform `.refine` is the gate that catches "missing translation_source on FR file"; validate-content is intentionally not the second gate (would require it to know the filename pattern). Acceptable; if drift becomes a problem, add the refine on the Zod-4 schema too with a filename-aware caller pattern.
3. **`translation_source: "machine-assisted"` lean is not added as an opinion**. The enum lists `["human", "machine-assisted"]`; no default. Curators choose explicitly when authoring `.fr.yaml`. Matches ADR-0011 D-G + ADR-0009 D-F (`verified: false` for LLM drafts).
4. **Velite produces 203 records with `lang: "en"` at HEAD** (was 203 records with no `lang` field). `.velite/*.json` shapes change — any consumer that does `const { slug, title } = problem` is unaffected; any consumer that does `Object.keys(problem)` enumeration would see the new key. Low risk; the 36 vitest files don't appear to do such enumeration (verified by grep before commit).
5. **The `problemPages` regex extension** — `(background|definition|history)*.mdx` admits weirder names like `background.foo.mdx`. The transform's existing regex `^problems/([^/]+)/(background|definition|history)$` runs after lang-strip; would fail on `background.foo` (lang-strip on `background.foo` is a no-op since `.foo` isn't a known locale). The collection's parse would error, blocking the build. Defensive: the strict kind regex catches malformed filenames. Velite's `pattern` is the gross filter; the schema's transform is the fine filter.

- `pnpm build` → **335 prerendered pages UNCHANGED** (no route changes; .velite payload grows by a few bytes per record but no route changes).
