**Why a NEW column instead of editing `users.image` directly**:
ADR-0016 D-A established the displayName/name partition pattern —
user-controlled override sits in a NEW column; GitHub-derived
original stays canonical. Same pattern Phase 16 inherits:
`imageOverride` (user-controlled) + `image` (GitHub-derived). The
fallback chain `imageOverride → image → fallback initials`
mirrors `displayName → name → githubLogin → fallback`. Preserves
auth-side vs user-side write separation.

**Why 512-char cap**:
- Absolute URLs up to 2048 chars are theoretically allowed in
  browsers, but in practice avatar URLs (Gravatar / GitHub avatars
  / Vercel Blob) cluster under 256 chars.
- 512 chars is conservative + accommodates Vercel Blob's URL
  format (`https://*.public.blob.vercel-storage.com/<random-hash>-<filename>`).
- Phase 17+ may expand if signal demands.

### D-4. Image storage architecture (load-bearing — ADR-0017's keystone decision)

This is the **load-bearing decision Phase 16 surfaces**. Three
viable storage paths, each with distinct operational, cost, and UX
shape:

#### Option A: Vercel Blob with file-upload pipeline (recommended for ADR-0017)

- **Upload flow**: user selects image file in `/profile` form →
  client-side validation (MIME + size) → server-action POSTs to
  Vercel Blob via `@vercel/blob` SDK → Blob returns public HTTPS
  URL → server-action writes URL to `users.imageOverride` →
  redirect with success banner. Old override blob (if any) deleted
  in same transaction.
- **Storage cost**: ~$0.02/GB stored; ~$0.36/M reads (Vercel's
  public pricing). MVP scale: 100 users × 200 KB avatars = 20 MB
  = ~$0.0004/month. Negligible.
- **Operational env var**: `BLOB_READ_WRITE_TOKEN` (Vercel-provided
  on project linkage). Q69 operational-gate candidate parallel to
  Q54/Q55.
- **New dependency**: `@vercel/blob@1.x` (~30 kB transitive;
  server-only).
- **Pros**:
  - First-party (project is on Vercel; Blob is tightly integrated).
  - No CORS / IAM setup (token-gated server-side ops).
  - Same token works in dev (via `vercel link` + `vercel env pull`)
    and production.
  - Public URLs immediately CDN-served.
  - Cleanup-on-replace is a single SDK call.
- **Cons**:
  - Vendor lock-in (migrating to S3 / R2 means rewriting
    `lib/storage/`).
  - Operational concern: abandoned blobs (if delete-on-replace
    fails partway, orphan blob persists). Mitigation:
    delete-on-replace inside try/finally; abandoned-blob cleanup
    script as Phase-16 Class B follow-on.
  - New env var adds operational gate (Q69 candidate).

#### Option B: External URL allowlist (simplest; no storage)

- **Upload flow**: user pastes URL in `/profile` form →
  server-action validates URL format + allowlist match
  (gravatar.com, avatars.githubusercontent.com,
  raw.githubusercontent.com, *.imgur.com OR similar narrow set) +
  optional HEAD-request MIME validation → writes URL to
  `users.imageOverride` directly.
- **Storage cost**: $0 (no storage; users host elsewhere).
- **Operational env var**: none.
- **New dependency**: none.
- **Pros**:
  - Simplest architecture; smallest blast radius.
  - No vendor lock-in.
  - No abandoned-blob cleanup concern.
  - Smallest phase scope (5-6 units instead of 9).
- **Cons**:
  - Most users have no usable image URL handy. Power-user-only
    feature.
  - External URLs can rot, 404, or change content (allowlist
    domains mitigate but don't eliminate).
  - User cropping / sizing happens before they paste the URL; no
    in-app affordance.
  - Phase 17+ would still need to add upload pipeline for
    non-power-users; net more total work split across phases.

#### Option C: S3 / R2 (cheapest at scale; complex)

- **Upload flow**: like Option A but using AWS S3 or Cloudflare R2
  via the appropriate SDK.
- **Storage cost**: S3 ~$0.023/GB stored + reads; R2 ~$0.015/GB
  stored + zero egress. Cheaper at scale than Vercel Blob.
- **Operational env vars**: AWS_ACCESS_KEY_ID + AWS_SECRET_ACCESS_KEY
  + S3_BUCKET + AWS_REGION (or equivalent for R2).
- **New dependencies**: `@aws-sdk/client-s3` (~120 kB transitive)
  OR R2 equivalent.
- **Pros**:
  - Cheapest at scale.
  - Provider-portable (S3 API is widely supported).
- **Cons**:
  - Most complex setup (IAM policy + bucket policy + lifecycle
    rules).
  - Multi-env-var operational gate.
  - Heaviest dependency.
  - Vercel Blob is already first-party for this project.

**Recommendation**: **Option A (Vercel Blob)**. Rationale:
- Phase 16 ships the **natural user-flow surface** (file upload
  from computer) rather than the power-user-only URL surface.
- First-party advantage on Vercel.
- Single env var; small dependency.
- Storage cost negligible at MVP scale.
- Abandoned-blob cleanup is a real but tractable follow-on (Class
  B item in Unit 16.6 hygiene).
- ADR-0017 D-? can document Option B (URL allowlist) as Phase 17+
  optional COMPOSE option (user can EITHER upload OR paste URL),
  but Phase 16 ships the upload path only to keep scope tight.

**ADR-0017 records full deferral matrix**. Override path (e.g.,
"do URL allowlist instead") regenerates Unit 16.1's ADR + Unit
16.2's schema (text column shape unchanged; storage column
semantics shift from "Blob URL" to "allowlisted URL").

### D-5. Upload pipeline scope (Phase 16; Vercel Blob path)

Per ADR-0017 D-? (anticipated):

1. **MIME validation**. Server-action checks `file.type` against
   `["image/jpeg", "image/png", "image/webp"]`. **SVG explicitly
   excluded** — XSS surface (SVG can embed `<script>` tags).
2. **File size cap**. 2 MB. Server-action checks before Blob
   upload. Vercel Blob accepts up to 500 MB but bandwidth at SSR
   time matters; 2 MB is generous for avatar use case (typical
   avatar PNG ~50-200 KB; 2 MB room for high-res photos
   pre-resize).
3. **Dimensions**. **NOT validated Phase 16**. CSS `object-fit:
   cover` + fixed display sizes (48 px in pill; 64 px on header;
   128 px on `/u/{handle}`) handle the rendering. Phase 17+ may
   add server-side resizing if signal demands.
4. **EXIF stripping**. **NOT performed Phase 16** — defer to Phase
   17+ as **Q70 candidate** (privacy concern for users uploading
   personal photos with embedded GPS metadata).
5. **Content moderation**. **NOT performed Phase 16** — couples to
   Q68 bio moderation. Phase 17+ if abuse signals.
6. **Cropping UI**. **NOT provided Phase 16** — user crops their
   image before uploading (existing OS / browser tooling). Phase
   17+ may add `react-easy-crop` or similar if signal demands.
7. **Delete-on-replace**. Old `imageOverride` URL (if any) →
   parse Vercel Blob storage key → call `del()` → only then write
   new URL. Wrapped in try/finally so failure to delete old blob
   doesn't block writing new URL. Orphan tolerated until cleanup
   script lands (Class B follow-on).
8. **Clear-by-empty-submit**. Submitting form with empty file
   input + no URL = clear override (writes NULL to
   `imageOverride`; deletes old blob). Mirrors Phase 15's "delete
   displayName by saving empty" pattern.

### D-8. Validation + sanitization model

Per ADR-0017 D-? (anticipated):

1. **MIME**: `image/jpeg`, `image/png`, `image/webp` only. SVG
   excluded. Server-action checks via `file.type` (browser-
   provided) AND first-bytes magic-number check (defense-in-depth
   against forged MIME type).
2. **Size**: 2 MB max. Client-side `file.size` check + server-side
   re-check.
3. **Length cap on URL** (post-upload): 512 chars. Same enforcement
   as text fields (`MAX_IMAGE_URL_CHARS` constant).
4. **URL format**: must start with `https://` AND match Vercel Blob
   public URL pattern `https://*.public.blob.vercel-storage.com/...`
   (when storage path is Blob).
5. **No regex content filter**. Phase 16 trusts the upload (storage
   is trusted via Blob token; content is unscrutinized). Phase 17+
   may add content moderation (Q68 expansion).
6. **No uniqueness check on `imageOverride` URL**. Multiple users
   can technically point to the same image (e.g., team-shared
   default). Acceptable.
7. **Delete-on-replace** wraps the blob lifecycle.
