- **Pattern stays XSS-validatable**: slug character class
  remains `[a-z0-9-]+` (the regex IS the validation per APPEND-

  href. `[[scalable-oversight|the alignment frontier]]` still

  authoring workflow benefits naturally from alias syntax —
  prose flows better with "the alignment frontier" or "this
  problem" than the raw kebab-case slug. Curator content with
  multiple wikilink references to the same problem in one

  evidence** of `[[slug|display]]` usage (Phase 38 noted same);
  Phase 46 is demand-signal-relaxed per the property-validation
  framing precedent set by Phase 42-44.
- **Closes the second-oldest remaining APPEND-D-L deferral**:

### D-3. Multi-anchor alias shape anticipated (Unit 46.1)

D-A. **`WIKILINK_PATTERN` regex evolution** (in-place edit of
`lib/markdown/extensions/wikilinks.ts`):

```ts
// Before (Phase 38 ship through Phase-45 close):
const WIKILINK_PATTERN = /\[\[([a-z0-9-]+)\]\]/g;

// After (Phase 46 ship):
const WIKILINK_PATTERN = /\[\[([a-z0-9-]+)(?:\|([^\]\n]+))?\]\]/g;
```

- Group 1 = slug (unchanged `[a-z0-9-]+`).
- Group 2 = optional display text (`[^\]\n]+`; any non-`]` non-
  newline chars; one or more). The `\n` exclusion ensures
  alias doesn't span a paragraph break + bounds prose-ambiguous
  cases.
- Non-capturing group `(?:...)? ` makes the whole alias clause
  optional. Backwards-compatible: every existing `[[slug]]`
  occurrence still matches identically with group 2 undefined.

D-B. **`rehypeResolveWikilinks` plugin body update**:

```ts
// Inside the match-processing loop:
const slug = match[1];
const display = match[2]; // NEW: may be undefined
if (slug === undefined) continue;

newNodes.push({
  type: "element",
  tagName: "a",
  properties: { href: `/problems/${slug}` },
  // Display defaults to slug when no alias provided. Whitespace
  // in alias preserved (no auto-trim; curator controls); empty
  // alias `[[slug|]]` falls back to slug (display ?? slug).
  children: [{ type: "text", value: display ?? slug }],
});
```
