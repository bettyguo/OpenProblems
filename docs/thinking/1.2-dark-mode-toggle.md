# Unit 1.2 — Dark-mode toggle (persisted)

> Stage 1 (THINK) artifact.

## Problem restatement

§13 Phase 1 deliverables list "Dark mode toggle, persisted." Today (Unit 0.4) dark-mode activation is `@media (prefers-color-scheme: dark)` — purely OS-driven. This unit:

1. Switches activation to a `.dark` class on `<html>`.
2. Adds an SSR-safe theme provider (system / light / dark) with localStorage persistence.
3. Adds a button that cycles `light → dark → system → light` with accessible labels.
4. Closes OPEN_QUESTIONS Q18 (dark-mode activation mechanism).

## Inputs

- §13 (Phase 1 deliverable list).
- §10.1 (dark mode is a "true second design," not a tinted invert).
- OPEN_QUESTIONS Q18 (Phase-0 follow-on: swap `@media` → `.dark` + initialization script).
- Unit 0.4 `app/globals.css` — `:root` for light, `@media (prefers-color-scheme: dark) :root` for dark. Both blocks define the same variable names; this unit shifts which selector activates the dark values.
- Unit 0.3 + 0.4 — `next-themes` is the conventional Next.js library; `class-variance-authority` and `lucide-react` already installed.

## Constraints

- **No FOUC.** SSR must render the correct class on `<html>` before paint. `next-themes` ships a small inlined script for this; `suppressHydrationWarning` on `<html>` is required.
- **System preference is the default.** First visit reads `prefers-color-scheme` and applies it without writing localStorage; only an explicit user choice persists.
- **Cycle, not toggle.** Per §10.1 + accessibility convention: `system → light → dark → system`. A binary toggle hides the system option and forces users into an explicit choice.
- **Keyboard accessible.** `<button type="button">` with visible focus ring (`focus-visible:ring-2 ring-ring`), `aria-label` reflecting current theme, lucide icons set `aria-hidden`.
- **Honour `prefers-reduced-motion`.** Unit 0.4's global guard catches this; no animation library involved in the toggle.
- **Don't add the toggle to every page yet.** The site header arrives in Unit 1.10 (landing). For Phase 1.2 the toggle sits on the home page only (smoke proof) and the same component will be reused in the header later.

## Alternatives considered

1. **`next-themes` + class strategy + 3-state cycle button** (chosen).
2. **Hand-rolled provider + localStorage hook + boot script.** ~80 lines of code that reinvents `next-themes` poorly. Rejected.
3. **CSS-only `@media` + `[data-theme]` override.** No persistence; defeats the "persisted" requirement of §13.
4. **System-only (no toggle).** Defeats §13 explicitly.

## Edge cases & failure modes

1. **`<html>` server-render mismatch.** Pre-hydration HTML doesn't know the user's chosen theme. `next-themes` injects an inline `<script>` that runs synchronously before paint, reads localStorage, and sets the class on `<html>`. Requires `suppressHydrationWarning` on `<html>` because React will see a different `className` than the server-rendered HTML.
2. **Storage-disabled browsers.** `next-themes` degrades gracefully — system preference wins. Toggle still works in-session; choice doesn't persist across refresh.
3. **Server components reading theme.** RSCs can't read localStorage. Components that need theme branching (e.g., the future RatingRadar's stroke color) consume CSS variables, not theme strings — so RSC vs CSC distinction doesn't matter.
4. **Two-tone foundation flicker.** Mitigation: the boot script runs *synchronously* before first paint; flicker only happens if a contributor adds a slow client component above `<body>`.
5. **Existing `@media (prefers-color-scheme: dark)` block.** Replace with `.dark { ... }`. `next-themes` with `attribute="class"` adds `.dark` to `<html>` based on user preference (or system if `system` is chosen).
6. **Test environment.** Vitest's jsdom doesn't have `matchMedia` by default. If a test ever exercises theme code, mock `matchMedia` — skip for Phase 1.2 (no tests for the toggle yet).
