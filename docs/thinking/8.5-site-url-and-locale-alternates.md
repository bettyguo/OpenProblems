### Constraints

- **Backward-compat for `SITE` importers**: `lib/sitemap/build-sitemap.ts` and `lib/digest/rss.ts` were the export points. Re-export `SITE` from these so existing `import { SITE } from "@/lib/sitemap/build-sitemap"` (in `app/robots.ts`, tests) keeps working. Future code can import directly from `lib/site-url`.
- **Env-fallback semantic preserved**: the new `SITE` value falls through to `"https://llm-openproblems.org"` when `NEXT_PUBLIC_SITE_URL` is unset (Node `??` operator). Existing tests that compare against the literal still pass.
- **Home-route URL formatting**: `app/[locale]/page.tsx` (the home shadow under Unit 8.1) renders at `/en` and `/fr` (no trailing slash); the bare canonical is `/` (trailing slash). The helper builds `${SITE}/en` + `${SITE}/fr` for the home alternate, matching the SSG output.
- **Methodology + contributing version filter (`m.lang !== "en"`)**: kept. Prevents duplicate URLs when FR siblings (`v1.fr.mdx`) join the Velite collection; the alternates block already covers the FR variant.

### Alternatives considered

- **Skip the consolidation; just add universal alternates**: rejected. SITE call-site count is at the 5+ threshold and the env-fallback divergence between lib and pages is a real foot-gun (preview deploys would inconsistently use placeholder vs real URL). Better to consolidate now while the surface is small.
- **Inline the alternates expansion into each route push without a helper**: rejected. Duplicates the `alternates: { languages: { en: ..., fr: ... } }` block ~10 times; the helper is a 4-line consolidation that reads better.
- **Explicit `LOCALE_ALTERNATE_ROUTES` Set as 8.0 prep framed it**: rejected. The set would be congruent with "all routes" today; an explicit Set adds maintenance overhead without expressive benefit. The helper function makes the universal-application invariant self-evident.
- **Move `SITE_URL` env-fallback pattern into `site-url.ts` AND keep `SITE` as the hardcoded variant**: rejected. Two near-identical exports invite divergence. Single `SITE` with env-fallback is the canonical pattern; `NEXT_PUBLIC_*` is a stable Next.js contract.

### Edge cases & failure modes

1. **`process.env.NEXT_PUBLIC_SITE_URL` set to a URL with trailing slash**: would double-slash in URL concatenation. Mitigation: document the contract (no trailing slash) in `lib/site-url.ts`. Could add a runtime check but kept simple.
2. **Test files import `SITE` and compare against literal `"https://llm-openproblems.org"`**: tests don't set the env var, so the fallback fires; tests pass. Preview-deploy testing would need a separate env-aware fixture.
3. **`app/robots.ts` and tests import `SITE` from `lib/sitemap/build-sitemap`**: re-export preserves this. No import-path changes required for existing consumers.
4. **Home route URL formatting (`/` vs no-trailing-slash)**: `entryWithAlternates("/")` produces canonical `${SITE}/` and alternates `${SITE}/en` + `${SITE}/fr`. The test explicitly covers this asymmetry.
5. **Citation URL semantic change in paper page**: was `${SITE_URL}/papers/${paper.id}`; now `${SITE}/${locale}/papers/${paper.id}`. The locale segment is correct post-Unit 8.1 (every URL is locale-prefixed under `localePrefix: "always"`). Curators citing papers from `/fr/` get a French-prefixed URL — accurate to the page they're on.
