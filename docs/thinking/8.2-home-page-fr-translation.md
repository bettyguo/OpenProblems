# 8.2 — `app/[locale]/page.tsx` FR hero translation + `messages.home.*` keys

Second code unit of Phase 8. Realizes D-8 + D-11 from [Unit 8.0 prep](8.0-phase-8-prep.md). Closes one of the Class B follow-ons surfaced in Unit 7.9: a locale-aware home page with FR-translated hero copy. Builds on Unit 8.1's bulk migration (the home route is now at `app/[locale]/page.tsx`).

## THINK

### Inputs

- **[Unit 8.0 D-8](8.0-phase-8-prep.md)** — home-page i18n message keys deferred to this unit.
- **[Unit 8.0 D-11](8.0-phase-8-prep.md)** — FR translation of hero copy; curator-track-pilot scope (small, focused, mirror of the methodology FR pilot from Unit 7.5).
- **[ADR-0011 D-A](../adr/0011-i18n-strategy.md)** — `next-intl` runtime; `messages/<locale>.json` lookup format; ICU MessageFormat for plurals/interpolation.

### Scope (this unit)

1. **`messages/en.json`** — add `home.*` namespace with hero h1, hero p, two CTA button labels, three section h2s, two "all X →" links, methodology blurb p, methodology CTA, and aria-labels (nav, ChartTableSwitch, DomainMap).
2. **`messages/fr.json`** — FR translations for every `home.*` key.
3. **`app/[locale]/page.tsx`** — replace hardcoded EN strings with `getTranslations("home")` lookups. Pass aria-labels as props to `ChartTableSwitch` / `DomainMap` / `<nav>`.
4. **No content delta** — `messages/*.json` are i18n catalogs (lib-side), not `content/` files. `translation_source` frontmatter does NOT apply (ADR-0011 D-G scopes provenance to `*.<locale>.{yaml,mdx}` content files only; per Velite's refine in `velite.config.ts`).

### Scope NOT in this unit

- **`components/recently-rated/`** internal strings ("No rating actions yet — see /contributing.") — component-side; component is rendered from multiple call sites; deferred to a component-specific i18n pass when that component gets a dedicated unit.
- **`StatusPill` localization** (open / partially-solved / converging / solved / retired) — component-side enum mapping; deferred.
- **Site-header nav labels** — already in `messages.nav.*`; SiteHeader currently uses hardcoded strings (the toggle's `useTranslations` constraint per Unit 7.6); will fold into Unit 8.4 alongside the LocaleToggle aria-label work + HTML shell migration.
- **Methodology v1.0 link label text** ("Read methodology v1 →") — the "v1" reference is locale-stable; the labels around it translate.
- **Tags / domain titles** — content-side; curator-track per Q51.

### Constraints

- **ICU MessageFormat compatibility**: simple string lookups only; no plurals or interpolation needed for the home-page copy.
- **Em-dash preservation**: hero p uses `—` (em-dash) as a parenthetical separator; preserved in both locales.
- **Arrow glyph preservation**: "→" stays in both locales (typographic, not lexical).
- **Aria-labels translated in target language**: `aria-label="Browse by domain"` becomes the FR equivalent under `/fr/`. Same applies to `ChartTableSwitch ariaLabel` and `DomainMap ariaLabel`.
- **Dimension names ("Difficulty, Saturation, Urgency, Value, Industry Call") translated**: these are user-visible UI strings on the home page hero. FR equivalents: "Difficulté, Saturation, Urgence, Valeur, Demande de l'industrie" — matches Unit 7.5's methodology FR translation choices.
- **"Browse problems" / "Read the methodology"** CTA labels translated.

### Alternatives considered

- **Leave hero copy hardcoded; only translate aria-labels**: rejected. The hero is the most prominent surface on the site; if it stays EN on `/fr/`, the Bilingual thread's rollout completion is unconvincing.
- **Use `useTranslations` (client-side) instead of `getTranslations` (server-side)**: rejected. The home page is a server component (no `"use client"`); `getTranslations` is the canonical server-side accessor.
- **Inline FR strings as a const in `page.tsx`**: rejected. ADR-0011 D-A pins `messages/<locale>.json` as the lookup format; inline strings would diverge from the contract.
- **Defer FR translation to a curator-track pass**: rejected. Unit 8.0 explicitly scoped this as Unit 8.2 D-11. The home page is a small, focused pilot that exercises the `getTranslations` server pathway end-to-end.

### Edge cases & failure modes

1. **`getTranslations("home")` returns a function that throws on missing keys**: the schema's strict mode would surface this at build time. Catch: validate every key used in `page.tsx` against `messages/{en,fr}.json` before commit.
2. **FR string contains characters that need escaping in JSX**: `'` (apostrophe) is common in French; JSX accepts it inside string literals from t() but not inside literal JSX text. Since strings come via `t()`, they're already inside expression contexts; no escaping needed.
3. **Hydration mismatch**: `getTranslations` is server-side; the client doesn't re-resolve. No hydration risk for these strings.
4. **`ChartTableSwitch ariaLabel` / `DomainMap ariaLabel` accept string props**: passing `t("...")` results works the same as passing literal strings. No interface change.
