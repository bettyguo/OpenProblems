  to prove the Phase-2 surface meets the §13 bar, not to add features.

  Unit 2.12 aggregate rollups) are *deliverables*, not *gate criteria*
  per §13. They land in separate units after this one.
- **CI integration of `audit-content` lands here.** Unit 2.11 wired
  the script; this unit promotes it from "advisory" to required by
  adding the `pnpm audit-content` step to `.github/workflows/ci.yml`
  alongside `validate-content`. The audit currently has 0 errors, so
  the CI gate passes immediately.
- **Visual-regression baselines: refresh the existing one, add one
  per new dynamic route.** The Phase-1 baseline (RatingRadar on
  `/problems/hallucination-reduction`) is captured; Phase-2 adds
  `/papers/[id]`, `/authors/[slug]`, `/institutions/[slug]` to the
  visual surface. Baselines for each go in
  `e2e/smoke.spec.ts-snapshots/`.
- **Per-platform baseline coupling carried over from Unit 1.12.** The
  Linux/CI baseline still must be captured on first CI run; the
  win32 baselines committed here are local-only. The same one-shot
  `--update-snapshots` follow-on plan applies.
- **Unit tests for the audit.** Unit 2.11 deferred audit tests to
  this unit. Add `lib/content/cross-link-audit.test.ts` with fixture
  cases covering each check + a "real `content/`" sanity test.

### Edge cases

- **Acceptance gate without 2.9 / 2.10 / 2.12 done.** §13's gate
  criteria don't require those deliverables — only the audit and the
  visual baselines. Note the gap in the summary so the reader knows
  Phase 2 closes "gate-met, three deliverables in flight."
- **Audit unit tests.** Use small synthetic fixtures under
  `test/fixtures/audit/`: one happy fixture (clean), one fixture
  with a dangling paper→problem ref, one with an asymmetric
  related_problems edge, etc. Mirrors the validate-content fixture
  pattern from Unit 0.7.
- **Lighthouse on `/papers` (the index).** Index pages with client-
  side filter+sort have a slightly different SEO profile than detail
  pages; should still pass ≥ 0.95 on perf/a11y/SEO given the bundle
  sizes from Unit 2.8.
- **Playwright nav assertion targets.** Need a stable selector on
  `/papers/[id]`'s page. The h1 (paper title) is stable; use it.
  `/authors/[slug]` and `/institutions/[slug]` are not in the §13 nav
  path — leave them out of the Playwright traversal, cover them via
  Lighthouse only.

### Alternatives considered

- **Wait for 2.9 / 2.10 / 2.12 before running the gate.** §13's gate
  criteria don't require them. Running the gate now produces a
  meaningful "Phase 2 acceptance" milestone, even with three
  deliverables still in flight.
- **Move audit-content from Phase-2 CI gate to Phase-2 acceptance-
  only.** Rejected — `audit-content` is fast (~1 s) and adds zero
  flakiness. Promoting to CI gate ensures regressions get caught.

### Tradeoffs / defensible defaults flagged

1. **Gate-met without 2.9/2.10/2.12.** Documented in the unit
   summary; not a deviation from §13.
2. **Visual baselines are win32-local; Linux baseline lands on
   first CI run.** Carried over from Unit 1.12's policy.
3. **No audit fixture coverage for every check yet.** Will add the 4
   highest-value cases: clean, dangling paper→problem, dangling
   author→institution, asymmetric related_problems. The full 7-check
   coverage is a smaller follow-on unit when fixture cost is justified.
