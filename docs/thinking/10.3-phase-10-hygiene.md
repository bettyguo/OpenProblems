## THINK

### Class A — In-flight Phase-10 items (2)

Phase-10-specific items the unit work surfaced but didn't close.

1. **Profile-page Playwright smoke test**. Unit 10.2 shipped the
   profile-page surface but no Playwright test exercising:
   - Signed-out → redirect to `/api/auth/signin/github?callbackUrl=...`
     (3xx response; `Location` header matches).
   - Signed-in (mocked session cookie) → page renders watched list +
     sign-out form + avatar.
   - Empty state → "Browse problems →" link present.
   Deferred to a future Phase-10.x or Phase-11 unit. The unit test
   coverage gap is acceptable in isolation because (a) `auth()` is
   exercised end-to-end by SiteHeader's `safeAuth()` in every existing
   route's prerender, (b) `getWatchedSlugs` is a thin Drizzle wrapper
   tested by the type system, (c) `<WatchlistToggle>` reuse is covered
   by Unit 9.6's tests. The visible-state Playwright test is the only
   "real" coverage missing.

2. **LHCI enrolment for `/en/profile` + `/fr/profile`**. The 19-URL
   `lighthouserc.json` is unchanged through Phase 10. Profile pages
   are `dynamic = "force-dynamic"` so LHCI would need to either (a)
   bypass the auth redirect (test the signed-out → 3xx path) or (b)
   inject a mock session cookie via `lhci collect --extra-headers` to
   measure the signed-in render. Both options have a Q54-class
   operational gate (need a deployed sign-in path or a CI-provisioned
   mock session). Defer until first observed LHCI run motivates the
   enrolment.

### Class B — Phase-10-specific follow-ons (8)

Items that survive Phase 10 because they're either scope-creep (out of
the "consolidation" framing per Unit 10.0 D-1), waiting on an
architectural unblock that wasn't granted this phase, or tactical
hygiene whose threshold hasn't been crossed.

1. **Public profile rendering at `/[locale]/u/[handle]`** (Unit 10.0
   D-3 alternative). Allows viewing another user's public-facing
   profile (curator-of-record attribution, public watchlist count,
   etc.). Phase 11+ scope; no observed ask yet.

2. **User-editable fields** — display-name override (today the user
   sees `session.user.name` from GitHub, no in-app edit affordance);
   user-defined bio; preferred locale override (currently driven by
   `NEXT_LOCALE` cookie / URL prefix); notification preferences
   (couples to Class C item 3 below).

3. **Per-user statistics surface** — rating actions authored (joins
   to file-system `rating.curator` via `users.githubLogin`); paper
   contributions (joins to file-system author records); comment
   activity (couples to Phase-6 Discussions). Phase 11+ scope when
   the curator-of-record file-system data needs a per-user
   dashboard view.

4. **Auth-aware middleware-based route protection**. Phase-9 Class B
   item 12 partially resolved in Phase 10 (server-component-level
   protection landed via `auth()` + `redirect()` in the profile page).
   Middleware-based protection stays deferred until 2+ protected
   routes exist (matches Unit 10.0 D-4 lean rationale). Phase 11+
   scope when (e.g.) `/admin` lands.

5. **Profile-page styling polish** — mobile layout audit (header may
   overflow at < 480px when display name is long); dark-mode avatar
   border tint; sign-out-button confirmation modal (no-confirm today;
   single-click sign-out). Visual-regression baselines for
   `/en/profile` and `/fr/profile` not captured (Playwright spec
   change required; deferred).

6. **Per-user discussion-activity surface** — display the user's
   recent Giscus comments on watched problems. Couples to Phase-6
   Discussions; would query the same GraphQL surface as Unit 6.5/6.6
   but filtered by `discussion.author.login === user.githubLogin`.
   Phase 11+ scope; out of "consolidation" framing.

7. **Profile photo upload**. Today's avatar is GitHub-sourced
   (`session.user.image`); custom upload requires blob storage
   (S3 / Vercel Blob / etc.) which is a Phase-12+ architectural
   commitment.

8. **Redundant `isWatched()` query inside reused `<WatchlistToggle>`
   on profile page**. Unit 10.0 D-7 accepted this redundancy as
   tactical-not-architectural (single indexed SQL lookup; small list
   size). Optimization landing pad: a `WatchlistToggle` prop
   `initialWatched?: boolean` that skips the lookup when the parent
   already knows the state. Phase 12+ optimization when profile-page
   latency is observable.

### Risk surface at HEAD `6ea7a4f`

- **`AUTH_SECRET` must be set in CI / Vercel / preview** (Phase-9
  Class A item 3; carried through Phase 10). Profile page joins the
  exposure since `auth()` is called inside the route handler.
- **End-to-end profile exercise blocked on Q54 + Q55** (Class C
  items 1 + 2). Architectural surface complete; operational unblock
  pending.
- **Profile-page redirect baked into prerender output**. Build summary
  lists `/en/profile` + `/fr/profile` under SSG ●; `dynamic =
  "force-dynamic"` overrides at runtime but the build artifact
  contains a cached-redirect HTML. Per Next.js convention this is
  request-time-overridden; no observable user impact.
- **GitHub avatar URLs are external**. Profile page renders bare
  `<img>` against `https://avatars.githubusercontent.com/...`. If
  GitHub rotates avatar URLs (rare; account-rename only), the image
  stays cached in user's browser; force-refresh recovers. No
  architectural impact.
