# 2.7 — Paper detail page `/papers/[id]`

## THINK

### Inputs

- **§13 Phase 2 deliverable.** "Paper / Author / Institution detail
  pages." Author + Institution shipped in Unit 2.3; this unit lands the
  Paper detail page.
- **2.0 plan.** "§9 four-block layout (header, contributions table,
  authors+institutions, BibTeX-style cite). Replaces Unit 0.10 stub."
  No formal §9 ten-block layout exists for papers (that spec belongs to
  the problem page); the four-block layout is this project's
  convention.
- **`loadPaper(id)` (Unit 2.2).** Returns `{ paper, authors,
  unresolvedAuthorSlugs, institutions, unresolvedInstitutionSlugs,
  contributions: [{ contribution, problem }] }`. The detail page is a
  thin RSC consumer.
- **`allPaperIds()` (Unit 2.2).** Powers `generateStaticParams` so all
  30 paper pages prerender at build time. Bundle stays small (≈ 1 kB
  per route over the shared baseline).

### Constraints

- **RSC only, no client JS.** Matches Unit 1.6's problem detail page
  and Unit 2.3's author/institution pages.
- **Unresolved slugs render visibly.** Both `unresolvedAuthorSlugs`
  and `unresolvedInstitutionSlugs` are first-class output of the
  loader. Render them as monospace text (not links), so the cross-
  link audit (Unit 2.11) can see them in screenshots without needing
  to traverse the data.
- **`contribution.problem === undefined` is the cross-link audit's
  problem to fix, not the page's.** Render the row with the raw
  `problem_slug` (mono, not a link) and a small "(unresolved)" tag.
- **Citation block convention from Unit 1.6.** Problem detail's
  Citation section uses a `<pre>` BibTeX-ish stanza. Paper detail
  should do the same, using the paper's `id`, `title`, `authors`,
  `year`, `arxiv_id`, `doi`. When `authors[]` is empty (the Phase-2
  norm — see Units 2.4–2.6), fall back to `"LLM OpenProblems contributors"`.
- **No `score:` rendering when absent.** §15.6 — render `—` for
  missing benchmark scores. Don't fabricate even in the absence of
  Velite-side validation.
- **No new tests required.** `loadPaper` is already tested in
  `lib/content/load-entities.test.ts`. The page is a thin display
  layer; visual regression on a representative paper is Unit 2.11/
  Unit 2.13's job.

### Four-block layout

1. **Header** — breadcrumb (`Papers`), title (h1), one-line metadata
   row (`{venue} · {year} · arxiv / doi / github` badges).
2. **TL;DR + contributions table** — TL;DR paragraph; table with
   columns Problem · Benchmark · Metric · Score · Evidence. Rows
   resolve to `/problems/[slug]` links when the problem exists; show
   raw monospace slug + `(unresolved)` otherwise. `score` column
   shows `—` when absent.
3. **Authors + Institutions** — two side-by-side lists. Resolved
   slugs link to `/authors/[slug]` and `/institutions/[slug]`;
   unresolved slugs render as monospace `@slug` with a small
   `(unresolved)` tag. Empty `authors[]` (the current norm) renders a
   single italic placeholder line directing readers to the upcoming
   author-backfill unit.
4. **Citation** — BibTeX-style `<pre>` with `@misc{op-paper-{id}, …}`,
   echoing Unit 1.6's problem-citation block. Includes the canonical
   `{NEXT_PUBLIC_SITE_URL}/papers/{id}` URL and the venue + year if
   present.

### Edge cases

- **`paper.authors[]` and `paper.institutions[]` both empty.** Current
  norm for 100% of Phase-2 papers. The block renders an explanatory
  italic line on each side rather than going blank.
- **`paper.tldr` is `"[TLDR pending human review]"`.** Render
  verbatim; the sentinel is a feature, not a bug.
- **`paper.doi` or `paper.github` absent.** Conditional rendering;
  skip the badge.
- **Unknown `paper.id`.** `loadPaper(id)` returns `null`;
  `notFound()` triggers the global 404.
- **Multiple contributions to the same problem.** Render all rows;
  the cross-link audit verifies the union sense.

### Tradeoffs / defensible defaults flagged

1. **No paper-side viz.** No SaturationCurve or per-paper chart in
   this unit — those land in Phase 3 with the rating-history work.
2. **No "Related papers" block.** Could be derived from
   `paper.contributions[*].problem_slug` shared with other papers,
   but that's a Phase-3 graph concern. The contributions table
   already surfaces the primary cross-link.
3. **No metadata customization.** Root-layout `metadata` propagates
   the project title + description; per-page `generateMetadata` lives
   in Unit 2.13 (acceptance gate, where SEO scoring matters across
   all dynamic routes).
4. **Static prerender all 30 papers.** Bundle impact: ≈ 30 × 1 kB
   route code = +30 kB of static HTML; negligible at the framework
   chunk level.
