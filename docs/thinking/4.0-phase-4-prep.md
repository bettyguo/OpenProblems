### Phase-4-blocking decisions resolved here

#### D-1. DomainMap rendering target

Decision: **SVG**, not Canvas, not HTML/CSS.

Rationale:
- All 4 existing vizes are SVG; operator familiarity + a11y plumbing precedent (`role="img"`, `aria-describedby`, `<desc>`) carries over.
- Node count is bounded: 5 domains × ~3 subdomains × 10 problems = ~30–40 visible nodes today. SVG handles 1000+ force-layout nodes comfortably; we're nowhere near the ceiling where Canvas wins.
- Canvas trades a11y for raw render speed; we don't need the speed and we do need a11y for the §13 acceptance gate.
- `d3-force` computes positions in JS regardless of paint target — the SVG-vs-Canvas choice is purely the render pass.
- HTML/CSS rejected: positioning force-layout nodes via CSS transforms is feasible but the line/curve edges between nodes require an SVG `<path>` regardless.

Recorded as ADR-0007 in Unit 4.11.

#### D-2. D3 import surface

Decision: **tree-shaken sub-packages**, not the full `d3` umbrella.

| Package         | Purpose                       | ~gz size (rough) |
|-----------------|-------------------------------|------------------|
| `d3-force`      | force simulation              | ~10 KB           |
| `d3-selection`  | declarative DOM updates       | ~5 KB            |
| `d3-scale`      | composite-rating → node radius | ~6 KB           |
| `d3-zoom`       | pan/zoom (optional; if scoped in) | ~7 KB        |

Plus `@types/d3-*` dev deps for each. Projected client-bundle bump:
**~20–25 KB gz across the 3–4 imported sub-packages** (~30 KB if zoom
ships). First Load JS shared chunk is 103 KB today → projected ~125–135
KB after Phase 4. Within the perf-budget envelope (Lighthouse perf ≥
0.95 on desktop has comfortable headroom at current bundle sizes).

Rejected: the umbrella `d3` package pulls ~150 KB gz of unused modules
(geo, hierarchy, axis, brush, transition for animations we don't need).

Implementation pattern: `import { forceSimulation, forceLink, forceManyBody, forceCenter } from "d3-force";` — same shape as other named-imports in the project.

#### D-3. Node taxonomy levels

Decision: **3-level hierarchy** (domain → subdomain → problem).

Visible-by-default state: subdomain nodes collapsed (showing only
domain + problem). Click on a domain expands its subdomains; click
again collapses. Keeps the default view at ~15 visible nodes
(5 domains + 10 problems with edges to their domain), and matches the
"brushable" framing in §11.

Reject alternative ("flat domain → problem only"): hides the
information from `TaxonomySchema` and loses the structural insight
that motivates §11's "hierarchy" framing.

#### D-4. Node sizing

Decision: **bubble area ∝ composite rating value, with `sqrt` scaling for area-fair perception.**

- Problem nodes: `radius = sqrt(composite_rating) × k` where `k` is
  tuned to fit the largest problem at ~24px and the smallest at ~6px.
- Domain nodes: `radius = sqrt(mean(child_problem_composites)) × k × 1.4` (1.4× makes parents visually larger than children).
- Subdomain nodes: same formula as domain, with a smaller `k × 1.1` multiplier.

Composite rating uses the §8.3 weights; the Recompose UI's URL search
params (Unit 3.10) DO NOT propagate to DomainMap in Phase 4 (Q36 lean:
`/problems` only). Phase 5+ could deep-link Recompose weights into
DomainMap as an enhancement.

#### D-5. Color encoding

Decision: **domain hue from existing `--chart-*` tokens.**

The 5 design-token chart hues (`--chart-difficulty`, `--chart-saturation`,
`--chart-urgency`, `--chart-value`, `--chart-industry-call` from Unit
0.4) double-purpose as domain hues 1–5, in declaration order. Subdomain
and problem nodes inherit their parent domain's hue at lower saturation
(70% / 50% respectively, applied via CSS color-mix or OKLCH lightness
adjustment).

Color is **decorative**, not the primary disambiguator — the visible
text label is the primary; `aria-describedby` → `<desc>` is the
screen-reader path; color is a tertiary cue. This passes WCAG SC 1.4.1
(use of color).

#### D-6. Interactivity surface

Decision: **5 interactions, ranked by importance:**

1. **Hover/focus**: highlight node + show problem title + composite rating in an SSR-rendered `<title>` element (native SVG tooltip). Also focusable via Tab.
2. **Click**: navigate to `/problems/<slug>` (problem nodes) or `/domains/<slug>` (domain nodes).
3. **Drag**: standard `d3-force` drag — pin a node, the simulation responds, releases reset.
4. **Filter chips** (above the map): one chip per domain; toggling dims non-matching nodes' opacity to 0.2. Multi-select. URL search param `?d=domain-a,domain-b` for deep-link (mirrors Recompose UI's URL persistence pattern).
5. **Zoom/pan**: scoped out of Phase 4. The graph fits in the viewport at current node counts; zoom adds bundle weight (`d3-zoom`) without clear payoff. Re-evaluate if Phase 5 expands the content surface.

#### D-7. /domains index — replace tile grid, keep as fallback

Decision: **DomainMap is the primary, the existing tile grid drops under a `<details>` table-fallback**, mirroring §13's "all charts have table-fallback toggles" criterion (closed in Unit 3.12).

Pattern reuse: `components/viz/_shared/chart-table-switch.tsx` from
Unit 3.12 wraps DomainMap. Table-fallback renders the same domain →
problem hierarchy as the current Phase-1/2 tile grid, just inside a
table structure.

#### D-8. Issue-template format

Decision: **GitHub form-based `.yml` templates** (not legacy `.md`).

Each template:
- Top-level `name` + `description` + `labels` + `body` (form fields).
- 4–6 required fields per template, all with `description` blocks linking to MASTER_PROMPT.md and the relevant THINK / runbook.
- Title prefix per type (e.g., `[New problem]`, `[New paper]`, `[Leaderboard entry]`, `[Rating challenge]`).

Form-based is the modern GitHub convention; structured fields validate
before submit, surface in the issue body cleanly, and trigger no
schema work in the repo itself (issues are GitHub-side, not git-side).

Detailed field schemas live in Units 4.7–4.10 THINK docs.

#### D-9. /contributing tone & content

Decision: **distilled CURATION_PROMPT.md + PAPER_INGEST_RUNBOOK.md, written for an external curator (not Claude)**, rendered from `content/contributing/v1.mdx` via the `/methodology` MDX pattern.

Sections:
1. Who can contribute (anyone with a GitHub account; no auth required for issues; PRs for direct edits).
2. The four contribution types (problem / paper / leaderboard entry / rating challenge), each linking to the corresponding issue template.
3. Editorial standards (primary-source rule, ROR / ORCID conventions, ADR-0005 immutability for rating actions).
4. PR review expectations (curator + 1 reviewer; CI must be green).
5. Versioned MDX: `v1` initial; future bumps as the workflow evolves (mirrors `/methodology` versioning).

Tone: same register as MASTER_PROMPT.md — direct, no marketing language, technical but not jargon-heavy.

### DB-migration trigger evaluation (§12)

§12 trigger: migrate to Postgres+Drizzle if `gzip(content-snapshot) > 5MB` **OR** auth is needed for submissions.

Measured at HEAD = `709679f`:
- `.velite/` snapshot uncompressed: **464,600 bytes (~454 KB)**.
- `tar -czf` of `.velite/`: **68,969 bytes (~67 KB)**.
- Threshold: 5 MB = 5,242,880 bytes.
- Current usage: **~1.3% of trigger.**

Auth-for-submissions trigger:
- §13 Phase-4 workflow: issue templates + PR review. **Not authenticated user submissions.** Auth ships in Phase 5+ (§5.8).

Conclusion: **DB migration stays deferred to Phase 5.** Documented in
detail in Unit 4.12's standalone note.

textarea). Detailed schemas decided in the per-template THINK docs.

pattern (Unit 3.10)? Lean: yes, `?d=domain-a,domain-b` for deep-link.

**Q39. DomainMap node accessibility on small viewports.** At the
mobile breakpoint (< 640px), 30+ nodes can overlap. Lean: viewport

**Q40. ADR-0007 scope.** Just SVG-vs-Canvas, or also covering the
D3 sub-package import policy? Lean: cover both — they're the same
decision-cluster ("how we render force graphs in this codebase").
