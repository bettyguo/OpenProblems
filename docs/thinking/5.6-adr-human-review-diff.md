# 5.6 — ADR-0009 (Human-review diff format for LLM-assisted drafts)

## THINK

### Inputs

- **§13 Phase-5 deliverable**: "LLM-assisted extraction of candidate leaderboard entries from a paper PDF (must produce a human-review diff; no auto-merge)."
- **Unit 5.0 D-6** (decided): unified patch in `drafts/`; compatible with `git apply`. ADR-0009 (Unit 5.6) records the realized format.
- **Unit 5.0 D-7**: drafts go to gitignored `drafts/` (top-level, unhidden).

### Decision drivers (in the ADR proper)

1. **§13 "no auto-merge"** is non-negotiable. The format must mechanically prevent direct `content/` writes by the LLM script.
2. **Git compatibility.** Curators already use `git apply` and `git diff`. The format must drop into that workflow without bespoke tooling.
3. **Auditability (ADR-0008 D-E).** Every LLM call must leave a reproducible audit trail. The sidecar `.meta.json` is the audit anchor.
4. **Two cases** — new-file (paper YAML) + modify-existing (entries.json append) — both need to ship in one format.
5. **Lifecycle hygiene.** Drafts are ephemeral; the `drafts/` directory is gitignored.

### Decisions (in the ADR proper)

**D-A. File-naming convention.**
```
drafts/<unit>-<isoTimestamp>-<slug>.diff
drafts/<unit>-<isoTimestamp>-<slug>.diff.meta.json
```
- `<unit>` ∈ {"5.3", "5.5", future "5.x"}.
- `<isoTimestamp>` is the script's UTC ISO time with `:` + `.` replaced by `-` for filesystem safety (e.g., `2026-05-15T17-30-00`).
- `<slug>` is the canonical identifier (arxivId for 5.3; `<arxivId>-<problemSlug>` for 5.5; future scripts pick a self-describing slug).
- Realized in `lib/curate/paper-draft.ts::buildDraftFilenames`.

**D-B. Diff format = unified, `git apply`-compatible.**
- **New-file case** (no target on disk): `--- /dev/null` + `+++ b/<target>` + `@@ -0,0 +1,N @@` + `+` line per body line. New file mode 100644. Realized in `paper-draft.ts::buildUnifiedDiff`.
- **Modify-existing case**: standard unified diff via `diff@9::createPatch(target, before, after, "", "")`. Realized in `entry-draft.ts::buildEntriesDiff`.

**D-C. Audit sidecar `.meta.json`** per ADR-0008 D-E:
```json
{
  "unit": "5.3" | "5.5" | ...,
  "script": "ingest-arxiv" | "extract-leaderboard" | ...,
  "arxiv_id": "...",
  "problem_slug": "..." (5.5 only),
  "existing_entries": N (5.5 only),
  "proposed_entries": N (5.5 only),
  "merged_entries": N (5.5 only),
  "model": "claude-sonnet-4-6" | "claude-opus-4-7" | ...,
  "input_tokens": N,
  "output_tokens": N,
  "cache_creation_input_tokens": N,
  "cache_read_input_tokens": N,
  "cost_usd_estimate": N,
  "prompt_sha256": "hex",
  "completion_sha256": "hex",
  "anthropic_request_id": "msg_..." | null (dry-run),
  "iso_timestamp": "YYYY-MM-DDTHH:MM:SS.sssZ",
  "dry_run": true | false
}
```
Script-specific extensions (e.g., 5.5's entry counts) are additive; the base shape is stable.

**D-D. No-auto-merge contract.**
- Phase-5 CLIs **NEVER** write to `content/`. They write to `drafts/` only.
- The curator runs `git apply drafts/<file>.diff` to materialize.
- The curator may edit the diff before applying.
- A future ADR can authorize auto-apply for a constrained subset (e.g., metadata-only refreshes), but ADR-0009 is the working contract for all LLM-drafted output today.

**D-E. `drafts/` lifecycle**:
- Gitignored top-level directory (added in Unit 5.2's `.gitignore` update).
- No retention policy enforced by tooling — curators delete locally when finished.
- Drafts can be safely deleted without applying; no on-disk state outside the script's CLI run depends on them.
- Cross-curator coordination: a draft in one curator's working tree is not visible to other curators; they re-run the relevant script themselves.

**D-F. Verified-flag discipline (curator-side flips).**
- Leaderboard-entry diffs (Unit 5.5) always have `verified: false` on every new row. The curator flips to `true` after independent verification against the paper's primary source.
- The merge layer (`entry-draft.ts::mergeEntries`) enforces this even if the LLM erroneously included `verified: true`.
- A future paper-detail script (5.x or Phase 6+) could allow LLM-proposed `verified: true` ONLY when the prompt + tool-use shape encodes a verification check — out of scope for ADR-0009.

### Considered options (rejected — in the ADR proper)

1. **Direct apply to `content/`** (no diff). Rejected — §13 explicitly forbids.
2. **Pull-request-style branch + auto-commit**. Rejected — auto-creating a branch + commit is still "auto-merge" in spirit; curator review feedback loops would have to happen in PR comments, fragmenting the workflow.
3. **Custom JSON-patch format** instead of unified diff. Rejected — curators already have `git apply` muscle memory; introducing a bespoke format adds learning cost and tooling.
4. **Inline annotations in `content/` files** (e.g., comments wrapping the LLM-proposed lines). Rejected — pollutes the canonical content, breaks `pnpm validate-content`.
5. **One diff per row** (entries.json fans out to N diffs per script run). Rejected — atomic-by-paper-run is the right unit; partial application is error-prone.

### Consequences (in the ADR proper)

- **Positive**: Standard tooling (`git apply`, `git diff`). Curators don't need a custom UI.
- **Positive**: Audit trail in `<diff>.meta.json` enables spot-audit + reproduction.
- **Positive**: `verified: false` discipline matches Q43 / Unit 4.9 framing — submitters never self-attest.
- **Positive**: Modify-existing case via `diff@9` handles entries.json append cleanly; new-file case via `paper-draft.ts` handles new paper YAML.
- **Negative**: Curators must run an extra `git apply` step vs. direct edits. Mitigation: documented in the `/contributing` workflow (Unit 4.5 — future v1.x bump to add the LLM-assisted path).
- **Negative**: Stale `drafts/` files can accumulate. Mitigation: gitignored; periodic manual cleanup or a future `pnpm clean-drafts` script.
- **Negative**: No anti-spoofing on the sidecar. A malicious actor could fabricate a `.diff.meta.json` to claim a fake LLM run. ADR-0009 documents that the sidecar is a curator-side artifact and is NOT trusted for verification — the actual `git apply` + manual review is the verification.
