- **MASTER_PROMPT §13 Phase 1 acceptance.** Four bullets:
  - All seed problems validate. ✓ already (`pnpm validate-content` reports 21 files; Unit 1.11).
  - Lighthouse perf / a11y / SEO ≥ 95 on `/`, `/problems/[any-slug]`, `/domains/[any]`.
  - Playwright smoke suite green: navigate landing → domain → subdomain → problem → leaderboard.
  - Visual regression baselines captured.
- **Existing scaffolding.**
  - [playwright.config.ts](../../playwright.config.ts) — chromium, webServer boots `pnpm build && pnpm start` (Unit 0.9).
  - [lighthouserc.json](../../lighthouserc.json) — currently asserts the four categories on `/` only (Unit 0.9).
  - [.github/workflows/e2e-lighthouse.yml](../../.github/workflows/e2e-lighthouse.yml) — Playwright + Lighthouse jobs already exist, both `continue-on-error: true` per OPEN_QUESTIONS Q27 ("advisory in Phase 0; promote to required at Phase 1").
  - [e2e/smoke.spec.ts](../../e2e/smoke.spec.ts) — Phase 0 smoke that asserts the home `<h1>` reads "LLM OpenProblems" plus the rating-dimension legend. **This spec is broken** on the post-Unit-1.10 home page (the h1 is now "Rated open problems …" and the legend was removed).
  - RatingRadar Storybook stories from Unit 1.5 (`components/viz/RatingRadar/index.stories.tsx`) — four states (high-confidence, low-confidence WATCH, all-zeros solved, size-120 variant). The Storybook-Vitest project (Unit 0.9) runs them as browser tests but does not snapshot them.

### Constraints

- **Acceptance gate ≠ feature work.** This unit's job is to prove the Phase 1
  surface meets the §13 bar, not to add new features. No new components
  beyond what tests need.
- **Reproducibility.** Playwright web-server boots a real production build,
  so screenshots are pinned to the same surface Lighthouse measures. Local
  + CI must agree.
- **Single-canonical exemplar per dynamic route.** §13 says "any-slug" /
  "any" — the Lighthouse and Playwright runs only need one representative
  URL per dynamic family. Pick `/problems/hallucination-reduction` (the
  fully-detailed Unit 1.4 seed; renders all 10 §9 blocks including the
  RatingRadar) and `/domains/deep-learning` (the densest hub; 12
  subdomains, 3 curated problems).
- **Visual-regression scope.** The handoff names "RatingRadar Storybook
  stories captured into Playwright snapshots." Two paths:
  1. Build Storybook static (`pnpm build-storybook`), serve it, and
     snapshot each story URL. Faithful to the literal handoff but adds
     a static-server orchestration step to CI.
  2. Snapshot the live RatingRadar inside `/problems/hallucination-reduction`.
     Snapshots what users actually see; covers the RSC render path the
     Storybook stories also exercise; one less moving piece in CI.
  Going with **path 2** in this unit — it satisfies the spirit of the gate
  (visual-regression on the radar) with the fewest moving parts. Path 1 is
  a clean Phase-2 follow-on once Storybook artifacts are part of CI.
- **Visual-regression snapshot stability.** Playwright `toHaveScreenshot()`
  is sensitive to font-rendering and animation. The RatingRadar entry
  animation (Unit 1.5) runs 250 ms scale 0.5→1; need to wait for it to
  settle, or pass `animations: "disabled"` to the screenshot call. Also
  set `prefers-reduced-motion: reduce` via Playwright's `colorScheme` /
  `reducedMotion` to cancel the animation entirely — matches what
  Unit 1.5's RatingRadar honours via the same media query.
- **CI promotion = Q27 close.** Drop `continue-on-error: true` from both
  jobs in `e2e-lighthouse.yml`. From Phase 1 onward, e2e + Lighthouse
  failures block PRs.

### Alternatives considered

- **Snapshot Storybook iframes via `pnpm exec storybook test` (storybook
  test-runner).** Cleaner integration with the Storybook story metadata
  but doubles the CI surface (build-storybook + serve + run). Defer to
  Phase 2 when more viz components ship.
- **Per-story snapshot via Playwright + Storybook static + http-server.**
  Same trade as above; would need to add http-server as a devDep and a
  webServer-array to playwright.config. Defer.
- **Run the Playwright spec without `prefers-reduced-motion`.** Would
  require a `page.waitForFunction` on a sentinel attribute to know when
  the animation is done. The reduced-motion path is shorter, more
  deterministic, and matches a real accessibility setting users have on.
- **Add a separate `e2e/visual.spec.ts` file vs. inline assertions in the
  smoke nav spec.** Inline keeps the Playwright run self-contained — one
  webServer boot per CI job, no spec ordering to worry about. Going
  inline; the snapshot lives at the natural step in the nav (the
  `/problems/hallucination-reduction` visit).

### Edge cases

- **Lighthouse perf depends on first-load JS.** Current bundles: `/` 162 B
  + 106 kB; `/problems/hallucination-reduction` 176 B + 106 kB;
  `/domains/deep-learning` 176 B + 106 kB — all well under §10.4's 180 kB
  problem-detail budget. Perf ≥ 95 should hold easily.
- **a11y on /problems/[slug].** The page composes `<StatusPill>`,
  `<RatingRadar>`, MDX prose, and tables. RatingRadar carries `<title>` +
  `<desc>` per Unit 1.5; MDX inherits semantic structure; the table-
  fallback (§10.2 / Q35) is keyboard-reachable. Should pass.
- **SEO on dynamic routes.** Each problem detail page sets metadata via
  Next 15's `generateMetadata`? **Need to verify.** If not, the SEO
  category may dock for missing meta-description. Spot-check during
  implementation; if missing, add `generateMetadata` for `/problems/[slug]`
  and `/domains/[domain]`.
- **Visual-regression baselines are platform-sensitive.** Playwright
  screenshots differ subtly across OS / font-rendering. Generate the
  baseline on the same OS that runs CI (Ubuntu) — so the local-Windows
  baseline will diverge on first CI run. Workaround: gate the visual
  assertion behind `process.env.CI` for now, with a `TODO(1.12-followup)`
  to capture an Ubuntu-native baseline by running the spec once with
  `--update-snapshots` in a CI job and committing the result.
  Alternative cleaner path: capture the baseline locally, mark the spec
  `test.skip` outside CI, and accept that the baseline will be re-captured
  on the first CI run. Going with the latter — the baseline ships from
  this unit but only gates in the same OS that produced it.
- **Nav spec on `…/leaderboard`.** That sub-route is still a Phase-0
  `<StubPage>` (Unit 0.10). The §13 wording says "leaderboard placeholder"
  — i.e., the assertion is just that navigation reaches the route and the
  stub renders, not that the leaderboard is implemented.

### Tradeoffs / defensible defaults flagged

1. **Visual-regression scope = live radar inside `/problems/[slug]`, not
   Storybook stories.** Documented above. Clean follow-on path to per-
   story snapshots in Phase 2.
2. **Visual baseline OS coupling.** Playwright snapshots committed from
   this unit will be the local-Windows baseline. CI's Ubuntu run will
   need a one-time refresh on first PR (commit the Ubuntu baseline, drop
   the Windows one). Note in the unit summary so the reader knows what
   the first CI run will surface.
3. **`generateMetadata` on dynamic routes if missing.** Add it as part of
   this unit if the SEO Lighthouse score docks on the spot-check. Do not
   speculatively add it before measuring.
4. **CI promotion is Q27's close.** No new ADR needed; OPEN_QUESTIONS
   entry flips to "decided".
