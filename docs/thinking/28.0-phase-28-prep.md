### D-1. First-thread recommendation: **multi-provider OAuth expansion via ADR-0020** (Phase-9 Class B item 8 carryover; 17-phase patience signal)

**Recommendation**: ship **multi-provider OAuth expansion** as
Phase 28's keystone thread, lifting ADR-0012 D-B's "GitHub OAuth
only" restriction via a new **ADR-0020** that documents the
multi-provider architectural commitment. **Closes Phase-9 Class B
item 8** (multi-provider OAuth expansion; carried **17 phases**
since Phase 9 closed) — the **strongest non-deferred-multi-phase
architectural patience signal** in the project.

**Rationale**:

1. **17-phase patience signal is the largest non-multi-phase
   carryover**. Phase-9 Class B item 8 has been deferred every
   phase since Phase 9 closed. Phase 28 entry signal = 17 phases;
   Phase 28 close = 18 phases. Larger than any other architectural
   patience signal except Phase-5 D-4 subscriber-list email
   (21+ phases — but that thread is genuinely multi-phase / cross-
   cutting and not a clean Phase-28 candidate; it requires
   `lib/email/` infrastructure + provider integration + verification
   + unsubscribe + templates + scheduler).
2. **ADR-0020 slot has been open 14 phases**. Since ADR-0019 shipped
   Phase 19. Each new operational-script-keystone phase + each
   inheritance-pattern phase has deferred the slot. Phase 28 closes
   it: **first new ADR since Phase 19**.
3. **First architectural patience-signal phase since Phase 19**.
   Phase 19's EXIF stripping (Q70) was the previous tightest
   patience-signal close. Phase 20-27 closed operational + UX
   follow-ons (low architectural surface change). Phase 28 returns
   to architectural-commitment shape.
4. **The "no user base" defensibility argument has 7+ phases of
   receipts**. At each prior phase, deferral was defensible
   because Q54/Q55/Q69 operational gates were pending + the
   project hadn't deployed. But: (a) the deferral cost grows
   linearly each phase; (b) the project should plan the
   architectural surface BEFORE deployment so the deploy + user-
   acquisition timeline isn't blocked on multi-provider work; (c)
   Q54 itself names "GitHub OAuth registration" as an operational
   gate — broadening to multi-provider lets the curator confront
   the OAuth surface uniformly across providers (single Q54 →
   single Q54-expansion + new Q73 for Google OAuth registration).
5. **No DB migration**. The `accounts` table already supports
   multi-provider per Auth.js v5 default schema (one row per
   `(provider, providerAccountId)`); `users.githubLogin` stays
   nullable; new sign-ins via non-GitHub providers leave it NULL.
   Migration count stays at **6**; **twelfth consecutive
   0-migration phase** anticipated (Phase 17-28; 15 of 19 phases
   since DB landed are 0-migration).
6. **Bundle invariant preservation** — `lib/auth/` is server-only;
   sign-in form lives in `components/auth-control/index.tsx`
   (already a server component using `signIn` server-action).
   Per-provider buttons add ~2 lines of HTML each + a small
   submit form (zero kB net client JS impact; server-side
   rendered). First Load JS UNCHANGED at 103 kB; middleware
   UNCHANGED at 160 kB.
7. **Editorial accountability model preserved**. `editorial.primary_curator`
   in file-system YAML still requires a GitHub login (per ADR-0012
   D-E + ADR-0005 + ADR-0011 D-G + Phase-9 Unit 9.6 `linkAccount`
   event). Non-GitHub users can submit challenges (Phase 11+
   submission flow) and edit their profile (Phase 15+) but
   **cannot be a curator-of-record**. Document this constraint
   in ADR-0020 D-D + as a follow-on Q-candidate (Q73-adjacent:
   "non-GitHub users as curators" — Phase 29+).
8. **Reverses the "pre-emptive" objection**. ADR-0012 D-B's
   "Phase 9 scope cap" was about not over-engineering before the
   first sign-in flow worked. Eighteen NON-§13 phases later, the
   sign-in flow is mature + tested + integrated across 5+ surfaces.
   Multi-provider expansion is now under-engineering relative to
   the rest of the project's maturity.

### D-3. Provider choice for ADR-0020

**Recommendation**: add **Google OAuth** as the second provider.
GitHub remains the only "curator-eligible" provider per D-D below;
Google expands the **submitter** + **profile-editing** surfaces to
non-GitHub users.

| Concern | Phase 28 scope | Notes |
|---|---|---|
| **Why Google specifically** | Broadest reach among non-GitHub users (academic users especially have Google accounts via institutional emails); `next-auth/providers/google` adapter is the most-tested non-GitHub Auth.js v5 provider; aligns with §5.8's broader "OAuth" recommendation when GitHub-only is too narrow. | |
| **Why not GitLab** | Smaller user base; closer to GitHub identity model (academic-Git users typically have both). Lower marginal value. Phase 29+ candidate. | |
| **Why not email-link / passkeys** | Requires SMTP infrastructure (email-link) or WebAuthn surface (passkeys); both are multi-phase scopes. Phase 29+. | |
| **Provider count after Phase 28** | 2 (GitHub + Google). | Phase 29+ can add a third if signal. |
| **Account-linking strategy** | Auth.js default: same email = same user (auto-link). | A user who signs in once with GitHub then later with Google (same email) gets the same `users.id` row with a new `accounts` row. `users.githubLogin` populated only via the GitHub-provider `linkAccount` event; Google sign-in leaves it NULL. |
| **First-sign-in flow** | Per-provider — no provider chooser modal. SiteHeader's `<AuthControl>` adds a second `<form>` posting to `/api/auth/signin/google`. Mirrors `signin/github` shape. | |
| **Account-merge UI** | NOT in Phase 28 scope. If a user has both providers and they have different primary emails, they get two `users.id` rows (Auth.js default behavior). Phase 29+ if curator demand. Documented in ADR-0020 D-F. | |

### D-4. ADR-0020 D-clauses

ADR-0020 pins six D-clauses (mirrors ADR-0012's five + adds one for
account-linking semantics):

- **D-A. Multi-provider runtime contract** — `next-auth/providers/*`
  adapters allowed; one entry per provider in the
  `providers: [...]` array of `NextAuth({...})`. Other auth
  runtimes (Clerk SDK, Lucia, Iron Session, etc.) remain
  forbidden per ADR-0012 D-A. ADR-0012 D-B's "GitHub OAuth only"
  is **lifted by this ADR** with an explicit cross-reference.
- **D-B. Second provider = Google OAuth** — explicit provider
  selection per D-3. Adds `next-auth/providers/google` to the
  `providers: [...]` array. Phase 28 ships exactly 2 providers;
  third+ requires a follow-on ADR amendment or new ADR.
- **D-C. Env-var convention** — Auth.js v5 canonical names
  (`AUTH_<PROVIDER>_ID` + `AUTH_<PROVIDER>_SECRET`) per ADR-0012
  D-A precedent. Phase 28 adds `AUTH_GOOGLE_ID` + `AUTH_GOOGLE_SECRET`.
  Env var count 7 → 9.
- **D-D. Curator-of-record gate remains GitHub-only** — file-system
  `editorial.primary_curator` (per ADR-0005 + ADR-0012 D-E) joins
  to `users.githubLogin`, which is populated only via the GitHub
  `linkAccount` event. Non-GitHub users cannot be curators in
  Phase 28; Phase 29+ candidate (Q73-adjacent: "non-GitHub users
  as curators" — would require a non-GitHub editorial-identity
  field on either users table OR a username-claim mechanism).
- **D-E. Account-linking strategy** — Auth.js default `allowDangerousEmailAccountLinking: false`
  per Auth.js security guidance. Users who sign in with two
  different providers using the same email get two separate
  `users.id` rows. Phase 29+ can add per-user account-merging UI
  if curator demand surfaces.
- **D-F. Sign-in UX is per-provider buttons** — `components/auth-control/index.tsx`
  renders one `<form>` per provider in the signed-out state. No
  chooser modal; no dropdown. Mirrors ADR-0012 D-D's
  redirect-to-provider pattern verbatim per provider. When the
  list grows beyond 3 providers, a chooser pattern may be
  introduced (deferred to Phase 30+ if signal).
