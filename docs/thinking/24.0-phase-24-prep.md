### D-3. Forward-path scope

| Concern | Phase 24 scope | Notes |
|---|---|---|
| **Sharp pipeline edit** | `sharp(buffer).rotate().resize({ width, height, fit: "cover" }).webp({ quality }).toBuffer()`. | Inserts two transforms between Phase-19's `.rotate()` and `.toBuffer()`; chain order matters (resize happens on rotated pixels; webp encoding happens last). |
| **Resize dimensions** | `width: 512, height: 512`. | Covers hi-DPI use cases (32-150px display @ 2-4x); plenty for avatar contexts; balance between fidelity + file size. |
| **Resize fit** | `"cover"`. | Avatars are square; cover scales + crops to fill the target dimensions while preserving aspect ratio. Other fits ("contain" would letterbox; "fill" would distort). |
| **WebP quality** | `85`. | Sharp's recommended default for balance; matches ADR-0019 D-F's example. ~70-80% smaller than equivalent JPEG/PNG at indistinguishable visual quality. |
| **Storage key extension** | Always `.webp` after Phase 24. | Output content is always WebP regardless of input format. Storage-key derives extension from output, not input. |
| **Content-type header** | `"image/webp"`. | Matches output format. Vercel Blob serves the URL with this Content-Type. |
| **Input MIME validation** | UNCHANGED from Phase 16 ADR-0017 D-B (`image/jpeg` / `image/png` / `image/webp` accepted). | Phase 24 doesn't relax or tighten upload validation; only changes output. |
| **`inferExt()` helper** | Removed from `putAvatar` call path (storage-key always `.webp`); kept in module if backfill needs it (it doesn't — backfill output is also `.webp`). Phase-24 lean: **remove `inferExt` entirely** since both forward + backfill always emit `.webp`. | Reduces dead code; simpler test surface. |
| **Existing test mocks** | Update sharp mock to include `.resize()` + `.webp()` returning the same fluent-chain instance. Update Vercel Blob `put` mock assertions to expect `image/webp` content-type + `.webp` extension regardless of input MIME. | ~5 existing tests changed shape. |

### D-6. Backwards compatibility + safety

- **Existing avatars Phase 16-23**: continue serving the
  original-format blob until backfill runs. Phase-24 forward
  path only affects NEW uploads.
- **Phase 24 + Phase 20 backfill interaction**: both can coexist.
  Running Phase-20 backfill on a Phase-24-resized blob yields
  the same `.webp` content (sharp's `.rotate().toBuffer()` is a
  no-op on EXIF-stripped + already-rotated images). Running
  Phase-24 backfill on a Phase-20-backfilled blob applies the
  resize+webp downsizing.
- **Re-run safety**: Phase-24 backfill is idempotent — re-running
  on already-resized blobs produces functionally-equivalent
  output (slight pixel difference from re-encoding rounding;
  acceptable per Phase-19 D-1 "re-encoding artifacts" caveat).
- **DB column unchanged**: `users.imageOverride` stays TEXT; no
  schema migration. Only the URL value changes (new blob, new
  filename).
