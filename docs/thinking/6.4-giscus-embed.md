# 6.4 — `components/discussions/GiscusEmbed.tsx` (client-only iframe wrapper + theme sync)

Third code unit of Phase 6. Lands the actual Giscus iframe wrapper that Unit 6.3's `app/problems/[slug]/talk/page.tsx` slot anticipates. Closes ADR-0010 D-A (embed = Giscus, `@giscus/react`) + the deferred D-7 (Giscus version pin) + D-11 (theme sync via `next-themes`).

## THINK

### Inputs

- **[ADR-0010 D-A](../adr/0010-discussions-backend.md)** — embed backend = Giscus via `@giscus/react`; lazy-loaded; iframe loads after page shell renders.
- **[ADR-0010 D-C](../adr/0010-discussions-backend.md)** — pathname-based mapping (`mapping="pathname"`); discussion lazily created on first comment.
- **[Unit 6.3 placeholder](6.3-talk-page-route.md)** — the `<section id="discussions">` slot in `app/problems/[slug]/talk/page.tsx` is positioned to receive this component as a direct child.
- **`components/theme-toggle/index.tsx`** — the existing `useTheme` consumer pattern: `"use client"` directive + `useState(false)` + `useEffect(() => setMounted(true))` placeholder shape to avoid pre-hydration flash. Mirror this exactly.
- **`next-themes`** is already a project dependency (`^0.4.6`); no install needed for the theme integration.
- **`@giscus/react@3.1.0`** — installed in this unit. Pure JS; no postinstall; no `pnpm-workspace.yaml` `allowBuilds` change. +8 transitive packages.
- **GitHub-assigned repo ID** — Giscus requires a `repoId: string` prop. The ID is generated at [giscus.app](https://giscus.app)'s config UI after Discussions is enabled on the repo (Q47). At HEAD time, the ID is unknown; this unit reads it from `NEXT_PUBLIC_GISCUS_REPO_ID` env (build-time-baked per Next.js convention) with a graceful fallback for the unset case.

### Scope (this unit)

Three file changes + one install:

1. **Install** `@giscus/react@^3.1.0` in `dependencies` (mirror ADR-0010 D-A; mirror Phase-5 `@anthropic-ai/sdk` + Unit 6.2 `@octokit/graphql` precedents — runtime imports go in `dependencies`, not `devDependencies`).
2. **`components/discussions/GiscusEmbed.tsx`** (new) — `"use client"` component with:
   - `useTheme()` integration: maps `resolvedTheme` ∈ `{"light", "dark"}` to Giscus's `theme` prop. Default mapping: `dark → "dark"`, anything else → `"light"`.
   - Pre-hydration placeholder (returns a loading-state element pre-mount; mirrors `components/theme-toggle/index.tsx`'s pattern to avoid hydration mismatches).
   - `NEXT_PUBLIC_GISCUS_REPO_ID` env-driven config: when unset (the HEAD state until Q47 lands), renders an "embed unavailable" message with a link to giscus.app config. When set, renders the actual Giscus component with the pinned props.
   - Pinned Giscus props per ADR-0010: `repo="bettyguo/OpenProblems"`, `mapping="pathname"`, `strict="1"` (exact path matching), `loading="lazy"` (per D-A; iframe loads after shell), `inputPosition="bottom"`, `lang="en"`, `reactionsEnabled="1"`, `emitMetadata="0"`.
   - **No** `category` / `categoryId` props — defaults to the repo's general category (Q49 / D-F: moderation is delegated to GitHub native; category selection isn't load-bearing for v1).
3. **`components/discussions/GiscusEmbed.test.tsx`** (new) — vitest tests for the no-env path + the placeholder behavior. The Giscus iframe itself isn't tested (cross-origin iframe; per ADR-0010 D-A + §14.2 "tests must not depend on cross-origin iframe contents loading"). At least 4 tests.
4. **Edit `app/problems/[slug]/talk/page.tsx`** — replace the placeholder `"Discussion thread loading…"` text inside the `<section id="discussions">` slot with the rendered `<GiscusEmbed />` component import.

### Constraints

- **§5.5 perf budget** — First Load JS shared chunk = 103 kB. The Giscus React wrapper adds bundle weight to the talk-page route chunk only (per ADR-0010 D-A: iframe doesn't count against shared chunk; the wrapper itself is ~3-5 kB). Expected: talk-page route chunk grows from 201 B (Unit 6.3 baseline) to ~5-10 kB; shared chunk unchanged.
- **§14.2 testing** — every page route gets a Playwright e2e smoke. The component itself gets vitest unit tests for the testable branches (no-env path + placeholder). The Giscus iframe's contents are intentionally NOT tested per ADR-0010 D-A's negative consequence.
- **`exactOptionalPropertyTypes`** — `process.env.NEXT_PUBLIC_GISCUS_REPO_ID` can be `undefined`; `?? ""` (string fallback) keeps it well-typed.
- **`"use client"` directive** — required because the component uses `useState` + `useEffect` + `useTheme`. Per Next.js App Router conventions.
- **Hydration safety** — render a stable placeholder pre-mount (mirror `components/theme-toggle/index.tsx`). Without this, server-rendered HTML would not match client-rendered HTML and the React reconciler warns / re-renders.
- **No additional ARIA work** — Giscus handles its own a11y inside the iframe; the page's `<section aria-label="Discussion thread">` from Unit 6.3 provides the landmark.

### Alternatives considered

- **Use Giscus's `<script>` snippet directly (no React wrapper)**. Rejected — would have to manage hydration + theme-sync + lifecycle by hand. `@giscus/react` is the React-canonical surface; 5-10 kB bundle weight is the cost of doing it right.
- **Hard-code a placeholder `repoId`**. Rejected — would produce a broken iframe at HEAD that LOOKS like it works but actually points at no real discussion. The env-driven fallback is more honest about the operational gate (Q47).
- **Render the embed only when `repoId` is set; otherwise render nothing**. Rejected — silent absence is worse than an explicit "unavailable" message. Curators reading the page should know the iframe is blank because of an operational gate, not because the slot is empty.
- **Move theme sync into a `useEffect` that posts `setConfig` messages to the iframe**. Rejected — `@giscus/react` handles this internally when the `theme` prop changes. No reason to re-implement.
- **Use `process.env.GITHUB_TOKEN`** (server-side) instead of `NEXT_PUBLIC_GISCUS_REPO_ID`. Rejected — the `repoId` is needed CLIENT-side (the iframe URL contains it). `NEXT_PUBLIC_*` is the right Next.js convention for build-time-baked client env vars.

### Tradeoffs / defensible defaults flagged

1. **`NEXT_PUBLIC_GISCUS_REPO_ID` env not yet set anywhere** — production deploys will need to set it once Q47 resolves and the curator runs giscus.app's config UI. Until then, all `/problems/<slug>/talk` pages show the "embed unavailable" placeholder. Acceptable — explicit > implicit; the page renders correctly.
2. **Pre-hydration placeholder is small** ("Loading discussion…" italic) — chosen to minimize layout shift. Not styled with explicit min-height; if the user wants a stable card-sized placeholder, that's a future refinement.
3. **Theme map is binary** (`dark` → Giscus `"dark"`, everything else → `"light"`). The full Giscus theme set is much wider (gruvbox, catppuccin, etc.) — none align with the project's OKLCH 170° palette. `light` / `dark` is the right minimum-viable mapping; custom-theme URL injection (per Giscus's `https://...` theme type) is a future possibility if the default themes don't blend well visually.
4. **No `id` prop on the Giscus component** — Giscus uses `id` for multiple-embeds-per-page disambiguation; we have at most one embed per page; omit.
5. **`emitMetadata="0"`** — Giscus's `discussion.created` / `discussion.metadata` postMessage events go unused; we have first-party GraphQL (Unit 6.2) for metadata. Save the postMessage chatter.
6. **`strict="1"`** — exact path matching. Without it, Giscus does fuzzy matching on the path (e.g., `/problems/X/talk?utm=...` matches `/problems/X/talk`). With it, only exact matches join the same discussion. Right call for a structured-URL project.
