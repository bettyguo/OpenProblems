**The Unit 5.5 tool path** is now operational for backfilling:
1. Pick a target problem (e.g., `mechanistic-interpretability`).
2. Pick a paper from `content/papers/` whose `contributions[]` includes that problem.
3. Run `pnpm extract-leaderboard <arxiv-id> --problem <slug>` (Opus 4.7 default).
4. Curator reviews the proposed entries.json in `drafts/`.
5. `git apply drafts/<file>.diff` if the diff passes review.
6. PR + merge.

**Why not in this unit**: the workflow is curator-mediated by design (§13 no-auto-merge + ADR-0009). Phase 5 ships the toolchain; populating the content is downstream editorial work — best done in a curator-driven session where source-finding judgement applies. A future hygiene-style unit (e.g., "5.11a — entries.json backfill on `mechanistic-interpretability`") can pick it up.

### Alternatives considered

- **Ask for explicit deletion authorization mid-session** via `AskUserQuestion`. Rejected — the "Continue" pattern suggests the user wants forward momentum; pausing mid-loop for a single-file deletion is more friction than the gain.
- **Skip Unit 5.11 entirely**; ship it as part of Unit 5.13's acceptance gate. Rejected — the Phase-5 plan in Unit 5.0 named 5.11 as a discrete unit; collapsing it into 5.13 erodes the rhythm.

Rejected — `--dry-run` produces a placeholder diff (no real entries); shipping that to `content/` would be pseudo-content, worse than no content. 3. **No `/contributing` v1.x bump here**: content authoring with the right tone is editorial work; better done as its own unit when DNS / email lands (Q2).
