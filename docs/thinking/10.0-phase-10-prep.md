### D-1. First-thread recommendation: **Profile page + Phase-9 UI polish (consolidation thread)**

**Recommendation**: pull the **profile page** thread from Phase-9 Class
B (Unit 9.7 item 1). Lands `/[locale]/profile` showing the signed-in

phase. **Consolidation-first**: lights up the auth-thread's unbuilt UI
surface (signed-in users have nowhere to see their state today) before
pulling a bigger second-write-path expansion.

**Rationale**:

1. **Smallest defensible scope** (~5-6 units; matches Phase 8's 9-unit
   delivery and Phase 9's 10-unit delivery on the smaller-side of the
   project's rhythm). Lower risk on a NON-§13 phase where the
   thread-selection isn't pre-anchored.
2. **Builds directly on Phase-9 surfaces**. Uses `lib/auth/` + `lib/db/` +
   `lib/watchlist/` + `components/watchlist-toggle/` + `events.linkAccount`
   (Phase 9 populates `users.githubLogin`; profile page surfaces the
   handle). No new architectural primitives.
3. **Closes the natural UX gap**. Phase 9 introduced sign-in but no
   destination for signed-in users. Today a user signs in via SiteHeader,
   sees the "Signed in" pill, and... has nowhere to go. Profile page
   closes that gap with a tangible "see what you're watching" surface.
4. **First protected route** lands in this phase — exercises the
   Phase-9 Class B item 12 ("auth-aware route protection"). Architectural
   shape: server-component-level `auth()` check + redirect to
   `/api/auth/signin/github?callbackUrl=...` if no session. NOT
   middleware-based protection (defer until 2+ protected routes exist).
5. **Tests the auth state propagation end-to-end**. Sign-in → DB session
   row → `auth()` reads session → page renders user-specific data → user
   can mutate state (unwatch) → DB row updated → page re-renders. Catches
   any auth-thread regressions before Phase 11+ expands the surface.
6. **Avoids the bigger Phase-10 candidates' marination cost**.
   Rating-challenge submission (~6+ units; needs draft + curator review

   multi-provider OAuth (~3-4 units; forbidden in Phase 9 per ADR-0012

### D-3. Profile route shape

Three viable shapes:

1. **`/[locale]/profile`** — "the signed-in user's profile" (singular;
   self-only). No handle in URL. Easy bookmark. Mirrors GitHub's `/me`
   or Linear's `/profile` pattern.
2. **`/[locale]/u/[handle]`** — handle-in-URL (e.g.,
   `/[locale]/u/bettyguo`). Public-vs-private surface: signed-in user
   sees own profile with edit affordances; others see public-only view.
   More architectural surface; future-proofs for "view another user's
   public profile" but Phase 10 has no public-profile-render surface.
3. **`/[locale]/me`** — short URL; same singular-self semantics as #1.

**Lean** (subject to Unit 10.2 implementation): `/[locale]/profile`.
Reasoning:
- Shortest non-singular URL with self-evident semantics.
- Future public-profile-render (`/[locale]/u/[handle]`) can land in
  Phase 11+ as a separate route without renaming `/profile`.
- Aligns with Phase-9 Class B item 1's `/[locale]/profile` naming
  precedent.
- Phase 10 doesn't need public profiles (no observed ask + curator-of-
  record file-system data is the canonical public-profile surface for
  now).

### D-4. Route protection strategy

Two viable strategies for protecting the profile route:

1. **Server-component check + redirect (chosen)**: page calls `auth()`
   at the top; if `!session?.user?.id`, calls
   `redirect("/api/auth/signin/github?callbackUrl=/${locale}/profile")`.
   Page-local; no middleware change.
   - Pros: smallest surface; no middleware bundle delta; locally
     reasonable; no architectural commitment for Phase 11+ routes (each
     decides for itself).
   - Cons: each protected route must remember to add the check; harder
     to audit "which routes are protected" from a single file.
2. **Middleware-based protection**: extend `middleware.ts` to check
   `req.auth` against a `protectedPaths` list; redirect if no session.
   - Pros: centralized; auditable; reusable across routes.
   - Cons: middleware bundle already 159 kB (~15% of Vercel Edge's 1 MB
     budget); adding a path-list adds parsing overhead per-request;
     architectural commitment (every future route uses the same
     pattern).

**Lean** (subject to Unit 10.2 THINK doc): **Server-component check +
redirect**. Reasoning:
- Phase 10 has ONE protected route (profile). Middleware-based
  protection is premature without 2+ protected routes.
- `safeAuth()`-style defensive wrappers established in Phase 9 don't
  apply here — the profile page MUST have a session (or redirect); a
  null result is a redirect signal, not a render-degradation signal.
- Phase 11+ can lift to middleware-based when 2+ protected routes
  exist (auth-aware route protection per Phase-9 Class B item 12);
  promotion is a small refactor.
- Locality preserved: the page's auth dependency is visible in its
  own file.

### D-6. Watchlist surface (profile-page primary content)

Three viable shapes for the watched-problems list:

1. **Dense list** — one row per watched problem; problem title +
   domain breadcrumb + last-curated date + inline unwatch button.
2. **Card grid** — problem title + status pill + rating snapshot
   (radar miniature?) + unwatch button. Heavier; closer to
   "miniature dashboard."
3. **Plain table** — title / domain / subdomain / unwatch. Most
   compressed; least visually engaging.

**Lean**: **Dense list** (#1). Reasoning:
- Reuses existing `Link` + breadcrumb patterns from problem pages.
- Smaller bundle / render cost vs card grid.
- Inline unwatch via reuse of `WatchlistToggle` (Phase-9 unit 9.6
  component).
- Empty state: "You're not watching any problems yet. **Browse
  problems →**".

Will need a new helper in `lib/watchlist/`:
`getWatchedSlugs(userId): Promise<string[]>` returning user's watched
slugs sorted by `createdAt` DESC. Joins to `problems` from
`#site/content` at the page level to assemble the dense-list rows.

### D-7. Watchlist-toggle reuse vs dedicated unwatch button

The Phase-9 `WatchlistToggle` component renders a star/unstar button
in the watched / not-watched state. On the profile page, every visible
slug is by-definition watched (the page lists `getWatchedSlugs`); we
only need the **unwatch** path.

Two viable approaches:

1. **Reuse `WatchlistToggle`** as-is. It renders the "Watching ★"
   button which becomes the unwatch CTA. Visual consistency with
   problem-detail pages. The only downside: it does its own
   `isWatched()` check (DB query) which is redundant given we already
   queried `getWatchedSlugs`.
2. **Dedicated `UnwatchButton`** component. Tiny server-action form
   that calls `removeFromWatchlist(userId, slug)` + `revalidatePath`.
   Skips the redundant `isWatched()` check.

**Lean**: **Reuse `WatchlistToggle`**. Reasoning:
- Visual consistency with problem-detail pages (same CTA shape).
- Redundant `isWatched()` is a single indexed SQL lookup; cost
  irrelevant.
- Simpler maintenance (one component to evolve).
- Avoids the "two components for the same logical action" pattern
  that breeds drift.

Re-evaluate if profile pages become hot enough that the per-item
`isWatched()` queries become observable in latency (Phase 12+
optimization).
