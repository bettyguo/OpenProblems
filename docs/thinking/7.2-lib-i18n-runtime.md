# 7.2 — `next-intl` runtime install + `lib/i18n/` infrastructure

First code unit of Phase 7. Lands the `next-intl` runtime dependency + the `lib/i18n/` config files + seed message catalogues + the Next.js plugin wiring. Per ADR-0011 D-A. Does NOT yet restructure routes or add middleware — those land in Unit 7.3 atomically so the site doesn't break between phase commits.

## THINK

### Inputs

- **[ADR-0011 D-A](../adr/0011-i18n-strategy.md)** — runtime = `next-intl@^3.x` (only i18n dependency allowed); JSON-per-locale message files at `messages/<locale>.json`; ICU MessageFormat.
- **[ADR-0011 D-B](../adr/0011-i18n-strategy.md)** — sub-path routing (`/en/...`, `/fr/...`); default locale = `en`; bare URLs 308-redirect to defaulted locale. Implementation: middleware in Unit 7.3.
- **[ADR-0011 D-D](../adr/0011-i18n-strategy.md)** — fallback chain `fr → en`; untranslated pages render EN content + switch hint.
- **`@anthropic-ai/sdk` + `@octokit/graphql` + `@giscus/react`** — precedents for runtime third-party deps in `dependencies` (not `devDependencies`).
- **`next.config.ts`** — currently has `reactStrictMode` + ESLint/TypeScript guards; needs the next-intl plugin wrapper applied.

### Scope (this unit)

Six file changes + one install:

1. **Install** `next-intl@^3.x` in `dependencies`. Pure JS (no postinstall; no `pnpm-workspace.yaml` `allowBuilds` change).
2. **`lib/i18n/routing.ts`** (new) — defines `locales = ["en", "fr"]`, `defaultLocale = "en"`, and an `isLocale(value: string): value is Locale` type-narrowing helper. Single source of truth for the locale list.
3. **`lib/i18n/request.ts`** (new) — the `getRequestConfig` callback that `next-intl` invokes per request. Loads `messages/<locale>.json` via dynamic import; falls back to `defaultLocale` when the requested locale is unrecognized (per ADR-0011 D-D); throws when neither locale's messages file is present (build-time invariant — messages files are required).
4. **`messages/en.json`** (new) — seed translation catalogue (site title + tagline + nav items + locale-toggle labels). Minimal but real; subsequent units consume + expand.
5. **`messages/fr.json`** (new) — FR mirror of the EN seed. Curator-authored translations for the seed keys (subject to v1 refinement at Unit 7.4's `/methodology` pilot).
6. **`next.config.ts`** edit — wrap the existing config with `createNextIntlPlugin(...)` from next-intl. Plugin's path argument points at `lib/i18n/request.ts`.
7. **`lib/i18n/routing.test.ts`** (new) — vitest tests for the routing helpers (≥ 4 tests covering: locales list, defaultLocale value, isLocale truthy/falsy cases).

### Scope NOT in this unit

- **`middleware.ts`** — Unit 7.3. The middleware enforces sub-path routing; activating it before the `app/[locale]/` segment exists would 308-redirect every URL to a non-existent route, breaking the site. Atomic restructure-with-middleware in 7.3.
- **`app/[locale]/` route segment** — Unit 7.3.
- **First FR-rendered page** — Unit 7.4 (`/methodology` pilot).
- **Velite sibling-file glob extension** — Unit 7.5.
- **Locale-toggle UI** — Unit 7.6.
- **Sitemap with locale variants** — Unit 7.8.
- **`useTranslations` usage on any page** — no page consumes the runtime in this unit. The runtime is installed + plugin-wired, but observable behavior is unchanged at HEAD. Routes still SSG at their existing paths.

### Constraints

- **§5.5 perf budget** — First Load JS shared chunk = 103 kB. `next-intl` adds ~30 KB ONLY when imported on a page. This unit imports `next-intl` zero times in `app/` or `components/`. The shared chunk should stay at 103 kB. The plugin wrapper around `next.config.ts` doesn't import next-intl into the page bundle.
- **§14.2 testing** — `lib/i18n/routing.ts` gets vitest unit tests (≥ 4). `lib/i18n/request.ts` is a Next.js request-config callback; testing it requires mocking the `headers()` / `cookies()` request context which is awkward. Skip vitest for request.ts; rely on the Unit 7.4 pilot's e2e to validate the loader end-to-end.
- **`exactOptionalPropertyTypes`** — `Locale = "en" | "fr"` is a string-literal union; no optional-prop concerns.
- **No client-bundle impact** at this unit's HEAD — verified by running `pnpm build` and noting the First Load JS shared chunk stays at 103 kB.
- **Messages JSON shape**: nested objects for namespacing (`site.title`, `nav.problems`). ICU MessageFormat allowed but not required for v1 seeds.

### Alternatives considered

- **Defer next-intl install to Unit 7.3** (combine install + middleware + restructure in one big unit). Rejected — three changes in one commit is hard to review + bisect. Splitting install into 7.2 makes the diff manageable.
- **Ship `middleware.ts` in 7.2 with `matcher: []`** (effectively no-op until 7.3). Rejected — a no-op middleware is dead weight at HEAD that future units could trip over. Cleaner to skip it entirely.
- **Inline messages JSON directly in `request.ts`** (vs. separate files). Rejected — separate JSON files are the next-intl-canonical pattern + lets curators edit translations without touching TypeScript.
- **Use `loadMessages()` from a JSON-import in request.ts** vs. a dynamic `import()`. Going with the dynamic import (next-intl's recommended pattern) — handles tree-shaking + lazy loading correctly.

### Tradeoffs / defensible defaults flagged

1. **Seed FR translations are minimal**. ~10 keys covering site + nav. Subsequent units expand. Acceptable — the seed is enough to validate the loader works.
2. **Plugin path is `./lib/i18n/request.ts`** (not the next-intl-default `./i18n/request.ts` at project root). Project convention: i18n config lives in `lib/i18n/`, not in a top-level `i18n/` directory.
3. **No `locales` exported from `routing.ts` as a TS const-assertion** at this stage — `as const` is implicit via the array literal pattern. If a future unit needs `typeof locales[number]` type derivation, refactor then.
4. **Messages files DON'T live in `content/`** — they're application-level translation data, not editorial content. Velite shouldn't see them. Top-level `messages/` directory keeps them separate from `content/` (which has its own validation pipeline).
5. **No `.gitkeep` in `messages/`** — both `en.json` + `fr.json` ship at this unit; the directory has real files immediately.
6. **`lib/i18n/request.ts` falls back to `defaultLocale` for unknown locales** — per ADR-0011 D-D. Hard-throw would be brittle (a URL typo `/xx/problems/...` would crash; instead it renders English).

### Post-install pnpm-workspace.yaml change

pnpm 11 flagged two new transitive deps (`@parcel/watcher`, `@swc/core`) for `allowBuilds` opt-in when installing `next-intl@^3`. Both are core Next.js infrastructure (file watching + JS compiler); the prior 3 entries (`esbuild`, `sharp`, `unrs-resolver`) set the precedent. Setting both to `true` matches that posture — allows postinstall scripts to build the optimal native bindings. No alternative path investigated; the precedent is decisive.

### Realized version note

ADR-0011 D-A pinned `next-intl@^3.x`; pnpm resolved to `3.26.5`. The latest next-intl major is `4.x` (`4.12.0` is the auto-pick without a constraint). Per the ADR-immutability rule (`docs/adr/README.md`), I downgraded the install rather than amend the ADR. A future ADR-0012 (if 4.x becomes load-bearing — e.g. for a new API surface 3.x doesn't expose) would supersede ADR-0011's version-pin clause specifically.
