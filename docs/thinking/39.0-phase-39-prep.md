### D-1. First-thread recommendation: **GFM tables per-reviewNotes via `schemaOverrides` — validates the Phase-37 framework's second slot**

`TablesExtensionRegistry` enables GFM tables on the `reviewNotes`

`<tbody>`, `<tr>`, `<th>`, `<td>` to the `reviewNotes`

**Why tables per-reviewNotes**:

1. **Validates the framework's `schemaOverrides` slot end-to-
   end** with a real consumer — Phase 38 only exercised
   `rehypePlugins`. After Phase 39 the framework has TWO real
   consumers exercising TWO of the three slots
   (`rehypePlugins` + `schemaOverrides`); only `remarkPlugins`
   remains as a Phase-40+ deferral.
2. **Second concrete Phase-37-framework consumer in project
   history** — validates the framework's multi-consumer
   composition. After Phase 39 the framework is consumed by
   2 distinct registries (`WikilinkExtensionRegistry` +
   `TablesExtensionRegistry`) for 2 distinct surfaces
   (actionRationale + reviewNotes) via 2 distinct slots
   (rehypePlugins + schemaOverrides).
3. **Per-surface differentiation across consumers** — Phase
   37 + 38 framework was per-surface within ONE consumer;
   Phase 39 demonstrates per-surface differentiation across
   TWO consumers (tables on reviewNotes; wikilinks on
   actionRationale; the dispatch arms are mutually exclusive
   Phase 39 — multi-value composition deferred Phase 40+).
4. **`remark-gfm` already parses tables** (Phase-17 pipeline);
   they're currently STRIPPED at sanitize because `<table>` is

   content currently uses tables (zero `|---|---|---|` patterns

   the architectural validation; curator authoring of tabular
   review notes is an enabling, not a forcing-function.
   **Trade-off flagged**.

### Demand-signal-first concern resolution

Phase-37-prep + Phase-38-prep both emphasized "demand-signal-
first" as gating heuristic. Phase 39 tables-per-reviewNotes has
**zero current content evidence** — DB-backed reviewNotes are
ephemeral with no `|---|` patterns visible at Phase-39 ship.

Why ship anyway:

1. **The framework's value is its composability**. Validating
   the second slot with a real consumer is a recurring
   architectural milestone — without it, the framework remains
   "1-of-3 slots demonstrated". Phase 39 makes it
   "2-of-3 demonstrated", strengthening the framework's
   claim to general reusability.
2. **Architectural validation has option value**. Even if no
   curator authors tabular reviewNotes Phase 39, the schema
   override is reversible (env-var unset = Phase-38 behavior
   verbatim); the cost of shipping is bounded (1 plugin-free
   extension; ~50-100 LOC + tests).
3. **Demand-signal-first applies to user-facing features**;
   the schema override path of the framework is an internal
   architectural surface that benefits from validation
   irrespective of curator authoring patterns. Same shape as
   ADR-0024 framework-only Phase 35 (Phase-35 shipped the
   moderation framework with `NoopModerator` default = zero
   user-visible behavior change; the framework's existence is
   the value).

**Trade-off**: passing on demand-signal-first discipline for
the second framework consumer specifically. If curator
authoring patterns suggest a different next consumer (e.g.,
@mentions, cross-surface wikilinks) Phase 39 implementation
can pivot in Unit 39.1 per phase-prep alternate-thread
flexibility.

### D-3. `TablesExtensionRegistry` framework consumer shape anticipated (Unit 39.1)

D-A. **Registry class**:

```ts
export class TablesExtensionRegistry implements MarkdownExtensionRegistry {
  constructor(private readonly enabledSurfaces: ReadonlySet<MarkdownSurface>) {}

  getExtensions(surface: MarkdownSurface): MarkdownExtensionSet {
    if (this.enabledSurfaces.has(surface)) {
      return { schemaOverrides: GFM_TABLE_SCHEMA_OVERRIDES };
    }
    return {};
  }
}
```

D-B. **`GFM_TABLE_SCHEMA_OVERRIDES`** = `Partial<Schema>`
adding `tagNames` (full replacement per APPEND-D-C override-
replace semantics) including the existing base allow-list PLUS
6 new table tags. Plus `attributes` shallow-merging:

```ts
const GFM_TABLE_SCHEMA_OVERRIDES: Partial<Schema> = {
  tagNames: [
    // Phase-17 base allow-list (copied verbatim per override-replace)
    "p", "strong", "em", "code", "pre", "a", "ul", "ol", "li",
    "h3", "h4", "h5", "h6", "blockquote", "hr", "del", "br", "input",
    // Phase-39 table additions
    "table", "thead", "tbody", "tr", "th", "td",
  ],
  attributes: {
    a: ["href"],
    input: [["type", "checkbox"], "checked", "disabled"],
    th: [["align", "left", "center", "right"]],
    td: [["align", "left", "center", "right"]],
    "*": [],
  },
};
```

D-C. **Override-replace semantics realized**: APPEND-D-C
documented that callers MUST supply the full base list when
overriding (no deep-merge). Phase 39 implementation exercises
this contract — `GFM_TABLE_SCHEMA_OVERRIDES.tagNames` includes
the full Phase-17 base list verbatim plus 6 additions; same
for `attributes`. **First real-consumer exercise of override-
replace semantics** in project history.

D-D. **Default-enabled-surfaces** = `Set(["reviewNotes"])` Phase
39. Other 3 surfaces (`bio` + `rationale` + `actionRationale`)
receive empty extension set per `TablesExtensionRegistry`'s
default-deny. Curator review notes (4000-char field, Phase-12
ADR-0014) is the natural surface for tabular content (COI
comparisons; multi-criterion scoring; structured editorial
reasoning).

D-E. **Factory dispatch arm** = `MARKDOWN_EXTENSIONS=tables` in
`getExtensionRegistry()`. **Mutually exclusive with `wikilinks`
Phase 39** — multi-value composition `MARKDOWN_EXTENSIONS=wikilinks,tables`
deferred Phase 40+ per Phase-38-prep D-11 deferral.

D-F. **Integration with existing pipeline** — Phase-37
`buildProcessor` already folds `schemaOverrides` over the
base schema via shallow merge per APPEND-D-C. No
`buildProcessor` edit needed; framework's "zero-rework
consumption" property validated for the second consumer.
