### Alternatives considered (per-D-clause)

#### D-A. Scheduler runtime — Vercel Cron (chosen)

- **Vercel Cron** (chosen): native to the deployment platform; no extra
  service to provision; standard `vercel.json` `crons` config; built-in
  `vercel-cron` header injection; `CRON_SECRET` env-var auto-injection
  per Vercel's documented pattern.
- **GitHub Actions scheduled workflow**: workflow_dispatch + cron `on:
  schedule`. Pros: zero new infra. Cons: would need to call a public
  endpoint (re-introduces the auth-secret problem) OR commit results
  back to the repo (rejected — file-system content invariant per ADR-0004
  + Vercel runtime filesystem is read-only). Vercel Cron is the cleaner
  shape.
- **Inngest / Trigger.dev / external orchestrators**: full event-driven
  job runtime; overkill for a single weekly cron; adds dependency +
  vendor account.
- **In-process timer / `setInterval`**: rejected outright; Next.js App
  Router does not have a long-running process model; Vercel functions
  are request-scoped.
- **External cron (cron-job.org, EasyCron, etc.)**: re-introduces the
  endpoint-auth problem with no upside vs Vercel Cron.

#### D-B. Schedule — weekly Monday 00:00 UTC (chosen; closes D-9)

- **Weekly Monday 00:00 UTC** (chosen): "fresh-week" framing matches
  RSS-side `lib/digest/build-digest`'s 7-day window default; Monday
  morning aligns with most readers' work-week start; UTC avoids
  per-region tz drift.
- **Daily**: too noisy; volume × 7 = saturates Resend free tier at
  ~107 subscribers vs ~750 weekly.
- **Bi-weekly / monthly**: too sparse; loses the "weekly cadence" framing
  the welcome email + ADR-0021 prepared subscribers for.
- **Per-subscriber-preferred-day**: deferred Phase 32+ (D-H; requires
  schema column + UI; out-of-scope for foundation).

#### D-C. Send semantics — per-domain × per-subscriber loop (chosen)

- **Per-domain × per-subscriber nested loop** (chosen): outer loop
  iterates taxonomy domains; inner loop iterates verified subscribers
  who opted into that domain (via `parseDomainSubscriptions(row.domainSubscriptions).includes(domain)`).
  One email per (subscriber, domain) pair. Mirrors RSS-side per-domain
  iteration shape.
- **Per-subscriber × per-domain inverted loop**: same set of (subscriber,
  domain) pairs; slightly easier to batch by recipient but harder to
  share the per-domain `DigestPayload` build cost across subscribers.
- **Single aggregated email per subscriber** (cross-domain summary):
  deferred Phase 32+ (Q76 sibling consideration; needs UX decision +
  email-template shape change).

#### D-D. Auth boundary — `CRON_SECRET` bearer-token (chosen; partial-resolves D-12)

- **`CRON_SECRET` HTTP `Authorization: Bearer` header** (chosen): standard
  Vercel Cron pattern documented at vercel.com/docs/cron-jobs/manage-cron-
  jobs. Vercel injects `Authorization: Bearer <CRON_SECRET>` automatically
  on cron invocations. Route reads `req.headers.get("authorization")` and
  compares to `Bearer ${process.env.CRON_SECRET}` via constant-time
  comparison (mirrors Phase-30 `lib/subscribers/safeCompareTokens`).
- **Vercel's built-in `vercel-cron` header check ONLY**: weaker; `vercel-cron: 1`
  is set on cron invocations but `Authorization` header is the canonical
  auth surface.
- **No auth (rely on path obscurity)**: rejected outright; cron endpoint
  triggers 3,000 emails/month if abused.

#### D-12 fully — both belt-and-suspenders

- **Both `Authorization` Bearer + `vercel-cron` header presence check**
  (chosen for D-12): defense-in-depth. If either fails the request, return
  401. Reasoning: the `vercel-cron` header costs nothing to check and adds
  a second layer that catches the edge case where `CRON_SECRET` was
  somehow leaked but the attacker doesn't know to set the `vercel-cron`
  header on their forged request. Both layers must pass.

#### D-10. Idempotency — `subscriber.lastDigestSentAt` column (chosen)

Three options weighed:

- **Option α: `subscriber.lastDigestSentAt` column extension** (chosen):
  add one nullable `integer("lastDigestSentAt", { mode: "timestamp_ms" })`
  column to the existing `subscriber` table. Migration `0007_subscriber_last_digest_sent_at.sql`.
  Send loop checks `if (row.lastDigestSentAt && row.lastDigestSentAt >
  weekStartMs) skip`. On successful send, `UPDATE subscriber SET
  lastDigestSentAt = now`. **Smallest write surface** (1 column, no new
  table). **Explicit per-row idempotency guarantee** against Vercel Cron
  retries / function timeouts / partial batches. **Preserves USER-STATE
  invariant** per ADR-0013 D-F.

- **Option β: new `digest_send` audit table**: richer history (rows per
  (subscriber, domain, sent_at) tuple); 1 migration; enables future
  analytics. **Over-engineering for Phase 31** (foundation phase ships
  minimal surface; analytics is Q78 candidate per prep doc + Phase 32+
  D-H deferral). Defer until reliability/observability signal.

- **Option γ: stateless send-window key**: derive an idempotency key from
  the ISO week + subscriber id; expect Resend dedup or accept double-
  sends as cost of zero-migration. **Fragile**: Vercel Cron retries on
  5xx and on function timeouts; Resend has no first-party dedup;
  rejected. Cannot ship without a real persistence layer for the dedup
  state.

**Rationale for α**: Phase 30 already broke the 13-phase 0-migration
streak; Phase 31 paying a second consecutive migration is not paying
a fresh "first migration" cost. Single column on existing table is the
smallest possible write surface. Audit-table richness is deferred Phase
32+ if reliability signal emerges (Q78 candidate). Stateless approach
is fragile under partial-failure conditions.

**Consequence**: **first non-zero-migration phase 2 phases in a row**
since Phase 15-16 (mirrors Phase-15-16 series — `displayName` + `bio`
Phase 15 → `imageOverride` Phase 16); confirms the migration-cluster
pattern. Migration count 7 → 8; DB table count UNCHANGED at 7 (column
extension, not new table).

#### D-E. Send template — plain template-literal `DigestPayload → HTML` (chosen; closes D-11)

- **Plain template-literal string builder** (chosen): mirrors Phase-30
  `verification.tsx` + `welcome.tsx` shape verbatim. Inputs: `DigestPayload`
  (the same type RSS already consumes) + `unsubscribeUrl` + `supportEmail`.
  Iterates `payload.items` and renders each as a list entry with title +
  date + description. Defensive HTML-escape on all string inputs.
  Mobile-friendly width (`max-width: 600px`; standard transactional-
  email pattern).
- **React Email template**: deferred Phase 32+ per ADR-0021 D-H (only
  worthwhile when template count grows beyond ~5; Phase 31 = 3 templates).
- **MJML**: rejected at ADR-0021 D-H; same rationale.
- **Richer RSS-mirror shape (with images/charts)**: digest items have no
  images per Phase 5 `DigestItem` schema (no `imageUrl` field); chart
  images would require server-side rendering + CDN; rejected for Phase
  31 foundation.

#### D-F. Failure handling — per-row try/catch + log/skip; no retry queue Phase 31 (chosen)

- **Per-row try/catch + structured log** (chosen): a single subscriber's
  send failure does NOT halt the loop. On error, log `{ subscriber_id,
  domain, error_message }` and skip; the row's `lastDigestSentAt` is NOT
  updated so the next week's cron will re-attempt naturally.
- **First-failure abort**: rejected; one bad recipient address blocks
  the entire batch.
- **Retry queue (e.g., BullMQ / Inngest)**: deferred Phase 32+ per D-H;
  premature for foundation phase + adds infrastructure.

#### D-G. Resend free-tier ceiling — document + monitor under Q75 (chosen)

- **Document in ADR-0022 + add monitoring sub-step to Q75**: free tier
  3,000/month; weekly cadence × 750 subscribers = 3,000 sends/month;
  cron-route response includes `sent: N` count for observability. Q75
  operational gate includes "monitor monthly send count against tier
  ceiling; upgrade or switch provider before ceiling hit."
- **Hard-cap in code at N subscribers per send**: rejected; partial-list
  sends create unfair experience (some subscribers get digest, others
  don't, deterministic by row order). Better to monitor + upgrade tier
  when needed.
- **Switch to per-domain rate-limit ceilings**: deferred Phase 32+; no
  signal yet.

#### D-13. `vercel.json` `crons` config shape (chosen)

Standard top-level Vercel Cron config:

```json
{
  "crons": [
    {
      "path": "/api/v1/cron/digest-send",
      "schedule": "0 0 * * 1"
    }
  ]
}
```

- `path` field = absolute path under the deployment root (Vercel injects
  the host).
- `schedule` field = standard cron expression. `0 0 * * 1` = "at minute
  0 of hour 0 on Monday" = Monday 00:00 UTC weekly per D-B.
- Single-entry `crons` array; future cron jobs (Phase 32+ stale-token
  cleanup B.15 item 4, retry queue, etc.) extend the array.

**`vercel.json` is a NEW top-level config file** — confirms the Phase-31
prep doc's "first `vercel.json` config file in project history" anticipation.

### Edge cases & failure modes

1. **Empty `DigestPayload`** (zero rating actions + zero entries + zero
   discussions in 7-day window for a domain): skip the domain entirely;
   do NOT send empty-content emails. Subscribers' `lastDigestSentAt`
   stays NULL for that week so next week re-attempts cleanly.
2. **Cron triggered with missing `CRON_SECRET`** (local dev test;
   misconfigured env): return 401; do NOT proceed with the loop.
3. **Vercel Cron retry after function timeout**: `lastDigestSentAt`
   check skips already-sent rows; remaining rows resume. Eventual
   consistency over single-week boundary.
4. **Resend API outage**: per-row try/catch logs failures; loop completes;
   failed rows retry next week (no in-week retry; Phase 32+ D-H).
5. **Subscriber unsubscribed mid-loop**: `getVerifiedSubscribersForDomain`
   filters `status = "verified"` at SELECT time; rows that flip to
   `unsubscribed` between SELECT and send are still sent (acceptable —
   single-week race window; their next-week's send is skipped).
6. **Resend free-tier exhaustion mid-loop**: API returns 429 / 402;
   per-row try/catch logs; remaining rows skipped this week. Q75
   monitoring catches before this happens in steady state.
7. **`taxonomy` change during deploy mid-cron**: cron route reads
   taxonomy at request time; if a domain was removed between subscriber
   subscription and cron execution, the inner loop yields zero
   subscribers and the domain is skipped cleanly.
8. **`subscriber.domainSubscriptions` contains a non-existent domain**:
   `parseDomainSubscriptions` returns the array; the outer loop only
   visits known taxonomy domains; the orphaned domain ID is simply
   never matched (idempotent skip).
9. **Concurrent cron invocations** (Vercel does NOT promise non-overlap):
   `lastDigestSentAt` write inside `UPDATE WHERE` is the atomic guard;
   last-write-wins on the timestamp is fine. The double-send case
   reduces to "two near-simultaneous cron runs each see `lastDigestSentAt
   < weekStartMs` and both UPDATE + send" — this is the theoretical
   race window. **Acceptable for Phase 31**: SQLite per-row writes are
   serialized; the second UPDATE will succeed but the send may have
   already happened. Mitigation Phase 32+: SELECT FOR UPDATE / row-
   level lock pattern. Documented as a known limitation.
