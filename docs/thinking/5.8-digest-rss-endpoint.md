# 5.8 — `app/api/v1/digest/[domain]/route.ts` (per-domain RSS endpoint)

## THINK

### Inputs

- **Unit 5.7 (`6c33ed9`)** ships `buildDigest({ domain, windowDays?, now? })` returning a typed `DigestPayload` with items + channel metadata.
- **Unit 3.5's `app/api/v1/rss.xml/route.ts`** sets the RSS-rendering precedent: `xmlEscape`, `toRfc822`, channel + atom:link self-ref + items with title/link/guid/pubDate/dc:creator/description, `application/rss+xml` content-type, `cache-control: public, max-age=300`.
- **Q44 (open)**: `<managingEditor>` is gated on Q33 + Q2 (DNS). Lean: ship without `<managingEditor>` (W3C warns, not errors) until DNS resolves.
- **§13 Phase-5 deliverable**: "Email\RSS digest: per-domain weekly summary."

### Route-path deviation from Unit 5.0

**Unit 5.0's plan**: `app/api/v1/digest/[domain].xml/route.ts` (URL ending `.xml`).

**Realized**: `app/api/v1/digest/[domain]/route.ts` (URL without `.xml`).

Rationale: Next.js 15 App Router's dynamic-with-dot-suffix segment convention (`[domain].xml/`) is at best fragile on Windows (NTFS path quoting + git's handling of `.` in folder names) and at worst unsupported. The simpler `[domain]/` convention works reliably, the URL is still RSS-typed via `content-type: application/rss+xml`, and RSS readers don't care about URL suffixes. The deviation is purely a path-convention adjustment; the contract (one RSS feed per domain) is unchanged.

Trade-off flagged: the discoverability nudge of `.xml` in the URL is lost. Mitigation: Unit 5.9's `/digest` hub page links each domain with explicit "RSS" labelling.

### Scope (this unit)

1. **`app/api/v1/digest/[domain]/route.ts`** — Next.js App Router dynamic route. `generateStaticParams()` enumerates `taxonomy.domains[].id`. `GET({ params })` calls `buildDigest({ domain })` and renders RSS XML. SSG (`export const dynamic = "force-static"`).
2. **`app/api/v1/digest/[domain]/route.test.ts`** — vitest tests rendering against fixture digest payloads (no Next.js runtime needed; we import the `GET` function directly).

### Constraints

- **Mirror Unit 3.5's RSS shape** for cross-feed consistency. Channel-level: title, link, atom:link self, description, language, lastBuildDate. Item-level: title, link, guid (isPermaLink=false), pubDate, description. No `<dc:creator>` (digest items have no single curator; the channel itself is the "author").
- **`xmlEscape`** mirrors 3.5's 5-entity escape (`&` `<` `>` `"` `'`).
- **`toRfc822`**: `Date.toUTCString()` is RFC-822-compliant.
- **`atom:link rel="self"`** required for W3C validator.
- **No `<managingEditor>`** per Q44 lean (gated on Q33 + Q2 DNS).
- **`SITE` constant** matches Unit 3.5's `https://llm-openproblems.org` placeholder (Phase-0 stub; awaiting Q2 DNS).
- **Cache-control**: same `public, max-age=300, s-maxage=300` as Unit 3.5 — 5-minute freshness window.
- **Static SSG** — `generateStaticParams()` lists all taxonomy domains; each becomes a prerendered XML file at build time. Route count grows by N (one per domain).
- **Empty-window valid feed**: when `buildDigest` returns `items: []`, the RSS document is still parseable — just has an empty channel + the descriptive `channelDescription`.

### Route surface

```
GET /api/v1/digest/<domain-slug>

Response:
  Content-Type: application/rss+xml; charset=utf-8
  Cache-Control: public, max-age=300, s-maxage=300

  <?xml version="1.0" encoding="UTF-8"?>
  <rss version="2.0" xmlns:dc="..." xmlns:atom="...">
    <channel>
      <title>LLM OpenProblems — <Domain> digest</title>
      <link>https://llm-openproblems.org/digest</link>
      <atom:link href="https://llm-openproblems.org/api/v1/digest/<slug>" rel="self" type="application/rss+xml" />
      <description>...</description>
      <language>en</language>
      <lastBuildDate>...</lastBuildDate>
      <item>...</item>
      <item>...</item>
    </channel>
  </rss>

GET /api/v1/digest/<nonexistent>
  Status: 404
```

### Item rendering

Each `DigestItem` → one `<item>`:

```xml
<item>
  <title>${xmlEscape(item.title)}</title>
  <link>${SITE}${item.link}</link>
  <guid isPermaLink="false">${xmlEscape(item.guid)}</guid>
  <pubDate>${toRfc822(item.date)}</pubDate>
  <description>${xmlEscape(item.description)}</description>
</item>
```

No `<dc:creator>` because digest items include both rating actions (which have a curator) and leaderboard entries (which don't have a per-row author — the paper's authors are the natural attribution, but they're a list, not a single name). Simpler to omit at the item level; the channel `description` carries the editorial framing.

### Alternatives considered

- **`[domain].xml/route.ts` route pattern** (matches Unit 5.0's plan). Rejected — see route-path deviation note above.
- **Single combined RSS feed** for all domains (one URL, items tagged by domain). Rejected — §13's "per-domain" wording is explicit. Multiple feeds also let RSS readers subscribe selectively.
- **HTML page that auto-discovers via `<link rel="alternate">`** instead of a dedicated API route. Rejected — Unit 5.9's `/digest` hub will add `<link rel="alternate">` tags to its `<head>` pointing at the per-domain feeds for auto-discovery; that's complementary to the route, not a replacement.
- **Use Unit 3.5's exact channel description**. Rejected — digest channel describes recent activity, not the full rating-action log. Different framing.
- **Per-item `<source>` element** linking back to the parent problem feed. Considered. Rejected for v1 — adds noise; the `<link>` already navigates to the problem page.

### Tradeoffs / defensible defaults flagged

1. **Route path = `[domain]` not `[domain].xml`**: deviation from Unit 5.0's planned shape per the Next.js + Windows + git compatibility analysis above.
2. **`<atom:link rel="self">` uses the full canonical URL** (not relative). RSS validators require absolute URLs in self-refs.
3. **No `<dc:creator>` at item level**: simpler than per-item-kind branching. Channel-level framing carries the editorial source.
4. **`lastBuildDate` = `generatedAt`** from `buildDigest`. Trade-off: if `buildDigest` is called multiple times in one SSG build (it won't be — Next.js memoizes per build), the lastBuildDate could drift. Phase-5 acceptance gate confirms it's stable.
5. **Static export at build time**: every domain's feed prerenders on every `pnpm build`. Trade-off: subscribers see content at build time, not real time. Real-time would require dynamic rendering + the auth/deploy infrastructure Phase 5 explicitly defers (D-4).

### Tests

`route.test.ts` (≥ 5 tests):

1. **GET on a valid domain** returns 200 + RSS body.
2. **Content-type is `application/rss+xml`**.
3. **Channel metadata** present: `<title>`, `<link>`, `<atom:link rel="self">`, `<description>`, `<language>en</language>`, `<lastBuildDate>`.
4. **Items render** with title/link/guid/pubDate/description when the digest has items.
5. **404 on unknown domain**.
6. **XML entities escaped** (titles with `&` / `<` / `>` produce escaped output).
7. **Self-link URL** matches the requested domain.
