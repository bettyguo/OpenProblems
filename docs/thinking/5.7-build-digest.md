# 5.7 — `lib/digest/build-digest.ts` (per-domain weekly summary builder)

## THINK

### Inputs

- **Unit 5.0 D-4**: Phase 5 ships **RSS-only**. Email defers to Phase 6+ alongside auth.
- **Unit 5.0 D-12** (deferred to per-unit): digest cadence = trailing-7-days; empty windows produce empty (but valid) RSS feeds.
- **§13 Phase-5 deliverable**: "Email\RSS digest: per-domain weekly summary."
- **Available sources**:
  - Rating actions (`#site/content::ratings`, via `lib/content/load-ratings.ts`) — have `problem_slug`, `date`, `dimensions`, `watchlist`.
  - Leaderboard entries (`content/problems/<slug>/entries.json`, via `lib/content/load-entries.ts::loadEntriesForProblem`) — have `paper_id`, `benchmark_id`, `score`, `date`, `protocol_notes`.
  - Papers (`#site/content::papers`) — have `year` only (not ingest-date); skip as a digest source for v1.
- **Domain scope**: digest is per-domain. Each item maps to a problem via `problem_slug` → problem → `domain` field.
- **`#site/content::problems`** exposes `domain` (slug) on every problem object.

### Scope (this unit)

Three new files:

1. **`lib/digest/build-digest.ts`** — pure function `buildDigest({ domain, windowDays?, now? }) → DigestPayload`. Composes recent rating actions + new entries for problems in the named domain.
2. **`lib/digest/build-digest.test.ts`** — vitest tests against the real content + injected `now`.
3. **`lib/content/load-entries.ts`** is the existing loader; this unit consumes it. No changes needed.

The `app/api/v1/digest/[domain].xml/route.ts` (Unit 5.8) consumes this. The `/digest` hub page (Unit 5.9) also consumes it (for the empty-state messaging + per-domain item counts).

### Constraints

- **Trailing-7-days window** (D-12 default; `windowDays` parameter overrides).
- **Anchor = `now`** parameter (defaults to `new Date()`). Distinct from MoversBoard's "most-recent-action-date" anchor (Unit 3.0 D-8) — RSS readers expect wall-clock cadence, so the digest cadence anchors to the harness clock. Tests inject a fixed `now`.
- **Empty-window valid output**: items = []; channel-level `description` notes "no activity this week". Unit 5.8 still produces a parseable RSS document with no items.
- **No LLM call** in this unit. Pure-function summarization of existing content.
- **Async**: `loadEntriesForProblem` reads from disk; `buildDigest` is async to handle that fan-out.
- **`exactOptionalPropertyTypes: true`** — optional fields use `field?: T` and omit-when-undefined.

### Data shape

```ts
export interface DigestItem {
  /** "rating-action" or "leaderboard-entry". */
  kind: "rating-action" | "leaderboard-entry";
  /** Channel-display title (e.g. "Hallucination Reduction — saturation 35 → 32"). */
  title: string;
  /** Site-relative link (consumer wraps in canonical URL). */
  link: string;
  /** ISO date (YYYY-MM-DD) of the item. */
  date: string;
  /** Sentence-level description; ~1-2 sentences. */
  description: string;
  /** Stable id, used as RSS GUID. */
  guid: string;
  /** Slug of the problem this item references. */
  problemSlug: string;
  /** Title of the problem (denormalized for the consumer). */
  problemTitle: string;
}

export interface DigestPayload {
  domain: string;          // slug
  domainTitle: string;     // display title
  windowDays: number;
  generatedAt: string;     // ISO timestamp of the build
  cutoffDate: string;      // ISO date (YYYY-MM-DD); items have date >= this
  items: DigestItem[];     // newest-first
  channelTitle: string;
  channelDescription: string;
}

export interface BuildDigestOptions {
  domain: string;
  windowDays?: number;
  now?: Date;
}

export async function buildDigest(options: BuildDigestOptions): Promise<DigestPayload>;
```

### Item-construction rules

**Rating actions**:
- Filter `allRatingActions()` to those whose `problem_slug` resolves to a problem in `options.domain`.
- Further filter by `action.date >= cutoff`.
- Title: `${problemTitle} — ${headlineDelta(action) ?? "initial rating"}`. Reuse `diffRatingAction` from `load-ratings.ts` if available; for v1, just title with the action's slug+date.
- Description: 1 sentence summarizing the action's primary delta or "initial rating posted".
- Link: `/problems/<slug>/ratings#<action-id>`.
- guid: the action's stable Velite-injected `id`.

**Leaderboard entries**:
- For each problem in `options.domain`, load `entries.json`. Filter by `entry.date >= cutoff`.
- Title: `${problemTitle} — ${benchmark_id} ${score}`.
- Description: 1 sentence including `paper_id` + `score` + (if present) `protocol_notes`.
- Link: `/problems/<slug>` (no per-entry anchor surface yet — Phase-6+ enhancement).
- guid: `entry:<problemSlug>/<paper_id>/<benchmark_id>/<date>` (a stable composite; entries aren't currently uniquely-id'd in `entries.json`).

### Alternatives considered

- **Anchor window to most-recent-action-date** (mirror MoversBoard / Unit 3.0 D-8). Rejected — RSS readers expect wall-clock cadence; subscribers seeing "this week" framing expect actual recent activity, not relative-to-data-time activity. Trade-off: with the current simulated forward-dated rating histories (2026-09, 2026-12), the digest may be empty for a curator running it on `2026-05-15`. Acceptable — empty windows produce valid empty RSS per D-12.
- **Include papers as a third source**. Rejected — papers have `year` only, no ingest date. Future: track `ingested_at` in `papers/*.yaml` (via Unit 5.3's ingest CLI) and surface in digest.
- **Sort items by relevance / composite-rating-delta**. Rejected — newest-first matches RSS-reader expectations.
- **Per-domain channel description includes the problem-count + benchmark-count**. Considered. Rejected for v1 — keeps the channel description stable across runs (RSS validators sometimes warn on every-call channel-meta changes).

### Tradeoffs / defensible defaults flagged

1. **Wall-clock `now` default** vs. data-time anchor: digest cadence reads as "real time" to RSS subscribers, which is the right framing for a per-week digest even if the test fixtures produce empty windows.
2. **Single `windowDays` parameter** — no separate window per item-kind. Acceptable simplicity for v1.
3. **No item-de-duplication** between rating-actions and entries within the same window. A paper's first-published entry + the rating action it triggered will both appear. That's signal, not noise — keep them both.
4. **`guid` shape stability**: rating-actions use the Velite-injected `id`; entries use a composite. Composite collision risk: two entries on the same `(problem, paper, benchmark, date)` are functionally identical anyway; collision = duplicate-row = curator error, not digest error.
5. **No empty-domain handling beyond domain-not-found error**: `buildDigest({ domain: "nonexistent" })` throws. Trade-off: tighter contract; consumers handle 404.

### Tests

`build-digest.test.ts` (≥ 6 tests):

1. **Empty domain throws** — `domain: "nonexistent"` rejects with a "domain not found" error.
2. **Window cutoff respected** — items with `date < cutoff` are excluded.
3. **Items include rating actions** for problems in the named domain.
4. **Items include leaderboard entries** for problems in the named domain.
5. **Items exclude actions/entries for problems outside the domain** (cross-domain isolation).
6. **Items sorted newest-first**.
7. **Empty window produces empty items + valid description** (`items: []`, descriptive `channelDescription`).
8. **`generatedAt` matches the injected `now`**.
