## THINK

### Decisions pinned (per Unit 14.0 D-13 anticipated shape)

**D-A. Public-profile field partition.** Pins which fields are public
on `/u/{handle}` (GitHub-derived + status-partitioned aggregate counts
+ curator-of-record badge); which are submitter-only (`rejected` +
`withdrawn` counts); which are NEVER public (`email`;
`LOP_CURATOR_LOGINS` membership). Inherits Phase-13 Unit 13.0 D-3's
status-gated visibility partition verbatim. Establishes the
**public-data invariant**: no new public-data category introduced —
every public field is already public elsewhere.

**D-B. Handle routing canonical case.** Case-preserved URL +
case-insensitive DB lookup + NO redirect on case mismatch. Matches
GitHub.com's pattern. Implementation: `LOWER(githubLogin) =
LOWER(?)`. Profile body renders the canonical case from
`users.githubLogin` regardless of URL case.

**D-C. User-editable fields (Phase 15+ deferral).** Phase 14 ships
**READ-ONLY**. `users.name` + `users.image` populated from GitHub
OAuth profile; no editing surface in Phase 14. Q63 candidate
flagged for Phase 15+ if usage feedback demands.

**D-D. Per-user privacy opt-out (Phase 15+ flag; Q64 candidate).**
Phase 14 ships **NO opt-out**. Rationale: per D-A's public-data
invariant, no NEW public-data category is introduced. Q64 candidate
flagged. Footer link to `/about` placeholder for now; Phase 15+
expands to `/about/privacy` explainer.

**D-E. Curator-of-record badge case-sensitivity.** Case-**sensitive**
comparison between `editorial.primary_curator` (YAML literal) and
`users.githubLogin` canonical case. Phase 15+ may relax to
case-insensitive if observed mismatches accumulate.

**D-F. SiteHeader integration shape.** SiteHeader gets "Your profile"
link (signed-in only) → `/u/{login}` (PUBLIC canonical). `/profile`
becomes the EDIT-mode surface. Avatar 16×16 + `@login` truncated 12
chars. Hidden when `users.githubLogin === null` (Phase-9 retrofit
edge).

### Why an ADR at all?

Two reasons:

1. **Per-user privacy model crosses §13 + ADR-0012 + ADR-0014 + Phase-13
   Q58 boundaries**. Pinning the field partition + opt-out posture +
   case-sensitivity contract in ONE document prevents downstream drift
   when Phase 15+ writes surface ships.
2. **Q58 lean #3 deferral was explicit** (Unit 13.0 D-9 + Unit 13.5
   promotion note): "per-user privacy model needs its own ADR". Phase
   13 deferred the per-user surface PRECISELY to give Phase 14 room to
   land this ADR. Not landing one would be a documentation regression.

Contrast Phase 13: Q58 was promoted to resolved WITHOUT an ADR
because (a) the lean had been documented across Unit 11.6 + Unit 12.0
D-10 + Unit 13.0 D-3 (three forward references converging on the
per-status partition) and (b) Phase 13's surfaces were small enough
(counter + listing) to absorb the lean directly. Phase 14's surfaces
are larger (route tree + helpers + SiteHeader integration +
curator-of-record scan) and the architectural contract spans
multiple D-clauses — ADR-warranted.

### Options + chosen rationale (mirrors ADR body)

Four options considered:

1. **GitHub-derived public + USER-STATE aggregate counts per Phase-13
   partition + read-only Phase 14 + case-preserved-URL + curator-of-
   record case-sensitive** (CHOSEN). Honors the four-phase honored-
   deferral lineage; no new public-data category; no new dependencies
   / env vars / migrations; scope-cap discipline (9 units).
2. **Full per-user-editable Phase 14**. REJECTED. Doubles scope
   (writes surface + new migration + XSS audit + image upload
   pipeline ADR). Phase 15+ writes surface is the natural home.
3. **Opt-in public profile (default private)**. REJECTED. Adds writes
   surface + breaks Phase-13 `@login` link future. No new public data
   is introduced; defaulting to opt-out is over-conservative.
4. **`/[locale]/users/[handle]` instead of `/u/[handle]`**. REJECTED.
   `/u/{login}` matches GitHub's pattern + ADR-0012 identity inheritance;
   `/authors/[slug]` already exists for paper authors (different
   identity system); conflating both would create routing confusion.

### Anticipated open questions surfaced (NOT promoted this unit)

Three Phase-15+ Q-candidates flagged in D-C + D-D + D-E + Cross-
references:

- **Q63 candidate** — user-editable profile fields (D-C deferral).
  Requires new writes API + new DB columns (`users.displayName`,
  `users.bio`, possibly `users.imageOverride`) + new migration +
  XSS sanitization + possible Vercel-Blob image-upload ADR.
- **Q64 candidate** — per-user privacy opt-out toggle (D-D deferral).
  Requires `users.profilePublic` boolean column + migration + writes
  surface + 404 response on `/u/{handle}` when opted-out.
- **Q65 candidate** — per-curator activity feed / contribution
  timeline (D-E expansion). Requires `getCuratorActions(handle)` helper
  + new `/u/{handle}/curated` sub-route + rating-action history join.

All three remain as flagged candidates, not promoted to `Status: open`
this unit. Phase-14 acceptance gate (Unit 14.8) will assess promotion
status; if no signals surface during Phase 14, the candidates stay
flagged for Phase 15+ kickoff to decide.
