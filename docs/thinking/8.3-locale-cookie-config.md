# 8.3 — `NEXT_LOCALE` cookie configuration on middleware

## THINK

- **[ADR-0011 D-F](../adr/0011-i18n-strategy.md)** — locale-toggle "persists via URL (no cookie needed for state; cookie used only for first-visit Accept-Language hint)."

### What the cookie does

next-intl's middleware sets a `NEXT_LOCALE` cookie on its response when a user is redirected to a locale-prefixed URL. The cookie holds the user's last-chosen locale. On future requests to bare URLs (which our `localePrefix: "always"` middleware redirects to a localized variant), the middleware reads the cookie first, falls back to `Accept-Language`, and finally to `defaultLocale`. This means a first-visit user lands on `/en/...` or `/fr/...` based on browser language; returning visitors keep their last-clicked-locale.

### Scope (this unit)

Edit `middleware.ts` to explicitly configure `localeCookie`:

| Field | Value | Reason |
|-------|-------|--------|
| `name` | `NEXT_LOCALE` | next-intl default; spelled here for clarity (no rename) |
| `maxAge` | `60 * 60 * 24 * 365` (1 year, in seconds) | Long enough that returning visitors keep their choice; not so long that stale browser state outlasts a reasonable curator turnaround. Matches next-intl's documented recommendation. |
| `sameSite` | `lax` | Required for top-level navigation cookies; the LocaleToggle navigates within-origin only. `strict` would break the `<Link>`-based toggle on cross-tab visits. |
| `path` | `/` | Cookie applies site-wide, not per-segment. |
| `secure` | `process.env.NODE_ENV === "production"` | `localhost` dev and `pnpm start` smoke tests need the cookie over HTTP. CI Lighthouse + Vercel previews run under HTTPS. |

`httpOnly` intentionally left at next-intl's default (false). Today only the middleware reads the cookie (server-side). Setting `httpOnly: true` would harden against XSS-driven cookie reads but pre-empt future client-side personalization (e.g., LocaleToggle reading the cookie to surface a "remember this choice" UX).

### Scope NOT in this unit

- **Custom cookie name** (e.g., `op_locale`): rejected. Conventional `NEXT_LOCALE` is widely understood by tooling and crawlers; deviating buys nothing.
- **Encrypted / signed cookie**: rejected. Cookie content is the locale code (`en` / `fr`), not a secret. Tampering forces a re-detection on the next request; no security risk.
- **Disabling the cookie entirely** (`localeCookie: false`): rejected. Without the cookie, every bare-URL visit re-runs Accept-Language detection — annoying for users whose browser language differs from their preferred site language.
- **Multi-locale tracking** (e.g., remembering FR for marketing pages and EN for docs): out of scope. The locale is a single global choice today; segment-scoped locale preferences would be a Phase-9+ concern coupled to user accounts.

### Alternatives considered

- **Leave defaults implicit** (Unit 8.1's `31ea2d5` state): rejected. Defaults work but the next-intl docs explicitly recommend pinning the configuration for posterity and review. ADR-0011 D-F said the cookie configuration would be "pinned in Unit 8.3" — this is that pin.
- **Configure via `next.config.ts` instead of inline**: not applicable. next-intl's cookie configuration lives on the middleware, not on the Next config.
- **Add a Storybook story / vitest test**: deferred. The cookie behavior is integration-level (browser ↔ middleware); the existing routing/`routing.test.ts` covers the locale-set-of-truth, and Unit 8.1's smoke pass validated middleware redirects end-to-end. A dedicated middleware test would mock `NextRequest` headers — useful but Class B follow-on, not a Phase-8 deliverable.

### Edge cases & failure modes

1. **First-visit user without `Accept-Language` header** (rare; misconfigured client): falls through to `defaultLocale = "en"`. Acceptable.
2. **User clears cookies between visits**: middleware re-detects via `Accept-Language` on next visit. Acceptable.
3. **Cookie set on `/api/...` requests** (server-to-server API calls): middleware matcher excludes `/api`, so the cookie isn't set on API responses. Correct — programmatic API clients shouldn't get the cookie.
4. **`secure: true` blocks the cookie on `localhost` HTTP**: avoided by `process.env.NODE_ENV === "production"`-gating the flag.
5. **Browser without cookie support** (private mode, restrictive settings): middleware falls back to `Accept-Language` on every request. Slight perf cost; no functional break.
