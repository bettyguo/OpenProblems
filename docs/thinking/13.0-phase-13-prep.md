### D-3. Q58 visibility policy (per-status)

Per Unit 11.6 Q58 lean + Unit 12.0 D-10 reaffirmation:

| Status         | Public on problem detail counter? | Public on per-problem listing? | Submitter-only on profile? |
|----------------|-----------------------------------|--------------------------------|----------------------------|
| `submitted`    | **YES** (in "pending" count)       | **YES** (with truncated rationale) | YES                   |
| `under_review` | **YES** (in "pending" count)       | **YES** (with truncated rationale) | YES                   |
| `accepted`     | **YES** (in "accepted" count)      | **YES** (with truncated rationale + `acceptedActionId` link) | YES |
| `rejected`     | **NO**                             | **NO** (preserves submitter privacy on editorial decisions) | YES (submitter sees on own profile) |
| `withdrawn`    | **NO**                             | **NO** (submitter chose this; preserve privacy) | YES (submitter sees on own profile) |

**Rationale**:

- `submitted` + `under_review` are "in flight" public records. Showing
  them publicly invites community awareness + accountability (the
  curator team knows the community sees the queue depth).
- `accepted` is editorial-record-shaped (the challenge contributed to
  a new rating action); fully public.
- `rejected` carries an editorial decision against the submitter's
  argument; making it public would create a "shame surface". Submitter
  sees it on own profile; non-submitters don't.
- `withdrawn` is submitter-initiated termination; revealing it
  publicly would expose the submitter's change of mind. Submitter-only.

**Submitter login visibility**:
- `submitted` / `under_review` / `accepted` public surfaces show
  submitter's `githubLogin` (i.e., the `@handle`). Reasoning: GitHub-OAuth
  sign-in inherently exposes the login publicly via problem-detail-page
  attribution + future public profile route. Hiding it on Q58 surfaces
  would be inconsistent.
- `rejected` / `withdrawn` private surfaces never reach non-submitters.

### D-6. New helper signatures (`lib/rating-challenges/`)

Phase-12 already shipped `getOpenChallengeCountByProblem(slug)` —
reused for counter (per D-4). Phase 13 adds:

```ts
export interface PublicChallengeRow {
  id: string;
  problemSlug: string;
  submitterLogin: string | null;   // null for Phase-9 retrofit edge
  submittedDate: string;            // ISO YYYY-MM-DD
  dimension: Dimension;
  proposedValue: string;
  rationale: string;
  status: "submitted" | "under_review" | "accepted";  // public subset
  acceptedActionId: string | null;  // non-null when status === "accepted"
}

export async function getPublicChallengesByProblem(
  problemSlug: string,
): Promise<PublicChallengeRow[]>;

export async function getAcceptedChallengeCountByProblem(
  problemSlug: string,
): Promise<number>;
```

Implementation joins `ratingChallenges` to `users` on `userId` to get
`submitterLogin` (matches existing `getLoginById` join shape).
Filters `status IN ('submitted', 'under_review', 'accepted')` —
excludes rejected + withdrawn per D-3. LIMIT 50 (mirrors
`CHALLENGES_LIMIT` in module).
