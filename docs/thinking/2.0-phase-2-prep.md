### Constraints

- **Acceptance gate (¬ß13).** `scripts/cross-link-audit.ts` must run
  green, and new Playwright visual regression baselines must be
  captured. The cross-link audit is the unit that gates the rest;
  every Phase-2 commit before it should be authored with the audit's
  rules in mind so the final unit is a script-and-run, not a
  back-fill.
- **Bidirectional `related_problems`.** OPEN_QUESTIONS Q33 (cross-
  link audit) was flagged in Unit 1.11 as a Phase-2 concern. The
  audit is the natural home: at Phase-1 close, several Unit 1.11
  problems have one-direction `related_problems` edges. The audit
  either enforces bidirectionality (require both ends to point at
  each other) or simply reports asymmetric edges as warnings.
  Lean: warn-on-asymmetry, not block ‚Äî enforcing bidirectionality
  forces every backref update to also stage the opposite end,
  which fights the parallel-curator workflow committed in
  `021bf7e`. Flag as a Phase-2 decision.
- **Slug discipline.** `author.slug` and `institution.slug` are now
  load-bearing ‚Äî Phase-2 content drops the author/institution
  slugs into `paper.authors[]` / `paper.institutions[]` (arrays of
  `slug`), and Velite joins on them. Slugs must be globally unique
  inside each entity (no two authors with the same slug, no two
  institutions with the same slug). Authors that share a personal
  name disambiguate by slug suffix (`jane-smith-mit`,
  `jane-smith-stanford`).
- **¬ß15.6 no-invent rule.** Paper TLDRs, scores, and contributions
  are all subject to the no-fabrication rule. A paper YAML may
  include a `TODO(curate)` score and a `[TLDR pending human review]`
  string per `docs/CURATION_PROMPT.md` ¬ß6. Authors and institutions
  may carry minimal facts (display_name + optional homepage / ROR /
  ORCID) where the public record supports them; otherwise the slug
  references stay valid even if the entity YAML body is sparse.
- **Velite + Zod 4 incompatibility (Q31).** Carries forward to
  Phase 2. Each new collection in `velite.config.ts` duplicates its
  schema body in `s.object({...})`. `scripts/validate-content.ts`
  remains the authoritative validator (it imports the canonical
  Zod-4 schemas from `lib/schemas/*`).
- **Font subset (Q16) is no longer "future work".** Phase 2 brings
  author names that carry diacritics (`≈Åukasz`, `M√ºller`, `Fran√ßois`,
  `Yejin Choi` is ASCII but `≈Åukasz Kaiser` is not). The visual
  register breaks the moment the first diacritic falls back to the
  system font. Lean: expand `next/font/google` `subset: ["latin",
  "latin-ext"]` in Unit 2.1 (the authors-and-institutions content
  unit). Cost: ~20% per font family, ~60 KB total at the project's
  three families.
- **Composite per-problem leaderboard ordering.** ¬ß13 says "sortable,
  filterable table" but does not fix the default sort. Lean: sort by
  `score` (with `metric_direction` honoured) within a chosen
  benchmark, then break ties by `date` desc. Flag for confirmation.
- **`entries.json` shape vs. per-paper `contributions[]`.** The
  PaperSchema already carries `contributions[]` with
  `{ problem_slug, benchmark_id, score, metric, rank_at_publication,
  evidence }`. The Phase 2 `entries.json` per-problem feed
  is the *denormalised* view (paper ‚Üí entry projections), kept in
  the problem's folder so the problem detail page can render the
  leaderboard without traversing the global paper index at request
  time. Source of truth lives on the paper YAML; the entries.json
  is regenerated from contributions. The cross-link audit verifies
  the two views agree.

### Alternatives considered

- **Author papers detail page (`/papers/[id]`) before any paper
  content lands.** Cleaner sequencing ‚Äî Phase-0-style stub-then-fill.
  Rejected ‚Äî without at least one real paper YAML the page renders
  no useful skeleton (the ¬ß9 four-block paper-detail layout depends
  on rendered `contributions[]` and `authors[]` joins). Author the
  first paper batch (Unit 2.4) at the same time as the detail page
  (Unit 2.7), so the detail page has live data to validate against.
- **Skip authors/institutions in Phase 2; render paper detail pages
  with author names as plain strings.** Saves Unit 2.1 and Unit 2.2
  for Phase 3. Rejected ‚Äî ¬ß13 explicitly names "Paper / Author /
  Institution detail pages" as Phase-2 deliverables, and the
  aggregate rollups (cumulative author impact, institution
  subdomain coverage) require author/institution slug-keyed pages
  to link to.
- **Run cross-link audit as a Velite build-time check rather than
  a separate script.** Velite collection schemas can express
  inter-collection FK constraints. Rejected ‚Äî Velite enforces at
  build time but does not produce the rich
  "paper X claims contribution to problem Y but problem Y's
  signals_considered does not name paper X" reports the ¬ß13
  acceptance gate calls for. Keep `scripts/cross-link-audit.ts`
  as the authoritative pass; Velite continues to enforce the
  per-collection schema.
- **Generate `entries.json` from `contributions[]` at build time
  (no committed file).** Argued for by ADR-0004 (file-first, no
  DB) ‚Äî `entries.json` is a derived view, not a source of truth.
  Rejected for Phase 2 ‚Äî committing the file keeps the
  `validate-content` pass cheap, makes `git blame` work on
  benchmark numbers, and lets a future curator edit a verified
  flag without re-running a build script. Re-evaluate at Phase 3.

### Edge cases

- **Paper `arxiv_id` vs `doi` as the filename.** PaperSchema makes
  both optional. Convention: filename is the `id` field (which the
  schema requires); for papers with an arXiv, `id == arxiv_id`. For
  papers without an arXiv but with a DOI, `id == "doi-" +
  slugified-doi`. Otherwise `id == kebab(title)`.
- **Author with no homepage / ORCID / Scholar.** AuthorSchema makes
  all three optional. A bare `{ slug, display_name, affiliations:
  [] }` validates. Phase-2 audit script should warn on bare authors
  (low signal) but not block.
- **Institution `ror_id`.** ROR is preferred but optional. Phase 2
  authors `ror_id` where I can find it (5 minutes of lookup per
  institution at curation time); skips otherwise.
- **A paper's listed authors include curators of this site.** ¬ß8.6
  COI rules apply at the rating-action level (a curator does not
  rate a problem they are co-author on). They do not block authoring
  the paper YAML itself. Flag in the paper's `notes:` field if it
  applies.
- **A benchmark cited in `contributions[].benchmark_id` is not in
  the parent problem's `benchmarks[]`.** This is a content bug and
  the cross-link audit should fail. Phase-2 curation discipline:
  add a benchmark to the problem before citing it on a paper.

### Tradeoffs / defensible defaults flagged

1. **Font subset ‚Üí `["latin", "latin-ext"]`** (Q16 resolved in
   Unit 2.1). Reason: author diacritics arrive now; visual register
   breaks otherwise. Cost: +60 KB across the three font families.
2. **Cross-link audit reports asymmetric `related_problems` as
   warnings, not failures** (Q33 lean). Reason: enforcing
   bidirectionality fights the per-slug-bounded parallel-curator
   workflow. Curators can fix backrefs in a follow-on pass.
   Surfacing to OPEN_QUESTIONS.
3. **`entries.json` is a committed file, not a build-time
   derivation.** Reason: cheap validation, git-blame on numbers,
   curator-editable verified flag. Costs an extra audit rule
   (entries.json must agree with contributions); the audit owns it.
4. **Sort default on leaderboards: `score` (direction-aware) then
   `date` desc.** Reason: matches what readers expect; can be
   re-sorted client-side. Surfacing to OPEN_QUESTIONS for
   confirmation.
5. **Author seed-set size: ~10 authors, ~8 institutions** to back
   ~30‚Äì40 papers. The ¬ß13 floor is 30 papers; 40 is the target.
   Authors-per-paper averages ~3 in this seed but the same authors
   recur across papers, so the unique author count is small.

### OPEN_QUESTIONS surfaced by Phase 2

- **Q32. Cross-link audit strictness on asymmetric
  `related_problems`** ‚Äî block or warn? Lean: warn.
- **Q33. Default sort order on per-problem leaderboards** ‚Äî score
  (direction-aware) then date desc? Other choices: date desc only;
  verified-only by default with a toggle.
- **Q34. Aggregate-rollups composite weighting on author pages.**
  Cumulative-impact score on `/authors/[slug]` needs a function
  over the author's contributions (sum of `score`? rank-weighted?
  problem-weighted by composite rating?). Phase 3's recompose UI
  may obsolete the choice; Phase 2 picks a simple default.
- **Q35. Verified-flag provenance.** Who can set
  `LeaderboardEntry.verified = true`? Phase 2 default: only the
  primary curator, set at author time. Phase 4+ when submissions
  open, requires a separate verification step.

(These are the four expected; new ones may surface as units land.)
