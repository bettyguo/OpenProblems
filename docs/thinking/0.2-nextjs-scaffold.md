# Unit 0.2 — Next.js 15 + React 19 + TS strict scaffold

> Stage 1 (THINK) artifact for Unit 0.2.

## Problem restatement

Land a minimal, buildable Next.js 15 App Router project that materializes the application-framework decision (ADR-0001) and the package-manager decision (Q12 → pnpm). The scaffold must be the foundation that subsequent units (0.3 styling, 0.5 schemas, 0.7 validate-content, 0.8 lint, 0.9 tests, 0.10 IA stubs, 0.11 CI, 0.12 deploy) layer onto without rework.

Definition of "minimal":

- `pnpm install` runs clean.
- `pnpm typecheck` passes.
- `pnpm build` produces a `.next/` directory and exits 0.
- A single placeholder route at `/` renders "LLM OpenProblems · Phase 0 scaffold" with zero JS shipped to the client (RSC-only).

This unit is explicitly **not** the place for Tailwind (Unit 0.3), shadcn/ui (Unit 0.3), fonts (Unit 0.3), design tokens (Unit 0.4), ESLint config (Unit 0.8), Vitest / Playwright / Storybook (Unit 0.9), or any of the §9 route stubs (Unit 0.10). Each of those is a separate unit with its own decisions.

## Inputs

- ADR-0001 (Next.js 15 App Router + React 19 + TypeScript strict, accepted).
- ADR-0003 (Zod 4 — pinned major). Zod is *not* installed yet; it lands in Unit 0.5. But the TS strict settings here must already support its inferred types (`noUncheckedIndexedAccess` and `exactOptionalPropertyTypes` per §14.1).
- MASTER_PROMPT.md §5.1 (framework rationale), §6 (folder layout: `app/`, `components/`, `lib/`, `content/`, `public/`, `scripts/`, `docs/`), §14.1 (TS strictness + no barrel files + component file structure).
- Q12 resolved: pnpm. pnpm 11.1.2 is installed.
- Node v22.14.0 (LTS Active) installed.

## Constraints

- **TypeScript strictness floor:** `strict: true`, `noUncheckedIndexedAccess: true`, `exactOptionalPropertyTypes: true` (per §14.1).
- **App Router only** — `app/` at repo root, no `src/` (per §6).
- **No Tailwind in this unit** — Unit 0.3 owns CSS choices. Use a minimal `globals.css` with a CSS reset only.
- **No ESLint config** — Unit 0.8 owns it. Suppress Next.js's built-in ESLint hint when present.
- **Lockfile committed.** `pnpm-lock.yaml` must be tracked (§5.9 mandates lockfile for reproducibility; my Q12 resolution states so).
- **Engines pinned.** Node `^22.x` in `package.json#engines`; pnpm pinned via `packageManager` field.
- **Path aliases.** `@/*` → repo root (so `@/lib/schemas/...`, `@/components/viz/...` work cleanly without barrels — §14.1 forbids barrel files).
- **No bundle leak.** The Phase 0 stub landing must ship 0 JS to the client (RSC-only). Lighthouse perf ≥ 95 is the Phase 0 acceptance gate (Unit 0.12); the scaffold should make hitting that gate easy.

## Alternatives considered

1. **Manual scaffold** (chosen): hand-author `package.json`, `tsconfig.json`, `next.config.ts`, `app/layout.tsx`, `app/page.tsx`, run `pnpm install`.
2. **`pnpm create next-app`** in a temporary directory, then merge non-conflicting files back into the repo.
3. **A starter template** (e.g., Vercel's official Next.js 15 + TS template, or shadcn/ui's `next.js` example).

Trade-offs:

- (1) is more work upfront but produces *exactly* the surface we want — no Tailwind, no ESLint, no `src/`, no example `globals.css` we'd just delete. We control every line.
- (2) is faster but creates clashing files (README, .gitignore, package.json) we'd have to triage. The merge step would re-introduce churn — and we'd still hand-edit tsconfig for `noUncheckedIndexedAccess` and `exactOptionalPropertyTypes` (neither is on by default in any template).
- (3) starter templates ship with opinions (Tailwind, ESLint config, sample components) that conflict with our phased unit plan. Stripping those is more work than starting fresh.

Chosen: **(1) manual scaffold.** ~6 files of hand-authored config is cheaper than triaging template debris.

## Edge cases & failure modes

1. **`pnpm install` network failure.** The user is on Windows-China; npmjs.org access may be slow. Mitigation: rely on pnpm's default registry; if it fails, fall back to `pnpm install --registry=https://registry.npmmirror.com` only as a one-off (do not commit a mirror in `.npmrc` — that's a user preference, not a project decision).
2. **React 19 + Next 15 compatibility.** Next 15 ships with React 19 by default; pin both to compatible minor versions. Use Next `15.x` latest and let it choose React.
3. **`exactOptionalPropertyTypes` strictness breaks third-party type defs.** Some packages (especially those targeting older TS) have lax optional types. Mitigation: keep this on from day one — the cost of turning it on later is much higher than the cost of dealing with it now. If a specific library blocks us, narrow with a `declare module` shim rather than dropping the flag.
4. **`next-env.d.ts` is auto-generated.** It's already in our `.gitignore` (Unit 0.0). Next will rebuild it on every dev/build; we never commit it.
5. **Phantom dependency on Tailwind / ESLint / others.** A future contributor running `pnpm install` should not get prompted by `create-next-app`-style postinstall scripts. Manual `package.json` avoids this.
6. **Lockfile drift on different OSes.** Optional `engines.strict` enforcement is too aggressive for a small project; rely on the `packageManager` field plus a `.nvmrc` to coordinate Node versions across contributors.
7. **App Router caching surprises in dev vs prod.** Stub page is pure RSC; no `fetch`, no `cache`, no `revalidate` — caching semantics are irrelevant in Unit 0.2 and become a real concern only at Phase 3 (rating-action publishing).
8. **CSS clash with Unit 0.3.** Unit 0.3 will introduce Tailwind v4 via `@import "tailwindcss"` in `app/globals.css`. The minimal `globals.css` we ship in 0.2 should be one Edit away from a Tailwind-ready version — keep it tiny: a CSS reset and a `:root` font stack only.

## Open questions surfaced (→ OPEN_QUESTIONS.md)

- **Q14. React Compiler.** Next 15 supports the React 19 Compiler experimentally. Opt in now (Phase 0) or later (Phase 1+)? Lean: later — the compiler's behavior on viz code (D3, react-force-graph) is undertested. Defer to Phase 1.
- **Q15. Node version policy.** Pin to Node 22 LTS Active across CI and contributor `.nvmrc`? Or follow Node Active LTS as it rolls? Lean: pin to 22 now; bump via ADR when 24 hits LTS Active. (`.nvmrc` will say `22`, `package.json#engines.node` will say `>=22 <24`.)

## Decision

Author the following files in this unit:

1. `docs/thinking/0.2-nextjs-scaffold.md` — this file.
2. `package.json` — `next@15`, `react@19`, `react-dom@19`, `@types/node@22`, `@types/react@19`, `@types/react-dom@19`, `typescript@5`. Scripts: `dev`, `build`, `start`, `typecheck`. `engines.node`, `packageManager`.
3. `tsconfig.json` — strict + `noUncheckedIndexedAccess` + `exactOptionalPropertyTypes` + `@/*` paths.
4. `next.config.ts` — minimal, TS-typed; explicitly disable Next's bundled ESLint integration so CI doesn't run two linters once Unit 0.8 lands.
5. `app/layout.tsx` — root layout with `<html lang="en">`, minimal head metadata.
6. `app/page.tsx` — placeholder route returning `"LLM OpenProblems · Phase 0 scaffold"`.
7. `app/globals.css` — CSS reset + `:root` font stack. No Tailwind directives.
8. `.nvmrc` — `22`.
9. `pnpm-lock.yaml` — generated by `pnpm install`, committed.

Stage 2 (DESIGN): the file shapes above are the design. Each is small enough that the THINK file list serves as the design artifact.

Stage 3 (CODE): write the files, run `pnpm install`, then `pnpm typecheck` and `pnpm build` as smoke tests. No production fetch/cache wiring; no client components.

Stage 4 (ITERATE): reviewer-mode checklist, CHANGELOG entry, surface Q14 + Q15 to OPEN_QUESTIONS.md, summary to human. This unit is not critical-path (per §15.4), so it may proceed without an additional checkpoint after the summary.
