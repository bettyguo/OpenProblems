# 4.9 — Issue template: leaderboard-entry

## THINK

### Inputs

- **Unit 4.0 D-8 + Q37**: form-based `.yml`; 3–5 required fields + 2–4 optional + free-text notes.
- **Units 4.7 (`de9460b`) and 4.8 (`1954b1d`)** establish the convention: form-based YAML, `[<Type>]` title prefix, `<type>` + `needs-triage` labels, intro markdown linking workflow docs + live data dirs, freeform inputs over enumerated dropdowns.
- **`LeaderboardEntrySchema`** (`lib/schemas/entry.ts`) required fields: `paper_id`, `benchmark_id`, `score`, `date`, `verified` (boolean). Optional: `protocol_notes` (free text — protocol description + source URL + model details).
- **Existing precedent in `content/problems/hallucination-reduction/entries.json`** (Unit 2.6h): 3 entries on SimpleQA, each with `paper_id: 2411.04368`, model details + source URL in `protocol_notes`. The protocol-notes format that's emerged is freeform but consistent: `Model: <name>. Source: <url>. <metric definition>.`
- **`benchmark_id` must match a benchmark declared in the parent problem's `benchmarks[]`** in `problem.yaml`. The submission form can't validate this client-side; helper text directs contributors to look it up.

### Scope (this unit)

`.github/ISSUE_TEMPLATE/leaderboard-entry.yml` — captures enough to draft an `entries.json` row. Verification happens during curator review; submitters don't self-attest.

### Constraints

- **`verified: boolean` is curator-side.** §15.6 (primary-source rule) + the curator's judgement determine whether an entry is verified. The submitter provides the source; the curator labels.
- **`benchmark_id` is a slug, not a display name.** Helper text instructs looking up the slug in the parent problem's `problem.yaml` `benchmarks[]` (each declared with an `id:` field).
- **Mirror Units 4.7 / 4.8** structural decisions: form-based, label scheme, GitHub URLs pinned to `github.com/bettyguo/OpenProblems`.
- **Quote any input/placeholder value that contains a `:`** to avoid prettier's YAML-parse failure (the Unit 4.8 quoting bug).

### Form-field schema

7 body items: 1 markdown intro + 6 form fields.

| # | Item                                  | Type       | Required | Notes                                                                  |
|---|---------------------------------------|------------|----------|------------------------------------------------------------------------|
| 1 | Intro markdown                        | markdown   | n/a      | Links MASTER_PROMPT.md (§9 leaderboard framing + §15.6 primary-source), `content/problems/` for benchmark-slug lookup. Notes the `verified` boolean is curator-side. |
| 2 | Problem slug                          | input      | yes      | Existing problem (`content/problems/<slug>/`). Identifies which `entries.json` the row lands in. |
| 3 | Benchmark id                          | input      | yes      | Must match a `benchmarks[].id` in the parent problem's `problem.yaml`. Helper text links the directory. |
| 4 | Paper id                              | input      | yes      | Paper that reports the score. arXiv ID typically. If the paper isn't yet ingested, helper text links the `[New paper]` template. |
| 5 | Score                                 | input      | yes      | Numeric. Helper text reminds about metric direction (`higher-is-better` / `lower-is-better` per the benchmark's declared `metric_direction`). |
| 6 | Date (ISO `YYYY-MM-DD`)               | input      | yes      | When the result was obtained / published. Format spec `YYYY-MM-DD` per `LeaderboardEntrySchema.date`. |
| 7 | Protocol notes                        | textarea   | no       | Freeform — model name, source URL, metric definition, eval setup. Established format: "Model: …. Source: …. <metric definition>." |

### Alternatives considered

- **Surface `verified: boolean` as a `checkboxes` field.** Rejected — the submitter isn't the verifier; presenting the option invites self-attestation that the schema would then need to override. Curator workflow owns the flip.
- **`dropdown` for `problem_slug` / `benchmark_id`** from a static list. Rejected — same staleness argument as 4.7 / 4.8 freeform pattern. Slug count is small (10 problems × 1–4 benchmarks ≈ 30 slugs total today) but evolves.
- **Single combined "Paper + score + date" field**. Rejected — the schema treats these as separate columns; structured form inputs map 1:1 to the row, simpler for the curator to translate.
- **`number` type for `score`**. Considered. GitHub Forms supports validation-free numeric inputs but doesn't have a built-in number type for `body[].type`; rendering `type: input` with helper text "numeric" is what's available. Curator parses on ingest.

### Tradeoffs / defensible defaults flagged

1. **5 required fields, not 3–5.** Q37's lean tops at 5; this template hits the ceiling because all 5 (`problem`, `benchmark`, `paper`, `score`, `date`) are load-bearing for the row schema. Lower-bound the friction by making `protocol_notes` optional + verbose-only.
2. **`source URL` not separated from `protocol_notes`.** The existing entries.json examples (Unit 2.6h) interleave the source URL into the protocol-notes prose. Separating would be cleaner but breaks the established freeform convention.
3. **Cross-validation deferred to ingest.** Benchmark-id-exists-on-problem, paper-id-already-ingested, and date-in-future checks all happen during curator review; no form-level enforcement.
