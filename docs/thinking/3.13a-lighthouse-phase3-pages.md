# 3.13a — Add Phase-3 pages to lighthouseci config

## THINK

### Inputs

- **Unit 3.13 acceptance gate** flagged criterion 3 (Lighthouse a11y ≥ 95
  with new charts) as **advisory-only** because the `lighthouserc.json`
  URL list doesn't yet enumerate the 4 routes Phase 3 added:
  - `/problems/[slug]/ratings` (Unit 3.3, 10 SSG paths)
  - `/problems/[slug]/history` (Unit 3.9, 10 SSG paths)
  - `/trending` (Unit 3.7)
  - `/ratings` (Unit 3.4)
- **Phase-1 contract (Unit 1.12 / Q27)**: the Lighthouse CI gate is a
  required cohort on every PR; the `assert.assertions` block enforces
  `categories:accessibility ≥ 0.95` as an `error`-class assertion.
- **Existing config** (`lighthouserc.json`) enumerates 6 URLs and uses
  `pnpm start` against the production build. Pattern: one canonical
  example per dynamic route, not all members.
- **A11y plumbing on the 4 new pages already shipped** per Unit 3.13:
  every Phase-3 viz has `role="img"`, `aria-label`, and
  `aria-describedby` → `<desc>` with full data prose. Table fallbacks
  (Unit 3.12) have `<caption className="sr-only">`. Pages should hit
  ≥ 95 today; this config change just makes the gate enforce it.

### Constraints

- **One canonical slug per dynamic route.** Existing config picks
  `hallucination-reduction` for `/problems/[slug]` and `deep-learning`
  for `/domains/[any]`. Reuse `hallucination-reduction` for the new
  `/problems/[slug]/ratings` and `/problems/[slug]/history` entries —
  it's the most-populated problem (3 rating actions, 2 q3/q4 revisions,
  the SimpleQA leaderboard entries) so it exercises the chart paths
  with non-trivial data.
- **No threshold changes.** Phase-3 viz a11y plumbing is already in
  place; the gate just needs the URLs.
- **JSON only.** No Velite or schema touches, no app code, no test
  changes. Zero collision risk with parallel-session content work.

### Alternatives considered

- **Enumerate all 10 slug × 2 (ratings + history) = 20 URLs.** Rejected
  — would bloat the Lighthouse run from 6 URLs × 3 runs = 18 collections
  to 26 × 3 = 78, tripling CI wall time. The same-slug convention used
  for `/problems/[any-slug]` is the precedent.
- **Add `/problems` (the recompose-weights index from Unit 3.10) too.**
  Considered. Skipped for this unit — `/problems` is a static index
  page; the recompose-weights interactivity is `"use client"` and
  doesn't change the prerendered DOM. The a11y surface is identical
  to the Phase-2 index. If a parallel session wants to add it, the
  diff is one line.
- **Lower the threshold for the new pages.** Rejected — Phase-3 viz
  a11y was a design-time requirement (§13 acceptance criterion 3).
  Treating it as soft would undermine the gate.

### Tradeoffs / defensible defaults flagged

1. **Slug choice**: `hallucination-reduction` for the per-problem
   ratings and history pages. Alternative would be a problem with the
   §8.2 saturation N/A escape hatch exercised — but none currently
   use it in HEAD (it's a v1.1 schema affordance per ADR-0006; no
   committed rating-action YAML hits it yet). Re-evaluate when the
   first v1.1 N/A action lands.
2. **No `numberOfRuns` change**. Stays at 3 per URL. 10 URLs × 3 = 30
   Lighthouse collections, ~5-7 minutes of CI on `ubuntu-latest`.
   Acceptable.
3. **Bug surface**: if the new routes a11y-fail on CI Ubuntu but pass
   on Win32 (paralleling the Q27 / Phase-2 baseline pattern), the
   first PR run is the canonical signal. No local Lighthouse run from
   this commit — the CI Ubuntu cohort is the source of truth, same
   pattern as the Q27 visual-regression baselines.

### Smoke gates

- `pnpm typecheck` — should be clean (no app changes).
- `pnpm test` — should remain 171/171.
- `pnpm validate-content` — 203 files green (unchanged).
- `pnpm build` — 198 routes (unchanged).
- Lighthouse CI itself runs in `.github/workflows/e2e-lighthouse.yml`
  on PR — that's where the new enumeration gets exercised.
