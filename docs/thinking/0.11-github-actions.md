# Unit 0.11 — GitHub Actions CI + PR template

> Stage 1 (THINK) artifact.

## Problem restatement

Convert the local pnpm scripts from Units 0.2–0.10 into a CI matrix that runs on every push to `main` and every PR, plus a PR template per §14.4. The CI is the binding gate for everything Phase 0 has built — typecheck, lint, format-check, vitest, validate-content, build, and the Layer-3 ADR-0005 enforcement. E2E and Lighthouse are slower and live in a separate workflow with their own triggers.

## Inputs

- §13 Phase 0 deliverable: GitHub Actions for `typecheck`, `lint`, `test`, `build`, `lighthouse`, `validate-content`.
- §14.4 (PR template ticks Schemas / ADR / CHANGELOG / Tests / Lighthouse / OpenQuestions).
- ADR-0005 § "Enforcement has three layers: 1. Filesystem convention. 2. Pre-commit hook (Husky). 3. CI gate."
- Unit 0.8 husky hooks (already enforce locally).
- Unit 0.9 Playwright + Lighthouse CI (configs ready, browsers need install in CI).

## Constraints

- **One canonical CI workflow** for the fast checks (typecheck, lint, format, test, validate-content, build, ADR-0005). Each as a step, not as separate jobs, to amortize the `pnpm install`.
- **A separate workflow** for the slow checks (Playwright with chromium, Lighthouse CI). These take 2–3 min each on a fresh runner.
- **pnpm via `pnpm/action-setup`** to honor the `packageManager` field. Node 22 via `actions/setup-node` with `cache: pnpm`.
- **No secrets needed** for Phase 0. Lighthouse CI uses `temporary-public-storage` upload target.
- **ADR-0005 CI check** must compare base branch to PR head (not staged diff like local). Use `git diff --name-status origin/${{ github.base_ref }}...HEAD`.
- **Required vs optional checks.** Required for merging: typecheck, lint, format:check, test, validate-content, build, ADR-0005. Lighthouse/e2e advisory in Phase 0 (perf can fluctuate on shared runners); promote to required at Phase 1 once baselines stabilize.
- **Path filters** so docs-only changes still run lint/format but not full build. Keep Phase 0 simple: all PRs run all fast checks.

## Alternatives considered

1. **Single workflow with one job + sequential steps** (chosen for fast CI). Cheapest install, clearest logs.
2. **Matrix job over Node versions (20, 22)**. Defer — we pin Node 22; multi-version is Phase 1+ if Vercel changes its runtime support.
3. **Run Playwright + Lighthouse in the same job as the fast checks.** Rejected — adds 4–5 min to every PR even when the change is docs-only.
4. **GitLab CI / CircleCI**. The project is on GitHub (Q3 confirmed). No reason to deviate.

## Edge cases & failure modes

1. **`pnpm install` slow on first run.** Use `actions/setup-node` cache to keep the pnpm store; cache hit rate should be >90% across PRs.
2. **`pnpm validate-content` exit on no problems**. Currently 1 file (taxonomy.yaml); script exits 0. Phase 1 will exercise the failure path.
3. **Husky `prepare` script on CI.** Husky installs hooks during `pnpm install` via the `prepare` script. On CI we don't need git hooks — `HUSKY=0` env var skips installation cleanly.
4. **ADR-0005 first commit on a branch.** Comparing `origin/main...HEAD` works for PRs but not for direct pushes to main. Restrict the check to `pull_request` events.
5. **Lighthouse score variance.** A budget of 0.95 on the home page is generous (the Unit 0.10 build shows the home shipping 165 B on top of 103 kB framework). If we see flakiness, run 3 times and assert against the median.
6. **Playwright browser install size.** `pnpm exec playwright install --with-deps chromium` downloads ~200 MB. Cache via `~/.cache/ms-playwright` keyed on the Playwright version.
7. **Storybook build job.** Defer — Phase 0 has no stories, so a `pnpm build-storybook` step would just produce an empty bundle. Reintroduce at Phase 1 when components arrive.

## Open questions surfaced (→ OPEN_QUESTIONS.md)

- **Q27. Required vs advisory checks.** Lighthouse / e2e are advisory in Phase 0. Phase 1 should promote them to required once baselines stabilize. Confirm the cutover at Phase 1 kickoff.
- **Q28. Branch protection.** GitHub branch-protection rules need to be configured on the repo (not in code). List the required checks somewhere the repo admin sees them. Defer to first GitHub setup pass.

## Decision

Files in this unit:

1. `.github/workflows/ci.yml` — fast-path CI:
   - Triggers: `pull_request`, `push` to `main`.
   - Single job `verify`. Steps:
     1. Checkout (`fetch-depth: 0` so the ADR-0005 base-branch diff works).
     2. Setup pnpm (action-setup, reads packageManager).
     3. Setup Node 22 with pnpm cache.
     4. `HUSKY=0 pnpm install --frozen-lockfile`.
     5. `pnpm typecheck`.
     6. `pnpm lint`.
     7. `pnpm format:check`.
     8. `pnpm test`.
     9. `pnpm validate-content`.
     10. `pnpm build`.
     11. ADR-0005 base-branch immutability check (only on PR events).
2. `.github/workflows/e2e-lighthouse.yml` — slow-path:
   - Triggers: `pull_request`, `push` to `main`.
   - Two jobs: `e2e` (Playwright) and `lighthouse` (lhci autorun). Both run in parallel after a shared install.
   - Continues on failure (advisory in Phase 0); Phase 1 flips this.
3. `.github/pull_request_template.md` — the §14.4 checklist.
4. `scripts/check-rating-action-immutability-ci.mjs` — reusable CI variant of the pre-commit script: takes the base ref as argv and exits non-zero on any M/D/R/C against `content/problems/*/ratings/`.
