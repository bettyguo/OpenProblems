### Integration shape per surface

#### Surface 1: Bio (`lib/users/index.ts` `updateProfile`)

```ts
if (typeof set.bio === "string" && set.bio.length > 0) {
  const decision = await getModerator().moderateText(set.bio, {
    surface: "bio",
    userIdOrEmail: userId,
  });
  if (!decision.ok && decision.severity === "block") {
    return decision.reasons[0] ?? "Bio refused by content moderation.";
  }
}
```

- Only calls moderator when `set.bio` is non-empty (clear-by-empty-
  submit short-circuits per ADR-0016 D-B).
- Returns the moderator's first reason string OR a generic fallback
  per ADR-0024 D-11 lean.
- Returns string = signals validation error to caller (same shape as
  `validateBio` errors).
- `displayName` is NOT moderated per ADR-0024 D-12 lean (short-form
  field; future provider can extend).

#### Surface 2: Avatar (`lib/users/index.ts` `updateProfileImage`)

```ts
const imageBuffer = Buffer.from(await file.arrayBuffer());
const imageDecision = await getModerator().moderateImage(imageBuffer, {
  surface: "avatar",
  userIdOrEmail: userId,
});
if (!imageDecision.ok && imageDecision.severity === "block") {
  return imageDecision.reasons[0] ?? "Image refused by content moderation.";
}
```

- Sits AFTER MIME / size / magic-byte validation (so moderation only
  runs on bytes that already pass cheap checks).
- Sits BEFORE the existing-image lookup + putAvatar upload (refused
  image never touches paid Vercel Blob storage; refused image never
  triggers delete-on-replace of the prior image either — atomic
  failure).
- Double-reads the file buffer (once for moderation, once inside
  putAvatar's sharp pipeline). File is ≤ 2 MB; double-read cost is
  microseconds. Avoids refactoring `lib/storage/putAvatar` to take a
  moderator dependency.

#### Surface 3: Rating-challenge (`app/api/v1/rating-challenges/route.ts`)

```ts
const moderationDecision = await getModerator().moderateText(rationale, {
  surface: "rating-challenge",
  userIdOrEmail: session.user.id,
});
if (!moderationDecision.ok && moderationDecision.severity === "block") {
  return moderationRefused("rationale", moderationDecision.reasons[0] ?? "...");
}
```

- Only `rationale:` is moderated per ADR-0024 D-12 lean.
- Returns HTTP 422 (new shape: `{ error: "moderation-refused", field,
  message }`) — distinguishes from 400 validation-failure shape so
  callers can render different UX (block-icon vs validation-warning-
  icon).
- Sits AFTER all other validation (problemSlug + dimension +
  proposedValue + rationale length) so moderation only runs on otherwise-
  valid requests.

#### Surface 4: Subscribe (`app/api/v1/subscribe/route.ts`)

```ts
const moderationDecision = await getModerator().moderateText(email, {
  surface: "subscribe",
  userIdOrEmail: email,
});
if (!moderationDecision.ok && moderationDecision.severity === "block") {
  return Response.json({
    error: "moderation-refused",
    field: "email",
    message: moderationDecision.reasons[0] ?? "...",
  }, { status: 422 });
}
```

- Moderates email-as-text per ADR-0024 D-13 lean. Noop default makes
  this costless. Future regex-wordlist provider can refuse slur-
  containing email addresses without further wiring change.
- `userIdOrEmail` is the email itself (subscribe surface is the only
  one where actor may not be signed in).
- Sits AFTER email validation but BEFORE the
  `createOrRefreshPendingSubscription` DB write — refused email
  never creates a DB row.

### i18n keys — none added Unit 35.2

The framework returns provider-specific reason strings; the API
responses surface them directly in the JSON `message` field. UI
consumers (the bio-edit server action's error toast; the rating-
challenge form's inline error; the subscribe form's error chip)
render the returned message verbatim under the existing error-
surface i18n styling. The noop default never produces a refusal
message, so no user-facing string is shipped Phase 35.

Per-surface custom i18n-keyed UX is Phase-36+ polish per ADR-0024
D-H + Phase-35-prep D-11 lean.
