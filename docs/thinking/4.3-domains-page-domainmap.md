### Constraints

- **`exactOptionalPropertyTypes: true`** — `composite` may be undefined for an unrated problem; build-domain-map must handle `composite | undefined` cleanly (skip the node? coerce? — see Tradeoffs).

- **`ChartTableSwitch` mirror of Unit 3.12 use**: same `chart={}` + `table={}` props, label `"View as table"`.
- **Domain Hue mapping is build-time-stable** (no UI-side reordering); persists across PRs unless `taxonomy.yaml` declares a new domain or reorders.
- **5 domains × 5 hues today**. A 6th domain (Phase 5+) would wrap hue to 1 — acceptable per Unit 4.2's color-decoration framing.

### Data shape

```ts
// build-domain-map.ts
export function buildDomainMap(): {
  nodes: DomainMapNode[];
  links: DomainMapLink[];
} {
  // For each domain (in taxonomy.domains order):
  //   - emit a "domain:<id>" node with hue = (index % 5) + 1
  //   - aggregate composite from descendant problems (mean; default 3.0 if no rated descendants)
  // For each (domain, subdomain) pair:
  //   - emit a "subdomain:<domainId>/<subId>" node
  //   - aggregate composite from rated problems in this subdomain (mean; default 3.0)
  //   - inherit parent hue
  // For each problem:
  //   - emit a "problem:<slug>" node
  //   - composite from getIndexedProblems()'s `composite`, or 3.0 fallback
  //   - inherit parent hue
  //   - href = "/problems/<slug>"
  //   - parent = "subdomain:<d>/<s>" (or "domain:<d>" if the problem somehow has no listed subdomain)
  // Links: each child → parent. Order: problems → subdomains → domains (no skipping levels).
}
```

### Composite-aggregation rule (problem → subdomain → domain)

- **Problem composite**: from `getIndexedProblems()` `.composite` (mean of dimension scores per §8.3). When undefined (no rating action committed yet for this problem), use **3.0** as the "midpoint placeholder". The DomainMap node still renders; the table fallback shows the placeholder transparently.
- **Subdomain composite**: arithmetic mean of child problems' (real or placeholder) composites. When the subdomain has zero problems, use 3.0.
- **Domain composite**: arithmetic mean of *all descendant problems*' composites (not subdomain means — flatter aggregation is more honest under uneven subdomain weights).

This is deliberately simple. Phase 5+ may revisit with weighted aggregation (e.g., weight by # of leaderboard entries), but the v1 rule is "mean of leaves".

### Hue assignment

```ts
const hueForDomainIndex = (idx: number): DomainMapHue =>
  (((idx % 5) + 1) as DomainMapHue);
```

Subdomain + problem nodes inherit their parent domain's hue. Domain order is `taxonomy.domains[]` order (taxonomy.yaml declaration order).

### Alternatives considered

- **Build nodes+links inline in `app/domains/page.tsx`**. Rejected — Unit 4.4 needs the same data; extracting to `lib/content/build-domain-map.ts` keeps the page focused on layout. The DRY savings (~50 LOC across two pages) outweigh the extra file.
- **Render DomainMap directly without `ChartTableSwitch`**. Rejected — §13 acceptance criterion still applies; the parallel session's Unit 4.2 explicitly shipped `DomainMapTable` for exactly this consumption pattern.
- **Compute composite-on-the-fly using §8.3 with current Recompose-UI weights**. Rejected — Recompose-UI scope is `/problems` only per Q36; cross-page propagation is a Phase-5 enhancement.
- **Skip placeholder composites for unrated problems** (filter them out). Rejected — incomplete coverage masquerades as "no problems here". Placeholder + transparent table-fallback is more honest.

### Tradeoffs / defensible defaults flagged

1. **3.0 placeholder composite** for unrated problems. Implication: visually identical to a "true 3.0" problem. The table-fallback distinguishes them via a `—` indicator in the score column (TBD: check if real composites can land exactly at 3.00; if so, add a separate `rated: boolean` field).
2. **Subdomain composite via mean-of-children, not weighted**. Phase 5 can swap formulas without changing the DomainMap viz contract.
3. **Hue wrapping at 6 domains** (Phase 5+). Color collision but not a correctness issue.
4. **Domain composite uses ALL descendant problems, not subdomain means**. Treats subdomains as taxonomic groupings, not weighted units. If Phase 5 ingest creates highly uneven subdomain populations (e.g., 1 vs. 20 problems), revisit.

### Tests

`build-domain-map.test.ts` (≥ 4 tests):

1. **Hue assignment is index-based and wraps modulo 5**.
2. **Every problem has a `parent` that resolves to a subdomain or domain node in `nodes`**.
3. **Every link's `source` and `target` exists in `nodes`**.
4. **Subdomain composite is the mean of its child problems' composites** (numeric-equality test with a 2-problem subdomain).
5. **Unrated problems render at composite 3.0** (placeholder).
