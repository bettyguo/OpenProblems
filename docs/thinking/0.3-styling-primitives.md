# Unit 0.3 — Styling primitives: Tailwind v4, shadcn/ui, fonts

> Stage 1 (THINK) artifact for Unit 0.3.

## Problem restatement

Land three styling primitives that subsequent units (Unit 0.4 design tokens, Unit 0.10 IA stubs, every Phase 1+ UI unit) depend on:

1. **Tailwind v4** — utility CSS engine. v4 is CSS-first (no `tailwind.config.ts` required), uses `@import "tailwindcss"`, and resolves theme tokens via `@theme` blocks. PostCSS-based for Next.js.
2. **shadcn/ui** — Radix-based, copy-in, fully owned component primitives. Provides `components.json`, `@/lib/utils.ts` (the `cn` helper), and a starter set of CSS variables that Unit 0.4 will override.
3. **Fonts via `next/font/google`** — Inter (UI), Source Serif 4 (body prose), JetBrains Mono (code/metrics). Variable fonts only (per MASTER_PROMPT.md §5.2).

This unit explicitly does **not** finalize the design tokens (Unit 0.4) or the brand accent (OPEN_QUESTIONS Q5). It picks defensible placeholders that Unit 0.4 will replace with one Edit.

## Inputs

- ADR-0001 (Next.js 15 App Router accepted).
- MASTER_PROMPT.md §5.2 (Tailwind v4 + shadcn/ui; Inter + Source Serif 4 + JetBrains Mono; variable fonts only; no Bootstrap, no MUI).
- §10.1 (visual language — two-tone foundation `#0B0D10` / `#FAFAF7`, single brand accent — vermilion or deep cyan, pending Q5; reading width ~70 ch on problem detail pages).
- §10.4 (perf budget: LCP < 1.8s on slow 4G; viz chunks lazy-loaded; problem-detail JS budget < 180 KB gz excluding viz).
- §14.1 (no barrel files; one component per folder structure once we have components — n/a for Unit 0.3).

## Constraints

- **Tailwind v4 idioms only.** No v3 holdovers like `@tailwind base / components / utilities`. Use `@import "tailwindcss"` and `@theme` blocks.
- **shadcn/ui CSS-variable mode.** Tokens live in CSS variables so Unit 0.4 can replace them without touching component code.
- **Variable fonts only.** Inter Variable, Source Serif 4 Variable, JetBrains Mono Variable — all available on Google Fonts. `next/font/google` handles self-hosting and `display: swap`.
- **No brand-accent commitment.** Pick a neutral base color (shadcn's `neutral`) so Unit 0.4 can swap to whichever Q5 resolves to (vermilion vs deep cyan) with a single `--primary` edit.
- **No example component pulls.** shadcn/ui init only — don't `pnpm dlx shadcn add button` yet. Components arrive where they're needed (Phase 1).
- **No additional pnpm-workspace.yaml entries.** This unit installs deps with no postinstall scripts (Tailwind, shadcn helpers, font packages all run clean).

## Alternatives considered

1. **Tailwind v4 + shadcn/ui + `next/font/google`** (chosen).
2. **Tailwind v4 alone** (no shadcn). Defer shadcn to Phase 1 when a component need surfaces.
3. **Plain CSS modules + custom design tokens** (no Tailwind). Rejected by ADR-0001's stack rationale; not revisitable here.
4. **Self-host fonts manually** (download `.woff2` files, place in `public/fonts/`). More control but more maintenance; `next/font` already self-hosts and inlines fallback metrics — strictly better for our use case.

Trade-offs:

- (1) front-loads shadcn so Unit 0.4 can drop tokens onto an already-wired Radix layer; otherwise Unit 0.4's tokens are theoretical until shadcn arrives.
- (2) is one less dependency now but pushes the same work into Phase 1, where it competes with content authoring.
- (4) is a Phase-5-or-later concern if Google Fonts becomes a privacy / latency issue. `next/font` already addresses both.

Chosen: **(1).**

## Edge cases & failure modes

1. **Tailwind v4 + Next.js 15 PostCSS integration.** v4 ships `@tailwindcss/postcss`, not the old `tailwindcss` PostCSS plugin. The postcss config must use `@tailwindcss/postcss`, *not* `tailwindcss`.
2. **shadcn/ui CLI interactivity.** The init asks ~6 questions. Use `--yes` / `--defaults` to run non-interactively. If a question has no flag, fall back to scripting answers via stdin.
3. **shadcn/ui modifying `app/globals.css`.** The init will append CSS variables (color tokens) to our existing `globals.css`. We accept this and let Unit 0.4 replace those variables with the actual two-tone foundation + accent.
4. **`next/font` network access on first build.** Google Fonts must be reachable at install/build time. On Windows-China, `fonts.googleapis.com` is often slow; mitigation is `next/font`'s caching — once downloaded into `.next/cache/fonts/`, subsequent builds work offline.
5. **Variable font axis support.** Inter Variable exposes `wght 100–900`. Source Serif 4 Variable exposes `wght 200–900` and `opsz 8–60`. JetBrains Mono Variable exposes `wght 100–800`. All three are configured to load the wght axis (and opsz for Source Serif 4); other axes are skipped to keep payload small.
6. **CSS variable name collisions.** shadcn's tokens (`--background`, `--foreground`, `--primary`, ...) sit in the same namespace where Unit 0.4 will land semantic tokens. Coordinate via the Unit 0.4 design — for Unit 0.3 we just accept shadcn's defaults.
7. **No JSX changes to existing routes.** `app/layout.tsx` and `app/page.tsx` are updated *only* to wire fonts and (optionally) apply the `@theme` font-family tokens. No new components.
8. **Build-time CSS scanning.** Tailwind v4 scans content paths automatically based on `@source` directives. Default heuristics catch `app/`, `components/`, `lib/` — sufficient for our layout.

## Open questions surfaced (→ OPEN_QUESTIONS.md)

- **Q16. Font subset choice.** Default `next/font` behavior loads `latin` subset. Project will eventually support FR (Q6) and possibly other Latin-extended characters in author names (diacritics). Lean: load `latin` only for Phase 0; expand to `latin-ext` when content with diacritics first arrives. Surface as a low-priority question.
- **Q17. shadcn/ui base color.** Init must pick one of `neutral`, `slate`, `gray`, `zinc`, `stone`. Picking `neutral` (true grays, no hue cast) for Unit 0.3 so Unit 0.4 can choose hue independently. Surface so user can override if they want a different placeholder.

## Decision

Files / actions in this unit:

1. `docs/thinking/0.3-styling-primitives.md` — this file.
2. `pnpm add -D tailwindcss @tailwindcss/postcss postcss` — install Tailwind v4.
3. `postcss.config.mjs` — single-line config pointing at `@tailwindcss/postcss`.
4. Update `app/globals.css` — replace minimal reset with `@import "tailwindcss"`, keep two-tone foundation in `@theme` (placeholder, replaced in Unit 0.4).
5. `pnpm dlx shadcn@latest init` with non-interactive flags — base color `neutral`, CSS variables on, RSC yes, `@/*` alias confirmed. This will create `components.json` and `lib/utils.ts` and update `globals.css`.
6. Wire fonts in `app/layout.tsx` via `next/font/google`: Inter Variable, Source Serif 4 Variable, JetBrains Mono Variable. Expose as `--font-sans`, `--font-serif`, `--font-mono` CSS variables on `<html>`.
7. Add Tailwind `@theme` aliases mapping `font-family.sans → var(--font-sans)`, etc.
8. Update `app/page.tsx` to use Tailwind utilities so we can prove the pipeline end-to-end.
9. Smoke test: `pnpm typecheck && pnpm build`.
10. Update `CHANGELOG.md` and `OPEN_QUESTIONS.md` (Q16, Q17).
11. Commit.
