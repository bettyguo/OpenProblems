## THINK

### What this unit ships

Two artifacts:

1. **`lib/db/schema.ts` (edit)** ÔÇö adds one nullable text column
   to the `users` table:
   - `imageOverride: text("imageOverride")` ÔÇö nullable; 512-char
     URL cap enforced app-level via Unit 16.3's
     `validateImageOverride`.

2. **`lib/db/migrations/0005_user_image_override.sql` (new)** ÔÇö
   generated via `pnpm db:generate --name user_image_override`.
   drizzle-kit output:

   ```sql
   ALTER TABLE `user` ADD `imageOverride` text;
   ```

   Clean ALTER ADD COLUMN; no FK; no `ON DELETE` clause; no
   manual inspection / correction needed. **drizzle-kit's
   nullable text ALTER emission is clean** at THIRD exercise;
   ADR-0014 D-E pattern fully validated.

### Why a SEPARATE column (not editing `users.image` directly)

ADR-0017 D-E pins the user-controlled override pattern that
Phase-15 ADR-0016 D-A established:
- `users.image` is GitHub-derived (populated by Auth.js v5 from
  the OAuth profile on first sign-in; not user-touched).
- `users.imageOverride` is user-controlled (set/cleared by the
  user via the `/profile` upload form Phase 16 ships).

Fallback chain `imageOverride Ôćĺ image Ôćĺ fallback initials
placeholder` mirrors Phase-15's `displayName Ôćĺ name Ôćĺ githubLogin
Ôćĺ fallback` semantically. Preserves auth-side vs user-side
write separation. Editing `users.image` directly would lose the
GitHub fallback (if user clears their override, re-fetching the
GitHub profile becomes complex; storing the override separately
keeps the GitHub-derived original always available).

### Why nullable text (not `notNull` with default empty)

- NULL = "no override; render fallback chain to GitHub `image`".
- Empty string vs NULL: collapsed to NULL in Unit 16.3's
  `clearProfileImage` (mirrors Phase-15 `updateProfile`'s
  empty-after-trim Ôćĺ NULL semantics). Storing NULL explicitly
  keeps DB shape consistent with "field never set" vs "user
  touched then cleared".

### Why 512-char cap

Per ADR-0017 D-F:
- Vercel Blob public URL format: `https://*.public.blob.vercel-storage.com/<random-hash>-<filename>`.
  Typical lengths ~80-160 chars for the project's avatar paths
  (`avatars/<userId>-<timestamp>.<ext>`; userId is a UUID ~36
  chars; timestamp ~13 chars; ext ~3-4 chars).
- 512 chars is conservative + accommodates long filenames + any
  future query-string suffixes (e.g., cache-busting parameters).
- Far below SQLite's TEXT length limit; this is an app-level
  defense to bound DB size + protect against malicious-input
  URL bloat.

Drizzle-kit's monotonic 0-indexed sequence (per ADR-0013 D-E

`0001_watchlist` + `0002_rating_challenges` +
`0003_rating_challenge_review` + `0004_user_profile_fields` +

### Snapshot + journal atomicity

drizzle-kit writes:
- `lib/db/migrations/0005_user_image_override.sql`
- `lib/db/migrations/meta/0005_snapshot.json`
- `lib/db/migrations/meta/_journal.json` (updated with new
  entry idx=5).

All three commit together. Per ADR-0013 D-B migration
immutability: **NEVER edit `0000`-`0004` to add the
`imageOverride` column**. Doing so would re-apply the migration
on databases that already have the original schema, producing
an inconsistent applied-migrations log.

### Forward-compat for existing rows

Existing `users` rows from Phase 9 onward get NULL
`imageOverride` (SQLite's natural default for ADDed nullable
columns). Application reads NULL as "no override; render
fallback chain" without special handling. **No data migration
needed.**

### Backward-incompat surface

SQLite ALTER TABLE supports `ADD COLUMN` natively for nullable
columns; does NOT support `DROP COLUMN` cleanly. Phase 16 makes
no column removals.

### Why land this ALONE (not bundled with Unit 16.3 helpers)

Two reasons:

1. **Atomic schema delivery for parallel safety**. If a parallel
   session is running while Unit 16.2 is in flight, the schema
   change must be self-contained (just the column adds) so that
   subsequent units pull from a consistent DB-side foundation.
2. **Migration immutability discipline (ADR-0013 D-B)**. Bundling
   helpers + schema would create churn risk on the migration file
   if helper development surfaces new schema needs mid-unit.
   Splitting the schema commit isolates it.
