# 15.2 — DB migration `0004_user_profile_fields` + schema edit (second ALTER migration)

Third Phase-15 unit; first code unit. Realizes
[ADR-0016](../adr/0016-user-editable-profile-fields.md) D-A schema
component. **Second ALTER migration in project history** (first was
Phase-12 `0003_rating_challenge_review` per ADR-0014 D-E).

## THINK

### What this unit ships

Two artifacts:

1. **`lib/db/schema.ts` (edit)** — adds two nullable text columns to
   the `users` table:
   - `displayName: text("displayName")` — nullable; 80-char cap
     enforced app-level via Unit 15.3's `validateDisplayName`.
   - `bio: text("bio")` — nullable; 280-char cap enforced app-level
     via Unit 15.3's `validateBio`.

2. **`lib/db/migrations/0004_user_profile_fields.sql` (new)** —
   generated via `pnpm db:generate --name user_profile_fields`.
   drizzle-kit output:

   ```sql
   ALTER TABLE `user` ADD `displayName` text;--> statement-breakpoint
   ALTER TABLE `user` ADD `bio` text;
   ```

   Clean ALTER ADD COLUMN; no FK; no `ON DELETE` clause; no manual
   inspection / correction needed (unlike Phase-12 Unit 12.2's
   `reviewerId` FK edge case which required in-place SQL correction).
   **drizzle-kit's nullable text ALTER emission is clean**;
   Phase-12 D-E pattern validates at second exercise without surprise.

### Why nullable text (not `notNull` with default empty)

- NULL = "not set; render fallback chain". Empty string = "user
  explicitly cleared the field". Distinguishable in DB queries
  (`displayName IS NULL` vs `displayName = ''`).
- Phase-15 lean: empty-string-after-trim collapses to NULL (Unit
  15.3's `updateProfile` behavior). Both NULL + empty render the
  fallback chain. So semantically only NULL surfaces; but storing
  NULL explicitly keeps the DB shape consistent with "field never
  set" vs "user touched then cleared".

### Why `displayName` and `bio` in same migration

Two reasons:

1. **Atomic schema delivery**. Phase-15 Unit 15.3 helpers need both
   columns to populate `PublicProfile` interface; landing them
   together avoids the "Unit 15.3 helpers reference a column that
   doesn't exist yet" sequencing issue.
2. **Single ALTER discipline pattern**. drizzle-kit groups multiple
   ALTER ADD COLUMN statements into one migration cleanly;
   user-facing migration history stays terse (one row per logical
   schema change, not one row per column).

### Migration filename + drizzle-kit's 0-indexed sequence

Drizzle-kit's monotonic 0-indexed sequence (per ADR-0013 D-E
clarification in Unit 9.8): `0000_initial_auth` + `0001_watchlist` +
`0002_rating_challenges` + `0003_rating_challenge_review` +
**`0004_user_profile_fields`**. Phase-15's `0004_*` slot is the
5th migration in project history; the `--name user_profile_fields`
argument names it descriptively.

### Snapshot + journal atomicity

drizzle-kit writes:
- `lib/db/migrations/0004_user_profile_fields.sql`
- `lib/db/migrations/meta/0004_snapshot.json`
- `lib/db/migrations/meta/_journal.json` (updated with new entry).

All three commit together. Per ADR-0013 D-B migration immutability:
**NEVER edit `0000`-`0003` to add the displayName + bio columns**.
Doing so would re-apply the migration on databases that already have
the original schema, producing an inconsistent applied-migrations
log.

### Forward-compatibility

Existing `users` rows from Phase 9 onward (which never had
`displayName` or `bio`) get NULL values in the two new columns;
application reads NULL as "not set; render fallback chain" without
special handling. **No data migration needed**.

### Backward-incompatibility surface

SQLite ALTER TABLE supports ADD COLUMN natively for nullable columns;
DOES NOT support `DROP COLUMN` cleanly (until recent SQLite versions
+ requires copy-table-and-drop dance). Phase 15 makes no column
removals; future column-drop concerns are Phase 16+ scope.

### Parallel-curator awareness

- This unit edits 1 file (`lib/db/schema.ts`) + adds 3 files (the
  migration + snapshot + journal). LOW-MEDIUM collision risk per
  Unit 15.0 D-? parallel-curator section:
  - `lib/db/schema.ts` was last edited in Phase-12 Unit 12.2;
    Phase-15 edit is additive (only adds 2 columns at the end of
    `users` definition).
  - Migration sequence is monotonic; if parallel session generated
    a different `0004_*` migration, both would conflict on the
    journal — but parallel session would have to be RUNNING a
    migration generation simultaneously, which is unusual.
