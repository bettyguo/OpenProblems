# 2.11 — Cross-link audit script

## THINK

### Inputs

- **§13 Phase 2 acceptance gate.** "Validation script
  `scripts/cross-link-audit.ts` green." This unit lands the script.
- **2.0 plan Q32.** Asymmetric `related_problems` are warnings, not
  errors. The audit's exit code only flips on errors; warnings appear
  in the report.
- **Existing data the audit will read.**
  - 30 papers in `content/papers/*.yaml` (Units 2.4–2.6)
  - 12 authors in `content/authors/*.yaml` (Units 2.1 + 2.4)
  - 14 institutions in `content/institutions/*.yaml` (Units 2.1 + 2.4 + 2.6b)
  - 10 problems in `content/problems/<slug>/problem.yaml` (Units 1.4 + 1.11)
- **Existing infrastructure.** `scripts/validate-content.ts` + 
  `lib/content/validate.ts` is the model: lib has the testable
  pure function returning a `Report`; the script is a thin CLI
  wrapper that exits 0/1 and prints.

### Constraints

- **Reads YAML directly from disk, not via Velite.** Velite output
  is only available after `pnpm velite`; an audit that requires a
  prior build is a worse developer-loop story than one that runs
  standalone against `content/`.
- **No new dependencies.** `yaml` is already a dep (used by
  `validate.ts`); use the same `parse` API.
- **No new tests required for this unit.** Adding integration
  fixtures for the audit would be a separate unit; for now the
  audit's own "no errors on the real `content/`" smoke is the test.
  Add unit tests for the audit logic in Unit 2.13 (acceptance gate)
  alongside the Lighthouse + visual-regression refresh.
- **Schema is not the audit's job.** `pnpm validate-content` already
  enforces shape. The audit only checks references and reciprocity
  between schemas. If a file isn't even schema-valid, the audit
  may emit confusing errors — fine, the validation gate runs first.
- **Per the 2.0 plan, `entries.json↔contributions` is a check.**
  Currently no `entries.json` files exist in `content/problems/<slug>/`,
  so the check is a no-op pending Unit 2.10. Stub the check so it
  runs cleanly today; activates automatically when entries land.

### Checks

| # | Name | Severity | What it verifies |
| - | ---- | -------- | ---------------- |
| 1 | `paper-problem-fk` | error | every `paper.contributions[].problem_slug` exists in `content/problems/` |
| 2 | `paper-author-fk` | warning | every `paper.authors[]` slug exists in `content/authors/` |
| 3 | `paper-institution-fk` | warning | every `paper.institutions[]` slug exists in `content/institutions/` |
| 4 | `author-institution-fk` | error | every `author.affiliations[].institution` exists |
| 5 | `related-problems-fk` | error | every `problem.related_problems[]` slug exists |
| 6 | `related-problems-symmetry` | warning | if A→B, then B→A (Q32) |
| 7 | `entries-contributions-agreement` | warning | for any `content/problems/<slug>/entries.json`, every `entry.paper_id` should appear in `paper.contributions[]` for that slug |

Severity rationale:
- `paper-problem-fk` (1) is error: a paper claiming a contribution to a
  non-existent problem is structurally broken.
- `author-institution-fk` (4) is error: the seed-author set is small;
  these should resolve cleanly.
- `related-problems-fk` (5) is error: problem.related_problems is curated
  by humans; a typo'd slug here means a broken cross-link.
- `paper-author-fk` (2) and `paper-institution-fk` (3) are warnings:
  the Phase-2 norm is empty `authors[]` and partially-populated
  `institutions[]`; the audit surfaces what needs backfill without
  blocking the build.
- `related-problems-symmetry` (6) is a warning per Q32 — the cardinal
  rule is the slug resolves, not that the graph is symmetric.

### Output format

```
Cross-link audit — <ISO timestamp>
  files read:      <N total>
  errors:          <X>
  warnings:        <Y>

ERRORS
  [check:paper-problem-fk] content/papers/<id>.yaml — …
  ...

WARNINGS
  [check:paper-author-fk] content/papers/<id>.yaml — …
  ...

SUMMARY
  papers: 30 (X dangling, Y resolved)
  authors: 12 (X dangling-referenced, Y orphaned)
  institutions: 14 (X dangling-referenced, Y orphaned)
  problems: 10 (X related-problems edges, Y asymmetric)
```

Exit code: 0 if `errors == 0`, 1 otherwise. CI treats warnings as
diagnostic, not blocking.

### Alternatives considered

- **Use Velite output instead of re-reading YAML.** Faster + the joins
  are pre-resolved, but adds a build dependency to the audit. The lib
  test in Unit 0.7 already shows direct YAML reads are fast at this
  scale (~600 ms for 70 files). Stay direct.
- **Emit machine-readable JSON in addition to the text report.**
  Defer to Unit 2.13 when the audit is wired into CI — JSON output
  there makes annotation easier. For now, text is enough.
- **Auto-fix mode** (e.g. add reciprocal `related_problems` entries).
  Out of scope. ADR-0005 forbids mutating rating files; the related-
  problems case is curator-judgment territory, not script-decidable.

### Tradeoffs / defensible defaults flagged

1. **`paper-author-fk` = warning, not error.** All 30 seed papers
   carry `authors: []` or partial author lists; this check would dump
   ~hundreds of warnings under Phase-2-norm conditions. Making it
   error-class would block every CI run until the entire author
   backfill lands. Better surfaced as informational pressure for the
   curator to backfill.
2. **`paper-problem-fk` = error.** Phase-2 papers all reference
   existing problem slugs (Units 2.4/2.5/2.6 used real slugs). If a
   future PAPER-INGEST run typos a slug, blocking CI is the right
   call.
3. **No `paper.contributions[]` "evidence URL reachability" check.**
   Live HTTP probes belong in a separate, async hygiene job (Phase 4+);
   the static audit only checks shape and references.
4. **Audit runs on the working tree, not git HEAD.** `pnpm
   audit-content` reflects pending edits. This matches the
   `validate-content` model and is what a curator wants during a
   PAPER-INGEST run.
