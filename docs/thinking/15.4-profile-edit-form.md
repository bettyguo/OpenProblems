# 15.4 — `/[locale]/profile` edit form + `messages.profile_edit.*` (atomic pre-add 12 keys × 2 locales)

Fifth Phase-15 unit; third code unit. Realizes
[ADR-0016](../adr/0016-user-editable-profile-fields.md) D-C (edit
surface route shape) + D-D (server-action) at the UI layer. First
write-form UI consumer of Unit 15.3's `updateProfile` helper.

## THINK

### What this unit ships

   net**). All keys pre-added upfront per Phase-14 atomic-i18n
   discipline.
2. **`app/[locale]/profile/page.tsx` (edit)** — adds:

   - `updateProfileAction` inline server action.
   - Edit-form `<section>` between header card and watchlist (Unit
     14.2 layout precedent reproduced; mirrors Phase-10's profile
     surface composition).

3. **Reuses Unit 15.3 helpers** verbatim — `updateProfile`,
   `MAX_DISPLAY_NAME_CHARS`, `MAX_BIO_CHARS` imported from
   `@/lib/users`.

### Why server-action (not REST + fetch)

Per ADR-0016 D-D: server actions for signed-in own-state mutations.
Mirrors Phase-10/11/12 precedent (sign-out form, withdraw challenge,
curator review buttons all use inline `<form action={...}>` with
`"use server"` directive). CSRF + cookie handling for free; no
client-island; no new dependency.

### Why URL search params for success/error feedback

Server actions can't easily return data to a server component in the
same render cycle without client-side state. The well-trodden pattern
is:

1. Server action does work.
2. On success: `redirect(/?saved=1)`.
3. On error: `redirect(/?error=<encoded>)`.
4. Server component reads `searchParams` and renders success / error
   banner conditionally.

This pattern:
- Preserves form-content history on success (no error string lingering
  in URL).
- Surfaces validation errors to the user without client islands.
- Allows the user to bookmark or share `/profile` without leaking
  error state into the canonical URL.
- Mirrors the Phase-11 `redirect("/profile?challenge=submitted")`
  precedent (challenge submission feedback).

### Why "saved" banner is `role="status"` and error is `role="alert"`

WCAG 2.2 AA:
- `role="status"` — polite announcement; SR users hear "Profile
  updated." after their save action completes without focus shift.
- `role="alert"` — assertive; SR users hear the error string
  immediately (length-cap-exceeded surface is uncommon but important
  to surface clearly).

### Why ICU placeholder `{login}` in description

The description text:

> "These fields are public on your /u/{login} page. Display name
> overrides your GitHub-derived name."

The `{login}` placeholder is the user's `githubLogin` value, allowing
the description to namedrop the user's actual URL ("your /u/BettyGuo
page" reads more concretely than "your /u/{login} page"). Falls back
to "you" if `githubLogin` is null (Phase-9 retrofit edge case).

### Why no DELETE / "clear all" affordance

ADR-0016 D-B clear-field semantics are achieved by submitting the
form with an empty text input — `updateProfile` collapses empty-
after-trim to NULL. A separate "Clear" button would be redundant.
Mirrors GitHub's pattern (delete displayName by saving empty).

### Why pre-populated `defaultValue` instead of controlled

Server-component form; no client state. `defaultValue` is the React
canonical pattern for non-controlled inputs with initial values; the
browser then takes over form state. Submission rebuilds the form
from `users` table on the next render. No client JavaScript involved.

### Why `maxLength` on both client + server

- **Client** (`<input maxLength={80}>` + `<textarea maxLength={280}>`):
  prevents accidental over-typing; UX feedback that the field has
  a limit.
- **Server** (`validateDisplayName` + `validateBio` returning error
  string): authoritative; defends against:
  - Power user editing HTML to bypass `maxLength`.
  - Form-data POST from external client (curl, custom script).
  - Browser bug.

Both layers needed; neither alone is sufficient.

### Layout choices

Edit-form section placement: between **header card** and **watchlist
section**. Rationale:

- The header card surfaces identity (avatar + display name + @login
  + sign-out); the edit form mutates that identity. Adjacency is
  semantically tight.
- Putting it BEFORE the watchlist preserves the existing watchlist +
  challenges sections (Phase 10 / 11 / 12) without renumbering or
  reorganizing.

`<section aria-label>` shape mirrors Phase-10's watchlist + Phase-11's
challenges aria-labels. Visual register matches existing form
register (border-input + rounded-md + 9-h button mirrors sign-out
button shape).
