### D-3. Provider choice

**Recommendation**: **Resend** as the email provider (ADR-0021
D-A).

**Rationale**:
- **Modern TypeScript SDK** (`resend@4+`); Next.js / Vercel-
  ecosystem-native; widely used in the Next.js community.
- **Simple API**: `resend.emails.send({ from, to, subject, html })`.
  No SMTP plumbing; no DKIM/SPF/DMARC management beyond domain
  setup (operational gate Q75).
- **Free tier**: 100 emails/day; 3,000/month. Sufficient for
  MVP-scale subscriber-list verification + welcome traffic at
  Phase 30; weekly digest send (Phase 31+) will need monitoring
  against the free-tier ceiling (subscriber count × 1 weekly send
  / 4 = per-day rate).
- **Alternative provider choices** considered:
  - **Postmark**: high deliverability; 100/month free; more
    expensive at scale; less modern API.
  - **AWS SES**: cheapest at scale; needs AWS account + IAM
    setup; operational complexity high for MVP.
  - **SendGrid**: mature but complex API; 100/day free; less
    Next.js-aligned ergonomics.
  - **Raw SMTP via nodemailer**: most flexible but requires
    operational mail-server setup or transactional-relay provider
    integration anyway. Skipped — adds an abstraction layer
    without buying flexibility at MVP scale.

**`resend@4+`** dependency added Phase 30. **First new runtime
dep since Phase 28** (Phase 28 added no new deps;
`next-auth/providers/google` was part of existing
`next-auth@5.0.0-beta.31`). **First new runtime dep with new

### D-5. Verification flow (double opt-in; ADR-0021 D-D)

Standard transactional email patterns:

1. User submits subscribe form on `/[locale]/digest` (Phase-30
   UI; new component).
2. Server creates `subscriber` row with `status =
   pending_verification` + `verificationToken` + 24-hour expiry.
3. Server sends verification email via Resend with link to
   `/api/v1/subscribe/verify/[token]`.
4. User clicks link → server validates token + expiry → updates
   status to `verified` + clears token + sets `verifiedAt` →
   redirects to a confirmation page (or returns JSON for API
   clients).
5. Server sends welcome email confirming subscription + listing
   subscribed domains + including unsubscribe link.

**Token semantics**:
- `verificationToken`: 32-byte URL-safe random; expires 24h
  after row creation; single-use (cleared after verify).
- `unsubscribeToken`: 32-byte URL-safe random; **never expires**;
  one-time unsubscribe (status → `unsubscribed`; doesn't delete
  row — preserves audit trail).

**Rate limiting**: Phase 30 ships no rate limiting. If
subscribe-spam emerges, Phase 31+ adds Vercel Edge rate-limit
config. Acceptable Phase 30 because: (a) no public user base
yet; (b) Resend free-tier daily cap acts as natural ceiling
against spam exfiltration.

### D-7. Unsubscribe flow (ADR-0021 D-E)

Single-click unsubscribe via token-based GET endpoint:

1. Every email includes `https://<host>/api/v1/subscribe/unsubscribe/[unsubscribeToken]`.
2. User clicks link → server validates token → updates status to
   `unsubscribed` + sets `unsubscribedAt` → renders confirmation
   page ("You've been unsubscribed; click here to re-subscribe").
3. Token NOT cleared (one-time use semantically; re-clicking the
   link is idempotent).
4. **Soft delete**: subscriber row preserved with status =
   `unsubscribed` (audit trail; prevents accidental email-list
   loss if re-imported from backups).

**Re-subscribe flow**: existing subscriber row with status =
`unsubscribed` can re-subscribe via the form; row's
verificationToken + token-expiry refreshed; status → `pending_verification`;
new verification email sent.
