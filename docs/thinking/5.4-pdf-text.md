# 5.4 — `lib/curate/pdf-text.ts`

## THINK

### Inputs

- **Unit 5.0 D-2** (decided-as-lean): start with `pdf-parse` (lightweight, pure JS, no native deps); escalate to `pdfjs-dist` if extraction quality is too lossy.
- **Q43** (decided-as-lean): cache extracted text to `.pdf-cache/<arxiv-id>.txt`. Mirrors the `.arxiv-cache/` strategy from Unit 5.2.
- **Unit 5.2's `ArxivMetadata.pdfUrl`** field is the canonical PDF URL.
- **Unit 5.2 rate limiter** is module-scoped to `lib/curate/arxiv-client.ts` — `export.arxiv.org` (Atom API) vs. `arxiv.org/pdf/` (PDF host) are different subdomains, so the same bucket doesn't cleanly apply. Phase-5 PDF fetches will be few-per-session (5–20); a separate, more permissive limiter is appropriate.
- **`pdf-parse` API**: `pdfParse(buffer: Buffer): Promise<{ text, numpages, info, metadata, version }>`. Pure JS, MIT.
- **Test seam**: same `fetchImpl` + `parseImpl` injection pattern as Unit 5.2.

### Scope (this unit)

1. **`lib/curate/pdf-text.ts`** — exports `extractPdfText(arxivId, options?)` returning `{ text, numPages, sourceUrl, fetchedFromCache }`. Internally: fetch PDF via the URL → run `pdf-parse` → cache text to disk → return.
2. **`lib/curate/pdf-text.test.ts`** — vitest unit tests with mocked `fetchImpl` + mocked `parseImpl` + tmp `cacheDir`.
3. **New devDep**: `pdf-parse` + `@types/pdf-parse` (devDep because `lib/curate/*` is only imported from `scripts/*`).
4. **No `.gitignore` change** — `.pdf-cache/` was preemptively added in Unit 5.2 alongside `.arxiv-cache/`.

### Constraints

- **`Buffer` is Node-only.** The function imports from `node:fs/promises`; not browser-compatible. Same shape as `arxiv-client.ts` — pure server/runtime use.
- **Cache write is text, not the original PDF binary.** The PDF binary itself is ephemeral (re-downloadable from arXiv). The text extraction is the expensive step we cache. Trade-off: re-fetching the PDF on a cache miss is cheap (~5-10s); re-running pdf-parse adds another ~1-2s. Caching just the text is the right granularity.
- **No `pdf-parse` types in DefinitelyTyped at a recent version** — the package ships `index.d.ts` in newer releases. Check + use `@types/pdf-parse` if needed.
- **Failure mode**: throw on HTTP error or parse error; do NOT cache partial / failed responses (same convention as Unit 5.2's arxiv-client).
- **No Anthropic SDK** in this unit. PDF text is the INPUT to Unit 5.5's leaderboard extractor; LLM calls live there.

### Rate limiter

A separate token bucket (also module-singleton) for PDF fetches:
- **Capacity**: 2 tokens.
- **Refill rate**: 1 token / 2000 ms.

More permissive than the Atom-API limiter because PDF fetches hit a different arXiv subdomain with separate published policies (arXiv's PDF host is more elastic than the metadata API).

### Cache shape

`.pdf-cache/<arxivId>.json` (not `.txt`) so we can store metadata alongside the text:

```json
{
  "arxivId": "2310.06770",
  "sourceUrl": "https://arxiv.org/pdf/2310.06770",
  "text": "...",
  "numPages": 18,
  "fetchedAt": "2026-05-15T19:50:00.000Z"
}
```

`fetchedAt` is informational; cache TTL = unlimited (same as arxiv-client; PDFs are immutable per version).

### Alternatives considered

- **`pdfjs-dist`** instead of `pdf-parse`. Deferred per Unit 5.0 D-2 lean. Heavier (~500 KB gz) but more faithful for academic PDFs.
- **Server-side OCR** for image-only PDFs. Out of scope. Arxiv PDFs are text-bearing in 99%+ of cases; OCR is a Phase-6+ enhancement.
- **Cache the PDF binary too.** Rejected — binary is ~MB; text is ~50KB. Caching the cheap-to-recompute part is wasteful disk.
- **Module-shared rate limiter** across arXiv-Atom + PDF fetches. Rejected — different subdomains, different limits.
- **Use `Buffer.from(await response.arrayBuffer())`**: yes, that's the standard pattern for Node's Buffer from a Web `Response`.

### Tradeoffs / defensible defaults flagged

1. **Separate rate-limit bucket** for PDF fetches. Trade-off: if a script makes both Atom + PDF calls, the two buckets are independent — could exceed combined-host expectations. Acceptable because PDFs and Atom hit different subdomains.
2. **`pdf-parse` warts**: known to emit `Warning: TT: undefined function: 22` console noise. Acceptable; pipe to stderr or swallow. Going with: silently ignore. The function returns the text successfully despite the warnings.
3. **No layout preservation**: pdf-parse gives plain text without column / table structure. Tables get linearised. Acceptable for Unit 5.5's leaderboard extraction — the LLM handles the cleanup. If quality is too lossy, ADR-style escalation: replace `pdf-parse` with `pdfjs-dist` + a custom layout-aware extractor.

### Tests

`pdf-text.test.ts` (≥ 6 tests):

1. **Fetches and parses on cache miss** — mock fetch returns a Buffer; mock parser returns `{ text, numpages }`; verify `extractPdfText` returns the right shape.
2. **Reads from cache on subsequent calls** — `fetchImpl` not called the second time.
3. **`noCache: true`** forces a re-fetch.
4. **Writes the cache as JSON** — verify the disk shape (`arxivId`, `text`, `numPages`, `sourceUrl`, `fetchedAt`).
5. **Defaults to `https://arxiv.org/pdf/<id>`** when `pdfUrl` is omitted.
6. **Custom `pdfUrl` overrides the default**.
7. **HTTP error throws** without writing the cache.
8. **`fetchedFromCache: true`** on a cache hit, `false` on a miss.
