# 9.1 — ADR-0012 (Auth provider: NextAuth.js v5 + GitHub OAuth)

First ADR of Phase 9. Pins the provider lean surfaced in [Unit 9.0 D-3](9.0-phase-9-prep.md). Mirrors the [ADR-0008](../adr/0008-llm-provider-anthropic.md) precedent — first ADR of a phase that introduces a new third-party runtime surface lands BEFORE the runtime install (Unit 9.4 will pull `next-auth@^5`). Docs-only.

## THINK

### Scope

Single output: [`docs/adr/0012-auth-provider.md`](../adr/0012-auth-provider.md) accepted on this commit; [`docs/adr/README.md`](../adr/README.md) index extended; cross-references threaded.

The ADR pins five sub-decisions:

- **D-A. Runtime = NextAuth.js v5 (Auth.js)** — `^5.x`; only auth library in `dependencies`; other libraries (Clerk SDK, Lucia, Iron Session) forbidden.
- **D-B. Identity provider = GitHub OAuth (initially the only one)** — multi-provider expansion deferred to Phase 10+ Q-promotion.
- **D-C. Session strategy = database-backed (Drizzle adapter)** — JWT-only sessions forbidden in Phase 9; revocability + auditability prioritized.
- **D-D. Sign-in UX = redirect-to-provider** — no modal / popup.
- **D-E. User identity model** — Drizzle `users` table schema; `githubLogin` column joins to file-system `editorial.primary_curator` (preserves ADR-0005 file-first curator-of-record; DB tracks sign-in identity).

### Why this ADR lands now (vs Unit 9.4 with the code)

ADR-0008 precedent: "first ADR of a phase that introduces a new third-party runtime surface lands BEFORE the runtime install." NextAuth.js v5 + the Drizzle adapter are the new runtime surfaces. Pinning the provider choice on a docs-only commit before Unit 9.3's DB scaffold + Unit 9.4's auth wrapper keeps the architectural decision separable from the implementation mechanics.

### Why GitHub-only initial provider

Multi-provider expansion (Google, GitLab, email-link, etc.) is **forbidden in Phase 9** per the ADR's D-B clause. Reasons:

1. The curator community is GitHub-native (per the existing Phase-6 Discussions integration + `/contributing` v1.x §1 — every existing issue template assumes GitHub).
2. Multi-provider expansion is one config block per provider; not a refactor. Adding Google later is cheap.
3. Single-provider scope keeps Phase 9 at 10 units (per Unit 9.0 breakdown); multi-provider would inflate by 2-3 units per provider (separate OAuth app registrations, separate env vars, separate test coverage).
4. The §5.8 "GitHub OAuth strongly preferred for community signal" guideline aligns.

Non-GitHub users have a Phase-10+ Q-promotion path (email-link provider would let arbitrary users sign in via magic-link emails, decoupling from the GitHub assumption).

### Why DB-backed sessions vs JWT

JWT sessions are stateless; revocation requires either short expiry or a JWT-blocklist. For a curator workflow where the curator-of-record needs to be auditable + revocable per-session, DB-backed sessions are the cleaner contract:

- Session record in `sessions` table holds the user-ID + expiry.
- Cookie holds the session ID (opaque token).
- Server-side `auth()` reads the DB on each request → returns the user + session.
- Revocation = `DELETE FROM sessions WHERE id = ?` (instant; no waiting for JWT expiry).
- Auditability = `SELECT * FROM sessions WHERE user_id = ?` (full session history visible).

Extra DB read per request is ~5ms on Turso edge (per ADR-0013 lean); acceptable on the 103 kB First Load JS budget.

### Considered alternatives (mirrored from ADR-0012)

- **Clerk SaaS**: rejected. Vendor lock-in; SaaS pricing curve; user data in third-party DB.
- **GitHub OAuth direct**: rejected. We'd own OAuth state, CSRF tokens, session refresh, sign-out cleanup — higher per-unit complexity for marginal benefit over NextAuth.js v5.
- **No auth (defer Phase 9)**: rejected. Closes nothing on §13; postpones DB-trigger flip that's already 5 phases overdue.

### What this ADR does NOT pin

- **DB choice** — Unit 9.2's ADR-0013 (Turso/libSQL + Drizzle lean per Unit 9.0 D-4). ADR-0012's D-C references "the Drizzle adapter" without naming the underlying DB; that's deliberate. ADR-0013 closes the DB axis.
- **Sign-in / sign-out UI placement** — Unit 9.5 D-11 deferred. Site-header right side next to LocaleToggle + ThemeToggle is the lean.
- **First write-path scope** — Unit 9.6 (watchlist toggle per Unit 9.0 D-5). ADR-0012 doesn't predetermine which features ship; it pins the auth runtime that EVERY write-path will use.
- **Multi-provider expansion timing** — explicitly Phase 10+ per D-B. Q-promotion path documented.
- **Profile / preferences page** — Phase 10+ scope.
