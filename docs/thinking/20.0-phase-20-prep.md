### D-3. Backfill scope

Per ADR-0019 D-E (already pinned):

| Backfill concern | Phase 20 scope | Notes |
|---|---|---|
| **Affected rows** | Rows in `user` table where `imageOverride IS NOT NULL` AND URL matches Vercel Blob pattern (`https://*.public.blob.vercel-storage.com/avatars/*`). | All Phase 16-18 uploads + Phase-19-uploads-pre-deploy-cutover. Treat Phase-19+ uploads conservatively: re-strip is idempotent (no EXIF present → no-op pixel-wise; same buffer round-trip). |
| **Download phase** | `fetch(url)` → `Buffer.from(await response.arrayBuffer())`. | Vercel Blob URLs serve publicly; no auth header needed. |
| **Strip phase** | `sharp(buffer).rotate().toBuffer()` — same pipeline as Phase-19 `putAvatar`. | Reuses ADR-0019 D-B + D-D contracts verbatim. |
| **Upload phase** | New `put()` call with new storage key (`avatars/<userId>-<new-timestamp>.<ext>`). | Cannot overwrite the original key — Vercel Blob immutable URLs. New key generates new public URL. |
| **DB-update phase** | `UPDATE user SET imageOverride = <new-url> WHERE id = <userId>`. | Single transaction per user; rollback on failure. |
| **Delete phase** | `del(<old-url>)` to remove the original Vercel Blob containing EXIF. | Best-effort; idempotent on already-deleted (Vercel Blob `del` is no-op). |
| **Dry-run** | `--dry-run` CLI flag: log all 5 phases without executing the strip / upload / DB-update / delete steps. | Curator inspects log first; production-safety check. |
| **Idempotency** | Safe to re-run; rows already migrated have new URLs (no longer match Phase 16-18 timestamp prefix — but pattern doesn't distinguish). Re-strip pass is a no-op on already-stripped images (sharp's `.toBuffer()` produces identical buffer when no EXIF present). | Operator-level safety: log-and-skip if `imageOverride` URL hash matches a recent-run sentinel; Phase-20 keep simple — accept idempotent re-process if curator runs twice. |
| **Error handling** | Per-row try/catch; failed rows logged + skipped; surface summary at end. | One bad row doesn't kill the batch. |
| **Rate limiting** | Phase 20 doesn't add rate limiting; user count low at MVP scale. | If Phase 21+ scales, add `setTimeout` delay between rows or batch concurrency limit. |
| **Concurrency** | Phase 20 ships serial-row processing (for loop). | Phase 21+ if curator demands parallelism. |

### D-4. CLI flag shape

Mirrors `ingest-arxiv.ts` / `validate-content.ts` posture:

```
pnpm backfill-exif-strip [--dry-run] [--user-id=<id>] [--verbose]
```

- **No args** → process ALL `users.imageOverride IS NOT NULL`
  rows matching Vercel Blob pattern; execute strip / upload /
  DB-update / delete.
- **`--dry-run`** → process all rows; log expected mutations;
  skip strip / upload / DB-update / delete. Curator inspection
  pass.
- **`--user-id=<id>`** → process a single user row (e.g., for
  surgical user-reported privacy retraction). Combined with
  `--dry-run` for safety check.
- **`--verbose`** → log per-row progress with byte counts +
  EXIF tag counts before/after.

Phase-20 minimal: ship `--dry-run` + no other flags. `--user-id` +
`--verbose` Phase 21+ if curator demands.

### D-5. Pipeline placement

Per ADR-0019 D-E (already pinned):

**Decision**: stand-alone CLI script at
`scripts/backfill-exif-strip.ts`. NOT a server action / NOT an
API route / NOT a cron job. Manual curator invocation only.

Why stand-alone CLI:
- One-shot operational task (run once Phase 20; re-run only if
  curator-driven retroactive privacy request).
- No user-triggered surface (avoids accidental backfill on
  user sign-in).
- No always-on listener (CRON / queue / webhook overkill at
  MVP scale).
- Mirrors `ingest-arxiv` / `extract-leaderboard` patterns
  (operational scripts).

Why reuse `lib/storage/putAvatar` flow elements:
- Same `sharp(buffer).rotate().toBuffer()` pipeline contract
  (ADR-0019 D-B + D-D).
- Same `avatars/<userId>-<timestamp>.<ext>` storage-key shape
  (ADR-0017 D-A).
- Same `put(key, buffer, { access: "public", contentType })`
  Vercel Blob call.
- **NOT identical reuse** — the script downloads from existing
  URL (not from a `File`), so the buffer-construction step
  differs (`fetch + arrayBuffer` vs `file.arrayBuffer`); the
  content-type comes from the URL extension (not from
  `file.type`); the storage-key timestamp uses `Date.now()` so
  the new key differs from the old.

### D-6. Backwards compatibility + safety

- **URL-pattern filter** narrows to `https://*.public.blob.vercel-storage.com/avatars/*`.
  Skips any `imageOverride` URLs that point elsewhere
  (theoretical; current Phase 16-19 code only writes Vercel
  Blob URLs, but the filter defends against future variations
  like S3/R2 swap).
- **No retroactive DB-row deletion**. Failed rows stay with
  original URL; curator can re-run after diagnostic fix.
- **Original-blob delete is last**. If the strip / upload /
  DB-update steps succeed and the delete fails, the user has a
  stripped avatar and an orphan blob — orphan-blob cleanup is
  the existing Phase-16 carryover (`scripts/cleanup-orphan-blobs.ts`
  unbuilt). Phase 20 doesn't deepen this gap.
- **`--dry-run` mandatory first run**. README + script header
  document this; the script exits with a one-line "did you mean
  to run --dry-run first?" prompt if 0 dry-runs precede an
  execution run within the operator's session (Phase 21+
  refinement; Phase 20 ships the warning only as a comment in
  the script header).

Phase 20 ships the backfill script primarily exercised via
`--dry-run` inspection rather than automated tests. Rationale:

- The script's correctness depends on:
  - Live DB connection (Turso) — not unit-testable without
    fixtures.
  - Live Vercel Blob storage — Phase-19 sharp-mock pattern
    breaks down for fetch-from-URL semantics.
  - Operator dry-run verification before each execution run.
- The `sharp` strip behavior is already exercised by Phase-19
  Unit 19.2 tests (564/564).
- The Vercel Blob `put` + `del` calls are tested at the
  `lib/storage/` layer (Phase-19 + Phase-16 tests).
- The DB-update SQL is a single-row update — drizzle-orm's own
  test suite covers `UPDATE` semantics.

**Phase 20 test posture**: ship the script + manual `--dry-run`
documentation; no new vitest tests (Phase-20 test count stays at

factor `backfillSingleUser(userId, blobUrl): Promise<{...}>`

### Phase-20-blocking decisions deferred to per-unit implementation

- **D-9. Drizzle query for affected rows** (Unit 20.1). Lean:
  `db.select({ id, imageOverride }).from(user).where(isNotNull(user.imageOverride))`
  then filter URL pattern in TS (drizzle doesn't have native
  regex predicate across SQLite/Turso).
- **D-10. Storage-key collision handling** (Unit 20.1). Lean:
  `Date.now()` differs from original timestamp (was at upload
  time; new is at backfill time); collision theoretically
  possible if multi-curator simultaneously running. Phase-20
  mitigation: append `-backfill` suffix to storage key for
  visual distinction in Vercel Blob console
  (`avatars/<userId>-<timestamp>-backfill.<ext>`).
- **D-11. Verbose log format** (Unit 20.1). Lean: one line per
  row — `[user-id] [original-url] → [new-url] (n bytes; m bytes
  saved)`.
- **D-12. Exit code convention** (Unit 20.1). Lean:
  - 0 → all rows processed successfully (or 0 rows matched).
  - 1 → at least 1 row failed (script still processes remaining
    rows; non-zero exit signals operator-attention).
- **D-13. Script invocation alias** (Unit 20.1). Lean: add
  `"backfill-exif-strip": "tsx scripts/backfill-exif-strip.ts"`
