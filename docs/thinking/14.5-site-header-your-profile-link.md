# 14.5 — SiteHeader "Your profile" link (signed-in only; ADR-0015 D-F)

Sixth Phase-14 unit; fourth code unit. Wires the **public profile
route discoverability** into the global chrome surface per
[ADR-0015](../adr/0015-per-user-privacy-model.md) D-F. Adds an
inline avatar + `@login` link in SiteHeader pointing to
`/u/{githubLogin}` (public canonical) when signed-in.

## THINK

### What this unit ships

Single artifact:

- **`components/site-header/index.tsx` (edit)** — adds a new
  `safeLogin()` defensive wrapper paralleling `safeAuth()`; calls it
  when a session is present; renders an inline `<Link>` with avatar
  16×16 + `@login` (truncated to 12 chars) → `/u/{githubLogin}` only
  when both session AND `githubLogin` are non-null. Hidden when
  `users.githubLogin === null` (Phase-9 retrofit edge from Unit 9.6's
  deferred `events.linkAccount` callback).

### Design decisions realized from ADR-0015 D-F

- **Target = `/u/{login}` (PUBLIC canonical)**, not `/profile` (edit
  mode). The link surfaces the user's PUBLIC identity surface as the
  primary "your profile" affordance. `/profile` remains accessible via
  the "Edit your profile" CTA on the public profile shell (Unit 14.3
  rendered conditionally when own profile).
- **Visual register**: avatar 16×16 + `@login` truncated to
  `PROFILE_LINK_LOGIN_TRUNCATE = 12` chars. Mirrors GitHub's
  navbar-identity pattern + Phase-10 Unit 10.2 avatar precedent.
- **Hidden when login is null**: a signed-in user who hasn't been
  through Unit 9.6's `events.linkAccount` callback has `users.githubLogin
  = null`; the public profile route can't resolve without a handle.
  Hiding the link is the right failure mode (better than rendering a
  broken `/u/null` link).
- **Defensive `safeLogin()` wrapper**: paralleling `safeAuth()`'s
  CI / fresh-clone defense pattern. Treats DB-read failure as "no
  login known" → link doesn't render. Production-mode normal-operation
  path is a no-op around success.

### Placement decision (Unit 14.0 D-18)

Unit 14.0 D-10 + ADR-0015 D-F specified "between 'Trending' nav link
and the right-side controls (search + locale + theme + auth)".
Reconciled at code time as: **prepend the profile link inside the
right-side controls cluster, before `SearchTrigger`**.

Rationale:
- The right-side cluster uses `ml-auto` to push everything to the
  right; anything floated "between" the nav and the right cluster
  would create a mid-page anchor with awkward spacing.
- Placing the profile link as the first item in the right cluster
  preserves the "user identity on the right" convention (GitHub's
  pattern; convention the auth-control already established).
- Visually adjacent to `AuthControl` (which renders the sign-out
  button) reinforces the "this is your account surface" affordance.

Alternative considered + rejected: append the profile link to the end
of `<nav>`. Mixed dense-text nav links with avatar-bearing identity
link breaks visual rhythm of the nav cluster.

### Truncation + class details

- `PROFILE_LINK_LOGIN_TRUNCATE = 12` chars; handles longer than 12
  show `@truncated…`.
- `hidden ... sm:inline-flex` — mobile-hidden per existing SiteHeader
  desktop-first pattern (the nav cluster is also `hidden ... sm:flex`).
- `font-mono text-xs underline-offset-2 hover:underline` — matches
  the per-problem listing `@login` register established by Unit 14.4.
- `aria-label` + `title` reuse `public_profile.aria_label` ("Public
  profile of @{login}") translation key from Unit 14.3's pre-added
  namespace. Avoids new keys.

### No new i18n keys

Reuses the `public_profile.aria_label` key pre-added in Unit 14.3 per
atomic-i18n discipline (Phase-13 Unit 13.2 + Phase-12 Unit 12.4
precedent). Unit 14.5 ships ZERO new messages.json edits — the entire
i18n surface was front-loaded.

### Edge cases handled

- **No session**: link absent. AuthControl renders the "Sign in"
  branch as today.
- **Session present, `users.githubLogin = null`**: link absent
  (Phase-9 retrofit edge). User sees AuthControl signed-in pill +
  sign-out form but no profile link.
- **`session.user.image = null`** (rare; some GitHub accounts lack
  avatars): avatar omitted but `@login` text renders. Link still
  navigates.
- **Long handle** (`>12` chars): truncated with ellipsis; full handle
  visible in `title` tooltip via the aria-label reuse.

### Phase-10 + Phase-12 protected-route threshold (no change)

Phase-9 Class B item 12 (middleware-based auth-route protection) lifts
when **3+ protected page routes** exist. Phase 10 = `/profile` (1);
Phase 12 = `/curator/challenges` + detail (2). Phase 14's new
`/u/[handle]` + `/u/[handle]/challenges` routes are NOT auth-protected
(any visitor can view); they don't count toward the threshold. **Lift
remains deferred** at Phase-14 close.
