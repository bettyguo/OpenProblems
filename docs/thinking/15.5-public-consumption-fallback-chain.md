# 15.5 — Public consumption: `displayName` fallback chain + bio section (3 surfaces + new helper)

Sixth Phase-15 unit; fourth code unit. Realizes
[ADR-0016](../adr/0016-user-editable-profile-fields.md) D-E
(public-consumption fallback chain) + D-F (bio display formatting)
across the three identity surfaces Phase 14 + Phase 10 established.

## THINK

### What this unit ships

Three surface changes + one helper extension:

1. **`lib/auth/login.ts` (extend)** — new
   `getUserMetadataById(userId)` helper alongside existing
   `getLoginById`. Returns `{ githubLogin, displayName } | null` in
   one query. Used by SiteHeader to avoid an extra round-trip when
   it needs both columns.
2. **`components/site-header/index.tsx` (edit)** — replaces
   `safeLogin` with `safeUserMetadata`; passes the resolved
   `displayName` to AuthControl via new prop.
3. **`components/auth-control/index.tsx` (edit)** — accepts new
   optional `displayName?: string | null` prop; updates signed-in
   pill fallback chain to `displayName → name → email → fallback`
   per ADR-0016 D-E (email permitted on own pill; never on public
   surfaces per ADR-0015 D-A invariant).
4. **`app/[locale]/u/[handle]/page.tsx` (edit)** — updates public
   profile fallback chain to `displayName → name → githubLogin →
   fallback` per ADR-0016 D-E + ADR-0015 D-A (no email on public
   surface). Adds bio section between header and activity; rendered
   ONLY when bio is non-null per D-17 lean.
5. **`app/[locale]/profile/page.tsx` (edit)** — updates own surface
   header card fallback chain to `displayName → name → githubLogin
   → email → fallback`. Reads `userRow.displayName` from the
   existing query (already added in Unit 15.4).
6. **`messages/{en,fr}.json` (edit)** — adds 1 new key per locale
   (`public_profile.bio_aria_label`) for the bio section's
   `<section aria-label>` (2 keys net; atomic pre-add).

### Why a sibling helper instead of extending `getLoginById`

`getLoginById` returns `string | null` (just the login). Phase-9 onward,
6 production callers + 1 test rely on this exact return shape. Adding
a 2nd return field would force changes to every caller, including
test mocks.

Cleaner: `getUserMetadataById(userId)` is a NEW sibling helper
returning `{ githubLogin, displayName } | null`. Existing callers
unchanged. New consumer (SiteHeader) is the first to use the new
shape; future per-user-attribution surfaces can adopt incrementally.

### Why `displayName` fallback differs between own surface + public surface

| Surface | Chain | Email surface? |
|---|---|---|
| `/u/{handle}` public | `displayName → name → githubLogin → fallback` | **NO** — ADR-0015 D-A invariant. |
| `/profile` own | `displayName → name → githubLogin → email → fallback` | **YES** — user sees own email. |
| AuthControl pill | `displayName → name → email → fallback` | **YES** — user sees own pill. |

Email surfaces only on the user's OWN signed-in surfaces; never on
the public profile route. The two-tier semantics preserve ADR-0015
D-A's "email never reaches `/u/{handle}`" invariant exactly.

### Why bio section omitted entirely when null

Per Unit 15.0 D-17 lean: omit the bio section (don't render an empty
placeholder). Rationale: profiles of users who haven't filled in a
bio shouldn't show clutter ("(no bio)"); the empty space is itself a
signal that the user hasn't customized.

Implementation: `{profile.bio && <section>...</section>}`.

### Why `whitespace-pre-wrap` and not `<pre>`

`whitespace-pre-wrap` preserves user-typed newlines + wraps long
lines (no horizontal scroll). `<pre>` would force monospace + no-wrap;
inappropriate for prose bio. The class matches Phase-12's curator
review notes rendering (`reviewNotes` block on `/profile`).

### Why no test changes needed

The fallback chain changes are pure UI render logic; no helpers
change behavior. Unit 15.3 added shape-extension tests for
`PublicProfile` (displayName + bio fields render through
`getPublicProfileByHandle`); those cover the data path.
`getUserMetadataById` is a thin Drizzle wrapper around an existing
query pattern; behavior parallels `getLoginById` (which has no
direct test today — both are too thin to test usefully).

The visible-behavior changes (UI bundle rendering) are covered by
the existing typecheck + build steps. No vitest tests target
JSX-rendered output for these surfaces in this codebase.

### Bio aria-label rationale

`<section aria-label="Bio">` gives screen readers a landmark to
navigate. The aria-label is locale-specific (EN: "Bio"; FR: "Bio" —
unchanged because the term is the same in both). Adding the key
upfront in both locales preserves the atomic-i18n discipline.
