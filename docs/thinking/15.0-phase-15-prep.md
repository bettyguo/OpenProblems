### D-1. First-thread recommendation: **Q63 promotion — user-editable profile fields** (natural Phase 14 → 15 progression)

**Recommendation**: pull **Q63 promotion** (user-editable profile
fields) as Phase 15's keystone thread. **Natural progression** from
Phase 14's read-only public profile to Phase 15's WRITE-able profile.
ADR-0015 D-C explicitly anticipated this: "Phase 14 ships READ-ONLY
public profile … Phase 15+ writes can extend the contract incrementally
with user-feedback data from Phase 14's read-only delivery."

~8-9 unit phase (matches Phase 12 + Phase 14's 9-unit shape; one new
ADR — editable profile field model — surfaces; second ALTER migration
in project history; first new writes surface for `users` table).

**Rationale**:

1. **Phase 14 → Phase 15 architectural inheritance**. Phase 14 shipped
   `users.name` + `users.image` from GitHub OAuth profile; Phase 15
   adds user-controlled overrides. Same DB table; same auth flow;
   same i18n namespace (extends `messages.profile.*` + `messages.public_profile.*`).
2. **ADR-0015 D-C explicit Phase 15+ deferral**. The deferral wasn't
   "indefinitely deferred"; it was "defer until Phase 14 read-only
   ships". Phase 14 shipped clean. Phase 15 is the natural follow-on.
3. **Largest UX-demand argument** among Phase-14-surfaced candidates.
   Users expressing identity preferences (custom display names + bios)
   is a basic identity-platform expectation. Q64 (opt-out) + Q65
   (per-curator activity feed) are less universally demanded.

   new ADR since Phase 14 / ADR-0015. Pins the writes-side contract
   for Phase 15+ identity surfaces. Anticipated D-clauses: field set,
   length/sanitization model, image-override deferral (Phase 16+),

### D-3. Editable field set (Phase 15 scope)

Per ADR-0016 D-A (anticipated) + Unit 14.0 ADR-0015 D-C deferral:

| Field | Phase 15 editable? | Max chars | Validation | Notes |
|---|---|---|---|---|
| `users.displayName` | **YES** | 80 | trim + length cap; no HTML | Falls back to `users.name` (GitHub-derived) on null. |
| `users.bio` | **YES** | 280 | trim + length cap; no HTML | Plain text only; whitespace preserved (newlines OK; no markdown rendering Phase 15). |
| `users.image` (override) | **NO** (Phase 16+) | — | — | Needs storage ADR; deferred. |
| `users.githubLogin` | **NO** (immutable post-OAuth-link) | — | — | GitHub-canonical per ADR-0012 D-E. |
| `users.email` | **NO** (never surfaces) | — | — | Per ADR-0015 D-A. |
| `users.name` (GitHub-derived) | **NO** (read-only; `displayName` overrides it) | — | — | Auth.js v5 populates from OAuth profile; not user-touched. |

**Rationale**:
- `displayName` + `bio` are the universally-demanded editable identity
  fields. Image / username / email overrides each surface a separate
  architectural concern (storage / canonical-identity / privacy) and
  belong in separate phases.
- Length caps: `displayName` 80 chars matches GitHub's display-name
  limit; `bio` 280 chars matches Twitter / X / Bluesky standard for
  short-form bios. Both are conservative; can be expanded Phase 16+.
- Plain text only: no markdown rendering Phase 15. Bio whitespace
  preserved via `whitespace-pre-wrap`. Markdown rendering would
  introduce XSS surface + new dependency (remark/rehype OR similar);
  defer until clear demand signal.

### D-4. Validation + sanitization model

Per ADR-0016 D-B (anticipated):

1. **Length cap** enforced both client-side (textarea `maxLength`) and
   server-side (return 400 if exceeded). Mirrors Phase-11 rationale
   field validation pattern.
2. **Trim whitespace** on save (no leading / trailing whitespace
   stored).
3. **No HTML / no markdown rendering**. Plain text only. Renders via
   `whitespace-pre-wrap` for newline preservation; React's escape
   handles XSS by default; no need for sanitization library.
4. **Empty/null treatment**: empty string after trim = clear the field
   (store NULL); displayed as fallback chain (custom → GitHub → "Anonymous").
5. **No regex content filter**. Phase 15 trusts users; Phase 16+ may
   add explicit content moderation if abuse signals appear.

### D-6. Server-action vs API route

Per ADR-0016 D-D (anticipated):

**Decision**: **server actions** for profile edit. NOT a `/api/v1/profile/edit`
REST endpoint.

**Rationale**:
- Server actions are the Phase-10 + 11 + 12 precedent for signed-in
  own-state mutations (sign-out form on `/profile`; submit challenge
  inline form; withdraw challenge button; curator review buttons).
- REST endpoint is the precedent for **external surfaces** that
  third parties might consume (watchlist API; rating-challenges API;
  curator review API). Profile edit is an internal UX-only surface;
  no third-party demand.
- Server actions get CSRF protection + cookie handling for free;
  REST would require explicit token handling.
- Mirrors Phase-10 Unit 10.2's sign-out form pattern verbatim:
  inline `<form action={async () => { "use server"; ... }}>`.
