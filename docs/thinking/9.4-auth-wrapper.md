### Why GitHub provider without explicit args

Auth.js v5 auto-discovers provider credentials from canonical env var names: `AUTH_GITHUB_ID` + `AUTH_GITHUB_SECRET`. Invoking `GitHub({ clientId: process.env.GITHUB_CLIENT_ID, clientSecret: process.env.GITHUB_CLIENT_SECRET })` runs afoul of `exactOptionalPropertyTypes: true` in `tsconfig.json` — the provider's config types `clientId: string` (not `string | undefined`), so passing potentially-undefined env values is a typecheck error.

Three solutions considered:

1. **Conditional spread** (`...(process.env.GITHUB_CLIENT_ID ? { clientId: ... } : {})`) — works; ugly.
2. **Non-null assertion** (`clientId: process.env.GITHUB_CLIENT_ID!`) — works; lies to the type system; would crash at runtime if env vars are missing.
3. **Auto-discovery** (`providers: [GitHub]` with no args) — Auth.js v5 reads `AUTH_GITHUB_ID` + `AUTH_GITHUB_SECRET` from env automatically.

Picked (3) — it's the framework's canonical pattern. `.env.example` + Q54 updated to use the auto-discovered names.

### Why `trustHost: true`

Auth.js v5 defaults to requiring an exact `AUTH_TRUST_HOST` env var match for the request's `Host` header, as a CSRF defense. For Vercel deploys this is normally `auto-trusted`, but the project's preview-deploy URLs (`https://op-pr-N.vercel.app`) shift per PR. Setting `trustHost: true` in code allows any host — acceptable because:

1. The OAuth callback is locked to the registered URI in the GitHub OAuth app; a malicious host can't complete the OAuth flow.
2. Vercel + GitHub OAuth's combination of `redirect_uri` validation + signed `state` parameter is the actual CSRF defense.
3. The project doesn't host arbitrary user-uploaded content that could exploit host confusion.

If Q54's OAuth app registration locks the production callback URL strictly, `trustHost: true` adds zero attack surface in production.

### Edge cases & failure modes

1. **OAuth credentials missing** (Q54 unmet): the `GitHub` provider initializes with `undefined` clientId; sign-in clicks redirect to `/api/auth/error` with a "Configuration" error code. Unit 9.5's sign-in/sign-out UI handles this gracefully (renders "Sign-in unavailable" instead of a broken Sign-in button).
2. **`AUTH_SECRET` missing**: Auth.js v5 requires it for signing CSRF tokens + session cookies. Auth.js will throw on initialization if unset. Documented in `.env.example`; missing-secret error is loud (clear stack trace).
3. **DB connection failure** during sign-in: Drizzle adapter throws; Auth.js surfaces as a server error to the user. Acceptable failure mode (rare; would indicate a deeper DB outage).
4. **Multiple sign-ins from same GitHub account**: Auth.js dedupes via the `accounts` table's composite primary key (`provider` + `providerAccountId`); subsequent sign-ins resolve to the same `user.id`. Correct behavior.
5. **`AdapterAccountType` import resolution**: requires `next-auth` installed (which it is post-Unit-9.3). Schema typecheck depends on this; deferring the cast to Unit 9.4 was the correct sequencing (could not have type-imported pre-install).
