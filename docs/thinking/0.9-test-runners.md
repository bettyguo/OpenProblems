# Unit 0.9 — Test runners: Playwright, Storybook, Lighthouse CI

> Stage 1 (THINK) artifact. Vitest already landed in Unit 0.5; this unit completes the §13 Phase 0 "Vitest, Playwright, Storybook, Lighthouse CI wired" deliverable.

## Problem restatement

Wire the three remaining test/QA runners so Phase 1+ units don't have to detour through "first set up X" loops:

1. **Playwright** — e2e smoke + visual regression. §5.9. Initial scope: one spec that boots the dev server and asserts the home page renders.
2. **Storybook 9** — component isolation. §5.9. Phase 0 has no components beyond `app/layout.tsx` + `app/page.tsx`; this unit installs the runner and registers a placeholder docs story so Phase 1 components can register stories from day one.
3. **Lighthouse CI** — perf / a11y / SEO / BP score gating. §13 Phase 0 gate is **perf ≥ 95 on the stub landing** (§13 Phase 1 raises this to 95 across perf+a11y+SEO+BP).

## Constraints

- **No real browser install in this commit.** Playwright requires a ~150 MB chromium download. Contributors run `pnpm exec playwright install chromium` once locally; CI image (Unit 0.11) pre-installs. Document this. The npm package itself is what we install in this unit.
- **Storybook 9 init is interactive** by default. Use `pnpm dlx storybook@latest init --yes` (non-interactive) if it exists; otherwise hand-author the minimal `.storybook/main.ts` + `.storybook/preview.ts` + a single MDX docs page.
- **Lighthouse CI runs against the production build** (`pnpm build && pnpm start`), not dev. CI auto-spins the server.
- **Coexistence with Vitest.** Playwright spec files use `*.spec.ts` (in `e2e/`); Vitest stays on `*.test.ts`. ESLint config (Unit 0.8) covers `test/**`; add `e2e/**` to the same node-globals override.
- **No conflicting `test` script.** `pnpm test` stays Vitest. `pnpm test:e2e` runs Playwright. `pnpm lhci` runs Lighthouse CI.
- **Storybook port collision.** Storybook defaults to 6006; safe (different from Next's 3000).

## Alternatives considered

1. **Playwright + Storybook 9 + Lighthouse CI** (chosen). The §5.9 stack.
2. **Cypress instead of Playwright.** Cypress is solid but Playwright won the React/Next ecosystem by 2025 and ships built-in cross-browser support, trace viewer, and visual diff. §5.9 explicitly names Playwright.
3. **Ladle / Histoire instead of Storybook.** Faster, leaner. But §5.9 names Storybook 9 explicitly, and the addon ecosystem (a11y, viewport, controls) is unrivaled.
4. **Skip Lighthouse CI; rely on Vercel's Lighthouse integration.** Vercel runs Lighthouse on every preview deploy automatically. But the §13 Phase 0 acceptance gate requires _enforcement_ on the threshold (≥ 95). Lighthouse CI gives us the gate in a way Vercel's built-in report doesn't.

## Edge cases & failure modes

1. **Playwright dev-server bring-up race.** `webServer` config polls; default timeout is 60s. Set to 120s for slow Windows-China network on first font fetch.
2. **Lighthouse against `pnpm dev`.** Don't — dev mode disables minification. Always run against `pnpm start` (production server).
3. **Storybook 9 build with Tailwind v4.** Storybook needs to load the same PostCSS pipeline as Next; configure via `viteFinal` or `addons: ["@storybook/addon-styling-webpack"]` depending on builder choice. Use Vite builder for faster HMR.
4. **Lighthouse CI on `prefers-color-scheme: dark` clients.** Lighthouse defaults to light mode emulation. Our two-tone design works in both; no special config needed for Phase 0.
5. **Playwright Reading on Windows file paths.** `playwright.config.ts` with `testDir: "./e2e"` works on both POSIX and Windows.
6. **Lighthouse perf score on bare landing**. With 102 KB First Load JS (RSC framework chunks), one h1, one paragraph — perf should easily land 100. Watch for regression once Phase 1 viz components arrive.
7. **CI vs local discrepancy.** Lighthouse scores fluctuate by 5–10 points between local and CI hardware. Setting threshold at 95 keeps headroom; CI workflow (Unit 0.11) runs 3 iterations and averages.
8. **Husky + Playwright slowness.** Pre-commit hook does NOT run Playwright (would be too slow). Vitest only. Playwright runs on PR via CI.

## Open questions surfaced (→ OPEN_QUESTIONS.md)

- **Q22. Visual regression baselines.** Where are Playwright screenshot baselines stored? Repo (`e2e/__screenshots__/`) is the default; OS-specific renderings (Windows-China devs vs Linux CI) drift. Lean: keep CI as the source of truth via dedicated Linux runner; ignore Windows-rendered baselines.
- **Q23. Storybook builder.** Vite (fast HMR) vs Webpack (mature Next-compat). Lean: Vite — Storybook 9's default — unless a specific Next integration forces Webpack.
