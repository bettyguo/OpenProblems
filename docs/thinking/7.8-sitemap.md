### Scope NOT in this unit

- **No `app/robots.ts`** — sister convention; complementary but out of scope here. Could be added in a follow-on once we have a `User-Agent` policy + `Disallow` patterns (no AI-scraping policy enumerated yet).
- **No `lastModified` per entry** — Next.js infers `Date.now()` at build time when omitted. Adding accurate per-entry timestamps requires plumbing `editorial.last_curated` (problems) / `methodology.date` (MDX) / etc., which adds 30-40 lines without changing crawler behavior meaningfully (every modern crawler treats lastModified as a hint, not authoritative). Defer.
- **No `changeFrequency` or `priority` per entry** — Google specifically [deprecated reliance on these](https://developers.google.com/search/blog/2023/04/sitemaps-lastmod-ping) in 2023; Next.js defaults are fine.
- **No locale-alternates for routes other than `/about`** — only `/en/about` + `/fr/about` exist at HEAD. Adding `alternates` blocks for the 22+ other routes would link to URLs that 404 today. Strict fail-closed: alternates only for routes that actually have a `[locale]/` shadow. Unit 7.3a expands this surface; this unit's `buildSitemap()` doesn't auto-expand — it explicitly lists the about-pilot exception.
- **No talk-page count limiting** — all 10 problem talk-pages are included (Q48 lean). If a future Phase introduces a `discussions_enabled: false` field, builder could skip; not relevant at HEAD.
- **No sitemap-index** — Next.js generates a single `/sitemap.xml`; for ~80+ URLs (well under the 50 k Google limit), no need to split into a sitemap-index file.

### Constraints

- **`app/sitemap.ts` is a Next.js convention** (same special-file pattern as `app/robots.ts` / `app/manifest.ts`). Default export must be a function returning `MetadataRoute.Sitemap`; arbitrary helper exports trigger build errors per the existing known wrinkle. Helpers live in `lib/sitemap/`.
- **`#site/content` import from Velite** — `app/sitemap.ts` is server-only; `#site/content` works there (it works in every server context in the project).
- **`exactOptionalPropertyTypes: true`** — `alternates` is optional on sitemap entries; only assign when defined. Matches the pattern.
- **Q2 / Q3 still open** — `SITE` is `https://llm-openproblems.org` placeholder; when DNS lands, all three sites (`lib/digest/rss.ts`, `app/api/v1/rss.xml/route.ts`, `lib/sitemap/build-sitemap.ts`) update together. Hardcoded duplication is acceptable for 3 sites (would extract to `lib/site-url.ts` if it grows to 5+).
- **No new dependencies** — `MetadataRoute.Sitemap` type ships with `next`; nothing else needed.

### Alternatives considered

- **Extract `SITE` into `lib/site-url.ts`** — Rejected for this unit. Would touch 3 unrelated files (digest/rss, api/rss.xml, plus new sitemap); same low collision risk but not in this unit's scope. Tracked as a deferred follow-on. Sites currently equal; drift would surface immediately in tests.
- **Generate sitemap entries auto-from Next.js's route discovery** — Rejected. Next.js doesn't expose a "list all routes" API; would need to scan `app/` directory ourselves. Velite collections + a hardcoded static-route list is simpler and more explicit (we control which routes get in).
- **Include locale alternates for *all* routes** — Rejected per "scope NOT in this unit" #4. Would link to 404s for routes that don't have a `[locale]/` shadow.
- **Use `s.path()` from Velite to enumerate URLs** — Rejected; Velite paths aren't 1-to-1 with site URLs (e.g., methodology MDX path → `/methodology/[version]`). Explicit route-template enumeration in the builder is clearer.
- **Ship sitemap-index + per-collection sitemaps** — Rejected; total URL count at HEAD is ~80 (10 problems × 5 routes + 30 papers + 126 authors + 14 institutions + 10 misc); well under 1 k. Single sitemap is fine.

### Tradeoffs / defensible defaults flagged

1. **Locale alternates only for `/about`** — narrow fail-closed scope. Once Unit 7.3a expands `/[locale]/<route>` coverage, a follow-on or 7.3a itself extends the alternates list. Builder shape supports an arbitrary route → alternates map; just one entry today.
2. **`SITE` hardcoded duplication** — three call sites; acceptable. Centralization is a deferred follow-on tracked here.
3. **No `lastModified`** — modern crawlers fetch the URL anyway and respect HTTP cache headers. lastModified is a heuristic hint; absence means "crawler decides freshness." Acceptable for a content-heavy site that updates infrequently. If Lighthouse SEO flags it, add per-entry timestamps in a follow-on.
4. **Builder is sync** — content collections are JSON-loaded at import time (no I/O at runtime). `app/sitemap.ts` can be sync but Next.js examples favor async; staying async for forward-compat with future I/O.
5. **`/_not-found` excluded** — Next.js doesn't render a not-found page at a discoverable URL; including it would be wrong. Same for framework routes (`/api/...`); excluded by builder design (no enumeration of `app/api/`).
