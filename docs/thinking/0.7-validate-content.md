# Unit 0.7 — scripts/validate-content.ts

> Stage 1 (THINK) artifact.

## Problem restatement

Build the CI gate that walks `content/` and validates every YAML / JSON file against the matching Zod schema from Unit 0.5. The gate is what MASTER_PROMPT §5.9 / §14.3 mean when they say "CI fails if any content file fails its schema." Once Unit 0.7 lands, every content commit (Phase 1 onward) is gated by it.

## Inputs

- ADR-0003 / ADR-0004 (Zod is source of truth; content is files).
- Unit 0.5 schemas, Unit 0.6 taxonomy.
- §6 (content layout: `content/taxonomy.yaml`, `content/problems/<slug>/`, `content/papers/<id>.yaml`, `content/authors/<slug>.yaml`, `content/institutions/<slug>.yaml`).
- §14.3 (CI fails on schema error).
- §13 Phase 2 ("Cross-link audit" lives in a _separate_ `scripts/cross-link-audit.ts` — Unit 0.7 does only single-document validation plus the simplest taxonomy FK check; everything else is Phase 2).

## Constraints

- **Single-document validation primary.** Each file checked against the schema appropriate to its location.
- **One cross-document FK check now**: `Problem.domain` must exist in the loaded taxonomy; `Problem.subdomain` must exist under that domain. Every other cross-FK (rating-action → problem, paper.contributions → problem + benchmark, paper.authors → author slug, etc.) is Phase 2's cross-link-audit job.
- **Tolerant of absent dirs.** Phase 0 content/ contains only `taxonomy.yaml`. The script must walk that case and exit 0 with `filesChecked: 1`.
- **Library/CLI split.** `lib/content/validate.ts` exports the logic (testable from Vitest). `scripts/validate-content.ts` is a tiny CLI wrapper around the library that exits 0/1 and prints a human-readable report.
- **Errors are structured.** Each error carries `file`, `schema`, and a list of `{path, message}` issues so the formatted output is greppable.
- **Run via `tsx`.** `"type": "module"` means a `.ts` script needs a TS runner. `tsx` is the de-facto standard in 2026 (esbuild-based, zero-config), small devDep.
- **No telemetry, no side effects other than logs + exit code.** This is a build-time gate.

## Alternatives considered

1. **Library + CLI split with `tsx`** (chosen).
2. Mono-file `scripts/validate-content.ts` with all logic inline. Rejected: cannot Vitest the validation logic without invoking the CLI's process.exit.
3. Compile-then-run with `tsc --outDir scripts-dist` then `node scripts-dist/validate-content.js`. Rejected: extra build step, no payoff over `tsx`.
4. Use Velite's CLI (`velite build`) to do the validation. Rejected: Velite isn't installed yet (Phase 1+), and we want validation working _before_ Velite arrives so it can also gate non-MDX content.

## Edge cases & failure modes

1. **Malformed YAML.** Parser throws; caught and converted to a structured error with `path: ""` and a `parse error: ...` message.
2. **JSON files.** `entries.json` per problem is JSON, not YAML. Branch on `.endsWith(".json")`.
3. **Entries.json arrays.** The file is an array of leaderboard entries; each entry validated individually, errors indexed (`entries.json[0]`, `entries.json[1]`, …).
4. **Empty `content/` subdirs.** `listEntries` returns `[]` if the dir doesn't exist; no error.
5. **Non-directory entries in `content/problems/`.** Skip non-dirs (e.g., a stray `.DS_Store`).
6. **Cross-FK with no taxonomy loaded.** If `taxonomy.yaml` fails to parse, skip cross-FK checks and continue. A taxonomy error is already its own error; we don't want every problem to also flag "domain not found because taxonomy is broken."
7. **Schema changes.** When a schema gains a required field, every existing content file fails validation. This is the _correct_ behavior — the migration is content-side work and should be explicit.
8. **`exactOptionalPropertyTypes` in test fixtures.** Fixture content can be authored as plain YAML — no TS typing — so the strict option doesn't affect fixture authoring.

## Decision

Files in this unit:

1. `lib/content/validate.ts` — `validateContent(contentRoot)`; walks the tree; returns `{ filesChecked, errors }`. Includes the problem→taxonomy FK check.
2. `lib/content/validate.test.ts` — three tests:
   - Real `content/` in repo: 0 errors.
   - Fixture `test/fixtures/content-valid/`: 0 errors with > 0 files checked.
   - Fixture `test/fixtures/content-invalid/`: > 0 errors.
3. `test/fixtures/content-valid/taxonomy.yaml` — minimal valid taxonomy.
4. `test/fixtures/content-invalid/taxonomy.yaml` — taxonomy with duplicate domain ids (a structural error TaxonomySchema's `superRefine` catches).
5. `scripts/validate-content.ts` — CLI. Calls `validateContent(${cwd}/content)`, prints results, exits 0/1.
6. Install `tsx` as devDep.
7. Add `pnpm validate-content` script (runs `tsx scripts/validate-content.ts`).
8. THINK artifact (this).
