### Design decisions realized

- **ADR-0015 D-A** (field partition): per-user listing uses the same
  `PUBLIC_CHALLENGE_STATUSES` filter as Phase-13 per-problem listing
  (server-side in `getPublicChallengesByUser`). `rejected` +
  `withdrawn` rows never reach the client.
- **ADR-0015 D-B** (case-insensitive handle lookup): page calls
  `getPublicProfileByHandle(handle)` for handle resolution + 404.
- **Unit 14.0 D-6** (mirror Phase-13 listing pattern): identical row
  shape, status pill color palette, 200-char rationale truncation
  precedent. Differs only in the column rendered alongside the row.

### Per-user vs per-problem rendering shape

The two listings share **most** row content:

| Element | Per-problem listing (Phase 13) | Per-user listing (this unit) |
|---|---|---|
| Top-left of row | Submitter `@login` (NEW: Link → `/u/{login}`) | Problem title (Link → `/problems/{slug}`) |
| Top-right of row | Submitted date | Submitted date |
| Second line | dimension → proposedValue + status pill | dimension → proposedValue + status pill |
| Body | Rationale truncated 200 chars | Rationale truncated 200 chars |
| Footer | `acceptedActionId` reference (when accepted) | `acceptedActionId` reference (when accepted) |
| Heading | "@login submitted N challenges to {problem.title}" | Per-user page title |

The "differ only on the column rendered" pattern justifies keeping the
two routes as separate page-level components (per Unit 14.0 D-6 lean).
Extracting a shared `<ChallengeRow>` component would save ~50 LOC but
couples the two listings' future divergence (e.g., per-user might
want a per-row "edit / withdraw" button when viewing own profile;
per-problem listing never has that surface).

### Why I navigate "← @{githubLogin}" back to the profile (not "← @login's profile")

The back-link is rendered as `← @{githubLogin}` — short and dense,
matches GitHub's nav back-pattern. The full handle is visible; the
"profile" framing is implicit since the page is titled "@{login}'s
rating challenges" right below.

### Per-problem listing `@login`-to-Link upgrade

Phase-13 Unit 13.3 rendered submitter login as:

```tsx
<p className="text-muted-foreground text-xs">
  {row.submitterLogin
    ? t("submitter_label", { login: row.submitterLogin })
    : t("submitter_unknown")}
</p>
```

Phase 14 upgrades to:

```tsx
<p className="text-muted-foreground text-xs">
  {row.submitterLogin ? (
    <Link
      href={`/u/${row.submitterLogin}`}
      className="hover:text-accent underline-offset-2 hover:underline"
    >
      {t("submitter_label", { login: row.submitterLogin })}
    </Link>
  ) : (
    t("submitter_unknown")
  )}
</p>
```

The `submitter_unknown` branch stays plain text — there's no profile
to link to when `submitterLogin` is null (Phase-9 retrofit edge from
Unit 9.6 deferred `events.linkAccount`).

Hover register: `hover:text-accent` + underline matches the page's
other clickable text patterns; preserves visual density.

### Empty state

When the user has submitted 0 public challenges, render the
bordered-dashed card with `@{login} has no public challenges yet` +
"Submit a challenge →" link to `/problems`. The empty CTA appears
unconditionally — even when viewing someone else's profile — because
the prompt is generic (anyone signed-in can submit; pointing other
visitors to `/problems` is a reasonable next-step). Phase-15+ may
refine to show CTA only on own profile.
