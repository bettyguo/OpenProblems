# 8.1 — Bulk page migration + middleware (`localePrefix: "always"`)

First code unit of Phase 8. Closes the open-ended deferral catalogued in Unit 7.9 Class A (the "bulk page migration + middleware" follow-on scope-cut from Unit 7.3 to a wiring proof). Lands `middleware.ts` enforcing `localePrefix: "always"` per [ADR-0011 D-B](../adr/0011-i18n-strategy.md); shadows every existing route under `app/[locale]/`; switches navigation across components + pages to the `next-intl` Link wrapper; extends `TALK_PATHNAME_REGEX` for locale-aware Giscus paths; updates `lighthouserc.json` URLs to the `/en/` prefix.

> **Scope drop (decided mid-flight)**: HTML shell migration (`<html lang={locale}>` move into `app/[locale]/layout.tsx`) was **dropped from this unit**. The existing `app/layout.tsx`-owns-`<html>` structure stays. The shell migration would have closed Unit 7.7's documented `html-has-lang` axe-rule risk; that risk **survives Unit 8.1 and now lives at Unit 8.4** (originally `useTranslations` for LocaleToggle aria-label; rescoped to absorb the shell migration since both depend on SiteHeader moving under `NextIntlClientProvider`).

### Why big-bang

**Big-bang is the only viable option** after the inventory surfaced these structural constraints:

1. **`localePrefix: "as-needed"` is not a clean transient state.** Under `as-needed`, `/en/about` 308-redirects to bare `/about` (because EN is the default locale and gets the bare canonical). This breaks Phase 7's LHCI cohort (`/en/about`, `/fr/about`, `/en/methodology`, `/fr/methodology` would all redirect away from the enrolled URLs).
2. **`localePrefix: "never"` doesn't preserve `/fr/...` paths.** Would break the FR pilot routes.
3. **`localePrefix: "always"` from the start is the only consistent semantic.** Requires every route to have a `/[locale]/` shadow before the middleware activates (otherwise bare-path requests redirect to non-existent `/en/<path>` and 404). The atomicity is forced.
4. The 22 `[locale]/<route>/page.tsx` shadows + middleware activation + bare-route deletion + `<Link>` import sweep all need to land in the same commit.

### Constraints

- **Sitemap test invariant** (line 56 of `build-sitemap.test.ts`): "does NOT attach locale alternates to routes without a `[locale]/` shadow." Currently checks `/problems` and `/digest`. **After Unit 8.1 lands**, every route has a `[locale]/` shadow, so the invariant becomes vacuously satisfied. Unit 8.5 will expand alternates to all routes.
- **TALK_PATHNAME_REGEX backward-compat**: existing pre-migration Giscus discussions are titled `/problems/<slug>/talk` (no locale prefix); the extended regex must still match these. Solution: optional capture group `/^\/(?:(en|fr)\/)?problems\/([a-z0-9-]+)\/talk$/` (the capture group index for slug shifts from `m[1]` to `m[2]` — `lib/digest/build-digest.ts` updated accordingly).
- **`app/not-found.tsx` stays at root.** Per Next.js convention, the root 404 page is rendered for unmatched routes regardless of locale. It is not under `[locale]/`. Locale-aware 404s could be added via `app/[locale]/not-found.tsx` in a future unit.
- **API routes stay at root.** `/api/v1/*` paths bypass middleware via the matcher exclusion. No locale prefix on API URLs.
- **First Load JS shared chunk = 103 kB**: must hold. The `lib/i18n/navigation` Link wrapper adds ~0.5 KB max (a thin re-export); `middleware.ts` is server-side only.

### Alternatives considered

- **Stage as 8.1 (foundations) + 8.1a (bulk migration)**: rejected because bare-route deletion + middleware activation + Link import sweep all need locale shadows in place simultaneously (otherwise routing breaks).
- **Stage as 8.1 (Link wrapper + middleware) + 8.1a (page migration)**: rejected for the same reason — middleware with `always` requires shadows; middleware with `as-needed` breaks `/en/...` LHCI URLs.
- **Land the HTML shell migration alongside the bulk migration** (original plan): rejected; existing `app/layout.tsx`-owns-`<html>` structure is preserved. Shell migration deferred to Unit 8.4.
- **Keep `next/link` everywhere and rely on middleware 308s**: rejected. Every internal click would hit the middleware and 308-redirect (1 extra HTTP round-trip per nav). Perf regression on a 103 kB-budget project. Also breaks deep-link prefetching.
- **Add `app/[locale]/not-found.tsx`** for locale-aware 404 copy: deferred. Out of scope; default Next.js root 404 is acceptable through Phase 8.

### Edge cases & failure modes

1. **A page references a hardcoded `/<route>` URL via `next/link` AND that page is missed in the migration sweep**: the link clicks would 308-redirect via middleware, working but inefficiently. Catch: grep sweep for `next/link` imports before commit.
2. **`generateStaticParams` on a per-slug page doesn't yet emit the cartesian product with locales**: build would error with "Page didn't generate any params for `[locale]`." Catch: each migration adds `locales.flatMap(...)` to the static-params function.
3. **`TALK_PATHNAME_REGEX` regression**: a pre-migration discussion title `/problems/x/talk` must still match. The capture-group index for slug shifts to `m[2]`. Catch: extend the existing 4 test cases in `lib/discussions/github-graphql.test.ts` + update `lib/digest/build-digest.ts` slug capture.
4. **`lighthouserc.json` URL-list contains 13 non-locale-aware entries that need `/en/` prefix**: missing one means LHCI runs against a redirect target (allowed but a perf cost). Catch: per-URL audit before commit; the diff itself is small enough to eyeball.
5. **`app/not-found.tsx` references `<Link href="/">`**: this lives outside `[locale]/`, so the locale-aware Link wrapper may not have provider context. Kept on `next/link`.
6. **`components/locale-toggle/index.tsx` uses explicit locale-prefix URLs**: the i18n wrapper would double-prefix. Kept on `next/link`.
