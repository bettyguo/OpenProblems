# 40.2 — Multi-value `MARKDOWN_EXTENSIONS` parsing in `getExtensionRegistry()` + 10 dispatch tests + 6 end-to-end composed tests

Phase 40 unit per `docs/thinking/40.0-phase-40-prep.md` (HEAD
`0bca3cb` = Unit 40.1 ship). Wires `CompositeExtensionRegistry`
into the factory via multi-value env-var parsing.

## THINK

### Problem restatement

Unit 40.1 shipped `CompositeExtensionRegistry` + 13 tests +
ADR-0018 D-G APPEND. The composite is not yet wired into the
factory — `getExtensionRegistry()` still rejects comma-
separated values via the `default` throw arm. Unit 40.2 adds:

- Multi-value parser: split on comma + trim + filter empty
  entries.
- Validation: reject `"default"` mixed with others; reject
  duplicates; reject unknown values.
- Construction: build component registries via a helper
  function; wrap in `CompositeExtensionRegistry` if ≥2
  components; single-value short-circuits to the direct
  registry.
- Updated JSDoc + error messages.
- 10 dispatch tests covering all parsing edge cases.
- 6 end-to-end composed tests verifying wikilinks + tables
  coexist on their respective surfaces.

### Inputs

- HEAD `0bca3cb` (Unit 40.1 ship).
- `lib/markdown/extensions/composite.ts` — Unit 40.1 class.
- `lib/markdown/extensions/index.ts` — Phase-39 single-value
  switch dispatch.

### Constraints

- **Backward compat**: existing single-value behaviors
  preserved (`""` / `"default"` / `"wikilinks"` / `"tables"`
  all unchanged).
- **Whitespace tolerance**: trim around comma-separated
  values for operator-friendly env-var syntax.
- **Single-value short-circuit**: `"wikilinks,"` → 1 entry
  after filtering → return direct `WikilinkExtensionRegistry`
  (not a 1-component composite).
- **Refactor extracted helper**: `buildSingleConsumerRegistry(name)`
  shared between single-value and multi-value paths;
  throw-on-unknown stays in one place.

### Alternatives considered

**Option 1: Comma-split + validate + compose (chosen)** —
the prep doc design. Standard ENV-VAR-with-multi-value
pattern.

**Option 2: Keep switch, add explicit case for
`"wikilinks,tables"` (rejected)** — N² combinations explode
as more consumers ship. Phase 41+ third consumer would need
3 new arms (`wikilinks,tables,X`, `wikilinks,X`, `tables,X`).
Not generalizable.

**Option 3: JSON env-var (rejected)** —
`MARKDOWN_EXTENSIONS={"wikilinks":true,"tables":true}`. More
expressive but worse operator ergonomics for a simple
on/off flag. Defer to a Phase 41+ if per-extension
configuration emerges (e.g.,
`MARKDOWN_EXTENSIONS_WIKILINKS_SURFACES=actionRationale,bio`).

### Edge cases & failure modes

1. **Empty after split** (`","`, `" "`, etc.): throw with
   clear message.
2. **`"default"` mixed**: throw — `default` is the empty-
   configuration; can't combine.
3. **Duplicates**: throw — defensive against env-var typos
   like `"wikilinks,wikilinks"`.
4. **Unknown entry in multi-value**: throw with the unknown
   value named.
5. **Single-value-via-comma**: `"wikilinks,"` → 1 entry →
   direct registry (not composite). Tested.
6. **Whitespace**: `"wikilinks , tables"` → 2 entries after
   trim. Tested.
7. **Singleton cache survives**: cached composite stays
   across calls within same env. Tested.
