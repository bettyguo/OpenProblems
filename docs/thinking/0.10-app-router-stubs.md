# Unit 0.10 — App Router stub IA

> Stage 1 (THINK) artifact. Critical-path per §15.4 — proceeding under the unit-rhythm rule.

## Problem restatement

Stand up every route from MASTER_PROMPT §9 as a Next.js App Router stub so the URL space is fixed in Phase 0 and Phase 1 content authoring can drop into already-routed paths. Stubs render a "Phase 1 content pending" placeholder with the right h1; dynamic routes echo their slug/id so the routing graph is verifiable. URL stability is non-negotiable per §6 ("Slugs are URL-stable. Renames go through `redirects.yaml`").

## Inputs

- MASTER_PROMPT.md §9 (the full IA table — 23 routes including api/v1/\*).
- §6 (URL stability rule, redirects.yaml as the only rename path).
- §14.1 (no barrel files; component file structure).
- §10.4 (perf budget — keep route code small, RSC by default).
- §15.5 reviewer checklist item: "Am I leaking implementation into the URL space?"
- Unit 0.4 design tokens (already wired).
- Unit 0.9 Playwright smoke spec (asserts home renders); will extend.

## Constraints

- **Every §9 route must exist** with a `page.tsx` (or `route.ts` for `api/v1`). Missing routes break the URL contract for Phase 1.
- **RSC-only by default.** No `"use client"` in this unit.
- **Async params for dynamic segments** (Next 15 made dynamic params async; type as `Promise<{ ... }>` and await).
- **API endpoints respond 501** (Not Implemented) with a JSON body identifying the endpoint. Avoid 200 with empty bodies — that would silently break consumers.
- **No content fetches yet.** Stubs render constants; Velite + content reads land in Phase 1.
- **Lighthouse stays ≥ 95.** Each stub is tiny: an `<h1>` + a `<p>`. Should not regress perf.
- **A11y.** Every page has a `<main>` landmark, a single h1, no orphan headings.
- **DRY.** Factor a `<StubPage>` (server component) and a `stubJsonResponse()` helper so the 22 page/route files don't duplicate boilerplate.

## Alternatives considered

1. **Per-route hand-authored page.tsx + a small `StubPage` helper** (chosen). Cleanest, lowest-duplication.
2. **One catch-all `[[...slug]]/page.tsx`** that branches on the path. Rejected: defeats the purpose of fixing the URL space; Next would dispatch the same page to every route.
3. **Generate stubs from a manifest file.** Overkill for 22 routes; would obscure the IA in a generator.

## Edge cases & failure modes

1. **Dynamic-segment name collisions across parents.** `[slug]` is used for problems / authors / institutions; `[id]` for papers; `[domain]` and `[subdomain]` for domain tree; `[version]` for methodology. Each scopes under its parent directory, so no collision.
2. **`[version]` route segment encoding.** §9 says `/methodology/v[N]` — the `v` prefix is part of the URL. Either name the segment `[version]` and require the URL to include the `v` (e.g., `/methodology/v1`), or name the segment with a literal `v` prefix and an enum-only `[version]`. Chose the cleaner option: route is `app/methodology/[version]/page.tsx`, URL is `/methodology/v1`, and the stub echoes the version literally.
3. **Trailing slash policy.** Next 15 defaults to no trailing slash. Don't override.
4. **`api/v1/rss.xml`** — `route.ts` returns `text/xml` with a 501 body. Phase 3 will replace with a real RSS feed.
5. **`generateStaticParams` not provided** for dynamic segments. Stubs render dynamically per request. Phase 1 will add `generateStaticParams` from content so they're prerendered.
6. **Sitemap discoverability.** Add a small `<nav>` section to the home page listing the top-level routes so reviewers (and Playwright) can crawl without hardcoded URLs.
7. **`not-found.tsx`.** Phase 0 doesn't add custom not-founds for stubs; Next's default suffices. Phase 1 may add per-segment 404s.
8. **`/about` vs `/contributing`** — neither is in the data model. They're prose-only. Same StubPage placeholder.
9. **API JSON envelope.** Phase 0 uses `{ endpoint, status: "not-implemented", phase: 0 }`. Phase 1 will define the canonical envelope (per §9 "Read-only JSON API"); document the deferral as Q25.

## Open questions surfaced (→ OPEN_QUESTIONS.md)

- **Q25. JSON API envelope.** Phase 1 needs a canonical response shape: top-level `{ data, meta }` à la JSON:API, or flat resource objects, or something else. Document and decide at Phase 1 kickoff.
- **Q26. `not-found.tsx` strategy.** Per-segment 404s (e.g., custom message under `/problems/[slug]/not-found.tsx`) vs single global 404. Defer to first user-facing not-found scenario.
