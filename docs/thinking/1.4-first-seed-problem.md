# Unit 1.4 — First seed problem: `hallucination-reduction`

> Stage 1 (THINK) artifact. Critical-path.

## Problem restatement

Author the **first** end-to-end seed problem (`hallucination-reduction`, §16 entry 1) so every later Phase 1 unit (1.5 RatingRadar, 1.6 problem detail page, 1.7 domain hubs, 1.8 problems index, 1.10 landing, 1.11 remaining seeds) has a real, schema-valid content folder to render against. This unit produces content files only; rendering is Unit 1.6's job. The Velite content pipeline is extended to surface the new collections.

## Inputs

- §16 entry 1 (hallucination-reduction): Domain `deep-learning` / Subdomain `large-language-models`. Benchmarks: HaluEval, TruthfulQA-2026, SimpleQA, FACTS. "Why now": Yang et al. NeurIPS 2025 — test-time-scaling increases hallucinations on knowledge-intensive tasks.
- Unit 0.5 schemas: `OpenProblemSchema`, `RatingActionSchema`. Velite config in Unit 1.1.
- §15.6 sticky rule: "Never invent leaderboard numbers or paper titles. If a number is unknown, write `TODO(curate)` and surface it in `OPEN_QUESTIONS.md`."
- §14.3: "All numbers carry a source. CI fails if a number appears in MDX without an adjacent `<Cite paper="...">` or footnote." Phase 1.4 authors *prose without specific numbers* — every quantitative claim either gets a `TODO(curate)` placeholder or is omitted.
- Unit 1.3 — `velite.config.ts` MDX pipeline is wired (KaTeX, Shiki). Problem MDX prose files reuse it.

## Constraints

- **Slug = directory name = `problem.yaml#slug` field.** All three must agree. `validate-content` (Unit 0.7) audits problem-level shape; Phase 2's cross-link audit will audit the dir-vs-slug match.
- **Zero fabricated benchmark scores.** Benchmarks list `id`, `name`, `dataset`, `metric`, `metric_direction`, optional `upper_bound` and `protocol_url`. The score-tracking is the leaderboard entries, which Phase 1.4 omits (no `entries.json`).
- **Initial rating action is the editorial baseline.** All five dimensions populated with `rationale` + `confidence`. Values are the curator's best read for May 2026; they will be revised. The action file is immutable (ADR-0005) — corrections come as new action files.
- **MDX frontmatter only.** Each of `background.mdx`, `definition.mdx`, `history.mdx` is a small frontmatter block (`title`, `summary`) plus prose. No raw numbers, no `<Cite>` components yet (Phase 2 lands the citation system).
- **Velite extends with problems + ratings + problem-pages.** Add three collections; keep the same `s`-based duplication pattern as Unit 1.1 (Q31).
- **Editorial fields point to `@jikun` placeholder.** Q7 (editorial governance) still open; placeholder for now.

## Alternatives considered

1. **Author all five files (YAML + 3 MDX + 1 rating) + extend Velite with 3 collections** (chosen).
2. **Author only problem.yaml + rating; skip the 3 MDX files.** Smaller unit; but §16 explicitly lists "background.mdx, definition.mdx, history.mdx" as required for each seed. Rejected — would only push the work to Unit 1.6.
3. **Author all 10 seeds in a single unit.** Far too big; the §15 unit-of-work rule explodes. Rejected.

## Edge cases & failure modes

1. **Velite + Zod 4 (Q31).** Same as Unit 1.1. Schemas in `velite.config.ts` use `s`; canonical schemas in `lib/schemas/` validate via `pnpm validate-content`.
2. **MDX without numbers.** §14.3's enforcement isn't wired yet (would be a future ESLint or MDX lint rule). For now, follow the rule by hand and surface a Q if a number genuinely needs to ship.
3. **`posed_year`.** This problem isn't dateable to a single year (hallucination as an LLM concern post-2020 GPT-3 era; "knowledge-intensive QA hallucination" as a benchmarked problem ~2022). Pick 2022 with a rationale in the rating-action.
4. **Subdomain id collision.** `large-language-models` is under `deep-learning`. The Q11 mapping handles this — the FK check (Unit 0.7) ensures the (domain, subdomain) pair resolves.
5. **`tags`.** Use 3–5 lowercase keywords; first tag conventionally maps to the most-specific aspect (e.g., "factuality").
6. **Pre-commit hook (ADR-0005).** The first rating-action file is a *new* file under `content/problems/*/ratings/`. New files are allowed; the hook only blocks modifications or deletions. Confirm by committing.

## Open questions surfaced

- **Q33. `<Cite paper="…">` component shape.** §14.3 mandates that every quantitative claim has an adjacent citation. The component is unimplemented. Lean: design in Phase 2 alongside the cross-link audit. Until then, Phase 1 MDX prose deliberately avoids numbers.
- **Q34. `entries.json` for the first seed.** Empty / absent? `[]` is valid against `LeaderboardEntry[]`. Lean: omit the file entirely in Phase 1.4; create it in Phase 2 when first verified entries land.

## Decision

Files:

1. `docs/thinking/1.4-first-seed-problem.md` — this file.
2. `content/problems/hallucination-reduction/problem.yaml` — fully-typed `OpenProblem`. 4 benchmarks (HaluEval, TruthfulQA-2026, SimpleQA, FACTS) with no scores, all `metric_direction: higher-is-better`. `status: open`, `posed_year: 2022`, `tags: ["factuality", "calibration", "llm", "evaluation"]`, `editorial: @jikun / 2026-05-14`.
3. `content/problems/hallucination-reduction/background.mdx` — frontmatter (`title`, `summary`) + 2–3 paragraphs framing the problem space. No numbers.
4. `content/problems/hallucination-reduction/definition.mdx` — frontmatter + a tight formal-ish definition (faithfulness vs. calibration vs. abstention).
5. `content/problems/hallucination-reduction/history.mdx` — frontmatter + 3–4 milestone paragraphs (knowledge-intensive QA → RAG → uncertainty estimation → test-time-scaling-makes-it-worse Yang et al. 2025). No specific numeric scores.
6. `content/problems/hallucination-reduction/ratings/2026-05-14-initial.yaml` — editorial baseline: Difficulty A · Saturation 35/100 · Urgency 5 · Value 5 · Industry Call 5. Each with confidence (0.6–0.8) and a one-sentence rationale.
7. `velite.config.ts` — extend with three collections: `problems` (problem.yaml per dir), `ratings` (ratings/*.yaml per dir), `problem_pages` (3 MDX files per dir, `kind` field discriminates). Same `s`-duplication pattern.
8. `OPEN_QUESTIONS.md` — Q33, Q34.
9. `CHANGELOG.md` — Unit 1.4 entry.

Stage 2 (DESIGN): the content shape is fully specified by the §7 / Unit 0.5 schemas + the §16 seed-list line for this problem. The MDX frontmatter shape for problem pages is `{ title, summary }` — no body schema needed.

Stage 3 (CODE): write the files, extend Velite, run validate-content + velite + typecheck + build.

Stage 4 (ITERATE): reviewer checklist, CHANGELOG, OPEN_QUESTIONS, commit.
