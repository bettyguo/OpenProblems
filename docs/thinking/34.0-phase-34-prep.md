### D-3. Widget design anticipated (no ADR; pure UX)

D-A. Selector helper = new pure-function-paired DB helper
`getSubscriptionsForUser(userId): Promise<Subscriber[]>` in
`lib/subscribers/`. Mirrors Phase-31 `getVerifiedSubscribersForDomain`
pattern (DB SELECT with simple WHERE filter; no in-JS reconciliation
needed since `userId` is indexed via FK constraint).

D-B. Widget placement = inside existing `/[locale]/profile` server
component, between the **Watching** section + the **Your rating
challenges** section. Mirrors existing widget-density pattern (multi-
section profile page Phase 9-33 precedent).

D-C. Display fields per row = domain title (taxonomy-derived) +
last-digest-sent label (either ISO date from `lastDigestSentAt` or
"Never" if NULL) + per-row "Unsubscribe" link (uses existing
unsubscribeToken; href = `/api/v1/subscribe/unsubscribe/{token}`).

D-D. Empty state = "You haven't subscribed to the weekly digest yet"
+ CTA "Subscribe to the digest →" linking to `/[locale]/digest`.

D-E. Status filtering = only `status === "verified"` rows shown
(pending_verification rows in flight + unsubscribed rows hidden;
matches the Phase-31 digest-send cron's verified-only filtering).

D-F. Multi-row-per-user handling = displays all rows where
`userId === session.user.id`; multiple anonymous email-only rows from
the same user (linked via per-email Phase-30 anonymous path) won't
appear since they have NULL userId. Acceptable — matches the email-as-
primary-identity Phase-30 model carried forward.

D-G. Phase 35+ deferrals: Profile B full-edit UX (add / remove
domains inline without re-submit); per-problem subscription UI; cross-
domain summary toggle; bulk-unsubscribe button.
