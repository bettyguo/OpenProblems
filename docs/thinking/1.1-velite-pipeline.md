# Unit 1.1 — Velite content pipeline

> Stage 1 (THINK) artifact. Critical-path per the Phase 1 plan; proceeding under the unit-rhythm rule.

## Problem restatement

Wire **Velite** (ADR-0002) as the build-time content pipeline so subsequent Phase 1 units (1.3 methodology page, 1.4 first seed problem, 1.6 problem detail page, 1.7 domain hubs, 1.10 landing) can `import { taxonomy, problems, … } from "#site/content"` and get fully-typed, Zod-validated content with MDX already compiled. The schemas authored in Unit 0.5 are the source of truth and must not be duplicated.

After this unit:

- `pnpm build` runs Velite first, emits `.velite/`, then runs `next build` against the generated artifact.
- `pnpm validate-content` (Unit 0.7) continues to work — Velite *and* the script both consume the same Zod schemas; neither owns the validation surface alone.
- One Phase 0 route is migrated to import from `#site/content` as a smoke proof.

## Inputs

- ADR-0002 (Velite chosen for the MDX content pipeline; Zod-native; build-time validated).
- ADR-0003 (Zod 4 is the schema source of truth — `lib/schemas/*` files).
- ADR-0004 (file-first storage; Velite emits a JSON snapshot the App Router consumes).
- MASTER_PROMPT.md §5.4 (KaTeX inline + display, Shiki code, Mermaid diagrams in MDX), §6 (folder layout under `content/`), §5.7 (snapshot-trigger rule).
- Unit 0.5 schemas: `Taxonomy`, `OpenProblem`, `Benchmark`, `RatingAction`, `Paper`, `LeaderboardEntry`, `Author`, `Institution`. Existing import alias `@/lib/schemas/...`.
- Unit 0.6 `content/taxonomy.yaml` — only content file in Phase 0.
- Unit 0.7 `lib/content/validate.ts` — still authoritative for `pnpm validate-content`.

## Constraints

- **No schema duplication.** Velite collections reference the existing Zod schemas via `schema: ZodSchema` (or `schema: (s) => MySchema` if Velite requires its `s` factory). The `lib/schemas/*` files don't change in this unit; if Velite requires a transform, do it inside `velite.config.ts`, not by editing the schemas.
- **`.velite/` is generated; not committed.** Add to `.gitignore` (Unit 0.0's `.gitignore` already excludes `.next/` and `dist/` — add `.velite/`).
- **Path alias `#site/content`.** TypeScript `paths` and Velite's emitted ESM both reference this. Avoids polluting the `@/*` namespace with generated artifacts.
- **Build ordering.** `pnpm build` must run Velite before `next build`. Two cleanest options: (a) `"build": "velite && next build"` script chain; (b) `next.config.ts` with a `VeliteWebpackPlugin` from `velite/dist/next` (when supported). Pick (a) — simpler, explicit, easier to debug.
- **Phase 0 routes keep working.** App Router stub IA (Unit 0.10) must continue to render. Migrating *one* route to consume `#site/content` is the smoke test, but I won't migrate all 23 stubs in this unit — that's Unit 1.7+ work.
- **MDX support enabled but unused.** Phase 1 will author MDX (methodology, problem prose) starting in Unit 1.3. Velite's MDX pipeline (with rehype/remark plugins for KaTeX + Shiki) is configured here but exercised in later units.
- **Mermaid deferred.** §5.4 lists Mermaid but no Phase 1 deliverable requires it. Skip the remark-mermaid plugin until a unit needs it; surface as OPEN_QUESTIONS for traceability.
- **Validate-content still runs independently.** A failure in `validate-content` or Velite must each produce a clear error. Don't collapse them.

## Alternatives considered

1. **Velite + Zod schemas as-is, `pnpm build = velite && next build`** (chosen).
2. **Velite with its own `s` schemas, mirroring our Zod schemas.** Rejected — violates ADR-0003 ("single source of truth").
3. **Skip Velite for taxonomy.yaml; only use it for MDX.** Tempting since Unit 0.6's `loadTaxonomy()` works. Rejected — uniform pipeline simplifies subsequent units and lets us share the snapshot across all collections.
4. **`next.config.ts` integration via Velite's Next plugin.** Cleaner DX but couples build ordering to Next internals; debugging an opaque webpack plugin is worse than debugging a one-line script chain. Rejected.
5. **Replace `lib/content/load-taxonomy.ts` with Velite imports immediately.** Rejected — keeps Unit 1.1 small. The loader stays alongside Velite for now; later units can converge on whichever is cleaner.

## Edge cases & failure modes

1. **Velite version churn.** Velite is sub-1.0 as of late 2025. Pin the minor; bump via ADR if API breaks.
2. **Zod 4 compatibility.** Velite was originally Zod 3; v0.3+ added Zod 4 support. If schemas don't accept, file an issue and fall back to wrapping each Zod schema with `z.preprocess(...)` or use Velite's `s` and re-derive types. Surface as OPEN_QUESTIONS if this triggers.
3. **MDX strictness.** Velite parses MDX through `@mdx-js/mdx` which has been picky about TS-strict React types. If a generated `.velite/index.d.ts` doesn't typecheck, add `.velite/` to `tsconfig.include` and `skipLibCheck: true` (already set).
4. **`.velite/` import resolution before first build.** Importing `#site/content` in a route file when `.velite/` doesn't yet exist fails the typecheck. Mitigation: run `pnpm velite` as a `prebuild` hook for `pnpm dev` too, and document that contributors run it once after fresh clone (also goes into `pnpm prepare` or similar). For Phase 1, simplest: `"dev": "velite && next dev"`.
5. **`.velite/` written during dev creates a noisy file watcher feedback loop.** Velite has `--watch` and Next.js Fast Refresh ignores `.velite/` by default — verify.
6. **CI ordering.** `pnpm build` in GitHub Actions (Unit 0.11) will pick up the new chained command automatically.
7. **Empty content collections.** Phase 1 starts with only taxonomy + (later) one problem. Velite collections with `pattern: "problems/*/problem.yaml"` will return `[]` if no problems exist yet — fine.
8. **Path alias collisions.** `#site/content` uses TypeScript's `#` prefix convention for internal modules; doesn't collide with `@/*`.

## Open questions surfaced (→ OPEN_QUESTIONS.md)

- **Q29. Velite MDX plugin set.** Phase 1.1 wires KaTeX + Shiki; defers Mermaid. Confirm — or pull Mermaid forward if a content author wants it.
- **Q30. Snapshot publishing.** The `.velite/` JSON is the eventual "static JSON snapshot the client consumes" (§5.7). Decide whether to expose it via `/api/v1/snapshot.json` (size-capped) for third-party consumers, or keep server-side only until the 5 MB trigger. Defer to Unit 1.10 (landing) or later.

## Decision

Files / actions in this unit:

1. `docs/thinking/1.1-velite-pipeline.md` — this file.
2. `pnpm add -D velite` — single new devDep (and Velite's transitive remark/rehype packages).
3. `velite.config.ts` — at repo root. Defines one collection now (`taxonomy`, single-file YAML) and *stubs* additional collections (`problems`, `papers`, `authors`, `institutions`, `methodology`) with their patterns and schemas — Phase 0 has no files matching the patterns, so the collections build to empty arrays. The stubs are zero-cost and avoid Unit 1.3/1.4 having to re-do this work.
4. `tsconfig.json` — add `"#site/content": [".velite"]` and `"#site/content/*": [".velite/*"]` paths.
5. `.gitignore` — add `.velite/`.
6. `package.json` — change `"build": "next build"` → `"build": "velite && next build"`; change `"dev": "next dev"` → `"dev": "velite && next dev"`; add `"velite": "velite"` standalone script; add `"velite:watch": "velite --watch"`.
7. One smoke-test route: pick `app/domains/page.tsx` (a route with no content yet; Phase 0 stub). Migrate it to `import { taxonomy } from "#site/content"` and render the 8 domain titles as a list. Proves the pipeline end-to-end without contradicting the Phase 0 stubs.
8. `CHANGELOG.md` — Unit 1.1 entry.
9. `OPEN_QUESTIONS.md` — Q29, Q30.

Stage 2 (DESIGN): `velite.config.ts` is the central design surface. Collection definitions are the design.

Stage 3 (CODE): install, configure, smoke. Run `pnpm velite`, then `pnpm typecheck`, then `pnpm build`.

Stage 4 (ITERATE): reviewer-checklist, CHANGELOG, OPEN_QUESTIONS, commit.
