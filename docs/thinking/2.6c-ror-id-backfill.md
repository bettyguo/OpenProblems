# 2.6c — ROR ID backfill for Unit-2.6b institutions

Phase-2 hygiene follow-on. Closes the explicit deferral in Unit 2.6b:
"`ror_id` deliberately omitted (no fabricated IDs per §15.6 — a follow-on
curator pass pulls them from `ror.org`)." This is that pass.

## THINK

### Inputs

- **5 institutions landed in Unit 2.6b without `ror_id`:**
  - `brown-university` (Brown University, Providence, RI, US)
  - `caltech` (California Institute of Technology, Pasadena, CA, US)
  - `nvidia` (NVIDIA Corporation, Santa Clara, CA, US)
  - `huawei` (Huawei Technologies, Shenzhen, CN)
  - `instadeep` (InstaDeep, Tunis, TN — now also UK/FR)
- **InstitutionSchema** (`lib/schemas/institution.ts`): `ror_id` is optional
  with a strict regex `^0[\da-z]{6}\d{2}$` (9 chars: leading `0` + 6
  alphanumeric lowercase + 2 digits) — matches the canonical ROR ID format.
- **Velite duplicate** (`velite.config.ts`, line 224): `ror_id` is also
  declared on the Velite `InstitutionS` mirror (as `s.string().optional()`
  without the regex — the regex is intentionally only on the Zod-4 side
  per `lib/schemas` source-of-truth contract from Q31).
- **Authoritative source**: the ROR public API at `https://api.ror.org/v2/`.
  Each ID below was retrieved from the v2 organizations endpoint and
  cross-checked against the location and founding year shown on the
  matching record.

### Constraints

- **4 YAML edits**: append `ror_id:` field at the end of each institution
  file, after `homepage:`, matching the schema field order in
  `lib/schemas/institution.ts`. Don't reorder existing fields.
- **InstaDeep is a documented exception, not a bug.** ROR's coverage of
  private AI/ML startups is incomplete; InstaDeep (founded 2014, acquired
  by BioNTech in 2023) is simply not registered. Per §15.6 no fabrication.
  If it lands a ROR record later, a future content commit adds it.
- **No schema changes.** Both schemas already accept `ror_id`. The regex on
  the Zod-4 side will reject malformed values on commit via
  `pnpm validate-content` (Unit 0.7 cross-validator).
- **No collisions.** This unit only touches `content/institutions/`. The
  parallel curator session (per `reference_parallel_curator.md`) operates
  on `content/papers/` and `content/problems/` for paper-ingest / new-
  problem flows; institution YAML edits are out of their hot path.
- **Scope discipline.** The other 9 institutions (committed pre-2.6b) also
  lack `ror_id`. They were NOT in the handoff's explicit follow-on list,
  so they stay out of scope here. A future hygiene unit can extend the
  same pattern across them.

### Alternatives considered

- **Backfill `ror_id` for all 14 institutions in one pass.** Rejected —
  the handoff explicitly scopes this follow-on to the 5 from Unit 2.6b.
  Expanding to 14 is defensible but it's a different unit; doing it here
  blurs the audit trail tying this commit back to Unit 2.6b's deferral.
- **Skip InstaDeep silently.** Rejected — the absence of a ROR record is a
  curation fact worth documenting, both in this THINK doc and in the
  CHANGELOG entry. Future curators looking for "why does this one
  institution still lack `ror_id`?" should find the trail without
  re-running the ROR query.
- **Tighten the Velite mirror regex.** Rejected — out of scope for this
  unit. The Zod-4 schema in `lib/schemas/institution.ts` is the source of
  truth (Q31 contract); cross-validation via `scripts/validate-content.ts`
  catches divergence on the next content commit. Tightening the Velite
  mirror is a separate Q31 follow-on.
- **Add a ROR-existence check to the cross-link audit (Unit 2.11).**
  Rejected — out of scope. The cross-link audit operates on local slug
  graphs; outbound ROR liveness checks would introduce a network
  dependency and a flake source. If wanted, that's a separate auditor.

### Tradeoffs / defensible defaults flagged

1. **InstaDeep `ror_id` omitted.** Documented; expected to remain omitted
   until and unless ROR registers the organization.
2. **Other 9 institutions still lack `ror_id`.** Out of scope for this
   commit by handoff scoping. A natural follow-on is "Unit 2.6d — ROR
   backfill for original 9 institutions" if a future audit asks for it.
3. **No collateral file changes.** Build surface, route count, and bundle
   envelope are all unaffected; this is a metadata-only addition.
