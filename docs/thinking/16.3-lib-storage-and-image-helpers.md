### Why `lib/storage/` is deliberately thin

Per [ADR-0017](../adr/0017-image-storage.md) Option 1 Pros: vendor
lock-in mitigated by "keep `lib/storage/index.ts` deliberately
narrow (~3 helpers) so the vendor swap to S3 / R2 stays
straightforward if Phase 17+ trigger fires."

The module exposes ONLY:
- `putAvatar(file, userId): Promise<string>` (URL).
- `delAvatar(url): Promise<void>`.
- Private `inferExt(mime)` helper.

No leaking of Vercel-Blob-specific types (no `PutBlobResult` /
`HeadBlobResult` reexports); no leaking of `BLOB_READ_WRITE_TOKEN`
env-var handling (the SDK auto-resolves it). The swap surface for
S3 / R2 is the body of these three functions plus an env-var
rename ŌĆö nothing else.

### Why `@vercel/blob@2.3.3` not `@1.x`

ADR-0017 D-A and Unit 16.0 prep anticipated `@vercel/blob@1.x`
based on the prep-doc-time SDK research. Actual install at Unit
16.3 time resolved to `@vercel/blob@2.3.3` (latest stable). The
v1 ŌåÆ v2 API for `put` + `del` operations is **API-compatible**
(both return `{ url, pathname, contentType, contentDisposition }`
for `put`; both accept URL for `del`). v2 adds new features
(client-token-based direct uploads, presigned URLs) that this
unit doesn't exercise. ADR-0017's substantive intent (use Vercel
Blob; thin wrapper; SDK-based; first-party Vercel) is
unaffected by the major-version drift.

This is a CHANGELOG-only correction; the ADR body stays
immutable per the ADR-0005 / ADR-immutability discipline.

### MIME validation + magic-byte defense-in-depth

Per ADR-0017 D-B + D-F:

- **MIME check** (browser-provided `file.type`): rejects anything
  outside `image/jpeg`, `image/png`, `image/webp`. SVG excluded
  for XSS surface; GIF / animated WebP excluded per D-H "static
  only".
- **First-bytes magic-byte check**: re-validates the MIME against
  the actual file contents. A malicious user submitting
  `file.type === "image/jpeg"` with PNG bytes (or vice versa)
  gets rejected. Defends against MIME-type forgery via curl POST
  / browser quirk.

Signatures matched (12-byte slice via `file.slice(0, 12).arrayBuffer()`):
- **JPEG**: `0xFF 0xD8 0xFF` (3-byte prefix; any third byte in
  the JPEG marker range like `0xE0..0xEF`).
- **PNG**: `0x89 0x50 0x4E 0x47` (`"\x89PNG"`; 4-byte prefix).
- **WebP**: `0x52 0x49 0x46 0x46 ?? ?? ?? ?? 0x57 0x45 0x42 0x50`
  (`"RIFF<size>WEBP"`; 12-byte check).

The check returns `false` (rejection) for any other MIME or
mismatched bytes.

### Why dynamic-import-free static imports

Considered + rejected: dynamic `await import("@/lib/storage")`
inside `updateProfileImage` to isolate mocks per-test. Rejected
because:
1. vitest's `vi.mock("@/lib/storage")` works equivalently with
   static imports (both go through Vite's module resolver).
2. Dynamic imports add runtime cost (cache lookup on every call)
   for no production benefit.
3. Static imports surface mock-misconfiguration at test-time more
   cleanly (failing fast instead of silent passthrough).

### Try/finally on delete-on-replace (delete failure ŌåÆ orphan tolerated)

Per ADR-0017 D-B: "Old Blob URL ŌåÆ parse storage key ŌåÆ `del()` ŌåÆ
only then write new URL. Wrapped in try/finally so failure to
delete old blob doesn't block writing new URL."

Implementation:
```ts
if (oldUrl) {
  try {
    await delAvatar(oldUrl);
  } catch {
    // Orphan tolerated; abandoned-blob cleanup script (Class B
    // follow-on per D-H) handles reconciliation.
  }
}
```

The user-facing flow (new image successfully shows up) takes
precedence over operational tidiness (old image cleaned up).
Abandoned blobs are tolerated until the cleanup script lands
(Phase-16 Class B follow-on per ADR-0017 D-H).

### Why `validateImageOverride` exists separately from `updateProfileImage`

Two reasons:

1. **Defense-in-depth**: server-action receives a FormData with
   `file` (binary upload) OR potentially `imageOverride` (URL
   paste ŌĆö Phase 17+ Option 2 path). If/when URL paste lands as
   a separate affordance, `validateImageOverride` is the
   appropriate gate; `updateProfileImage` handles file uploads
   only. Pre-shipping the URL validator keeps the surface
   forward-compat without changing the helper count.
2. **Tested independently** ŌĆö validation logic is pure; testing
   against bad/good URLs doesn't require mocking the storage layer.

Phase 16 ships `updateProfileImage` (file-upload path) as the
primary affordance; `validateImageOverride` is available but not
yet exercised by a server action. Phase 17+ may wire it into a
URL-paste affordance per ADR-0017 Option 2 extension.

### `imageOverride: null` defaults in existing fake-row builders

Five places in `lib/users/index.test.ts` had fake-row builders
asserting against `getPublicProfileByHandle`'s output. Each now
includes `imageOverride: null`:
1. "returns the row when DB matches the handle exactly".
2. "preserves canonical case when URL case differs".
3. "returns null when the matched row has a NULL githubLogin".
4. PublicProfile shape extension "includes displayName + bio".
5. PublicProfile shape extension "preserves null fields".

Without this update, vitest's `toEqual` could mismatch on the
expanded `PublicProfile` shape (the helper returns the new
`imageOverride` field; test fixtures need to match).
