## THINK

### Why ship the framework + ADR together this unit (vs split)

The framework-only-ADR is a NEW architectural pattern for this project
(every prior ADR pinned a concrete provider in the same ADR). The ADR
text + the framework code are mutually-referential: the ADR cites the
`ContentModerator` interface shape; the interface comments cite the
ADR. Splitting them across two units creates a stale-citation window
between them. Phase 22 Unit 22.1 + Phase 30 Unit 30.1 established the
precedent: ADR + first-code-shipping-the-ADR-decisions land together
when the code is the realization of the ADR's interface choice.

### ADR-0024 framework shape (anchored from Phase-35.0-prep D-3)

D-A. **Interface** = `ContentModerator` in `lib/moderation/types.ts`
with two methods: `moderateText(text, ctx)` and `moderateImage(buffer,
ctx)`. Single interface (not split text/image) per Phase-35-prep D-9
lean: parallels Resend SDK single-client + factory simplicity;
providers map their internal text-vs-image dispatch to this surface.

D-B. **`ModerationResult` shape** = discriminated union per Phase-35
TypeScript `exactOptionalPropertyTypes: true` discipline:

```ts
type ModerationResult =
  | { ok: true }
  | { ok: false; reasons: ReadonlyArray<string>; severity: "block" | "warn" };
```

Two-state severity (`block | warn`) per Phase-35-prep D-10 lean.
Numeric provider scores get mapped to this two-state by the provider
implementation; centralized two-state simplifies the caller side
(API route surfaces 422 only on `block`; UI may render warning chip
on `warn`).

D-C. **`ModerationContext` shape** = `{ surface, userIdOrEmail }`
where `surface` is a literal union over the four wired surfaces
(`"bio" | "avatar" | "rating-challenge" | "subscribe"`). The context
lets future providers tune per-surface (e.g., higher threshold for
public-bio than for subscribe-email; different policy version per
surface). The noop default ignores context; future providers may key
on it.

D-D. **Default provider** = `NoopModerator` in `lib/moderation/noop.ts`.
Both methods return `{ ok: true }` wrapped in `Promise.resolve()` for
async-shape consistency. Zero API cost. Zero latency. Zero false
positives. This is the default when `process.env.MODERATION_PROVIDER`
is unset, empty-string, or literal `"noop"`.

D-E. **Factory** = `getModerator()` in `lib/moderation/index.ts`.
Reads `process.env.MODERATION_PROVIDER`; dispatches on value; caches
singleton per-process. Throws clear `Error` on unknown value
(consistent with Phase-9 `safeAuth()` "fail loud on config error"
posture for production builds). Unknown-value case is unreachable on
default deploys (env unset → noop default); only fires when a curator
sets `MODERATION_PROVIDER` to a value the current build doesn't
recognize, which is a real misconfiguration worth surfacing loudly.

D-F. **Test-only reset hook** = `__resetModeratorForTests()` mirroring
the `lib/email/` `__resetResendClientForTests()` pattern Phase 30
established. Vitest tests need per-suite env-var control; the reset
hook clears the singleton between suites.

D-G. **No env var added this unit**. The env var
(`MODERATION_PROVIDER`) is documented in this unit's ADR + factory
docs but is NOT added to `.env.example` until Unit 35.2 (when actual
integration surfaces start reading it). This keeps Unit 35.1 pure-
framework: no `.env.example` modification = no other-unit collision.

D-H. **What this ADR does NOT decide** (carried verbatim into
ADR-0024 Negative Consequences): which provider to use; what
moderation policy text is; appeal mechanism; audit-trail retention;
bulk-rescan cadence. All Phase-36+ + curator-track.

### Considered options for ADR-0024

1. **Framework-only with `NoopModerator` default** (chosen). Ship
   interface + noop + factory; defer provider commitment.
2. **Ship with committed OpenAI Moderation API provider**. Single-
   provider lock-in like ADR-0021 Resend; concrete enforcement
   ships Day 1.
3. **Defer entirely to Phase 36+** when usage signals accumulate.
   Carries the current posture forward.
4. **Provider abstraction + ship regex-wordlist as default**. Mid-
   ground: real enforcement without API cost.

### Pros / Cons table (will be in ADR proper)

| Option | Pros | Cons |
|---|---|---|
| **1. Framework-only + noop default** (chosen) | Closes Q68 expansion architecturally; zero cost/latency/FP; multi-surface wiring lands once; provider swap is single env+file change; demand-signal-first concern dissolved | No real enforcement until provider commits; future provider may want richer context shape than D-C |
| 2. Ship with OpenAI Moderation | Real Day-1 enforcement; concrete policy enforced | Vendor lock-in; API cost per call; CI requires API key; OpenAI's policy may not match project's |
| 3. Defer entirely | No risk; matches existing posture | Q68 expansion accumulates phases without closure; future multi-surface refactor cost grows |
| 4. Regex-wordlist default | Real enforcement at zero API cost; deterministic | Wordlist authoring is curator-track; false positives on legitimate technical terms; needs `lib/moderation/wordlist.ts` + initial list authored |

### Why Option 1 wins

- Dissolves the demand-signal-first concern that deferred Q68 12+
  phases. No-op cost = no pre-pay for unobserved abuse.
- Multi-surface wiring is a one-time cost (Unit 35.2); future
  provider commitment costs only the new provider file + env-var
  flip.
- ADR-0024's framework-only-without-provider pattern becomes the
  playbook for Q66 / Q70 / similar future framework-first decisions.
- Establishes the project's first architectural pattern that
  separates "what's the shape" from "which provider implements it".
