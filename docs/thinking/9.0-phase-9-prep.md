### D-3. Auth provider choice (deferred to Unit 9.1 ADR-0012)

Per §5.8: Clerk or NextAuth.js v5; GitHub OAuth strongly preferred. Three viable approaches:

1. **Clerk** ([clerk.com](https://clerk.com)) — SaaS auth provider; drop-in UI components (`<SignIn />`, `<UserButton />`); managed user store; multi-provider OAuth out-of-box; pricing: free tier 10k MAU, paid tiers thereafter. Pros: fast wiring; UI included; multi-provider; strong DX. Cons: vendor lock-in; user data lives in Clerk's DB (separate from our DB); pricing scales with users; less control.
2. **NextAuth.js v5** (a.k.a. Auth.js) — open-source auth library; provider abstraction (GitHub, Google, etc.); session managed by us (Drizzle adapter exists); free; well-maintained. Pros: free; we own the data; integrates cleanly with our Drizzle DB; widely adopted; good docs. Cons: more wiring; we own session storage; no pre-built UI (we build sign-in components).
3. **GitHub OAuth direct** — implement OAuth 2.0 flow ourselves against GitHub's `/login/oauth/authorize` + `/login/oauth/access_token` endpoints; store sessions in our DB. Pros: minimal abstraction; community-signal-rich (everyone has GitHub); zero third-party dependency beyond GitHub itself. Cons: we own EVERYTHING (OAuth state, CSRF tokens, session refresh, sign-out cleanup); higher per-unit complexity; multi-provider expansion requires re-implementation.

**Lean** (subject to ADR-0012 in Unit 9.1): **NextAuth.js v5** with **GitHub OAuth** as the only provider initially. Reasoning:
- §5.8 lists NextAuth.js v5 as a candidate.
- "GitHub OAuth strongly preferred for community signal" — every curator + contributor has a GitHub account (per the existing Phase-6 Discussions integration; per `/contributing` v1.1 §1).
- We own the user data (lives in our Drizzle DB; cleaner separation than Clerk's SaaS).
- Free; no SaaS pricing curve.
- Drizzle adapter is well-supported (mature for v5).
- Multi-provider expansion later (Google, GitLab) is incremental, not a rewrite.

ADR-0012 will pin the choice with Pros/Cons per the README convention.

### D-4. DB choice (deferred to Unit 9.2 ADR-0013)

Per §5.7: SQLite via Turso/libSQL OR Postgres via Neon, plus Drizzle ORM. Three viable approaches:

1. **Turso/libSQL** (SQLite-as-a-service; Vercel-native edge replica) — Pros: edge-native (low latency from Vercel edge runtime); SQLite ergonomics (single-file local dev); free tier generous; Drizzle libsql adapter mature. Cons: SQLite-feature-subset; row count limits on free tier; less full-text-search support than Postgres.
2. **Neon** (Postgres serverless; branching) — Pros: Postgres feature-complete (JSON, GIN, FTS, etc.); branching for preview deploys; free tier 0.5 GB storage + 191 compute-hours; Drizzle pg adapter mature. Cons: cold-start latency on serverless; pricing curve at scale; more setup complexity.
3. **Vercel Postgres** (Vercel-managed Neon) — Pros: zero-config from Vercel UI; same Postgres as Neon. Cons: vendor coupling; pricing higher than Neon-direct.

**Lean** (subject to ADR-0013 in Unit 9.2): **Turso/libSQL** with Drizzle. Reasoning:
- SQLite ergonomics match the project's local-first / file-first ethos (`content/` is YAML files; pairing with SQLite-flat-file model is consistent).
- Edge-native replicas reduce latency for the read+write API endpoints we're shipping in this phase.
- The auth thread doesn't NEED Postgres features today (JSON column for session metadata is the only "Postgres-y" thing, and SQLite has JSON1 extension).
- Free tier amply covers project scale (10 problems / 30 papers / 126 authors — even if user count reaches 10k, fits comfortably).
- Drizzle libsql adapter is well-maintained.

ADR-0013 will pin the choice with Pros/Cons. Note: §5.7 says "**or** Postgres via Neon"; the choice between the two is the ADR-0013 scope.

### D-6. Session shape (deferred to Unit 9.4)

Two viable session-storage strategies:

1. **Database-backed sessions** (NextAuth.js v5 default with Drizzle adapter) — session record in DB; cookie holds the session ID; server reads DB on each request. Pros: revocable (delete the row); auditable (session history visible). Cons: extra DB read per request.
2. **JWT sessions** — session encoded in a signed JWT cookie; no DB read needed. Pros: stateless; fast. Cons: revocation requires a JWT-blocklist or short expiry; harder to audit; rotation complexity.

**Lean** (subject to Unit 9.4): **Database-backed sessions**. Reasoning:
- We have a DB anyway (Drizzle); the extra read is cheap and edge-cached.
- Auditability matters for the curator workflow (can answer "who's signed in right now").
- Revocation is straightforward (delete the row).
- NextAuth.js v5's Drizzle adapter default.
