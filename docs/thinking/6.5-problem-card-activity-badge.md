# 6.5 — `components/problems-index/` activity badge (closes Unit 6.0 D-5 read-side surfacing)

Fourth code unit of Phase 6. Surfaces Discussion activity counts on problem cards (the listing surface) + upgrades the "Discuss this problem →" link on the problem detail page (Unit 6.3) with a parenthesized count. Consumes Unit 6.2's `getDiscussionByPath` at SSG build time via a new env-safe wrapper.

## THINK

- **Unit 6.0 D-10** (decided-as-lean): "Discussion-card SSR vs CSR — discussion counts can stale within a single SSG build window. Acceptable for v1; revisit if the digest feels stale." Pins SSR (build-time) population — matches the existing problem-card data shape (everything is build-time).
- **Unit 6.2** ships `lib/discussions/github-graphql.ts` with `getDiscussionByPath(pathname, options)` returning `DiscussionMetadata | null`. The function calls `defaultClientFactory()` on cache miss → throws when `GITHUB_TOKEN` env is unset.
- **Unit 6.3 (`f0dbff1`) commitment**: "Unit 6.5 will UPGRADE this link to include the count badge later." The "Discuss this problem →" link in `app/problems/[slug]/page.tsx` is the upgrade target.

### Scope (this unit)

Six file changes + one helper:

1. **`lib/discussions/github-graphql.ts`** — add `tryGetDiscussionByPath(pathname, options)` env-safe wrapper. Catches errors (missing `GITHUB_TOKEN`, network failures, GraphQL errors) and returns `null`. Lets SSG builds proceed without crashing when the operational env is missing.
2. **`lib/discussions/github-graphql.test.ts`** — add 3-4 tests for `tryGetDiscussionByPath`: returns the underlying metadata on success; returns null when env is unset (factoring through `defaultClientFactory`); returns null when the inner call throws for any reason.
3. **`lib/content/load-problems-index.ts`** — extend `IndexedProblem` with three new optional fields:
   - `discussionCount?: number` — set when a discussion thread exists with comments.
   - `discussionUrl?: string` — set when a thread exists; the GitHub Discussions URL.
   - `discussionLastActivityAt?: string` — ISO 8601 timestamp of most recent activity.
   Keep `getIndexedProblems()` synchronous and unchanged (other consumers — `build-domain-map.ts`, `load-author.ts`, `load-institution.ts` — would otherwise need cascading async refactors). Add a new `getIndexedProblemsWithDiscussions()` async function that wraps the sync one + fans out `tryGetDiscussionByPath(`/problems/${slug}/talk`)` calls via `Promise.all`, populating the three new fields when the result is non-null AND `commentCount > 0` (skip 0-comment threads — empty signal isn't worth surfacing).
4. **`app/problems/page.tsx`** — convert `ProblemsPage` from synchronous to `async function`; `await getIndexedProblemsWithDiscussions()`. Other call sites of `getIndexedProblems()` stay sync + unchanged.
5. **`components/problems-index/index.tsx`** — render the badge in the card. Position: inline next to the existing composite/last-curated metadata row. Format: "💬 N" (or "N comments" — TBD). Wraps in a `<Link>` to the talk page.
6. **`app/problems/[slug]/page.tsx`** — upgrade the "Discuss this problem →" link with the count when available. Call `tryGetDiscussionByPath` from the page (already `async`). Format: "Discuss this problem (N) →" when count > 0; fall back to "Discuss this problem →" otherwise.

### Scope NOT in this unit

- **Digest-pipeline extension** — Unit 6.6.
- **`/talk` Lighthouse enrolment** — Unit 6.7.
- **Real-time count freshness within a deploy** — Unit 6.0 D-10 explicitly accepts per-build staleness.

### Constraints

- **§5.5 perf budget** — First Load JS shared chunk = 103 kB held since Phase 1. This unit adds string data to the prop and a tiny rendering branch in the client component; no new client deps. Should not move the shared chunk.
- **§14.2 testing** — `tryGetDiscussionByPath` gets vitest tests; the client component's render-when-data-present branch is implicitly tested by the build itself (the prop shape changes; TS catches mismatches; the existing component tests stay valid). No new component test file added — testing the branch would require either Testing Library + jsdom (not in the project) or substantially more setup than the value justifies.
- **Build-time env behaviour**:
  - **`GITHUB_TOKEN` set + cache cold**: 10 GraphQL calls × 1 per problem = 10 build-time API calls (rate limit: 5000 points/hour; comfortable margin). Cache populated.
  - **`GITHUB_TOKEN` set + cache warm**: 0 API calls; serves from `.github-cache/`.
  - **`GITHUB_TOKEN` unset + cache cold**: 10 silent fall-throughs to null; build succeeds; no badges. CI without the token still produces a working site.
  - **`GITHUB_TOKEN` unset + cache warm**: serves from cache; badges appear as last cached. Acceptable.
- **`exactOptionalPropertyTypes`** — new fields are optional and use the "assign-only-when-defined" pattern in the loader.
- **Parallel-curator awareness**: the new fields are additive on `IndexedProblem`; no breaking shape change. Existing consumers (the index component) ignore the new fields if not surfacing them.

### Alternatives considered

- **Keep `getIndexedProblems` synchronous; fetch discussions in `app/problems/page.tsx` directly**. Rejected — splits the data-assembly across two files; the loader is the natural single source of truth for `IndexedProblem`'s shape.
- **Fetch discussions client-side via a `useEffect`** in the index component. Rejected — couples the discussion fetch to the visitor's network (every page view triggers 10 GraphQL calls × N visitors) instead of the SSG cache; breaks ADR-0004's file-first contract (would need a runtime API endpoint to proxy the token).
- **Show "0 comments" as a real badge** when a thread exists but has no comments. Rejected — empty signal isn't worth surfacing; "no badge" cleanly indicates "no real activity yet."
- **Show the badge even when the fetch fails / env unset**. Rejected — silently rendering "0 comments" or "—" everywhere would mislead the curator into thinking discussions are functioning. The absence of badges IS the signal.
- **Add a `tryGetDiscussionByPath` only as a private helper in the loader**. Rejected — Unit 6.6 (digest extension) will want the same env-safe shape; exporting from `lib/discussions/` is the right home.
- **Pre-warm `.github-cache/` from a CI job**. Rejected — out of scope; CI runs the production build with the token automatically.
- **Render the badge as an icon** (e.g., a chat bubble from lucide-react). Rejected for v1 — text is screen-reader-clean; icon-only would need an aria-label. Text + parenthetical count matches the existing metadata-row register on the cards (`composite N.NN (XX%)`).

### Tradeoffs / defensible defaults flagged

1. **`tryGetDiscussionByPath` returns null on any error** — coarse but defensible. A future unit could distinguish "env unset" from "rate limit" from "network failure" if curator debugging needs it. For v1, silent fallback is the right shape.
2. **`getIndexedProblems()` becomes async** — minor breaking change to the loader signature. Only one call site (`app/problems/page.tsx`); easy to migrate. No external consumers (this is project-internal).
3. **`Promise.all` for fan-out** — 10 problems in parallel hit the GraphQL API simultaneously. GitHub's rate limit (5000 points/hour) accommodates this comfortably. If problem count grows past ~100, a `Promise.allSettled` + chunked-concurrency pattern would be more disciplined.
4. **Cache hit rates dominate cost** — after the first build of the day, subsequent builds serve from `.github-cache/`. Per-build cost: 0–10 API calls.
5. **The badge surfaces commit-time activity** — visitors see counts as-of the most recent deploy. Acceptable per D-10; visitors who want live counts click into the talk page (iframe loads live from GitHub).
6. **Detail-page link upgrade is "additive"**: existing markup unchanged except the link text adds `(N)` when known. If the count is unknown (env unset, no thread), the link text stays as it was in Unit 6.3.
7. **No new dependency**: this unit ships purely on Phase-6-already-installed deps (`@octokit/graphql` from Unit 6.2; no `@giscus/react` needed here).

### Refactor notes (debug → fix)

First attempt converted `getIndexedProblems()` to async directly. Typecheck immediately surfaced 3 other consumers (`build-domain-map.ts`, `load-author.ts`, `load-institution.ts`) that would have needed cascading async refactors — those sites only care about non-discussion fields, so the cascade was unjustified overhead. Refactored to a **two-function shape**: sync `getIndexedProblems()` stays as the canonical loader; new async `getIndexedProblemsWithDiscussions()` wraps it + adds the fan-out. Only `app/problems/page.tsx` uses the async wrapper; everything else stays sync. Cleaner blast radius; same end-state.
