### D-3. `rating_challenges` table shape

The table stores user-submitted challenges to existing rating dimensions.
Per ADR-0013 D-F (USER-STATE only) + Q56-precedent (no FK on file-system
slug references) + ADR-0004 (file-first for content):

```ts
export const ratingChallenges = sqliteTable("ratingChallenge", {
  id: text("id")
    .primaryKey()
    .$defaultFn(() => crypto.randomUUID()),
  userId: text("userId")
    .notNull()
    .references(() => users.id, { onDelete: "cascade" }),
  problemSlug: text("problemSlug").notNull(),
  dimension: text("dimension").notNull(),       // enum-string at app level
  proposedValue: text("proposedValue").notNull(), // dimension-shape varies; TEXT
  rationale: text("rationale").notNull(),
  status: text("status").notNull().default("submitted"),
  createdAt: integer("createdAt", { mode: "timestamp_ms" })
    .notNull()
    .default(sql`(unixepoch() * 1000)`),
});
```

**Lean** (subject to Unit 11.1):
- **UUID PK** (matches `users.id` UUID strategy). Drizzle's
  `$defaultFn(() => crypto.randomUUID())` generates per row.
- **userId FK** with cascade (matches `watchlist` precedent).
- **problemSlug** plain text, no FK (matches Q56 lean +
  watchlist precedent).
- **dimension** as text (not Drizzle enum since SQLite lacks native
  enum); app-level validation against `["difficulty", "saturation",
  "urgency", "value", "industry_call"]`.
- **proposedValue** as text (TEXT column; app-level interpretation per
  dimension — letter grade for difficulty / 0-100 for saturation /
  0-5 for stars-based). Alternative (5 typed columns: proposed_grade,
  proposed_saturation, proposed_stars_urgency, ...) rejected as
  sparse + brittle + bloating the row size.
- **rationale** required TEXT; app-level length validation
  (min 50 chars; max 2000 chars).
- **status** as text (default `"submitted"`). Phase 11 ships ONLY
  this value; future statuses (`"under_review"`, `"accepted"`,
  `"rejected"`, `"withdrawn"`) land in Phase 12+ with the curator
  review pipeline.
- **createdAt** timestamp_ms (matches `users.createdAt` +
  `watchlist.createdAt` precedent).
- **NO curator-review columns** (`reviewedAt` / `reviewerId` /
  `reviewNotes` / `acceptedActionId`). Per ADR-0005's
  immutability-and-explicit-evolution ethos: don't add speculative
  columns; future migrations add them when the surface lands. Cleaner
  schema for Phase 11; one extra migration cost in Phase 12+ is
  acceptable.
- **NO composite primary key** (single UUID PK is the right shape —
  per row uniqueness needs no compound semantic). Unlike watchlist
  (where `(userId, problemSlug)` is the natural unique constraint),
  a user can submit MULTIPLE challenges per problem (one per
  dimension OR multiple per dimension over time as their thinking
  evolves).
- **Indexes**: per-user lookup via `(userId, createdAt DESC)`
  benefits from a composite index; Phase 11 won't add one explicitly
  (queries are small; LIMIT 50; SQLite plans the query fine without
  an index for Phase-11 scale). Phase 12+ when challenge count grows.

### D-4. Status enum values

Phase 11 ships ONLY `"submitted"` as the default status. Future
statuses defer to Phase 12+ curator review pipeline:

- `"submitted"` — Phase 11 default; user submitted the challenge.
- `"under_review"` — Phase 12+; curator has begun reviewing.
- `"accepted"` — Phase 12+; curator accepted; a new rating action

- `"rejected"` — Phase 12+; curator rejected with notes.
- `"withdrawn"` — Phase 12+; user withdrew their challenge.

App-level validation in Phase 11: INSERT statements always set
`status = "submitted"`; no SELECT logic reads other values. Phase 12+
adds the validation table + state machine.

### D-5. Dimension enum values

5 values matching `RatingActionSchema.dimensions` keys:

- `"difficulty"` — letter grade (S/A/B/C/D/E)
- `"saturation"` — number 0-100 (or "N/A" per ADR-0006)
- `"urgency"` — integer 0-5 stars
- `"value"` — integer 0-5 stars
- `"industry_call"` — integer 0-5 stars

App-level validation: enum check on the dimension field; per-dimension
`proposedValue` format check (regex / numeric range) on the rationale's
sibling field.

### D-6. `proposedValue` typing (single TEXT column)

Stored as TEXT; app-level interpretation per dimension:

| Dimension       | Stored format examples                  | Display format       |
| --------------- | --------------------------------------- | -------------------- |
| `difficulty`    | `"S"`, `"A"`, `"B"`, `"C"`, `"D"`, `"E"` | letter pill          |
| `saturation`    | `"42"`, `"N/A"`                          | percentage / N/A pill |
| `urgency`       | `"3"`, `"5"`                             | stars                |
| `value`         | `"4"`                                    | stars                |
| `industry_call` | `"2"`                                    | stars                |

Alternative considered: 5 typed columns (proposed_grade text /
proposed_saturation int / proposed_stars_urgency int / etc.). Rejected
as sparse + brittle + bloating the row size; only 1 column per row is
populated.

Alternative considered: JSON column (`proposedValue` as JSON blob with
`{ type, value }` shape). Rejected because SQLite lacks JSONB; would
require app-level parse/serialize on every read; no win over plain TEXT.

### D-7. Rationale validation

- **Required field** (`NOT NULL` at SQL layer).
- **Min length**: 50 chars (app-level validation; SQL CHECK constraint
  redundant since the app validates before INSERT).
- **Max length**: 2000 chars (app-level; SQL allows up to TEXT max
  but UX-side cap prevents abuse).
- **No HTML / Markdown rendering** in Phase 11 — display as plain text
  with line-break preservation. Phase 12+ may add light formatting if
  curator-review UX needs richer rationale.

### D-8. Submission UX (problem-detail-page placement)

Two viable placements:

1. **Inline collapsible** on `/[locale]/problems/[slug]` under section
   #8 ("Recent rating actions") — `<details>` element revealing the
   form. User has problem context; form is in-line.
2. **Separate route** `/[locale]/problems/[slug]/challenge` —
   dedicated page. More architectural; deeper URL; SSG-able shell.

**Lean** (subject to Unit 11.3): **Inline collapsible**. Reasoning:
- Smaller surface (1 file edit; no new route).
- User has context (problem-detail-page rating radar is right above).
- Phase 11 scope discipline (~7 units total; new route would inflate
  to ~8).
- Submission is rare per-user (not every visit results in a challenge);
  inline collapsible is the right "default-closed; opt-in to expand"
  affordance.
