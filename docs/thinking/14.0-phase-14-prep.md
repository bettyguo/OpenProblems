### D-3. Handle-routing strategy

`users.githubLogin` is the project's join key per ADR-0012 D-E. Phase
14's public profile route MUST resolve a handle to a `users` row.
Routing shape:

```
/[locale]/u/[handle]/page.tsx           # shell page (this phase)
/[locale]/u/[handle]/challenges/page.tsx # challenges sub-route (Q58 lean #3)
```

**Resolution**:
- `[handle]` segment matches against `users.githubLogin` (unique
  column per ADR-0012 D-E + Unit 9.3 schema).
- Match → render profile shell.
- No-match → `notFound()` from `next/navigation` (Next.js renders the
  default 404 page; respects locale-prefixed URL via App Router
  conventions).
- **Case-insensitivity**: GitHub logins are case-insensitive on
  registration but case-preserved on display; `users.githubLogin`
  stores the case from the GitHub profile. Phase 14 lookup is
  case-insensitive via `LOWER(githubLogin) = LOWER(?)`; URL canonical
  case TBD (D-13).
- **No fallback to email / name search**: handle is the only stable
  identifier; ADR-0015 will pin this.

**Helper signature** (`lib/users/index.ts` new):

```ts
export interface PublicProfile {
  userId: string;
  githubLogin: string;       // canonical case from users row
  name: string | null;       // from users.name (GitHub-provided)
  image: string | null;      // GitHub avatar URL from users.image
  createdAt: Date;           // join date (signal of seniority)
}

export async function getPublicProfileByHandle(
  handle: string,
): Promise<PublicProfile | null>;
```

Implementation: single SELECT against `users` by lowercased
`githubLogin`. Returns `null` on no-match (caller invokes `notFound()`).

### D-8. Curator-of-record badge

Per D-4: scan `content/problems/*/problem.yaml#editorial.primary_curator`
for matches against the handle. Implementation route:

- `#site/content` Velite output exposes the `problems` array at build
  time with all problem.yaml fields parsed + validated.
- `getCuratorOfRecordSlugs(handle)` filters `problems.filter(p =>
  p.editorial.primary_curator === handle).map(p => p.slug)`.
- At build time `problems` is static (Velite-built); the function is
  O(n_problems) — 10 problems today; cheap enough to call on every
  profile request.
- Match → render badge: small inline pill labeled "Curator of N
  problem(s)" with hover-tooltip listing slugs (or expansion link to
  `/u/{handle}/curated` in Phase 15+).
- No-match → omit badge entirely.

**Edge case**: `editorial.primary_curator` is case-sensitive in the
YAML schema (raw string). `users.githubLogin` is also case-preserved
(GitHub returns the canonical case). Phase 14 lean: case-sensitive
comparison; curators using a different case in YAML vs GitHub profile
get a no-match (rare but possible; ADR-0015 anticipates). Fix is a
single-line case-insensitive comparison if surfaced.
