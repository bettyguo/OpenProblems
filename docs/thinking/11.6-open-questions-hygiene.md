### Q57 — Curator review pipeline shape

The Phase-12+ design question with the largest architectural surface
of the Phase-11 follow-ons. Pinning the lean here keeps the design
explicit through the Phase 11 acceptance gate.

**Question**: how does a curator review a rating-challenge submission?
What state transitions happen, who's authorized to make them, and how
does the project's existing file-first content posture interact with
the DB-stored challenge?

**Surfaces to design** (Phase 12+):

1. **Status transitions** — Phase 11 ships `status` TEXT default
   `"submitted"`. Phase 12 adds: `submitted → under_review` (curator
   begins review); `under_review → accepted` (challenge has merit; a
   new rating-action YAML file lands); `under_review → rejected`
   (challenge lacks merit; curator notes why); `submitted → withdrawn`
   (user-initiated; out-of-band from curator review).
2. **Curator-review columns** (Unit 11.0 D-3 deferral) — migration
   `0003_challenge_review_columns` adds nullable columns:
   `reviewedAt`, `reviewerId` (FK to `users.id` with `SET NULL` on
   delete), `reviewNotes` (text), `acceptedActionId` (text; points at
   the resulting rating-action YAML file's filename or slug; allows
   joining the DB state to the file-system rating-action that the
   challenge produced).
3. **Curator UI** — `/[locale]/curator/challenges` (admin-protected
   route; middleware-based protection lift becomes justified at 2
   protected page routes — Phase-9 Class B item 12). List view with
   filters (status / dimension / problem); detail view for individual
   challenge with `<form>` actions: "Mark under review" / "Accept &
   commit rating-action YAML" / "Reject with notes" / "Mark withdrawn".
4. **Authorization model** — who is a "curator"? Today the
   curator-of-record concept lives in `editorial.primary_curator` on
   each problem's YAML; the user with `githubLogin === primary_curator`
   has authority over that problem's ratings. For the curator UI: the
   simplest first cut is "any signed-in user whose `githubLogin` is in
   a curator allowlist" (admin-track config; Phase 12+ Q-surface);
   gradual expansion to per-problem ACLs as the project scales.
5. **File-first preservation** — accepting a challenge MUST produce a
   new rating-action YAML file under
   `content/problems/<slug>/ratings/<date>-<slug>.yaml` (file-first
   per ADR-0004; immutable rating-action chain per ADR-0005). The
   curator UI generates the YAML body from the challenge fields +
   curator's review notes; commits via git (Phase 12+: integrated
   commit; Phase 13+: maybe a draft-PR workflow); DB row's
   `acceptedActionId` then points at the new file. The DB tracks the
   submission event + review state; the file system tracks the
   editorial decision.

**Lean** (working position; not blocking Phase 11):
- Phase 12 ships the curator-side data layer + columns migration
  (`0003_challenge_review_columns`) + curator dashboard route
  (`/[locale]/curator/challenges`) with **status transitions limited
  to `submitted → under_review → accepted | rejected`**. Withdrawal
  is a separate sub-phase (Class B item 5 from Unit 11.5).
- Phase 13+ ships the file-system YAML commit integration (the
  curator types in the UI; the server-action emits a YAML file +
  optionally creates a PR).
- Authorization: allowlist via env var `LOP_CURATOR_LOGINS` (CSV of
  GitHub logins) for Phase 12; per-problem ACL Phase 13+.

This lean is documented in `OPEN_QUESTIONS.md` Q57 (added in this
unit). Not promoted to `decided-as-lean` yet because the design needs
ADR-level validation before commit; Q57 stays `open` until either an
ADR-0014 lands pinning the design OR Phase 12's first curator-review
unit's THINK doc anchors it.

### Q58 — Challenge visibility to non-author users

Smaller Phase-12+ question than Q57. Today the profile-page list
shows only the submitter's own challenges; no other surface displays
challenge information.

**Question**: who should see challenges on a problem detail page?
What's the visibility policy?

**Options**:

1. **No public surface** — challenges stay submitter-only forever
   (only the curator dashboard sees them in aggregate). Cleanest
   privacy posture; loses the "transparent debate" framing §3.1's
   "ratings are revisable" implies.
2. **Counter only** — problem detail page shows
   "N active rating challenges" + (optional) "View full list" link;
   no per-challenge content. Compromise: signals activity without
   revealing rationale text.
3. **Full public list** — anyone can see all submitted challenges
   with full rationale + submitter handle. Most transparent;
   aligns with §3.1 + §8.5 (rating action log is public).
4. **Counter + lazy-load aggregated stats** — counter on problem
   detail page + a separate "rating action archive" page that
   lists challenges alongside historical rating actions.
5. **Status-gated visibility** — `submitted` challenges stay
   submitter-only; `under_review` / `accepted` / `rejected` /
   `withdrawn` become public. Aligns the visibility transition
   with the curator-review transition; clean state machine.

**Lean** (working position; not blocking Phase 11): **Option 5**
(status-gated visibility). Reasoning: until the curator engages, a
challenge is just a submission — no editorial signal. Once the
curator promotes the status to `under_review` or beyond, the
challenge becomes part of the rating debate and warrants public
view. Aligns with §8.5's "rating action log" ethos: the curator's
review IS the editorial signal that triggers publicness.

This lean is documented in Q58 (added in this unit). Stays `open`
pending the curator-review pipeline (Q57) decisions in Phase 12+;
both questions resolve together.
