### D-3. className-styling shape anticipated (Unit 65.1)

D-A. **`rehypeResolveWikilinks` plugin body evolution** (Unit
65.1):

```ts
// Before (Phase 63 ship through Phase-64 close):
newNodes.push({
  type: "element",
  tagName: "a",
  properties: { href: buildHref(slug, entityType) },
  children: [{ type: "text", value: display }],
});

// After (Phase 65 ship):
newNodes.push({
  type: "element",
  tagName: "a",
  properties: {
    className: ["wikilink"],
    href: buildHref(slug, entityType),
  },
  children: [{ type: "text", value: display }],
});
```

**HAST property convention**: HAST uses `className` (camelCase)
as the property name for class; the value is an array of
strings (per `hast-util-properties` / `property-information`
HTML class attribute convention). `rehype-stringify` renders
`className: ["wikilink"]` as `class="wikilink"` at compile
time.

**Backwards-compat preserved**: every Phase-38-through-Phase-64
wikilink shape continues to emit the same `<a>` tag + same
`href` value; the ONLY addition is the `className` property.
External markdown links (non-wikilink) continue to emit
unclassed `<a>` tags via `remark-rehype`'s standard
transformation. Surfaces that don't enable wikilinks
(`MARKDOWN_EXTENSIONS=default` and the `DefaultExtensionRegistry`)
continue to emit unclassed `<a>` tags throughout.

D-B. **Sanitize bypass mechanics** (Unit 65.1; verified
against `lib/markdown/index.ts:213-215`):

The Phase-37 framework pipeline order:

```
.use(rehypeSanitize, schema)        ← sanitize FIRST
.use(rehypeStripUnsafeHrefs)        ← strip-unsafe-hrefs
.use(rehypeExtensions)               ← registry's rehypePlugins (wikilinks HERE)
.use(rehypeStringify);
```

Per APPEND-D-D plugin-order-after-default discipline, the
registry's `rehypePlugins` (which include `rehypeResolveWikilinks`)
run AFTER `rehype-sanitize`. The `<a class="wikilink">` element
the wikilinks plugin splices into the post-sanitize tree enters
under explicit framework intent — sanitize has already run and
cannot strip the class attribute. **No schemaOverrides change
required.**

This is the **plugin-only emit pattern** — the first plugin-
body realization to exploit the post-sanitize bypass for
attribute emission. Prior plugin-body realizations all emit
attributes already in the base schema allow-list (`href` on
`<a>` for wikilinks/arxiv/doi/pubmed/orcid/biorxiv; text-only
children for alias display); Phase 65 is the first to emit an
attribute NOT in the base schema allow-list (`class` on `<a>`
is NOT in `bioSchema.attributes.a`, which only allows
`["href"]`).

**XSS-safety preserved**: the className value is hardcoded
`"wikilink"` (no curator input; no slug/entityType
interpolation; no opportunity for attribute injection). The
attribute value is a fixed literal, not curator-controllable.

D-C. **External markdown links UNCHANGED** (Unit 65.1):

External links via standard markdown `[text](url)` syntax
continue to emit `<a href="url">text</a>` (no className) via
`remark-rehype`'s default transformation. Only wikilinks
(resolved through `rehypeResolveWikilinks`) receive the
className. This is the desired behavior: curators can style
`a.wikilink` distinct from unclassed external `a` links.

D-D. **No factory arm change** (Unit 65.1 / Unit 65.2):

The className is emitted by ALL invocations of
`rehypeResolveWikilinks` regardless of how the plugin was
invoked. Both existing arms inherit automatically:
- `MARKDOWN_EXTENSIONS=wikilinks` (Phase 38+; bare-form emit
  with `DEFAULT_BUILD_HREF`) — emits className AND className.
- `MARKDOWN_EXTENSIONS=wikilinks-cross-entity` (Phase 63+;
  tuple-form emit with `CROSS_ENTITY_BUILD_HREF`) — emits
  className AND uses cross-entity routing.

**`MARKDOWN_EXTENSIONS` single-value arms UNCHANGED at 10**
(no new arm). Phase 65 differs from Phase 63 + Phase 64 in that
the new behavior is automatically enabled for ALL existing arms
that use wikilinks rather than gated behind a new opt-in arm.
Defensible because: (a) the className addition is purely
additive — does not change `href` semantics, does not affect
sanitize, does not require curator opt-in; (b) the className
is the FIRST step toward consistent visual differentiation of
wikilinks across all surfaces; gating it behind a new arm would
create curator-opt-in friction without architectural benefit.
