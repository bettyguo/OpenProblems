# 2.12 — Aggregate rollups

## THINK

### Inputs

- **§13 Phase 2 deliverable.** "Aggregate rollups: each author shows
  cumulative problem-impact score; each institution shows ranked
  subdomain coverage."
- **Existing surfaces.** Unit 2.3's author page renders
  `problemsTouched: Problem[]`; Unit 2.3's institution page renders
  `subdomainCoverage: SubdomainCoverage[]`. The Unit 2.3 page-level
  footer says "Cumulative author impact (composite over
  contributions[]) lands in Unit 2.12." — that's the operational
  scope.
- **`composite()` helper** ([lib/ratings/normalize.ts](../../lib/ratings/normalize.ts)).
  Takes a `RadarPoint[]` and applies §8.3 default weights. Already
  used by `getIndexedProblems()` (Unit 1.8).
- **`getIndexedProblems()`** (Unit 1.8) already pre-computes
  `IndexedProblem.composite` per problem. The aggregate rollups can
  build a `Map<slug, composite>` from that snapshot, then sum across
  problemsTouched.

### Constraints

- **§8.3 "composite never shown alone."** Phase 1 problem detail
  rendered the composite *alongside* its dimension breakdown. For
  the aggregate roll-up, the per-author cumulative-impact number IS
  composed of multiple per-problem composites — so it's a sum of
  composites, not a single one. The rule should be respected by
  always showing the list of underlying problems alongside the sum
  (already done — problemsTouched section is right next to it).
- **`cumulativeImpact?: number` semantics.** Undefined when the
  author has zero problemsTouched OR when none of the touched
  problems has a composite (no rating action). Show "—" on the page
  when undefined; show `.toFixed(2)` of the sum otherwise.
- **§13 wording matters.** "Cumulative problem-impact score" not
  "weighted average". A sum (not a mean) reflects the "more problems
  touched = more impact" framing the §13 deliverable implies. Caveats
  flagged for human review at the page level.
- **Institution rollups: same `cumulativeImpact` + add
  `problemsTouched: Problem[]` for consistency with the author
  loader.** The institution loader currently surfaces
  `subdomainCoverage` (paperCount per subdomain); the missing piece
  is the explicit set of problems an institution's papers contribute
  to. Same shape as `LoadedAuthor.problemsTouched`.
- **No content changes, no new pages.** Unit 2.12 only touches
  `lib/content/load-author.ts`, `lib/content/load-institution.ts`,
  `app/authors/[slug]/page.tsx`, `app/institutions/[slug]/page.tsx`,
  plus tests. Pure derivation over existing data.

### Implementation sketch

1. Build a `Map<problem_slug, composite>` once at module load using
   `getIndexedProblems()`. Both loaders consume this map.
2. `LoadedAuthor.cumulativeImpact?: number` — sum of
   `problemComposite.get(slug)` over `problemsTouched`, undefined
   when zero contributions resolve to a composite.
3. `LoadedInstitution.cumulativeImpact?: number` + 
   `LoadedInstitution.problemsTouched: Problem[]` — same shape as
   author.
4. Page rendering:
   - `/authors/[slug]`: replace the Unit-2.3 placeholder line with
     "Cumulative impact: X.YZ across N problems". Drop placeholder
     when undefined.
   - `/institutions/[slug]`: add a parallel "Cumulative impact" line
     above the subdomain-coverage section. The existing subdomain
     coverage block already exists per Unit 2.3.

### Alternatives considered

- **Use the §8.3 composite as a per-paper score directly.** The
  composite is a *problem* score, not a paper score. Aggregating
  paper-by-paper would double-count when one author has multiple
  papers contributing to the same problem. Use the deduped
  problemsTouched set instead.
- **Weighted sum where each problem's composite is weighted by
  count of author's contributions to it.** Possible but adds
  curatorial complexity; defer until a real use case requests it.
  The simple sum-of-deduped-composites is honest about its
  derivation.
- **Add the cumulative-impact number to the authors index.** No
  authors index page exists yet — that's a Phase-3 surface. Skip.

### Edge cases

- **Author with empty `papers[]`.** `problemsTouched = []`,
  `cumulativeImpact = undefined`. Page renders an italic placeholder.
- **Problem with no rating action.** `problemComposite.get(slug)`
  returns undefined; that problem is silently excluded from the sum.
  Counted in problemsTouched.length so the "N problems" denominator
  reflects total touches.
- **Institution touching the same problem via multiple papers.**
  Deduped via Set; counted once in the sum.

### Tradeoffs / defensible defaults flagged

1. **Sum, not mean.** §13 says "cumulative" — sum is the right
   reading. A weighted mean masks volume.
2. **No display of the per-problem composite contributions on the
   author page yet.** That's a hover/tooltip-grade detail; defer
   unless requested. The page already lists problemsTouched as
   linked pills; clicking through to the problem reveals composite.
3. **No bundle-size analyzer pass.** Both loaders gain ~15 lines;
   the page renders gain ~5 lines each. No new dependencies.
