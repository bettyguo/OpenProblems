### Scope deferred to future units

- **Drag interactivity** (Unit 4.0 D-6 listed it). Drag requires `"use client"` + live d3-force simulation on the client. Deferred to a Unit 4.2a follow-on or Phase-5 enhancement. Click navigation (via `<a>` wrapping each node) and hover tooltips (via native `<title>`) ship in this unit — they don't need client JS.
- **Filter chips** (Unit 4.0 D-6). Lives at the page layer per Unit 4.0's per-unit breakdown — Unit 4.3 (`/domains`) and Unit 4.4 (`/`) own the chips. The viz exposes a `dimmedIds?: Set<string>` prop that the page can pass to highlight a subset; chip-state-management is page-owned.
- **Subdomain expand/collapse** (Unit 4.0 D-3). Collapse-on-click requires client state. Static rendering of all 3 hierarchy levels at once is the Phase-4-light call: ~30 nodes total at current content scale (5 domains + ~12 subdomains + 10 problems) — within the readability envelope of a single SVG without collapse.
- **Recompose-weight propagation** (Unit 4.0 D-4 + Q36). DomainMap consumes a static composite value from props; if the consumer page wants Recompose-weighted composites, the consumer recomputes upstream. No DomainMap-internal Recompose UI in Phase 4.

### Constraints

- **No `"use client"`** in any DomainMap file. Phase-3 vizes are SSR-only; this unit doesn't break that pattern.
- **`d3-force` only.** `d3-selection` ships in deps from Unit 4.1 but is not imported in 4.2 (it's a client-DOM-manipulation library; static SSR doesn't need it). Tree-shaking drops it from the client bundle.
- **Deterministic layout.** d3-force normally uses `Math.random()` for initial node positions, which would produce different output on every SSR pass. Mitigation: seed initial positions deterministically — place each node on a circle around the center, indexed by node id's position in the input array. With deterministic initial positions + deterministic forces (forceLink + forceManyBody + forceCenter are all deterministic given fixed positions), the simulation converges deterministically. No hydration mismatch surface (and Next.js SSG caches the output anyway).
- **`exactOptionalPropertyTypes: true`** per tsconfig. Optional fields use `field?: T` and consumers omit-when-undefined, never assign-`undefined`.
- **`noUncheckedIndexedAccess: true`** per tsconfig. Array access guards (`nodes[i]!` / `nodes.at(i) ?? fallback`) follow Phase-3 viz precedent.
- **Velite schemas unchanged.** No new fields on Problem / Domain / Subdomain — the data DomainMap reads is already in the existing collections.

### Alternatives considered

- **Run d3-force on the client (hydration-as-simulation)**. Rejected for Phase 4 — adds `"use client"`, hydration cost, +bundle weight; defers static-SVG benefits. Drag-as-Phase-5 lets us reconsider.
- **Pre-compute positions at build time (script in `scripts/`) and check them into the repo as JSON**. Rejected — adds a build-time script + a serialized state file the curator would never look at. Module-load-time simulation runs in <50ms for ~30 nodes; SSG caches the result.
- **Skip d3-force entirely, use a manual radial layout**. Considered. Rejected — force-directed graphs are what makes the §11 catalog item 4 different from the existing static tile grid on `/domains`. Force layout reveals cluster structure that radial layout doesn't.
- **Use ELK or Cytoscape-style hierarchical layout**. Rejected — different library, more deps, more learning curve. d3-force is already installed.
- **Render Canvas via OffscreenCanvas on the server**. Rejected per Unit 4.0 D-1.

### Tradeoffs / defensible defaults flagged

1. **No drag in 4.2.** Listed in Unit 4.0 D-6 as interaction #3. Pushing it to a follow-on lets 4.2 ship as a pure-SSR component and unblock 4.3 / 4.4. If drag turns out to be load-bearing for §11 "brushable" — re-evaluate.
2. **All 3 hierarchy levels visible at once** (no collapse). At ~30 nodes this is readable. If content scales 3× (Phase-5 ingest), revisit subdomain collapse as a client-interaction add-on.
3. **`viewBox` sized for desktop primary use case** (Lighthouse-CI runs `preset: desktop`). Responsive sizing via `width="100%"` keeps mobile usable; Q39 (small-viewport table-fallback default) is a separate decision in the consumer page units.
4. **Deterministic simulation iterations.** 300 ticks is the d3-force default for `forceSimulation(...).tick(n)`. Empirically sufficient for ~30 nodes. Documented; can tune in Unit 4.11's ADR.
5. **Color tokens reused.** 5 chart-token hues, one per domain in declaration order. If a 6th domain lands (Phase 5+), the 6th domain wraps around to hue 1 — visual collision but not a correctness issue (color is decorative per D-5).
6. **No `key` on simulation nodes**. d3-force is purely functional once positions are computed; the SVG output uses node `id` as React `key`. No re-render concerns.

### Data shape (the consumer contract)

```ts
// types.ts
export interface DomainMapNode {
  /** Stable id across the graph. Convention: `${kind}:${slug}` (e.g., "domain:deep-learning"). */
  id: string;
  kind: "domain" | "subdomain" | "problem";
  /** Display text on the node. */
  label: string;
  /** Composite rating in [1, 5]. Drives node size via radius = sqrt(composite) × k. */
  composite: number;
  /** 1–5 indexing the chart-token hue family. */
  hue: 1 | 2 | 3 | 4 | 5;
  /** Optional navigation target. Problems link to `/problems/<slug>`; domains link to `/domains/<slug>`. */
  href?: string;
  /** Parent node id (used to assign subdomain/problem hues from parent domain). */
  parent?: string;
}

export interface DomainMapLink {
  source: string; // node id
  target: string; // node id
}
```

The consumer (Unit 4.3's `/domains` page) builds these arrays from `taxonomy.yaml` + `problems.json` + the §8.3 composite formula. The viz itself is data-source-agnostic.

### Force-simulation tuning constants

Initial values (tune in 4.2 against the realized fixture; record final values in Unit 4.11 ADR-0007):

| Constant     | Value     | Rationale                                                                                  |
|--------------|-----------|--------------------------------------------------------------------------------------------|
| `linkDistance` | 60        | Edges short enough that hierarchy is visible; long enough that node circles don't overlap. |
| `chargeStrength` | -180    | Negative = repulsion; tuned to push nodes apart without flying them out of the viewport.   |
| `centerStrength` | 0.05    | Soft pull toward viewport center; prevents drift but doesn't override link forces.         |
| `nTicks`       | 300       | Default for `forceSimulation(...)`. Empirically converges for ~30 nodes.                   |
| `viewBox`      | 600 × 420 | Desktop primary; aspect ratio ≈ 1.4:1 fits the typical hub-page card width.                |

### A11y plumbing

Mirrors Phase-3 viz pattern:
- `<svg role="img" aria-label={...} aria-describedby="domainmap-desc">`.
- `<desc id="domainmap-desc">` enumerates: total node count by kind, then a per-domain summary including child counts and composite rating. Example: `"Domain map: 5 domains, 12 subdomains, 10 problems. Deep Learning (composite 3.8, 3 subdomains, 4 problems): includes Hallucination Reduction (composite 3.9), …"`.
- Each `<a>`-wrapped node group has `<title>{node.label} — composite {composite}</title>` for hover tooltip + native focus tooltip.
- Edges use `aria-hidden="true"` (they're decorative — the parent/child relationship is in the `<desc>`).
