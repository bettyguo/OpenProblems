### Scope (this unit)

A single GitHub form-based template at `.github/ISSUE_TEMPLATE/new-problem.yml`. The template's job is **not** to capture a complete YAML — it's to give a curator enough signal to draft one. Final YAML lands via a follow-on PR by the curator.

### Constraints

- **GitHub Forms YAML schema** (https://docs.github.com/en/communities/using-templates-to-encourage-useful-issues-and-pull-requests/syntax-for-issue-forms). Required top-level fields: `name`, `description`, `body[]`. Optional: `title` (auto-prefix), `labels[]`, `assignees[]`, `projects[]`. Body items have `type` ∈ {`markdown`, `input`, `textarea`, `dropdown`, `checkboxes`}.
- **`title`** auto-prefix: `[New problem] ` (matches §10's editorial-tone register; bracketed type tag for triage).
- **`labels`**: `["new-problem", "needs-triage"]`. The `new-problem` label is type-specific (used by curator workflows / filters); `needs-triage` is a generic state label.
- **No assignees / projects.** Out of scope; default behaviour is fine.
- **`required: true`** on only the load-bearing fields. Forms with too many required fields deter contribution.
- **No taxonomy-coupled dropdowns** for `domain` / `subdomain`. Taxonomy evolves; a stale dropdown is worse than freeform input. The template links to `content/taxonomy.yaml` so contributors can self-check.

### Form-field schema

7 body items: 1 markdown intro + 6 form fields.

| # | Item                                  | Type       | Required | Purpose                                                                  |
|---|---------------------------------------|------------|----------|--------------------------------------------------------------------------|
| 1 | Intro markdown                        | markdown   | n/a      | One paragraph + bullet links to MASTER_PROMPT.md + CURATION_PROMPT.md.   |
| 2 | Proposed slug                         | input      | yes      | kebab-case `[a-z0-9-]+` per `lib/schemas/_primitives.ts`.                |
| 3 | Proposed title                        | input      | yes      | 5–120 chars per `OpenProblemSchema.title`.                               |
| 4 | Domain / subdomain                    | input      | yes      | Freeform "<domain> / <subdomain>" or "<domain>" if unsure. Helper text links to `content/taxonomy.yaml`. |
| 5 | Problem statement                     | textarea   | yes      | One-paragraph statement. The §6.1 "What makes this an open problem" frame. |
| 6 | Primary contributing paper(s)         | textarea   | no       | arXiv IDs, DOIs, or URLs. Helper text mentions §15.6 primary-source rule. |
| 7 | Additional context                    | textarea   | no       | Anything else — related problems, benchmarks, status nuance, watchlist signal. |

### Alternatives considered

- **Enumerate domains in a `dropdown`.** Rejected — taxonomy evolves; stale options are worse than freeform. (When taxonomy stabilises in Phase 5+, revisit.)
- **Require `posed_year` upfront.** Rejected — non-load-bearing for triage; curator can fill from the contributing paper.
- **Require a primary contributing paper.** Considered (would enforce §15.6 primary-source rule at the form layer). Rejected as required — sometimes problems are well-known without a single canonical source. Kept as optional with a §15.6 reminder in helper text.
- **Use legacy `.md` templates with front matter.** Rejected per Unit 4.0 D-8.
- **Single combined template covering all 4 contribution types via `dropdown`.** Rejected — §13 explicitly names 4 templates; per-type forms get clearer titles + labels + triage.
- **Include a `checkboxes` "I read MASTER_PROMPT.md" gate.** Considered. Rejected — feels coercive and adds friction without enforcement. The markdown intro explicitly links the doc; reader compliance is on the contributor.

### Tradeoffs / defensible defaults flagged

1. **Freeform domain/subdomain field.** Trades input quality for taxonomy-evolution safety. Triage curator confirms or corrects.
2. **No `posed_year` field.** Saves a required field; curator infers from primary paper. If a contributor wants to declare it, the "Additional context" textarea catches it.
3. **No `tags[]` field.** Tags are post-acceptance editorial; surfacing them in the submission form invites speculation. Curator authors tags during YAML drafting.
4. **No `benchmarks[]` field.** Schema-required in `OpenProblemSchema.benchmarks`, but a new-problem submission may legitimately predate a benchmark. Curator can add benchmarks during YAML drafting; status `open` allows `benchmarks: []` if needed (the schema requires the array but not non-emptiness — verify in 4.7 implementation).
   - **Verify**: `OpenProblemSchema.benchmarks: z.array(BenchmarkSchema)` — `z.array(...)` allows `[]` by default. Confirmed.
5. **`labels` not enabled by default if labels don't exist in the repo.** GitHub silently ignores labels that don't exist. The first issue opened from this template after the labels are created will surface them; until then, no error.
