## THINK

### Schema choice = boolean column with `default true`

**`profilePublic: integer({ mode: "boolean" }).notNull().default(true)`**.
SQLite stores boolean as integer 0/1; drizzle-orm's `mode: "boolean"`
runtime-marshals between JS boolean and SQLite integer transparently.
`notNull().default(true)` backfills existing rows to `profilePublic =
true` = public-by-default = matches Phase 9-35 behavior verbatim.

The drizzle-kit-generated migration is a single clean ALTER TABLE:

```sql
ALTER TABLE `user` ADD `profilePublic` integer DEFAULT true NOT NULL;
```

No manual SQL edit needed (unlike Phase 33's manual `ON DELETE
CASCADE` add for FK; default-value column emission is reliable).
**Fourth ALTER migration adding a column to `users`** and the first
since Phase 16's `0005_user_image_override` (intervening migrations
were `subscriber` table extensions).

### Filter shape = single AND-clause on existing WHERE

**`getPublicProfileByHandle`** was the only existing helper that
needed updating. Added `profilePublic` to the SELECT list + a
sentinel check `if (!row || !row.githubLogin || !row.profilePublic)
return null` in the post-fetch null-guard.

Why the post-fetch sentinel rather than a SQL `AND profilePublic =
1` WHERE clause: keeps the existing SQL shape stable (no new WHERE
predicate); the row's `profilePublic` value is needed in scope to
support future Phase-37+ extensions (e.g., friend-only profiles
where the answer depends on the viewer-identity). Performance
parity at single-row scale.

`profilePublic` is NOT exposed on `PublicProfile` return type — by
definition, a returned `PublicProfile` is a public profile; the
boolean is gating, not data. Future Phase-37+ may surface it on
the user's OWN-surface profile (e.g., to render the toggle's current
state on `/[locale]/profile`).

### Helper shape = single-flag set

**`setProfilePublic(userId, isPublic): Promise<void>`**. Mirrors
the existing `clearProfileImage`-shape (`Promise<void>` for
write-only fire-and-forget DB UPDATE). The server action on
`/profile` (Unit 36.2 deliverable) calls this with the form-
submitted boolean.

`Promise<void>` instead of `Promise<string | null>` (the
`updateProfile`-shape) because there's no validation step to fail —
the boolean is either true or false from the form's checkbox state.
Server action wraps validation if needed (e.g., session check).

### Drizzle moderation-mock typing fix (carried from Unit 35.2)

Unit 35.2 shipped `vi.mock("@/lib/moderation", ...)` mocks that
returned `{ ok: true }` and `{ ok: false, severity: "block",
reasons: [...] }` literals. Pre-commit hook ran vitest (which
passed) + commitlint + prettier + eslint, but NOT typecheck —
TypeScript strict-mode rejected the `{ ok: boolean }`-inferred
return types against the `ModerationResult` discriminated union
expectation.

Phase-36 explicit `pnpm typecheck` invocation surfaced the latent
errors. Fix: add `as const` to every `ok: true` / `ok: false` /
`severity: "block"` literal so TS narrows them to the discriminated-
union variants. 5 sites updated across `lib/users/index.test.ts`
+ `app/api/v1/rating-challenges/route.test.ts`.

**Recommendation for Phase 36+ hygiene unit**: consider adding
`pnpm typecheck` to the pre-commit hook chain so latent type
errors surface at commit time rather than at next-invocation.
(Not landing this hygiene change in Unit 36.1 to keep the unit
focused.)
