# Unit 0.4 — Design tokens

> Stage 1 (THINK) artifact. Critical-path per §15.4 — proceeding without checkpoint per project feedback rhythm.

## Problem restatement

Replace the placeholder OKLCH neutrals shipped in Unit 0.3 with the actual project palette and the complete set of design tokens that every later UI unit (Unit 0.10 IA stubs, Phase 1 RatingRadar + problem detail, Phase 3 SaturationCurve / MoversBoard, Phase 4 DomainMap) will consume. The token surface must be:

- **Stable in names**, free in values. Future visual reworks change OKLCH values; component code never changes.
- **Semantic**, not raw. Components read `text-foreground` and `bg-accent`, never `text-neutral-900`.
- **Light + dark first-class** (§10.1: "Not a tinted invert — a true second design"). Each token has both modes declared.
- **WCAG 2.2 AA** at minimum on every text-on-background pair (§10.3). Chart hues meet 3:1 against the background, sufficient for graphical-object contrast.
- **One brand-accent placeholder** that the project can swap with a single-line edit when Q5 resolves (vermilion vs deep cyan).

## Constraints

- **OKLCH everywhere.** Linear lightness + chroma + hue. Better than HSL for both perceptual uniformity (so a token "0.65" lightness reads consistently across hues) and for dark/light pair generation (flip lightness, keep chroma + hue).
- **Two-tone foundation must match §10.1 exactly.** `#0B0D10` → `oklch(0.149 0.006 254)` (near-black, slight cool cast); `#FAFAF7` → `oklch(0.984 0.003 96)` (near-white, slight warm cast). Tokens encode these via OKLCH so derived shades interpolate cleanly.
- **Single source of truth.** Every token is a CSS variable in `:root` (light) and `@media (prefers-color-scheme: dark) :root` (dark). The `@theme inline` block aliases each variable into the Tailwind utility namespace (`text-foreground`, `bg-accent`, `text-chart-1`, …). No raw hex / OKLCH in `@theme`.
- **Variable-name stability.** Once a token is named, future units may change its *value* freely but must never rename the variable in `:root`. The aliasing layer in `@theme` is the only place where renames are allowed, and only by superseding ADR.
- **Brand-accent placeholder = deep cyan.** Q5 is unresolved; pick the more defensible default (deep cyan reads more "academic-industrial" — §10.1's stated register — than vermilion, and is better separated from chart hues 3 + 5 which lean warm). Flipping to vermilion is a single OKLCH edit on `--accent` and `--ring` in both light and dark blocks.
- **No persisted dark-mode toggle yet.** MASTER_PROMPT §13 Phase 1 owns the persisted toggle. Phase 0 follows `prefers-color-scheme` only.
- **No motion tokens above 250 ms.** §10.1: "Subtle. 150ms ease. No bouncing."

## Alternatives considered

1. **OKLCH everywhere, semantic CSS variables, `@theme inline` aliases** (chosen).
2. **HSL with shadcn's "neutral" palette** (Unit 0.3 placeholder). Rejected: §10.1 mandates a specific two-tone foundation that's *not* the shadcn neutral; HSL also interpolates poorly across hues for dark-mode pairs.
3. **Per-component Tailwind classes (no CSS variables)** — `<div class="bg-[#fafaf7] dark:bg-[#0b0d10]">`. Rejected: kills the "rename never" invariant; every visual tweak ripples to every component.
4. **A JS-driven theme provider** (e.g., `next-themes` from Phase 1). Rejected for Phase 0: persisted toggle is explicitly a Phase 1 deliverable; for Phase 0 `prefers-color-scheme` is sufficient.

## Edge cases & failure modes

1. **Accent flip from cyan to vermilion changes chart-hue contrast.** Vermilion (~30°) sits between `--chart-urgency` (~50°) and `--chart-industry-call` (~10°). If Q5 lands on vermilion, the chart-3 and chart-5 hues should be nudged ~15° away to preserve separation. Documented as a follow-up note in OPEN_QUESTIONS Q5; not a blocker.
2. **OKLCH browser support.** Safari 15.4+, Firefox 113+, Chrome 111+ all ship OKLCH. Below that, the variable falls back to its declared value but renders as a fallback color (browser-specific). Phase 0 contributors are on evergreen browsers; production users in 2026 will be. No fallback chain needed.
3. **`oklch()` inside `var()` substitution.** Tailwind v4's `@theme inline` resolves CSS variables at build time only for utility generation; the variables themselves remain CSS `var()` calls in the emitted CSS. Light/dark switching works at runtime via the media query updating `:root`.
4. **Chart hue collision with accent.** Listed five chart hues at ≥ 50° hue separation. With deep cyan accent at ~215°, no chart hue is within 30°. If accent flips to vermilion (~30°), chart-3 (50°) and chart-5 (10°) get close — addressed in (1).
5. **Reduced-motion + motion tokens.** Motion tokens (`--duration-base: 150ms` etc.) sit alongside the `prefers-reduced-motion` guard already in Unit 0.3. Components that animate must consume the duration tokens *and* expect the global reduced-motion rule to override.
6. **Token namespace pollution.** Each token gets a domain-prefixed name where ambiguity exists (e.g., `--chart-difficulty` not `--chart-1`; the `--color-chart-1` alias in `@theme inline` is the Tailwind-utility-friendly shorthand and is intentionally less semantic).
7. **Print stylesheet.** Not in scope for Unit 0.4. Phase 1+ may add.

## Decision

Replace `app/globals.css` with the full token sheet:

- **Foundation:** `--background`, `--foreground` (two-tone, both modes).
- **Semantic:** `--primary`, `--primary-foreground`, `--accent`, `--accent-foreground`, `--muted`, `--muted-foreground`, `--border`, `--ring`.
- **Rating-dimension hues** (five, both modes): `--chart-difficulty`, `--chart-saturation`, `--chart-urgency`, `--chart-value`, `--chart-industry-call`. Aliased as `--color-chart-1..5` in `@theme inline` so Tailwind's `bg-chart-1` etc. utilities work.
- **Radii:** `--radius` base (0.5 rem); `--radius-sm`, `--radius-md`, `--radius-lg`, `--radius-xl` derived.
- **Motion:** `--ease-out`, `--ease-in-out`, `--duration-instant`, `--duration-fast`, `--duration-base` (150 ms — §10.1), `--duration-slow`.

Update `app/page.tsx` to demonstrate the token layer end-to-end: a small "five-dimension legend" using the chart hues to prove the wiring works. This is a Unit 0.10 / Phase 1 concern long-term, but a stub-level demo here is the cheapest way to validate the tokens before they're consumed by real viz.

Smoke test (`pnpm typecheck && pnpm build`), update CHANGELOG, commit.
