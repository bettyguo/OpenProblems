### Why TWO sibling forms (upload + clear) instead of one form with a flag

Per ADR-0017 D-B "clear-by-empty-submit": the helper layer
supports empty-file → clear semantics. But UX-wise, users
expect an explicit "Remove" button rather than submitting an
empty file (most browsers don't even submit `<input type="file">`
when empty).

Two forms maps cleanly to two server actions; no flag-passing
inside FormData; "Remove" only renders when an override exists
(no-op UX). Cleaner separation than one-form-many-modes.

### Why the `"use client"` boundary is small + page-scoped

Per Unit 16.0 D-14 + ADR-0017 §10.4 perf-budget concern:
- `ProfileImageUploadField` is the ONLY Phase-16 client-side
  component. ~50 lines; uses `useState` + `useId` + DOM events.
- It's NOT in the SiteHeader / AuthControl / WatchlistToggle
  shared client surfaces.
- React's client runtime + the component land in the
  `/profile` route bundle only.
- Result: `/profile` First Load JS = **117 kB** (was 108 kB at
  Phase-15 close; **+9 kB page-scoped**); **shared chunk
  UNCHANGED at 103 kB** end-to-end through every Phase 9-16
  unit. Other routes unaffected.

The +9 kB cost is the unavoidable React client runtime cost
the page incurs to flip to "use client" mode for any boundary.
Adding more client components to the same page would amortize
across the same runtime cost; subtraction is not possible.

### Why explicit `?saved=image` etc. instead of generic `?saved=1`

Phase-15 Unit 15.4 introduced `?saved=1` for text-save success.
Phase 16 adds two new save shapes (image upload + image clear).
Generic `?saved=1` would conflate three distinct user actions
into one banner message; localized messages per action are
clearer UX:
- `saved=1` → "Profile updated." (text fields).
- `saved=image` → "Profile picture updated."
- `saved=image-cleared` → "Reverted to GitHub avatar."

The page reads three independent `boolean` flags; renders the
matching banner.

### Why we DON'T enforce file size client-side via JS

`<input type="file">` doesn't natively support a `maxsize`
attribute. Implementing client-side size check would require
the file `onChange` to compare `file.size`; on too-large,
either:
1. Block the upload (won't submit).
2. Show a warning but still submit.

Both add complexity. Server-side check (Unit 16.3's
`updateProfileImage`) is authoritative + returns a clear error
message via `?error=<encoded>`. Phase 16 ships server-side only;
client-side size warning may follow Phase 17+ if UX feedback
demands.

### Why we DON'T enforce MIME client-side via JS

Same rationale: the `accept="image/jpeg,image/png,image/webp"`
attribute is a hint to the file picker (filters the chooser);
it does NOT prevent a determined user from submitting other
MIME types. Server-side check is authoritative. Belt-and-
suspenders client-side check is Phase 17+.

### `currentAvatar` semantics

Per ADR-0017 D-E own-surface fallback chain:
```ts
const currentAvatar = userRow?.imageOverride ?? session.user.image ?? null;
```

- If `imageOverride` non-null: render the uploaded avatar.
- Else if `session.user.image` non-null: render the GitHub-
  derived avatar (Phase 9-15 behavior preserved).
- Else: omit the `<img>` entirely (existing Phase 9-15 behavior;
  fallback initials placeholder is Phase 17+).

The `ProfileImageUploadField` component receives `currentAvatar`
as its `currentSrc` prop — server-rendered; the preview state
is purely client-side, layered on top.

### Form submit + redirect-with-status pattern

Mirrors Phase-15's `updateProfileAction` pattern verbatim:
- Server action validates session (re-checks `auth()`).
- Reads FormData / file.
- Calls helper.
- On error: `redirect(?error=<encoded>)`.
- On success: `redirect(?saved=<flag>)`.
- `revalidatePath` invalidates cached fragments before the
  redirect (so the next render picks up the new state).

This pattern is the canonical Phase 10-16 "signed-in own-state
mutation" shape. New server actions inherit it without
deviation.
