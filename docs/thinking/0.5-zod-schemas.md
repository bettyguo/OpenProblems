# Unit 0.5 — Zod schemas (lib/schemas/)

> Stage 1 (THINK) artifact. Critical-path; proceeding without checkpoint per feedback rhythm.

## Problem restatement

Land the eight Zod schemas in `lib/schemas/` that ADR-0003 names as the source of truth for every content file the project will ever validate, and the matching unit tests that §14.2 mandates ("Every Zod schema: at least 3 tests (valid, invalid, edge)"). Once these land, every later unit and every Phase 1+ content commit must pass through them.

Schemas (one file per entity):

1. `taxonomy.ts` — `Taxonomy` (root) + `Domain` + `Subdomain`.
2. `benchmark.ts` — `Benchmark` (referenced by Problem).
3. `problem.ts` — `OpenProblem`.
4. `rating-action.ts` — `RatingAction` (with nested 5-dimension object).
5. `paper.ts` — `Paper` (with nested `Contribution`).
6. `entry.ts` — `LeaderboardEntry`.
7. `author.ts` — `Author`.
8. `institution.ts` — `Institution`.

The test files are co-located: `lib/schemas/<name>.test.ts`. Vitest is installed in this unit (minimal config) so the tests are runnable today; Playwright / Storybook / Lighthouse-CI still belong to Unit 0.9.

## Inputs

- ADR-0003 (Zod 4 pinned).
- ADR-0005 (rating-action immutability — informs the schema's structure but doesn't change it; immutability is enforced by filename + hook + CI, not the schema).
- MASTER_PROMPT §7 (canonical data-model sketches).
- §14.2 ("at least 3 tests per schema").
- §14.1 ("no barrel files" — every consumer imports the schema directly from its file).
- §6 (`content/` layout: one folder per problem; ratings as immutable files; papers/authors/institutions as YAML).
- OPEN_QUESTIONS Q11 (subdomain id collisions across domains — affects taxonomy validation rules).

## Constraints

- **Zod 4 idioms.** `z.string().date()` for ISO dates (Zod 4 adds this), `z.discriminatedUnion` where applicable, `z.infer<typeof X>` for derived TS types.
- **No barrel re-exports.** No `lib/schemas/index.ts`; every consumer imports the specific schema (`@/lib/schemas/problem`).
- **Slugs are kebab-case** — `/^[a-z0-9-]+$/` everywhere a slug appears (problem slug, author slug, institution slug, domain id, subdomain id).
- **Cross-document FKs are not enforced at schema level.** Schemas validate one file at a time. The `domain` field on a problem is `z.string()` (not `z.enum(['applications', ...])`), because cross-referencing a value against the live taxonomy is the validate-content script's job (Unit 0.7). This keeps schemas portable and content-pipeline-agnostic, and avoids a circular dependency where the taxonomy itself must already be loaded before any other schema can be type-checked.
- **Dates are ISO strings**, not `Date` instances. YAML serializes strings, not Date objects; the schema enforces the string format and downstream code converts to Date when needed.
- **`posed_year` upper bound is dynamic.** MASTER_PROMPT §7.1 uses `.lte(new Date().getFullYear())` — must evaluate at validation time (refine), not module load.
- **Numbers are bounded where the doc says so** (`confidence ∈ [0, 1]`, `saturation ∈ [0, 100]`, `stars ∈ {0,1,2,3,4,5}`, `posed_year ∈ [1950, currentYear]`).
- **Optional vs nullable.** Use `.optional()` for "field may be absent." Never `.nullable()` unless YAML can legitimately encode `null` for that field (we don't have any such case yet — all "missing" is "absent").
- **`exactOptionalPropertyTypes` (tsconfig).** Optional fields cannot be assigned `undefined` explicitly. Test fixtures must omit absent fields, not pass `undefined`.

## Alternatives considered

1. **One file per entity, no barrel** (chosen). Matches §14.1.
2. **Single `lib/schemas/index.ts` with all schemas.** Rejected: §14.1 explicitly forbids barrel files; one mega-file would also balloon and be re-imported widely.
3. **Schemas as part of Velite's `velite.config.ts`.** Rejected: schemas must be reusable outside the build pipeline (the public API in Phase 3, the validate-content script in Unit 0.7, the search-index builder). Velite _consumes_ these schemas; it doesn't own them.
4. **Branded types for slugs** (`type ProblemSlug = z.infer<…> & { __brand: "problem" }`). Rejected for Unit 0.5: adds friction with no payoff until many cross-references exist. Revisit at Phase 2 when the cross-link audit lands.

## Edge cases & failure modes

1. **Empty taxonomy.** `Taxonomy.domains.min(1)`; `Domain.subdomains.min(1)`. Empty arrays would silently allow no-op content; cheap to forbid.
2. **Duplicate ids inside a single domain's subdomain list.** Cannot be enforced by Zod's array; must be checked by `superRefine` on `Domain`. Add.
3. **Subdomain id collisions across domains** (OPEN*QUESTIONS Q11 — `representation-learning` in both `deep-learning` and `general-ml`). Per Q11 the chosen URL strategy is domain-scoped (`/domains/[domain]/[subdomain]`). At schema level, \_one* domain's subdomain list may not contain duplicates, but two domains _may_ share a subdomain id. The taxonomy schema enforces (a) per-domain uniqueness; the validate-content script (Unit 0.7) will additionally check (b) that Problem.subdomain exists under Problem.domain. No global subdomain-id uniqueness rule.
4. **Future-dated `posed_year`.** Catch via dynamic refine.
5. **`tldr.max(400)`** — encourages short summaries. A 500-character TLDR fails validation, surfacing the curator to either shorten or split. The number is per MASTER_PROMPT §7.3.
6. **Status enum drift.** MASTER_PROMPT §7.1 lists `open / partially-solved / converging / solved / retired`. Adding a status later is a schema change → ADR-class.
7. **Rating-action `dimensions` field has all five sub-fields required.** Strict: a rating action must address every dimension explicitly, even if "unchanged" — the absence of a sub-field would create silent gaps in the audit log. Per ADR-0005 (immutability), each action is a complete snapshot, not a delta.
8. **`difficulty.grade` enum.** S/A/B/C/D/E. Case-sensitive uppercase.
9. **`saturation.value` 0–100.** A null/N/A case is described in §8.2 ("When no ceiling exists, mark Saturation = N/A and use a qualitative band"). Schema choice: `z.number().min(0).max(100)` strict — N/A is encoded by a future field (`saturation: { value: null, qualitative_band: 'Low'|'Medium'|'High' }`). For Unit 0.5 we keep the strict numeric form; Phase 3 may extend via a superseding ADR.
10. **`signals_considered` may include arxiv IDs or URLs.** Schema is `z.array(z.string())` — looseness is intentional, the curator's prose `rationale` field is the actual provenance.
11. **`prior_action`** is a path relative to the rating-actions folder. Schema validates non-emptiness but not file existence; cross-doc validation is Unit 0.7's job.
