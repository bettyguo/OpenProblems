# 14.3 — Public profile shell page at `/[locale]/u/[handle]` + `messages.public_profile.*` + curator-of-record badge

Fourth Phase-14 unit; second code unit. Largest UI consumer of Unit
14.2's `lib/users/` module. Establishes the **first per-USER read-side
public surface** for the project per
[ADR-0015](../adr/0015-per-user-privacy-model.md) D-A / D-B / D-E / D-F.

## THINK

### What this unit ships

Three artifacts:

1. **`messages/{en,fr}.json` (edits)** — adds the `public_profile.*`
   namespace with **16 keys per locale** (32 total) per Unit 14.0 D-9
   atomic-i18n discipline. **All keys pre-added upfront**; Unit 14.4
   consumes a subset without further messages edits. Mirrors Phase-13
   Unit 13.2 + Phase-12 Unit 12.4 precedent.

2. **`app/[locale]/u/[handle]/page.tsx` (new)** — public profile shell
   page. Consumes `getPublicProfileByHandle` (404 on null per ADR-0015
   D-B) + `getProfileActivity` (3 activity counts per D-A) +
   `getCuratorOfRecordSlugs` (case-sensitive badge per D-E).

3. **Curator-of-record badge surface** — small inline pill in the
   header card; renders only when `curatedSlugs.length > 0`; uses
   `accent` color tokens (matches StatusPill register); ICU plural
   for "Curator of {count} problem(s)"; hover-tooltip shows slug list.

### Design decisions realized from ADR-0015

- **D-A (field partition)**: page renders ONLY public fields —
  `name` + `image` + `githubLogin` + `createdAt` + 3 publicly-visible
  activity counts + curator-of-record badge. `email` never accessed
  (not even from `users` table query — helper drops it at the
  Drizzle SELECT layer). `rejected` + `withdrawn` counts never
  surface (helper doesn't return them).
- **D-B (case-preserved URL, case-insensitive lookup, no redirect)**:
  `getPublicProfileByHandle(handle)` does the case-insensitive lookup;
  profile body renders `profile.githubLogin` (canonical case) regardless
  of URL case the user typed. No `redirect()` call; URL bar preserves
  user-typed case.
- **D-E (curator-of-record case-sensitive)**: passes
  `profile.githubLogin` (canonical case from `users` row, NOT raw URL
  handle) into `getCuratorOfRecordSlugs`. Badge appears only when the
  YAML literal `editorial.primary_curator` matches exactly.
- **D-F (SiteHeader integration + "Edit your profile" CTA)**: when
  `session.user.id === profile.userId`, renders "Edit your profile"
  → `/profile` link. Other visitors don't see the CTA. SiteHeader
  integration itself is Unit 14.5.

### Layout decisions (D-14 through D-17 from Unit 14.0 prep)

- **D-14 avatar fallback**: bare `<img>` with `loading="lazy"` +
  fixed 64×64 dimensions; mirrors Phase-10 Unit 10.2 D-10 precedent.
  No `next/image` (avoids `remotePatterns` config surface for one
  avatar URL).
- **D-15 activity count rendering**: ICU plurals via next-intl. Three
  separate list items: watching count alone (one line) + pending +
  accepted on same line separated by middle dot (`·`) + "View all"
  link on a third line (only when challenges exist).
- **D-16 display-name fallback chain**: `name → githubLogin →
  translated fallback`. Mirrors `/profile/page.tsx` pattern minus
  email (email never surfaces on public route).
- **D-17 curator badge placement**: inline below `@login` line; small
  pill with `accent` color tokens; ICU-plural label; hover tooltip
  shows slug list via `title=` attribute (Phase 15+ may upgrade to
  an expansion link to `/u/{handle}/curated`).

### Empty-state handling

The Phase-14 prep doc D-5 anticipated three sub-cases:

1. **Profile exists with full activity**: render all 3 counts +
   "View all challenges →" link (when any challenge exists).
2. **Profile exists with zero activity** (`watchedCount === 0 &&
   pendingChallengeCount === 0 && acceptedChallengeCount === 0`):
   render the "@login hasn't watched any problems or submitted any
   challenges yet" empty-state card. Skips the count list entirely.
3. **Profile exists, partial activity** (some counts > 0): render
   all counts. The "View all challenges →" link appears only when
   `pendingChallengeCount + acceptedChallengeCount > 0` (no point
   linking to an empty challenges list).

### Parallel data fetching

After resolving the profile (which requires sequential await for the
404 short-circuit), the page does **2 parallel calls**:

```ts
const [activity, session] = await Promise.all([
  getProfileActivity(profile.userId),  // 3 parallel COUNT queries inside
  auth(),
]);
const curatedSlugs = getCuratorOfRecordSlugs(profile.githubLogin); // pure; sync
```

`getProfileActivity` internally parallelizes 3 COUNT queries via
`Promise.all`; combined with `auth()` at the outer level, the
request does **4 DB ops in parallel** (3 counts + 1 session lookup)
+ 1 sync file scan. Render begins after the slowest of those.
