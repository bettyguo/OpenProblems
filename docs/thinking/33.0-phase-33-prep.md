### D-1. First-thread recommendation: **Q76 Option A — Per-user-account subscriptions via FK column extension** as new **ADR-0023** (2-phase Q-carryover; closes ADR-0021 D-C Phase-31+ anticipation; subscriber-list-email arc closes Phase 30 → 31 → 32 → 33 = 4-phase complete-feature pair)

**Recommendation**: ship **per-user-account subscriptions Option A**
(FK from `subscriber.userId` to `users.id` with ON DELETE cascade,
nullable for anonymous email-only) as Phase 33's keystone thread via
new **ADR-0023**. Closes **Q76 architectural carryover** at **2-phase
age** (Phase 30 → 31 → 32 → 33 = 3-phase carryover; Q76 has been the
architectural Q-carryover most directly tied to the subscriber-list-
email arc). **Closes ADR-0021 D-C Phase-31+ anticipation explicitly**:
the D-C clause anticipated "per-user-account subscriptions" as the
**first Phase-31+ candidate**; Phase 33 = realization. **Subscriber-
list-email arc completes Phase 30 → 31 → 32 → 33 = 4-phase complete-
feature pair** (foundation → scheduler → cleanup → authenticated-user
linkage).

**Rationale**:

1. **Closes ADR-0021 D-C anticipated Phase-31+ first candidate**.
   The Phase-30 ADR explicitly named per-user-account subscriptions
   as the Q76 architectural candidate to land Phase 31+. Phase 31
   shipped the scheduler instead (B.15 item 1 tight follow-on);
   Phase 32 shipped the cleanup-cron (B.15 item 4 follow-on);
   Phase 33 closes the architectural D-C anticipation.
2. **Migration cadence is alternating-with-pause, not 3-consecutive
   cluster**. Phase 30 + 31 = 2-phase migration cluster; Phase 32 =
   0-migration; Phase 33 picks up migration again = "alternating with
   pause" pattern (not 3-consecutive). The Phase-32 prep doc's "third
   consecutive migration phase would be unprecedented" concern
   evaporates after Phase 32 broke the consecutive run. Mirrors
   Phase-15-16-cluster shape (Phase 15 + 16 = 2-phase migration
   cluster; Phase 17+ = 13-phase 0-migration streak; Phase 30 + 31 =
   2-phase migration cluster; Phase 32 = 0-migration; Phase 33 = mig
   again = NEW shape — first "migration-cluster-resume-after-gap"
   in project history).
3. **Subscriber-list-email arc completes** as a 4-phase pair:
   - **Phase 30** = foundation (ADR-0021; `lib/email/` + `lib/subscribers/`
     + `subscriber` DB table + verify + welcome + subscribe form).
   - **Phase 31** = production-send (ADR-0022; Vercel Cron + digest
     template + cron route + idempotency column).
   - **Phase 32** = operational cleanup (no new ADR; cleanup-stale-
     tokens cron via Phase-31 reuse).
   - **Phase 33** = authenticated-user linkage (ADR-0023; `subscriber.userId`
     FK + auto-populate-on-form-submit + cascade-on-user-delete).
   Together the 4 phases = **complete subscriber-list-email feature
   delivery**; closes the "Email/RSS digest: per-domain weekly
   summary" Phase-5 deliverable to production-ready state with
   authenticated-user accountability for signed-in users.
4. **Single-column schema extension; smallest possible migration**.
   `subscriber.userId text references users(id) on delete cascade`
   — single nullable FK column; migration `0008_subscriber_user_id.sql`.
   No new table, no new index beyond the FK constraint, no new
   columns aside from the one.
5. **Auto-populate-on-signed-in-form-submit** is the smallest UX
   change. Subscribe route POST handler checks `auth()` for session;
   if signed-in, populates `subscriber.userId = session.user.id` on
   row create; if not, populates as `null`. **No UI change**;
   **no new i18n keys**; **no new env var**; **no new operational
   gate**.
6. **Cascade-on-user-delete** is automatic via the FK constraint.
   When the future account-deletion flow (Phase 34+ candidate; not
   shipped Phase 33) deletes a `user` row, the FK constraint cascades
   to soft-deleting the associated `subscriber` rows. Phase 33 SHIPS
   the constraint; future account-deletion path benefits automatically.
7. **No new operational gate Phase 33**. Reuses existing Q54
   (GitHub OAuth) + Q73 (Google OAuth) + Q75 (Resend) + Q77 (Vercel
   Cron). **Operational gate count UNCHANGED at 6**.
8. **No new runtime dependency Phase 33**. Server-only stack reused
   (Drizzle + NextAuth.js + Resend; all Phase-30+ infra carried).
9. **First Load JS preservation**. New auth-check inside subscribe
   POST handler is server-only; no client surface delta. **First
   Load JS expected UNCHANGED at 103 kB** (44th consecutive unit
   if preserved).
10. **Reuses Q76 Option A documented in OPEN_QUESTIONS**. Three
    resolution options were documented Phase 30 Unit 30.4 alongside
    Q76 promotion; Option A is the cleanest path (FK extension +
    auto-link). ADR-0023 pins Option A as the chosen path; Option
    B (separate `user_subscriptions` table) + Option C (anonymous-
    only) become rejected alternatives.

### D-3. ADR-0023 D-clauses anticipated

D-A. Per-user-account-subscriptions model = **Option A FK extension
verbatim per Q76 documentation**. Rejects: Option B separate table /
Option C anonymous-only-indefinitely.

D-B. Subscriber row shape = `subscriber.userId text references users(id)
on delete cascade` nullable; preserves anonymous email-only path
(NULL `userId` = anonymous subscriber; NOT NULL = authenticated).

D-C. Subscribe-route POST handler behavior = checks `auth()` for
session; if signed-in, populates `userId = session.user.id` on row
create; if not, populates as `null`. **No new UI flow**; **no opt-in
toggle**; **no opt-out** (signed-in users are auto-linked).

D-D. Cascade-on-user-delete = automatic via FK `on delete cascade`.
Phase 33 ships the constraint; future account-deletion flow (Phase
34+ candidate; not in Phase 33 scope) benefits automatically without
additional code.

D-E. Re-subscribe semantics (existing `unsubscribed` row + new signed-
in submit) = update existing row's `userId` to current session
user's id; preserves the Phase-30 D-D step 3 case-(d) "existing
unsubscribed → re-use row + set status = pending_verification + fresh
tokens" with the added userId update.

D-F. Cross-user re-subscribe semantics (existing row with `userId =
other_user_id` + new signed-in submit from different user) =
**preserves existing `userId`**; treat as ambiguous "is this same
email shared between accounts?" case. The form-submit user's
session-id does NOT overwrite a pre-existing different userId.
Conservative posture; Phase 34+ may revise if user-merge signal
emerges.

D-G. Anonymous → authenticated migration semantics (existing
anonymous row with `userId = NULL` + new signed-in submit from a user
with the same email) = update `userId = session.user.id` (no row
ambiguity; the existing anonymous row "becomes" the signed-in user's
subscription naturally).

D-H. Phase 34+ deferrals: "manage my subscriptions" page on
`/[locale]/profile` (display subscribed domains + manage / unsubscribe
buttons); curator analytics dashboard ("which signed-in users are
subscribed"); per-problem subscriptions extension; subscription-
preference editing UX.
