paper-title audit triggered by Unit 2.6e's HyenaDNA discovery
(arXiv-ID-vs-claimed-title mismatch). Audited all 30 paper YAMLs
against their arXiv canonical titles; found one additional title
canonicalization bug beyond the HyenaDNA case.

## THINK

For each of the 30 papers in `content/papers/`, compare:

- the `title:` field in the YAML, against
- the canonical title returned by the arXiv API
  (`http://export.arxiv.org/api/query?id_list=<id>`) — or, for the
  rate-limited subset, against the `https://arxiv.org/abs/<id>`
  HTML page.

### The fix

One line in `content/papers/2305.10160.yaml`:

```diff
- title: "Stop Uploading Test Data in Plain Text: Practical Strategies for Mitigating Data Contamination by Large Language Models"
+ title: "Stop Uploading Test Data in Plain Text: Practical Strategies for Mitigating Data Contamination by Evaluation Benchmarks"
```

Identifies the same paper (Jacovi, Caciularu, Goldman, Goldberg;
EMNLP 2023 — `https://arxiv.org/abs/2305.10160`). The substitution
is semantically wrong — "by Evaluation Benchmarks" identifies the
mechanism (benchmarks themselves end up in pretraining corpora and
contaminate; the title prescribes strategies for benchmark
authors); "by Large Language Models" mis-attributes the agency.

### Constraints

- **§15.6 — no fabrication.** The corrected title is verbatim from
  the arXiv canonical record. The current YAML title appears to be
  a paraphrase error introduced when the paper was seeded in Unit
  2.6, not an alternate venue title (no NeurIPS/EMNLP camera-ready
  variant matches the "Large Language Models" subtitle).
- **No collisions with parallel session.** The parallel session
  is editing `content/papers/2306.15794.yaml` (Unit 2.6e). This
  unit edits `content/papers/2305.10160.yaml`, which is
  untouched in the working tree.
- **Pure metadata fix.** No code, schema, route, or bundle changes;
  no other field on the paper YAML is affected (authors stays `[]`
  consistent with the rest of batch 3 pending batch-3 author
  backfill).

### Alternatives considered

1. **Defer until batch-1 arXiv verification clears.** Rejected —
   the 2305.10160 mismatch is independently confirmed via the
   `arxiv.org/abs` HTML page and doesn't depend on batch-1 status.
   Sitting on a known correctness fix because part of the audit is
   blocked is the wrong trade.
2. **Land both 2305.10160 and the batch-1 re-verification in one
   commit.** Rejected — batch-1 verification might require ~10 min
   of rate-limit cooldown plus retry. The 2305.10160 fix is
   independently ready and shouldn't wait.
3. **Defer to the parallel session's Unit 2.6e.** Rejected — the
   parallel session's THINK doc scopes 2.6e narrowly to the
   HyenaDNA correction. Extending their unit cross-paper would
   blur the audit trail.
4. **Reword the title to the EMNLP camera-ready (if different).**
   Rejected — no evidence the EMNLP-published title differs from
   the arXiv canonical. The Anthology entry would be the source of
   truth for the camera-ready form, but the project's seed
   convention (Unit 2.4 / 2.5 / 2.6) uses arXiv canonical titles.

### Surfaced findings (for the user / future curator)

1. **Batch-1 arXiv verification is incomplete.** 10 papers
   (TruthfulQA, HaluEval, SimpleQA, FActScore, SWE-bench, τ-bench,
   OSWorld, RE-Bench, Large Language Monkeys, Scaling LLM
   Test-Time Compute) are recall-verified only. A future unit
   should retry the arXiv API once the rate-limit cooldown
   clears.
2. **Audit-content does not check title-vs-arXiv.** A hardening
   of the cross-link audit could add an opt-in network-dependent
   check that re-fetches arXiv titles and compares — but adding
   network I/O to the audit script makes CI flakier. The
   trade-off is for a separate scope decision.
3. **The 2305.10160 mismatch was a paraphrase error, not a
   wrong-arXiv-ID assignment.** Same class of issue as a typo;
   different class than the 2306.15794 / HyenaDNA bug (where the
   YAML's claimed paper genuinely wasn't the paper at that arXiv
   ID).
