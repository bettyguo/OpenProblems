### D-3. Route shape

**Path**: `/[locale]/u/[handle]/challenges/[id]`.

Fits the existing Phase-14 per-user namespace
(`/[locale]/u/[handle]` + `/[locale]/u/[handle]/challenges`).
Adds `[id]` as the third dynamic segment.

**Params**: `{ locale, handle, id }`.
- `locale`: validated via existing `isLocale` helper.
- `handle`: resolves to `userId` via `getPublicProfileByHandle`.
- `id`: UUID; passes through to `getChallengeById`.

**Visibility logic**:
1. Resolve handle to public profile. 404 on no-match.
2. Fetch challenge by id. 404 on no-match.
3. Verify `challenge.userId === profile.userId`. 404 on
   mismatch (prevents enumeration via handle/id pair shopping).
4. Verify `challenge.status Ōłł PUBLIC_CHALLENGE_STATUSES`
   (submitted / under_review / accepted). 404 on private
   states (rejected / withdrawn) ŌĆö privacy preservation per
   Phase-13 Unit 13.0 D-3.

**Render**:
- Status badge (matches Phase-14 listing styling).
- Submitter (`@handle`) + submitted date.
- Problem title (linked to `/problems/<slug>`).
- Dimension label (translated; e.g., "Difficulty" / "Saturation").
- Proposed value (plain text per dimension format).
- Full rationale (plain text per Phase-11 schema; preserves
  `whitespace-pre-wrap` per Phase-13 listing precedent; NO
  markdown rendering ŌĆö would be Phase-27+ via thread F).
- **When status = accepted**:
  - Reviewer (`@reviewerLogin`) + reviewed date.
  - Full reviewNotes rendered as markdown (via Phase-18
    `renderReviewNotesMarkdown` helper).
  - `acceptedActionId` filename + link to rendered rating-
    action (if served via existing `/problems/<slug>/ratings`
    surface; otherwise just the filename text).
- **When status Ōłł {submitted, under_review}**: no reviewer
  info / no reviewNotes / no acceptedActionId (deliberation
  in progress per Phase-13 D-3 semantics).

### D-4. Visibility helper

New helper `getPublicChallengeDetailById({ id, profileUserId })`:

```ts
export interface PublicChallengeDetail extends PublicChallengeRow {
  reviewerLogin: string | null;
  reviewedAt: Date | null;
  reviewNotes: string | null;
}

export async function getPublicChallengeDetailById(opts: {
  id: string;
  profileUserId: string;
}): Promise<PublicChallengeDetail | null>;
```

- Joins `ratingChallenges` + `users` (submitter) + `users`
  (reviewer; aliased).
- Filters: `id = opts.id` AND `userId = opts.profileUserId`
  AND `status Ōłł PUBLIC_CHALLENGE_STATUSES`.
- Returns null when any filter fails (single 404 path on the
  page).
- Reviewer fields (`reviewerLogin` / `reviewedAt` /
  `reviewNotes`) are populated only for accepted status; null
  for submitted / under_review (server-side filter applied per
  D-3 visibility).

**Why a new helper rather than reusing `getChallengeById`**:
- Single query (no separate reviewer lookup).
- Server-side visibility filter (defense-in-depth vs UI
  filtering).
- Returns null for any privacy-denied case (single 404 path
  on page).
