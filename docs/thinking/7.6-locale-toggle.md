# 7.6 — `components/locale-toggle/` site-header UI

Fourth code unit of Phase 7. Implements the site-header locale-toggle per [ADR-0011 D-F](../adr/0011-i18n-strategy.md). Independent of the routing-layer work (Unit 7.3a's deferred bulk migration + middleware), so safe to land in parallel with Unit 7.5's methodology FR pilot. Picked over restarting Unit 7.5 / 7.3a because the parallel-curator session that authored Unit 7.4 (`f315458`) appears active in this repo and may already own 7.5; locale-toggle scope is fully disjoint (touches only `components/locale-toggle/` + a one-line `components/site-header/` insertion).

## THINK

### Inputs

- **[ADR-0011 D-F](../adr/0011-i18n-strategy.md)** — site-header placement; `"use client"`; stable placeholder pre-hydration; click cycles to next locale; aria-label describes next action.
- **[ADR-0011 D-B](../adr/0011-i18n-strategy.md)** — sub-path routing (`/en/...`, `/fr/...`); locale is in the URL.
- **[`components/theme-toggle/index.tsx`](../../components/theme-toggle/index.tsx)** — UX precedent: cycling button with current state as icon, aria-label describing next action.
- **[`components/site-header/index.tsx`](../../components/site-header/index.tsx)** — insertion site; LocaleToggle lands next to ThemeToggle in the right-aligned `<div>`.
- **[`lib/i18n/routing.ts`](../../lib/i18n/routing.ts)** — `locales`, `defaultLocale`, `isLocale` consumed by the toggle's locale-detection helper.

### Scope (this unit)

Three new files + one edit:

1. **`components/locale-toggle/index.tsx`** (new) — `"use client"` component:
   - Reads the current URL via `usePathname()` from `next/navigation` (NOT `useLocale()` from next-intl — see "Constraints" below).
   - Splits the path into segments; if the first segment is in `locales`, treats it as the current locale and computes the `rest` of the path.
   - For bare paths (no locale prefix — the current state of every route except `/en/about` and `/fr/about`), the component renders `null`. This is the transitional-state behavior; once Unit 7.3a's middleware lands, bare paths redirect to `/en/...` and the toggle becomes visible everywhere.
   - For locale-prefixed paths, renders a `<Link>` to the equivalent path under the next locale (cycle: `en → fr → en`); current locale label displayed (`EN` or `FR` as text); aria-label + title describe the next action in the *target* language ("Passer au français" when switching to FR; "Switch to English" when switching to EN).
   - Visually mirrors `ThemeToggle`: `size-9` square button styling, border, hover, focus-visible ring.
2. **`components/locale-toggle/index.test.tsx`** (new) — vitest cases:
   - Pure-helper tests for the path-parsing function (`computeToggle(pathname)` → `{ currentLocale, targetHref } | null`).
   - Render tests using `@testing-library/react` + a `usePathname` mock: on `/en/about` renders link to `/fr/about` with EN label; on `/fr/about` renders link to `/en/about` with FR label; on `/about` renders nothing.
3. **`components/site-header/index.tsx`** (edit) — one-line: import `LocaleToggle` and insert it BEFORE `<ThemeToggle />` in the right-aligned controls cluster. Both occupy `size-9` buttons; visual order is `SearchTrigger → LocaleToggle → ThemeToggle`.
4. **`docs/thinking/7.6-locale-toggle.md`** (this doc).
5. **`CHANGELOG.md`** — Phase-7 Unit 7.6 entry.

### Scope NOT in this unit

- **No locale cookie**. ADR-0011 D-F mentions `NEXT_LOCALE` cookie as a "preferred locale" hint set by middleware for first-visit redirects; this unit doesn't ship middleware (deferred to 7.3a), so no cookie write either. Cookie integration lands with middleware.
- **No locale-aware messages on the toggle itself**. The toggle label is plain text "EN" / "FR" + the aria-label is a fixed pair of strings ("Switch to English" / "Passer au français"). Using `useTranslations` would require the toggle to live under `NextIntlClientProvider`, which the current `app/layout.tsx` doesn't provide for the SiteHeader (provider is only in `app/[locale]/layout.tsx`, below SiteHeader). This is a known intermediate-state pattern; once 7.3a moves SiteHeader under [locale] the toggle can adopt useTranslations.
- **No third-locale support**. Locales = `["en", "fr"]`; the cycle has 2 stops. ADR-0011 D-F anticipates a third locale; the cycle logic uses `locales.indexOf(current) + 1 mod locales.length`, so adding a third locale to `lib/i18n/routing.ts` extends the cycle automatically — but no UI work is needed in this unit.
- **No route-transition animation**. The toggle is a plain `<Link>` (Next.js client-navigation handles the SPA-style transition); no view-transitions API; no fade.
- **No `app/[locale]/page.tsx`** (root home page under locale). `/en/` and `/fr/` still 404; the toggle hides on those routes naturally because they 404 before render. Unblocked by Unit 7.3a.

### Constraints

- **SiteHeader renders OUTSIDE `NextIntlClientProvider`**. The provider is in `app/[locale]/layout.tsx`; SiteHeader is in `app/layout.tsx` (the root layout, above [locale]). This means `useLocale()` / `useTranslations()` from `next-intl` would throw when called inside SiteHeader children. Workaround: read locale from the URL via `usePathname()` from `next/navigation`, which works regardless of provider context. Pure string parsing; no provider dependency.
- **`usePathname()` is SSR-stable**. Next.js provides the URL at SSR time; the component renders the same content SSR and post-hydration. No mounted-guard placeholder needed (unlike `ThemeToggle`, which guards against next-themes resolving theme post-hydration). This deviates from the ADR-0011 D-F line "Stable placeholder pre-hydration (no layout shift)" — but the deviation is *equivalent* (the component doesn't layout-shift because its SSR output matches its client output exactly). If a future redesign introduces a placeholder, it'd be a no-op for hydration; intentionally skipped here.
- **Bare-path behavior**. The toggle renders `null` on bare paths to avoid offering a navigation target that doesn't exist (clicking through to `/fr/about` from `/about` works since /fr/about exists; but `/fr/problems` would 404 at HEAD because `app/[locale]/problems/page.tsx` doesn't exist). Hiding is safer than offering 404 links. Acceptable for the intermediate state; once 7.3a creates every `/[locale]/<route>/` shadow, the toggle becomes visible everywhere.
- **`exactOptionalPropertyTypes: true`** — the component accepts `{ className?: string }`; matches ThemeToggle's pattern verbatim.
- **No new dependencies**. Pure first-party plus existing `next/navigation` + `lib/i18n/routing` imports.

### Alternatives considered

- **Render two separate buttons (`[EN] [FR]`) with current one styled active** — Rejected per ADR-0011 D-F "Click cycles to the next locale" — cycling is pinned in the ADR. Two buttons is a different UX shape that would need an ADR revision.
- **Use `<Link>` from `next-intl/navigation` for locale-aware routing** — Rejected. `next-intl/navigation` Link expects a non-localized pathname (just `/about`) and prepends the current locale automatically; useful when middleware is wired and consistently localizes URLs. At HEAD, the bare-vs-localized URL distinction is real and routes don't have `/[locale]/` shadows; using next-intl's Link would mask the intermediate-state asymmetry. Direct `<Link href="/fr/about">` is clearer.
- **Inject `LocaleToggle` into `app/[locale]/layout.tsx` (under the provider)** — Rejected. SiteHeader is a global header; duplicating SiteHeader under [locale] would create two render-trees and conflict on `/en/about` (root layout renders SiteHeader, then [locale] layout would render a second one). The toggle living in the global SiteHeader is the ADR-compliant placement; reading pathname is the workaround for the provider-scope mismatch.
- **Skip the toggle entirely until 7.3a** — Rejected. The toggle has independent value: on `/en/about` and `/fr/about` (the FR pilot routes shipping in 7.5), the toggle gives readers a visible affordance to switch between language variants. Without it, switching requires manual URL edits. Shipping the toggle now also de-risks 7.3a (the toggle is already built when middleware lands).

### Tradeoffs / defensible defaults flagged

1. **Toggle is hidden on bare paths** — During the intermediate state (every route except `/en/about` + `/fr/about` is bare), the toggle only shows on 2 routes. This is the correct fail-closed behavior (clicking would 404 on non-pilot routes) but UX-wise means the toggle is rarely visible until 7.3a + 7.5 expand the bilingual surface.
2. **No `useTranslations` for the aria-label** — aria-label is a fixed pair of strings hardcoded in the component. When a reader visits in EN, they see "Passer au français" (the next action's language); when in FR, they see "Switch to English". Self-evident for bilingual readers; opaque for monolingual readers, but the visible button label ("EN"/"FR") is the primary signal. Reconsider when SiteHeader moves under the provider.
3. **Text label instead of icon** — ADR-0011 D-F says "Three icons: EN, FR (and a third for future locales)". The "icons" are interpreted here as 2-letter locale codes rendered as text (matching `lucide-react`'s use of textual `Globe` is too generic; `Languages` icon doesn't disambiguate which language). Text labels are the most informative and consistent with the cycling-button precedent.
4. **No URL-encoding of `rest`** — the toggle computes `${target}${rest}` where `rest = "/about"`. If a route has query params or fragments, `usePathname()` strips them (returns pathname only); the toggle navigates to `/fr/about` without preserving `?foo=bar` or `#section`. Acceptable: no Phase-7 route currently uses query params; if it becomes a problem, switch to `useSearchParams` + reconstruction. Deferred.
5. **Cycle order is `en → fr → en`** — alphabetical; matches `locales` array order. When a third locale lands (e.g., `zh-CN`), order becomes `en → fr → zh-CN → en`. Curator chooses the array order; this unit makes no opinion beyond "cycle in array order."
