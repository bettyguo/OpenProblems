# 6.1 — ADR-0010 Discussions backend (Giscus embed + first-party GraphQL read-side)

Phase 6's first code-gating ADR. Pins the Discussions integration backend that subsequent Units 6.2 → 6.6 build on. Closes [OPEN_QUESTIONS Q46](../../OPEN_QUESTIONS.md#q46-discussions-backend-giscus-embed-vs-first-party-graphql) (currently decided-as-lean from Unit 6.0). Docs-only unit; ADR file + this THINK doc + index update + CHANGELOG.

## THINK

### Inputs

- **Unit 6.0 (`ada448b`)** named D-3 (Giscus + GraphQL split lean), D-4 (pathname-based per-problem mapping lean), D-5 (read-side surfacing on cards + digest) as the three sub-decisions this ADR pins. Also names D-6 (talk-page route shape) but that's a route-layout detail, not a backend decision — landed in Unit 6.3 not here.
- **MASTER_PROMPT.md §13 Phase 6+ bullet 1**: "GitHub Discussions integration for problem talk pages." Verbatim; no further enumeration.
- **§14.2 testing convention**: "Every page route: Playwright smoke (renders, no console errors, key landmarks present)." Constrains the talk-page implementation more than this ADR — but mention it so the Giscus iframe doesn't break the e2e smoke (it loads async; the page-render assertion must not depend on iframe contents).
- **§5.5 perf budget**: First Load JS ≤ ~110 kB has been held at 103 kB since Phase 1. Giscus is iframe-embedded — its bundle does NOT count against First Load JS shared chunk. The wrapper component (`@giscus/react`) adds ~3-5 kB to the talk-page route's per-page JS but stays off the shared chunk.
- **ADR-0008 D-C precedent** for env-driven token discipline (`ANTHROPIC_API_KEY` → `GITHUB_TOKEN` mirror).
- **ADR-0009 D-E precedent** for filesystem-cache pattern (`.arxiv-cache/`, `.pdf-cache/` → `.github-cache/` mirror).
- **Phase-4 Unit 4.5 / 4.6 `/contributing` framing**: "The site stores no user accounts." That contract holds under Giscus (auth-via-GitHub happens inside the embed; no first-party identity storage).

### Scope (this ADR)

ADR-0010 pins six concrete contracts:

1. **D-A. Embed backend = Giscus** (`@giscus/react` for the React wrapper). Reasons in §Decision Outcome.
2. **D-B. Read-side metadata = first-party GitHub GraphQL** (`lib/discussions/github-graphql.ts` — Unit 6.2 deliverable). Separate from the embed; cached at build time.
3. **D-C. Per-problem mapping = pathname-based** (`og:url` strategy in Giscus; lazy creation on first comment).
4. **D-D. Token + scope** = `GITHUB_TOKEN` env, `public_repo` minimum scope; never write.
5. **D-E. Caching** = `.github-cache/<query-hash>.json` (gitignored; mirrors `.arxiv-cache/` + `.pdf-cache/`).
6. **D-F. Moderation routing** = defer to GitHub Discussions native; no first-party moderation queue in Phase 6.

### Considered options

1. **Giscus embed + first-party GraphQL read-side** (chosen).
2. **Giscus embed only** (no read-side metadata; problem cards don't show discussion activity).
3. **First-party GraphQL build** (server-rendered thread; no Giscus).
4. **Utterances** (Issues-based, not Discussions; older).
5. **Disqus** (third-party, non-GitHub).
6. **No comments — defer to existing GitHub Issue templates** (Phase-4 `/contributing` framing).

### Order rationale

ADR lands FIRST in Phase 6 before any code (Unit 6.2+) because the backend choice gates:
- Which dependency lands in package.json (`@giscus/react` vs the heavier `@octokit/graphql` only, or both).
- Which env vars the README + CI need to know about (`GITHUB_TOKEN`).
- Whether `.github-cache/` needs to be added to `.gitignore` (Unit 6.2's setup).
- Whether ADR-0004 (file-first; no DB) survives or starts to flex (Giscus doesn't touch our DB; first-party-only GraphQL would still survive ADR-0004 with the build-time cache).

### Scope NOT in this ADR

- **Component shell** (`components/discussions/GiscusEmbed.tsx`) — Unit 6.4 owns. Wrapper details (theme sync, loading state) are implementation, not architecture.
- **Card-badge surfacing** — Unit 6.5 owns.
- **Digest-pipeline extension** — Unit 6.6 owns.
- **Lighthouse enrolment** — Unit 6.7 owns.

### Constraints

- **ADR-0004 (file-first; no DB)** holds. Build-time cache to `.github-cache/` is sidecar JSON, not a DB. Re-eval at Phase 7+.
- **ADR-0009 (human-review diff)** doesn't apply — Discussions aren't LLM-drafted content; they're community-authored. The audit-sidecar pattern doesn't transfer.
- **§5.5 perf budget**. Giscus iframe is below the fold + lazy-loaded; doesn't impact First Load JS shared chunk (103 kB). The React wrapper adds a small per-page increment to `app/problems/[slug]/talk/page.tsx`'s route chunk only.
- **§14.1 TypeScript strict + exactOptionalPropertyTypes**. `@giscus/react` is typed; tested against the strict mode at Unit 6.4 time.
- **§14.2 testing**. Every page route needs Playwright smoke. Talk-page smoke must NOT depend on iframe contents loading (cross-origin; flaky). Smoke asserts the SHELL renders + landmark a11y holds; iframe loads async out-of-test.
- **§15.6 primary-source rule**. Doesn't apply directly to comments (which are opinion-class, not editorial-claim-class). The rule applies to curator-side editorial actions; community discussion sits outside the rated content surface.
- **`exactOptionalPropertyTypes`** for any new GraphQL-derived types — careful with optional `lastCommentAt` etc.

### Alternatives considered (depth notes for the ADR)

#### Option 2 — Giscus embed only (no read-side metadata)

- Simpler: only one dependency (`@giscus/react`); no GraphQL client; no `.github-cache/`; no token env var; no rate-limit handling.
- Drops the activity-badge surface on problem cards (D-5 in Unit 6.0). Cards stay static; user has to click into talk page to see if there's activity.
- Drops the digest-pipeline extension (Unit 6.6's discussions-in-weekly-RSS). Digest stays as-is from Phase 5.
- Rejected: the activity badge is a load-bearing surface for the rating-agency framing — "Which problems have active community engagement?" is a question the cards should answer at-a-glance.

#### Option 3 — First-party GraphQL build (server-rendered thread; no Giscus)

- Pros: full SSR control over comment rendering; comments are crawlable; consistent theme without iframe theme-sync friction; smaller per-route JS (no embed).
- Cons: we build the comment UI ourselves (input box, threading, reactions, moderation actions). Estimated 5-8 additional units (Phase 6 grows from 11 to ~18). Worse: we have to handle WRITES (POST a comment), which means GitHub OAuth flow, which means first-party auth, which BREAKS the "site stores no user accounts" pact of Phase-4 `/contributing`. That cascades into a DB-trigger flip + the auth thread.
- Rejected: the cost (auth thread cascade + 5-8 extra units) buys cosmetic gains. Giscus's iframe is the right abstraction at this maturity.

#### Option 4 — Utterances

- Backed by GitHub **Issues**, not Discussions. Issues are a poorer model for talk threads (one author opens, others comment in a flat list; no nested threading; no "answer" / "discussion" / "Q&A" category split).
- Older project (less active maintenance than Giscus).
- Issues clutter the repo's issue tracker — would mix talk threads with the 4 Phase-4 issue templates (`new-problem`, `new-paper`, etc.).
- Rejected: §13 explicitly names "Discussions"; Utterances doesn't deliver.

#### Option 5 — Disqus

- Third-party, NOT GitHub. Breaks the "site stores no user accounts" pact only weakly (Disqus handles its own auth) but introduces an external dependency on a company with a known privacy posture (ads + tracking).
- Free tier injects ads; paid tier costs money.
- Rejected: third-party privacy posture + ads incompatible with the rating-agency framing's editorial integrity.

#### Option 6 — No comments; defer to existing GitHub Issue templates

- Pros: zero new code; zero new deps; preserves Phase-4 framing perfectly.
- Cons: §13 explicitly names "GitHub Discussions integration for problem talk pages." Skipping the deliverable IS a Phase-6 thread choice (e.g., bilingual-first) — not a "Discussions thread with Option 6."
- Rejected for this ADR specifically: Option 6 IS "redirect Phase 6 to bilingual/auth instead of Discussions." If the human picks Option 6, Unit 6.0's recommendation flips; Unit 6.1 regenerates around the new thread. Within the Discussions-thread choice, Option 6 is a contradiction.

### Tradeoffs / defensible defaults flagged

1. **Two dependencies in package.json**: `@giscus/react` (for the embed) + `@octokit/graphql` (for the read-side client). Could consolidate via `@octokit/rest` (REST + GraphQL combined) but the GraphQL endpoint is the right fit for the discussions-by-repo query shape.
2. **`.github-cache/` lifecycle**: gitignored, ephemeral per CI run. Build-time SSG fetches once per discussion category; production deploys hit fresh data each rebuild. Stale at runtime (between deploys) but acceptable for v1 — discussions evolve on minute scale; deploys on hour scale.
3. **Pathname-based mapping with lazy creation**. The cost: a problem with no comments has no discussion at all (vs. an empty seeded thread). Acceptable — empty threads look worse than absent ones, and the GraphQL read-side returns 0 counts cleanly for the "no thread yet" case.
4. **Theme sync friction**: Giscus's `data-theme` attribute syncs via `postMessage`. The wrapper component (Unit 6.4) needs a `useEffect` watching the existing `useTheme()` hook and posting the theme change to the iframe. Implementation detail; called out in the ADR's Consequences so Unit 6.4 has the contract.
5. **Indexing**: Q48's lean (sitemap-include talk pages) holds. Search engines won't index iframe contents — but the talk-page shell itself IS indexed (problem name, "Discuss" heading). The talk-page URL becomes a stable inbound link; the iframe contents are accessory.
6. **`GITHUB_TOKEN` scope**: `public_repo` (read-only) is the minimum. The GraphQL `repository.discussions` query works under `public_repo`. We never write — embedding Giscus delegates writes to GitHub's own OAuth flow inside the iframe.
