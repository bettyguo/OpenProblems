# 1.10 — Landing page v1

## THINK

### Inputs

- **MASTER_PROMPT §13 Phase 1 deliverable.** "Landing page v1: hero, 'Recently rated'
  carousel, 'By domain' tile grid, methodology link."
- **§9 IA.** `/` is "Landing — movers, hot, trending, search". The first three
  arrive in Phase 3 (MoversBoard / SaturationCurve / `/trending`). For Phase 1
  the landing surfaces what the MVP can: recent rating actions and the
  taxonomy entry-points. Search is already in the global header (Unit 1.9).
- **§10.4 perf budget.** "LCP < 1.8s on slow 4G for the landing." Implies
  RSC-only with no client islands above the fold.
- **§10.1 design language.** "Restrained, academic-industrial. arXiv ergonomics
  meets Linear polish meets Bloomberg data density." No marketing-style hero
  illustration; a tight typographic header with a tagline and two CTAs.
- **Existing Phase 1 surfaces to reuse.**
  - `getIndexedProblems()` ([lib/content/load-problems-index.ts](../../lib/content/load-problems-index.ts))
    already joins `problems × ratings × taxonomy`; the only gap is "latest
    rating-action date" which is what §13 says to sort the carousel by.
  - The `/domains` tile grid ([app/domains/page.tsx](../../app/domains/page.tsx))
    landed in Unit 1.7. The handoff for Phase 1 explicitly calls for the
    landing page to **reuse** that section — extract it into a shared RSC
    rather than re-author the grid.
  - `<StatusPill>` (Unit 1.6).
  - `<SiteHeader>` (Unit 1.9) — already provides the global nav, search
    trigger, and dark-mode toggle. No header work belongs in this unit.

### Constraints

- **Pure RSC.** No `"use client"` anywhere on `/`. Every section must be
  build-time-rendered HTML so the LCP candidate (the hero `<h1>`) hits the
  network as static markup. The site header already mounts a client-only
  `<SearchTrigger>` and `<ThemeToggle>` — those are the only client islands
  on `/` and they live below the fold of the LCP element on a typical
  viewport.
- **No new client deps.** §13 Phase 1 says "carousel" but with ≤ 5 records
  in this phase a carousel UX is overkill — a static 1-2-column responsive
  grid surfaces the same content with zero JS. Track this as a deliberate
  Phase-1 simplification (see *Tradeoffs*); a real interactive carousel can
  land in Phase 2 once there are enough rating actions to warrant it.
- **Reuse, don't fork.** The "By domain" section must be identical to
  `/domains` so the design is consistent and one fix updates both. Extract
  `components/domain-tile-grid/` and call it from both places.
- **§13 sort key.** "Recently rated" → sorted by **latest rating-action
  date desc**, NOT `editorial.last_curated`. With one curated problem in
  Phase 1 these dates are coincidentally the same; add the field anyway so
  Unit 1.11's seeds slot in correctly.
- **`exactOptionalPropertyTypes: true`.** Optional fields on `IndexedProblem`
  must never be assigned `undefined`; only set them when present (existing
  pattern in `load-problems-index.ts`).

### Alternatives considered

- **Render the existing `/domains` tile grid via an iframe / inline next/link
  card.** Rejected — duplicates layout, drifts visually over time. Extracting
  a shared component costs one file.
- **Add `embla-carousel-react` for a real horizontal carousel.** Rejected
  for Phase 1 — adds ~9 kB gz and is not justified by the data density.
  Note this in the THINK doc so Phase 2 can revisit when MoversBoard +
  ratings dynamics arrive.
- **Sort by `composite` desc instead of by latest-rating date.** Rejected —
  §13 says "Recently rated", which is a recency surface, not a top-rated
  surface. The `/problems` index already supports composite sort.
- **Inline a one-shot `RatingRadar` thumbnail per row.** Rejected — radars
  are dense and don't read at thumbnail scale; a status pill + composite
  chip + tag row is what the IndexedProblem rows already use on `/problems`
  (Unit 1.8) and matches that convention.

### Edge cases

- **Zero rated problems.** `getIndexedProblems()` returns rows whose
  `points`/`composite`/`latestRatingDate` are all `undefined`; the
  "Recently rated" section filters on `latestRatingDate != null` and
  renders an empty-state message ("No rating actions yet — see
  `/contributing`"). Won't happen in Phase 1 (Unit 1.4 seeded one) but
  protects the page from going blank during pre-1.4 / future re-seeding.
- **Ties on `latestRatingDate`.** Stable secondary sort by title asc so
  the order is deterministic across builds (visual-regression baselines
  Unit 1.12 will lock).
- **Carousel = grid in Phase 1.** When Phase 2 adds 30+ rating actions,
  the section will need either pagination or a real horizontal scroller.
  Keep the markup simple now so the upgrade is local.
- **Methodology link.** Goes to `/methodology` (latest version) — the
  versioned-snapshot pages live at `/methodology/[version]` and shouldn't
  be the front-page link.

### Tradeoffs / defensible defaults flagged

1. **No real carousel widget in Phase 1.** Static grid sized for ≤ 5 rows.
   §13 reads "carousel" literally; the cost-benefit doesn't favour adding a
   client lib for so few rows. Surface in the unit summary so the user can
   redirect if they want the literal carousel UX before Phase 2.
2. **`latestRatingDate` precision.** ISO date string (`YYYY-MM-DD`),
   matched against `RatingActionSchema.date`. Ratings authored on the same
   day fall back to title order. Acceptable for Phase 1 where rating
   actions are rare; revisit if Phase 3 introduces sub-day cadence.
3. **No new tests.** The two new RSC components are thin compositions over
   already-tested helpers (`getIndexedProblems`, `taxonomy`). Add Storybook
   coverage in Phase 2 once the visual-regression tier is paying for
   itself; for now keep the unit tight.
