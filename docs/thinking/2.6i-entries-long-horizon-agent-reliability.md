### Sourcing decision per benchmark

#### `tau-bench` ✓ landed (3 entries)

- **Primary source**: [`sierra-research/tau-bench`](https://github.com/sierra-research/tau-bench)
  README, fetched this session via the raw GitHub URL. Sierra Research is
  the authoring org (Shunyu Yao); the README's leaderboard tables are the
  canonical rolling reference for the benchmark.
- **Metric selected**: `pass^4` (every-trial success across 4 independent
  attempts). The benchmark's declared metric in
  [`problem.yaml`](../../content/problems/long-horizon-agent-reliability/problem.yaml)
  is `pass^k`; `k=4` is the highest k the README reports across all rows,
  and the problem.yaml notes pin "k = 4–8" as the regime of interest.
  Choosing `k=4` makes the entries comparable on a single column and
  preserves the "central artefact" the problem highlights (the pass^1 →
  pass^k gap).
- **Entries** (3, from the retail and airline domain tables):
  - `claude-3-5-sonnet-20241022` on **retail**: pass^4 = **0.462**
  - `claude-3-5-sonnet-20241022` on **airline**: pass^4 = **0.225**
  - `claude-3-5-sonnet-20240620` on **retail**: pass^4 = **0.387**
- **Score scale**: decimals (0–1), preserving the README's verbatim
  format. The problem.yaml benchmark declaration doesn't pin a scale;
  preserving primary-source format is the §15.6 defensible default
  (consistent with the 2.6h precedent of preserving OpenAI's
  percentage format for SimpleQA).
- **Date semantics**: model snapshot date (when those specific weights
  became publicly available). Both claude variants embed the date in
  the model name (`20241022`, `20240620`).
- **`verified: true`**: each entry has explicit `protocol_notes`
  identifying model, domain, strategy, and the README URL. The
  protocol-notes branch of the leaderboard page's inclusive-OR
  definition is met. Scores are Sierra-attested, not independently
  replicated.
- **`paper_id: "2406.12045"`**: the τ-bench paper. Same convention as
  2.6h — `paper_id` identifies the benchmark-defining paper, with
  `protocol_notes` carrying model + source attribution.
- **Why two retail rows for the same model variant aren't included**:
  the README only reports one retail row per (model, strategy);
  re-running the same configuration is not what produces multiple rows.
  Two retail entries would be a contradiction with the source.

#### `swe-bench-verified` — deferred

- **Primary source candidate**: [swebench.com](https://www.swebench.com/)
  hosts the canonical leaderboard. `WebFetch` on this URL this session
  returned a JS-rendered shell with truncated leaderboard data; the
  numerical rows are loaded dynamically and not present in the static
  HTML.
- **Fallback candidates** (deferred):
  - Anthropic's model cards / blog posts — `WebFetch` on
    `https://www.anthropic.com/claude/sonnet` returned marketing prose
    with the benchmark image referenced but no inline numerics.
  - The SWE-bench paper's own tables (paper 2310.06770) — the paper
    itself reports baseline numbers from 2023 that are well below
    current SOTA; not the "headline" frontier scores users expect.
  - The `princeton-nlp/SWE-bench` GitHub repo's results README — same
    story, results lag the live leaderboard.
- **Unblocking condition**: a Playwright / Puppeteer-style scrape of
  `swebench.com/verified.html` (the leaderboard sub-page identified
  this session), or a direct fetch of the JSON endpoint the page
  loads from. Either is outside the scope of a single-session
  curator pass; a Phase-5-style leaderboard-ingest tool would do it.

#### `osworld` — deferred

- **Primary source candidate**: the OSWorld paper (2404.07972)
  abstract attests **"the best model achieves only 12.24% success"**
  against a human baseline of 72.36%. This number IS attestable to
  the primary source.
- **Why deferred anyway**: the abstract identifies the score with
  the generic "best model" / "state-of-the-art LLM/VLM-based agents"
  — no specific model name (e.g. "GPT-4o", "Claude 3.5 Sonnet") is
  given. The schema has no `model_name` field, but the protocol_notes
  field is where 2.6h carries model attribution. Minting an entry
  with `protocol_notes: "best model (unspecified) per paper abstract"`
  would leak a model-attribution gap into the rendered leaderboard.
  The same paper's full tables (or the
  [os-world.github.io](https://os-world.github.io/) leaderboard,
  also JS-rendered) would resolve this; both required for a clean
  attestation.
- **Unblocking condition**: same as swe-bench-verified — a JS-rendered
  leaderboard scrape, or full-PDF table read of the paper.

#### `re-bench` — deferred

- **Primary source candidates** examined this session:
  - The METR blog post `metr.org/blog/2024-11-22-...` and the paper
    abstract (2411.15114). Both report only **relative comparisons**:
    "best AI agents achieve 4× human score at 2-hour budgets" and
    "humans achieve 2× the top AI agent score at 32-hour budgets".
    No absolute normalized scores for any named model.
- **Why deferred**: the schema requires `score: z.number()`, not a
  ratio. Minting `score: 4.0` with `protocol_notes: "ratio vs human
  baseline at 2-hour budget"` is a metric-scale mismatch with the
  declared `success-rate` metric in problem.yaml. The full RE-Bench
  paper presumably has named-model normalized scores; the abstract
  alone does not.
- **Unblocking condition**: full-PDF read of paper 2411.15114, or a
  METR-published model-attributed score table.

### Constraints

- **§15.6 — no fabricated scores.** Every score in the entries.json
  file traces to a primary source. The 3 τ-bench scores are verbatim
  copies of cells in the `sierra-research/tau-bench` README's
  leaderboard tables; no other entries are landed.
- **`exactOptionalPropertyTypes: true`** in tsconfig. `protocol_notes`
  is set on every entry, so no key-vs-undefined ambiguity arises.
- **Date format**: schema requires `YYYY-MM-DD`. Both claude snapshot
  dates comply: `2024-10-22` and `2024-06-20`.
- **JSON, not YAML**: `content/problems/<slug>/entries.json` is JSON
  per the [load-entries.ts](../../lib/content/load-entries.ts) loader
  (`JSON.parse`). No YAML.
- **Cross-link audit must stay green.** All 3 entries have
  `paper_id: "2406.12045"`, which exists in `content/papers/`. The
  `entries-contributions-agreement` check
  ([lib/content/cross-link-audit.ts:273-292](../../lib/content/cross-link-audit.ts#L273-L292))
  verifies that paper 2406.12045's `contributions[]` includes a row
  for `problem_slug: long-horizon-agent-reliability` — it does (one
  row, `benchmark_id: tau-bench`). Check passes.
- **No collision with parallel session.** Parallel session has
  `content/papers/2306.15794.yaml` modified-but-not-staged (Unit 2.6e,
  HyenaDNA correction in progress). This unit creates
  `content/problems/long-horizon-agent-reliability/entries.json` —
  a different directory entirely. No file overlap.

### Alternatives considered

1. **Land all 4 benchmarks (one entry per benchmark, using whatever
   number is in the abstract).** Rejected per the 2.6h precedent: a
   `verified: false` flag does not lower the §15.6 bar; an unverified
   entry still needs an attestable source. Generic "best model" /
   ratio-only / abstract-mentioned scores fail attestability at the
   level a verified-flag-aware leaderboard should publish.
2. **Use `pass^1` instead of `pass^4` for the τ-bench column.**
   Rejected — `pass^1` is the easy column where models hit 0.46–0.69
   on retail; the problem.yaml itself says "pass^k (every-trial
   success) collapses well below pass@1 by k = 4–8" — the whole
   point of declaring `pass^k` as the benchmark metric is that
   `pass^k` for `k ≥ 4` is the meaningful figure. Shipping the easy
   column would undercut the problem's editorial framing.
3. **Use `pass^k` mid-range like `pass^3`.** Considered. `pass^4` is
   the highest k the README reports for every row I included; using
   `pass^4` makes the entries directly comparable. `pass^3` would
   also work but is one step less discriminating.
4. **Include both retail and airline rows for both claude versions
   (4 entries total instead of 3).** Considered. Trimmed to 3 to
   mirror 2.6h's "3-entry range coverage" pattern and to keep the
   leaderboard table compact. The two claude variants in different
   domains give the user a sense of both the model-snapshot drift
   (retail 0.387 → 0.462) and the domain spread (airline 0.225 vs
   retail 0.462) without overloading the table.
5. **Add a `gpt-4o` entry.** Rejected on attestation grounds: the
   τ-bench README rows label only `TC (gpt-4o)` without a date or
   snapshot identifier. The τ-bench paper was submitted 2024-06-17;
   `gpt-4o-2024-05-13` was the only public GPT-4o variant at that
   date, but locking that inference into a `date:` field would mix
   primary-source attestation with my own inference. Per the 2.6h
   precedent, drop entries where attribution is ambiguous.
6. **Use `osworld` despite the model-name gap (since the score IS
   attestable to the paper).** Rejected — the leaderboard page
   renders `protocol_notes` as the only model attribution; a row
   that reads "best model (unspecified) — 12.24%" is editorially
   weak. Defer until a named-model attestation lands.

### Surfaced findings (for future curators)

1. **3 of this problem's 4 benchmarks are blocked on JS-rendered
   leaderboard scrapes.** swebench.com, os-world.github.io, and
   (likely) METR's RE-Bench page all dynamically load their leaderboard
   data; `WebFetch` returns the static shell only. A Phase-5-style
   leaderboard-ingest tool with a headless browser would unblock all
   three at once.
2. **`gpt-4o` rows in the τ-bench README lack snapshot dates.** When
   the README's authors update model labels (e.g. `gpt-4o` →
   `gpt-4o-2024-08-06`), those rows become entry-eligible. Open a
   small follow-on watching the README diff.
3. **The τ-bench paper itself reports the `pass^k` table** (the README
   numbers come from the paper). Future curator passes could
   double-source the same entries by reading the paper PDF as a
   cross-check, but it's redundant for §15.6 — the README is the
   rolling source the benchmark's own authors maintain.
4. **OSWorld paper abstract DOES attest 12.24% / 72.36%.** A future
   curator who's willing to land an `unverified` entry with explicit
   "best model (unspecified) per arXiv abstract" attribution could
   ship the OSWorld row; this unit's bar is stricter. Documented here
   so the option is visible.
5. **RE-Bench paper's full tables likely include named-model scores.**
   A full-PDF read (or a fetch of the METR results landing page once
   it's not JS-rendered) would land a `re-bench` entry. The 4× / 0.5×
   ratios in the abstract are not directly mintable as `success-rate`
   numbers.
