# 16.5 — Public consumption: `imageOverride` avatar fallback chain + `getUserMetadataById` extension (all Phase-16 surface delivery complete)

Sixth Phase-16 unit; fourth (and final) code unit. Realizes
[ADR-0017](../adr/0017-image-storage.md) D-E (public consumption
fallback chain) across the two identity surfaces (`/u/{handle}` +
`/profile`) and extends Phase-15's `getUserMetadataById` helper
forward-compat for future Phase-16+ avatar-pill consumers. **All
Phase-16 surface delivery complete after this unit**; Units 16.6
– 16.8 are hygiene + acceptance.

## THINK

### What this unit ships

Three artifacts:

1. **`app/[locale]/u/[handle]/page.tsx` (edit)** — public profile
   avatar fallback chain updated to `imageOverride → image →
   omit` per ADR-0017 D-E.
   - New `currentAvatar = profile.imageOverride ?? profile.image
     ?? null` derived before render.
   - `<img>` switch from `profile.image` to `currentAvatar`.
   - `object-cover` CSS class added (handles non-square uploads).
   - **Email never surfaces on public surface** — ADR-0015 D-A
     invariant preserved (`profile.imageOverride` is
     user-controlled override of an already-public-on-github.com
     field).

2. **`lib/auth/login.ts` (edit)** — `getUserMetadataById` return
   shape extended:
   - SELECT clause gains `imageOverride: users.imageOverride`.
   - Return type extended from `{ githubLogin, displayName }` to
     `{ githubLogin, displayName, imageOverride }`.
   - Doc comment notes Phase-16 forward-compat for the
     Phase-14 Class B SiteHeader avatar-dropdown follow-on; no
     current consumer renders the image override in SiteHeader /
     AuthControl.

3. **`components/site-header/index.tsx` (edit)** — `safeUserMetadata`
   wrapper return type extended to match the new
   `getUserMetadataById` shape:
   - `Promise<{ githubLogin, displayName, imageOverride } | null>`.
   - No render-path changes (SiteHeader doesn't render avatar in
     Phase 16; image-override is plumbed through for future
     consumers).

### Why `/profile` avatar was already updated in Unit 16.4

The own-surface (`/profile`) avatar fallback chain landed in Unit
16.4 alongside the upload form (header `<img>` updated to
`currentAvatar = userRow?.imageOverride ?? session.user.image ??
null`). Unit 16.5 focuses on the public surface
(`/u/{handle}`) which Unit 16.4 didn't touch. The two surfaces
share the same architectural pattern but live in different files;
splitting the edits per surface preserves single-file-per-unit
hygiene.

### Why SiteHeader + AuthControl pill are UNCHANGED Phase 16

Per ADR-0017 D-E: "SiteHeader + AuthControl pill UNCHANGED
Phase 16 (no avatar pill currently; defer until SiteHeader
avatar-dropdown lands as Phase-14 Class B item)."

The Phase-14 Class B catalog flagged "SiteHeader avatar-dropdown
menu" as a follow-on. Phase 16 doesn't add an avatar there
because:
1. The current SiteHeader is text-only ("LLM OpenProblems" +
   nav links + ThemeToggle + LocaleToggle + AuthControl pill).
2. Adding an avatar pill would change the header's visual
   weight; warrants its own design review.
3. The avatar-dropdown follow-on couples to a broader UX
   refactor (mobile-nav variant; nav-link expansion) which is
   Phase 17+ scope.

`safeUserMetadata` plumbs `imageOverride` through anyway as
forward-compat: when SiteHeader gains an avatar pill (Phase 17+
or via the Class B item), it'll have the value available without
another DB round-trip.

### Why no test extension for `getUserMetadataById`

The `lib/auth/login.ts` module has NO test file (`lib/auth/login.test.ts`
doesn't exist). Phase 15 added `getUserMetadataById` without a
dedicated test file — the precedent was "thin Drizzle wrappers
don't get unit tests; their consumers do." Phase 16 follows that
precedent.

The new `imageOverride` field surfaces transitively through
SiteHeader → AuthControl typecheck (TS build verifies the
extended shape compatible with all consumers). End-to-end smoke
on the surface happens at build-time + manual exercise.

### Why no AuthControl pill update

The Phase-15 pill chain was `displayName → name → email →
fallback`. The pill renders TEXT only (no avatar). Phase 16
doesn't change the pill chain — image override is irrelevant
to text-only rendering. AuthControl pill stays Phase-15 shape
verbatim.

### Two-tier email-fallback semantics (preserved from Phase 15 + extended for Phase 16)

| Surface | displayName chain | image chain |
|---|---|---|
| `/u/{handle}` public | displayName → name → githubLogin → fallback | **imageOverride → image → omit** |
| `/profile` own | displayName → name → githubLogin → email → fallback | **imageOverride → session.user.image → omit** |
| AuthControl pill | displayName → name → email → fallback | N/A (text-only) |

**ADR-0015 D-A invariant**: email NEVER on public surface;
**ADR-0017 D-E invariant**: imageOverride supersedes
GitHub-derived image; if both null, render no avatar (omit).
