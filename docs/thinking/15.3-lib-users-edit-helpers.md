# 15.3 — `lib/users/` extension: edit helpers + validation + tests

Fourth Phase-15 unit; second code unit. Realizes
[ADR-0016](../adr/0016-user-editable-profile-fields.md) D-A field
set + D-B validation/sanitization contract at the helper layer.
Anticipated dependency for Unit 15.4 (edit form) + Unit 15.5 (public
consumption).

## THINK

### What this unit ships

Three new exports + two new constants in `lib/users/index.ts`, plus
`PublicProfile` interface extension to include the two new fields.
All tested in `lib/users/index.test.ts` (17 new tests; 30 total in
file; 497/52 file-suite total).

**New exports**:

```ts
export const MAX_DISPLAY_NAME_CHARS = 80;
export const MAX_BIO_CHARS = 280;

export function validateDisplayName(value: string): string | null;
export function validateBio(value: string): string | null;

export interface UpdateProfileInput {
  displayName?: string;
  bio?: string;
}
export async function updateProfile(userId: string, input: UpdateProfileInput): Promise<string | null>;
```

**`PublicProfile` interface extension**: adds `displayName: string |
null` + `bio: string | null` to existing 5 fields (userId, githubLogin,
name, image, createdAt).

**`getPublicProfileByHandle` extension**: SELECT clause adds the two
new columns; row mapping passes them through.

### Why this exact shape

**`validateDisplayName(s)` + `validateBio(s)` return null-or-string**:
mirrors Phase-11's `validateProposedValue` + `validateRationale`
pattern verbatim. Pure functions; auth-agnostic; DB-agnostic. The
edit-form server action (Unit 15.4) calls them before invoking
`updateProfile`.

**Empty-string-after-trim collapses to NULL**: per ADR-0016 D-B's
"empty string after trim = clear the field (store NULL)" contract.
`updateProfile` does:

```ts
const trimmed = input.displayName.trim();
set.displayName = trimmed.length === 0 ? null : trimmed;
```

Both NULL and empty render the fallback chain — but storing NULL
explicitly keeps DB queries clean (`WHERE displayName IS NULL`
distinguishable from `displayName = ''`).

**`updateProfile` returns `null | string`**: null on success, error
string on first failing field. Mirrors `validateProposedValue` +
`validateRationale` shape. Caller can surface the error string
inline; the validation error pattern Phase 11/12/13 already uses
for form-error rendering.

**Partial update semantics** (`undefined` field skipped, empty
field cleared):

| Input | Behavior |
|---|---|
| `{}` | no-op (returns null; no DB hit) |
| `{ displayName: "X" }` | set `displayName = 'X'`; `bio` untouched |
| `{ displayName: "" }` | set `displayName = NULL`; `bio` untouched |
| `{ displayName: "X", bio: "Y" }` | set both |
| `{ displayName: "a".repeat(81) }` | validation error; NO DB hit |

The atomicity guarantee: BOTH fields validate before the UPDATE
SET clause is built; partial-write-on-validation-failure avoided.

### Why `updateProfile` mocks `db.update` (not `db.transaction`)

Phase 9-12 helpers all use direct `db.update` (no transactions). The
update is a single SQL statement (`UPDATE user SET ... WHERE id = ?`);
no atomicity gain from wrapping in a transaction. Mirrors the watchlist
toggle + rating-challenges helpers.

### Why no `getOwnEditableProfile(userId)` helper

Considered + rejected. The edit form (Unit 15.4) already has the
session `userId` from `auth()` and uses Phase-12's `getLoginById` +
existing `users` SELECT in the profile page. The form pre-populates
the textareas from the existing `getPublicProfileByHandle` shape via
`displayName ?? ""` + `bio ?? ""` defaults. No new helper needed.

### Why NOT a separate `applyProfileEdit` server-action wrapper

The server action lives in `app/[locale]/profile/page.tsx` (Phase
10/11/12 precedent — server actions inline on the surface route).
Wrapping `updateProfile` in another layer adds indirection without
benefit; the surface route's own action handler is small and clear.

### `PublicProfile` interface extension

ADR-0016 D-A says displayName + bio are user-visible (the only
two fields the user controls). Adding them to `PublicProfile` is
required for Unit 15.5's public consumption to read them; signed-in
own consumption (Unit 15.4's preview surface) also reads them via the
same interface.

**Backward-compat for existing callers**: Phase-14's
`getPublicProfileByHandle` callers (`/u/{handle}` page + tests) read
fields via property access; adding two new optional-ish fields
(`string | null`) won't break consumers that don't read them. They'll
just receive an extended object.

### Test surface

17 new tests in `lib/users/index.test.ts`:

- **`validateDisplayName`** (4 tests): accepts empty, accepts normal,
  accepts exactly MAX, rejects MAX+1.
- **`validateBio`** (4 tests): accepts empty, accepts with newlines,
  accepts exactly MAX, rejects MAX+1.
- **`updateProfile`** (7 tests):
  - Returns null + writes both fields when both supplied.
  - Trims whitespace before storing.
  - Stores NULL when trimmed value is empty.
  - Updates only the supplied field when other is undefined.
  - Returns validation error + skips UPDATE when displayName too long.
  - Returns validation error + skips UPDATE when bio too long.
  - Is a no-op when no fields are supplied.
- **`PublicProfile` shape extension** (2 tests):
  - Returns displayName + bio fields from `getPublicProfileByHandle`.
  - Preserves null fields for users who haven't edited.

Plus existing Phase-14 tests (13) updated for the new `PublicProfile`
shape: fake row builders extended with `displayName: null` + `bio:
null` defaults.

Total file: **30 tests passing** (was 13).
