# 5.9 — `/digest` HTML hub page

## THINK

### Inputs

- **Unit 5.7 (`6c33ed9`)** ships `buildDigest({ domain, windowDays?, now? })` returning a typed `DigestPayload` per domain.
- **Unit 5.8 (`3a08fac`)** ships `/api/v1/digest/[domain]` RSS endpoints (one per taxonomy domain). Unit 5.8 noted that the deviation from the `.xml`-suffixed URL convention costs URL-side discoverability — Unit 5.9 is the compensation surface.
- **`app/ratings/page.tsx`** is the structural precedent for a feed-hub page: server component, exported `metadata`, container with header + content, Tailwind design-tokens.
- **Next.js metadata `alternates.types`** supports declaring per-feed `<link rel="alternate" type="application/rss+xml">` tags in `<head>` for RSS-reader auto-discovery.

### Scope (this unit)

1. **`app/digest/page.tsx`** — server component. Renders:
   - Header (title + description + per-page-level "Subscribe via RSS" copy).
   - Per-domain section: one block per `taxonomy.domains[]` entry showing the channel title, item count for the trailing-7-day window, top items (truncated to ~3), link to the RSS feed and link to each domain hub.
   - `metadata.alternates.types["application/rss+xml"]` declares one `<link rel="alternate">` per domain in the `<head>` block — RSS readers auto-discover.
2. **No tests**: page-level rendering tests aren't a Phase-3/4/5 pattern (existing precedents `app/ratings/page.tsx` / `app/trending/page.tsx` ship without dedicated tests). The underlying `buildDigest` + `renderDigestRss` are tested in Units 5.7 + 5.8.

### Constraints

- **Server component, no `"use client"`** — mirrors `app/ratings/page.tsx` / `app/methodology/page.tsx` / `app/digest`-adjacent hub-pattern.
- **Iterate over `taxonomy.domains` once** at module load. `buildDigest` is async; the page is async and `await`s a `Promise.all` over the domains.
- **Wall-clock `now`** for the digest builder — matches Unit 5.7 D-12 default. Page rebuilds at deploy time; subscribers see the most-recent build's snapshot.
- **`metadata.alternates.types`** computed from `taxonomy.domains` at module load (no `generateMetadata` needed since the route is static).
- **Top-N item preview per domain**: 3 items. Truncates the rest with a "View N more in the RSS feed →" call-to-action.
- **Empty-window message** when a domain has 0 items: render the `channelDescription` text directly, no "view more" link.
- **No client interactivity.** Pure SSR.

### Alternatives considered

- **Static enumeration of feeds** (no inline item preview). Rejected — defeats the purpose of an HTML hub; "preview + click through" is the canonical RSS-hub UX (e.g., Substack's `/archive` pages).
- **Cross-domain feed combination ("everything") link**. Considered. Rejected for v1 — §13's "per-domain" framing is explicit; a combined feed adds API surface without clear demand. The existing `/api/v1/rss.xml` (Unit 3.5) is already an everything-feed for rating actions, which subsumes most of the use case.
- **Include the digest builder's `cutoffDate` + `generatedAt` in the page** for transparency. Yes — added as a small footer line per domain.
- **Static `metadata` instead of `generateMetadata`**: static is sufficient because `taxonomy.domains` is evaluated at module load; no dynamic params needed.
- **Lighthouseci addition for `/digest`**: appropriate but **deferred to Unit 5.13** (acceptance gate) — bundling lighthouseci changes with the page itself splits the gate work.

### Tradeoffs / defensible defaults flagged

1. **Default window = 7 days** with wall-clock `now`. With Phase-3's simulated forward-dated rating actions (2026-09 / 2026-12), the page may show "no activity" for all domains on a 2026-05-15 build. That's the correct cadence framing per Unit 5.7's D-12 trade-off; an empty hub is honest.
2. **Top 3 items per domain in the preview**. Bigger N inflates the page; fewer N hides too much. 3 is a Twitter-card-norm.
3. **No item-kind icon / badge** in the preview. The `kind` is part of `DigestItem` but rendering it adds visual noise; the `title` already encodes the source (rating action vs. leaderboard entry are distinguishable by phrasing). Re-evaluate if user-research signals demand it.
4. **No accumulated-stats header** ("12 items across 5 domains this week"). Considered. Rejected for v1 simplicity; the per-domain counts cover it.
