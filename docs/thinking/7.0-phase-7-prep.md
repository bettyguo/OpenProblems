### D-3. i18n strategy (deferred to Unit 7.1 ADR-0011)

Three viable approaches for Next.js 15 App Router i18n:

1. **`next-intl`** ([next-intl-docs.vercel.app](https://next-intl-docs.vercel.app)) — third-party; mature; canonical App Router recommendation; sub-path routing built-in; ICU MessageFormat support.
2. **Paraglide.js** ([inlang.com/paraglide](https://inlang.com/paraglide)) — newer; TypeScript-first; tree-shakable per-message bundle; smaller bundle than next-intl.
3. **Native Next.js i18n routing** + custom translation system. No third-party; we own the translation lookup layer.

Lean (subject to ADR-0011 in Unit 7.1): **`next-intl`** for the translation runtime + **sub-path routing** (`/en/...`, `/fr/...`) for the URL convention. Reasoning: next-intl is the App Router-canonical recommendation; sub-path routing is crawler-friendly + bookmarkable; we don't need bundle-size heroics for a 2-locale project. Deferred to ADR-0011 for the realized contract.

### D-4. Bilingual content-storage shape (deferred to Unit 7.1 ADR-0011)

Three viable content-storage shapes for translated content:

1. **Sibling files**: `problem.yaml` + `problem.fr.yaml` in the same directory. Loader picks by locale.
2. **Frontmatter `lang:` field**: single file per (slug, locale) combo with a `lang: "en" | "fr"` discriminator.
3. **Sub-tree mirror**: `content/fr/problems/...` mirroring `content/problems/...`.

Lean (subject to ADR-0011 in Unit 7.1): **Sibling files** with a `.<locale>.yaml` / `.<locale>.mdx` naming convention. Reasoning: preserves directory layout (a curator reviewing all hallucination-reduction artifacts sees them in one place); makes the FR file's absence explicit at the file-system level; Velite globs accommodate the `*.fr.yaml` pattern via collection extensions. Deferred to ADR-0011 for the realized contract.

### Phase-7-blocking decisions deferred to per-unit implementation

- **D-7. `next-intl` version pin** (Unit 7.2). Stable major; pin via package.json.
- **D-8. Translation lookup format** (Unit 7.2). next-intl uses JSON-per-locale files (`messages/en.json`, `messages/fr.json`) by default; we adopt that.
- **D-9. Locale fallback chain** (Unit 7.2). Default: `fr` falls back to `en`. No partial-translation rendering (a page missing FR content renders the EN version with a locale-switch hint, not garbled mix).
- **D-10. Velite glob extension** (Unit 7.5). `*.fr.{yaml,mdx}` joins the existing collection globs; loader filters by locale at SSG time.
- **D-11. URL slug strategy for translated content** (Unit 7.3). Lean: slugs stay English-canonical (e.g. `/fr/problems/hallucination-reduction`); titles + body translate. Locale-specific slugs deferred — would require a slug-alias table.
- **D-12. SEO `<link rel="alternate" hreflang="...">`** (Unit 7.4). Add to layout when both locales of a page exist.
