### D-1. First-thread recommendation: **B.15 item 1 — Weekly digest scheduler + send template** via new ADR-0022 (Phase-30 B.15 carryover; tightest 1-phase follow-on; foundation→production-send loop closes)

**Recommendation**: ship **weekly digest scheduler + digest-send
email template** as Phase 31's keystone thread. Closes **Phase-30
B.15 item 1** at **1-phase age** — mirrors the Phase 21/23/25/27/29
tight-follow-on close pattern verbatim. Closes the foundation →
production-send gap that Phase 30's "weak UX without scheduler"
boundary statement flagged.

**Rationale**:

1. **Tightest follow-on close pattern**. Phase 30 surfaced B.15
   item 1 as the immediate next-step item; Phase 31 closes it at
   1-phase age. Matches Phase 21/23/25/27/29 cadence
   (Phase 20 → 21; Phase 22 → 23; Phase 24 → 25; Phase 26 → 27;
   Phase 28 → 29; Phase 30 → 31). Foundation phases consistently
   followed by their production-completion phase.
2. **ADR-0022 candidate slot claimed at 1-phase age**. Opened
   Phase 30 by ADR-0021 landing; Phase 31 claims it. **First new
   ADR since Phase 30** = 1-phase ADR-cycle interval (tied-tightest
   gap in project history; matches Phase 17→18 + 18→19 + 28→30).
3. **Closes Phase 30's "foundation-only weak UX" boundary**. The
   Phase 30 acceptance gate flagged: *"foundation-only Phase 30
   has weak UX without scheduler — subscribers can opt in but
   won't receive digest emails until Phase 31+ ships the
   scheduler."* Phase 31 closes that gap; Phase-30 subscribers
   start receiving weekly digests once Q75 + the Phase-31
   scheduler land.
4. **Reuses Phase 5 + 30 surfaces verbatim**. `lib/digest/build-digest`
   already builds the `DigestPayload` per domain (Phase 5 RSS
   path); `lib/email/sendEmail` ships the transactional email
   wrapper; `lib/subscribers/` ships subscriber row management.
   Phase 31 = compose these three surfaces into a scheduled
   per-domain send loop.
5. **No DB migration anticipated** (D-2). Scheduler state can be
   stateless (Vercel Cron triggers per cron expression; idempotency
   via send-window-key in subscriber-row `lastDigestSentAt`
   column — wait, that needs a migration). **TBD in Unit 31.1**:
   does the per-subscriber-per-domain send loop need a tracking
   column (`lastDigestSentAt` on subscriber, OR a new `digest_send`
   audit table) to prevent double-sends? Either way: 0-1 migration
   anticipated.
6. **One new env var anticipated**: `CRON_SECRET` (HTTP
   authorization shared secret between Vercel Cron and the
   `/api/v1/cron/digest-send` endpoint; standard Vercel Cron
   pattern). Env count 11 → 12.
7. **Bundle invariant preservation**. New cron endpoint is
   server-only API route; email-send is server-side; digest-send
   template is plain string (matches Phase 30 template pattern).
   **First Load JS expected UNCHANGED at 103 kB** (40th
   consecutive unit if preserved).
