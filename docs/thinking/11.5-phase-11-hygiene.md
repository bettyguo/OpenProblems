### Class B — Phase-11-specific follow-ons (12)

Items that survive Phase 11 because they're either scope-creep (out
of the Phase-11 "rating-challenge submission MVP" framing per Unit
11.0 D-5), waiting on an architectural unblock that wasn't granted
this phase, or tactical hygiene whose threshold hasn't been crossed.

1. **Curator review pipeline** (Q57 — to be surfaced in Unit 11.6).
   Status transitions (`submitted → under_review → accepted | rejected
   | withdrawn`) + the corresponding curator-review columns
   (`reviewedAt`, `reviewerId`, `reviewNotes`, `acceptedActionId`)
   land in Phase 12+ migration. Phase 11 ships submissions only; the
   curator workflow is out-of-band today (curator can `pnpm db:studio`
   or query the DB directly to inspect submitted challenges; reviewing
   is a manual offline activity until the pipeline lands).

2. **Public challenge visibility** (Q58 — to be surfaced in Unit 11.6).
   Today only the submitter sees their challenges (on `/[locale]/profile`).
   Public visibility — e.g., a counter on the problem detail page
   ("3 active rating challenges") OR a section listing all open
   challenges with rationale snippets — is Phase 12+. The architectural
   shape is straightforward (the DB query already supports it via
   `getUserChallenges` generalized to `getChallenges({ problemSlug })`);
   only the policy decision (who-sees-what) is gated.

3. **Form-state preservation on validation error** (Unit 11.3 tradeoff).
   Today a server-action validation failure `redirect()`s back to the
   problem page with `?challenge_error_field=...&challenge_error_message=...`,
   clearing the form's input values. The user retypes. Phase 12+
   migration: `useActionState` client island + form-state echo in the
   error response — would cost ~3-5 kB client bundle but materially
   improve the rare-error UX. Defer until usage volume signals it.

4. **Per-dimension dynamic input format** (Unit 11.3 design call).
   Today a single text input accepts any string + the server validates
   per-dimension format on submit. A dimension-driven conditional
   input (select for difficulty grades; number input with min/max for
   stars-based; toggle for saturation N/A) would catch errors
   client-side before submit + improve UX. Couples to item 3 (both
   need a client island); same defer-until-volume framing.

5. **Withdraw-own-challenge UI** on profile page. Today the user can
   see their submitted challenges but cannot withdraw one. The
   architectural shape: `POST /api/v1/rating-challenges/[id]/withdraw`
   (or `DELETE /api/v1/rating-challenges/[id]`) + inline withdraw
   button on the profile-page row + status transition
   `submitted → withdrawn` (which means item 1's status enum lands as
   a partial). Phase 12+; the withdraw surface is small enough to be
   a 2-3 unit sub-phase.

6. **Per-challenge detail page** (`/[locale]/profile/challenges/[id]`
   or similar). Today the profile-page list shows a truncated
   rationale preview (`RATIONALE_PREVIEW_CHARS = 200`); the full
   rationale text isn't surfaced anywhere. Phase 12+; small route
   addition once the per-challenge URL space is needed (e.g., for
   curator review URLs that link to specific challenges).

7. **Search/filter on profile-page challenges list**. Premature without
   volume — Phase 11 has at most a handful of challenges per user.
   Phase 12+ if the list grows past ~20 items.

8. **Email notifications when challenge is reviewed**. Couples to the
   email subscriber-list thread (Phase-5 D-4 punt). The notification
   trigger (status change from `submitted → reviewed`) requires both
   item 1's curator review pipeline AND `lib/email/` infrastructure.
   Phase 12+ at the earliest.

9. **Rate-limiting on `POST /api/v1/rating-challenges`** (Unit 11.0
   D-10 explicit deferral). Per-user surface; auth-gated; per-user
   damage bounded. Phase 12+ when broader unauthenticated surfaces
   land (e.g., public submission forms).

10. **Composite index on `(userId, createdAt DESC)` for
    `ratingChallenge`** (Unit 11.0 D-3 deferral). Premature without
    volume; SQLite plans the LIMIT 50 query fine without an explicit
    index at Phase-11 scale. Phase 12+ when challenge count per user
    grows.

11. **Orphan-row cleanup script for `ratingChallenge`** (mirrors
    `watchlist` orphan policy per ADR-0013 D-F). Phase 11 inherits the
    same cleanup-deferred policy: rows pointing at deleted
    `content/problems/<slug>/problem.yaml` files are tolerated until a
    cleanup script lands. The script can handle both tables in one
    pass (same JOIN-NOT-IN shape per table). Phase 12+ curator-track
    or ops-script unit.

12. **Curator dashboard for reviewing pending challenges**. Out of
    Phase-11 scope. The curator's review surface — likely
    `/[locale]/curator/challenges` or similar — needs item 1's status
    transitions + an admin-protected route. Phase 12+ when the curator
    review pipeline lands.
