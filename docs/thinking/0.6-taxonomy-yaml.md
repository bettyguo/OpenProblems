# Unit 0.6 — `content/taxonomy.yaml` full population

> Stage 1 (THINK) artifact. Critical-path per §15.4 — proceeding under the unit-rhythm rule.

## Problem restatement

Author `content/taxonomy.yaml` covering all eight top-level domains from MASTER_PROMPT.md §4 in the **expanded** form (every subdomain as `{ id, title }`), validated by `TaxonomySchema` (Unit 0.5). Two of the eight domains were given in expanded form in §4; six were given as compact id arrays — those need title expansion now.

The taxonomy is **stable for Phase 1** (§4). Future renames go through `redirects.yaml`. Schema and `validate-content` (Unit 0.7) enforce shape; this unit produces the only canonical instance.

## Inputs

- MASTER_PROMPT.md §4 (eight domains + subdomains, mixed expanded/compact form).
- `lib/schemas/taxonomy.ts` (Unit 0.5) — `TaxonomySchema`, `DomainSchema`, `SubdomainSchema`. `superRefine` on `DomainSchema` rejects duplicate subdomain ids within a domain; on `TaxonomySchema` rejects duplicate domain ids. Cross-domain subdomain-id collisions are *allowed* (OPEN_QUESTIONS Q11).
- §6 (rules: slugs URL-stable; renames go via `redirects.yaml`).

## Constraints

- Match §4 ids verbatim — these are URL primitives, breaking them later is breaking URL stability.
- Titles for compact-form subdomains are inferred from the id and §4 context. Where §4 used short hyphenated ids that compress two concepts (e.g., `online-active-bandits`, `transfer-multitask-meta`), expand the title to spell the concepts out.
- "Everything Else" appears once per domain as `<domain-id>-other`. Title is "Everything Else" uniformly.
- Use the verbose YAML form (`- id: foo\n  title: Bar`) consistently. No compact inline `{id: foo, title: Bar}` mixing — author tooling and diffs are cleaner.
- Quote any title containing a comma or `&` to avoid YAML parser ambiguity.

## Alternatives considered

1. **Expanded `{ id, title }` for every subdomain** (chosen). Matches schema (which requires both fields).
2. **Compact id-only form**, keeping the schema flexible to accept either. Rejected: schema flexibility on this canonical file invites drift; one form is one form.
3. **Generate `content/taxonomy.yaml` from a TS source** (`lib/taxonomy.ts`). Tempting, but YAML is the master prompt's specified source format (§4, §6) and round-tripping through TS introduces a translation step the curators don't need.

## Edge cases & failure modes

1. **Cross-domain subdomain-id collisions** (`representation-learning` ∈ `deep-learning` + `general-ml`; `robustness` ∈ `deep-learning` + `social-aspects`). Schema allows; the duplicate-detection superRefine is *within-domain* only.
2. **`theory` is both a top-level domain and a subdomain id** under `deep-learning`. The domain has id `theory`; the subdomain has id `theory` under `deep-learning`. URL space `/domains/[domain]/[subdomain]` keeps these distinct.
3. **Title escapes.** Several titles contain `,`, `&`, or `:`. YAML requires quoting around `:` and around values beginning with `&`. Use double-quoted strings everywhere a literal would be ambiguous.
4. **`social-aspects` has its own `robustness` subdomain** distinct from `deep-learning/robustness`. Both kept; cross-link is a content-authoring concern.
5. **§4 has typos in the compact arrays** (none that I can see, but the ids must match §4 byte-for-byte — verified during authoring).
6. **Diff stability.** Each subdomain on its own line so future additions show as one-line additions, not block edits.

## Open questions surfaced

None new. Q11 (subdomain-id collisions) remains; this unit ships the configuration that depends on its current resolution.

## Decision

Write `content/taxonomy.yaml` with all 8 domains, each subdomain `{id, title}` on its own pair of lines. Install `js-yaml` + `@types/js-yaml` (devDeps; Unit 0.7 needs them anyway) and add a fixture-style test under `lib/schemas/taxonomy.test.ts` that loads the actual `content/taxonomy.yaml` from disk and validates it against `TaxonomySchema`. The test fails loud if either (a) the YAML is malformed or (b) the YAML drifts from the schema — making this the first piece of file-level cross-validation in the project.

Stage 2 (DESIGN): the file layout is the design — there is no architecture in this unit, only careful transcription.

Stage 3 (CODE): write the YAML, install js-yaml, write the fixture test, run vitest.

Stage 4 (ITERATE): reviewer-checklist, CHANGELOG, commit.
