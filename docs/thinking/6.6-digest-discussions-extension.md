# 6.6 — `lib/digest/build-digest.ts` extension (fold discussion threads into per-domain RSS)

Fifth code unit of Phase 6. Extends the Phase-5 `buildDigest` (Unit 5.7) to include recent Discussion-thread activity alongside the existing rating-action + leaderboard-entry items in the per-domain weekly RSS payload. Consumes Unit 6.2's `getRecentDiscussionActivity` via a new env-safe wrapper.

## THINK

### Inputs

- **Unit 6.0 D-5** committed to "extend Phase-5 Unit 5.7 to include new discussion threads in the weekly RSS roll-up." Half-fulfilled by Unit 6.5 (cards); this unit fulfills the digest half.
- **Unit 6.2** ships `getRecentDiscussionActivity(since, options)` returning `RecentActivityItem[]` — repo-global (not per-domain). The activity items carry `title` which (per Giscus's `mapping="pathname"` from ADR-0010 D-C) IS the pathname `/problems/<slug>/talk` at creation time.
- **Unit 6.5 precedent**: `tryGetDiscussionByPath` env-safe wrapper. Same pattern transfers — wrap the bulk activity fetch in `tryGetRecentDiscussionActivity` returning `[]` on any error.
- **`lib/digest/build-digest.ts`** (Phase 5 Unit 5.7) ships a clean `BuildDigestOptions` shape + `DigestItem` discriminated union. The extension adds a third `kind`.
- **`lib/digest/build-digest.test.ts`** uses fixture-driven testing against the real-data Velite output. Adding a `discussionsLoader` injection lets the test exercise the discussion-path without a real GraphQL call.
- **`lib/digest/rss.ts`** + **`app/api/v1/digest/[domain]/route.ts`** consume `DigestItem` and render to RSS. The shape is stable; adding a new `kind` value works without RSS-side edits as long as the item carries `title`/`link`/`date`/`description`/`guid` (which it will).

### Scope (this unit)

Three file changes + one helper add:

1. **`lib/discussions/github-graphql.ts`** — add `tryGetRecentDiscussionActivity(since, options)` env-safe wrapper mirroring `tryGetDiscussionByPath`. Returns `[]` on any error (missing `GITHUB_TOKEN`, network, GraphQL).
2. **`lib/discussions/github-graphql.test.ts`** — add 2-3 tests for `tryGetRecentDiscussionActivity` (identity passthrough, env-unset → `[]`, error → `[]`).
3. **`lib/digest/build-digest.ts`** — extend the digest builder:
   - Extend `DigestItem["kind"]` to include `"discussion"`.
   - Add `discussionToItem(activityItem, problemTitle)` helper that maps `RecentActivityItem` → `DigestItem`. Item shape: `title = "<problemTitle> — N new comment(s) in discussion"`, `link = "/problems/<slug>/talk"`, `date = lastActivityAt` (or `updatedAt`), `description = "Discussion thread for <problemTitle>: N comments; last activity <date>"`, `guid = "discussion:<discussionId>"`.
   - Add `discussionsLoader?: (since: Date) => Promise<RecentActivityItem[]>` to `BuildDigestOptions`. Default `() => tryGetRecentDiscussionActivity(since)`. Tests inject fixtures.
   - Inside `buildDigest`: call the loader with the cutoff date as `since`. For each returned item, parse the title against `^/problems/([a-z0-9-]+)/talk$` regex. If the slug matches AND is in the domain's problem set AND the item's `updatedAt` is in the window, include it. Skip otherwise.
   - Update `channelDescription` to mention discussion items when present in the channel-non-empty branch.
4. **`lib/digest/build-digest.test.ts`** — add 3-4 tests for the discussion-flow path:
   - With injected loader returning a matching activity item → item appears in `items` with `kind: "discussion"`.
   - With injected loader returning an item for a problem outside the requested domain → filtered out.
   - With injected loader returning an item whose title doesn't match the pattern (e.g., a manual non-problem discussion) → filtered out.
   - Default (no injection) → no crash; works with the real `tryGetRecentDiscussionActivity` which returns `[]` without env.

### Scope NOT in this unit

- **Per-thread comment-count cap on the digest item** (e.g., "5 new comments since last digest") — would require tracking per-build state. Out of scope; the digest item shows last-activity-at + total count, not the delta.
- **Real-API integration smoke for the digest path** — same Q47 gating as Unit 6.2/6.5; deferred to Unit 6.10 acceptance gate.
- **`<managingEditor>` on the RSS feed** — Q44 still gated on Q2 (DNS / project email). Unchanged.
- **Subscriber list / email delivery** — Phase-5 5.0 D-4 deferral still holds.

### Constraints

- **§5.5 perf budget** — server-side only; no client-bundle impact.
- **§14.2 testing** — new tests use the `discussionsLoader` injection pattern; mirror the Phase-5 `fetchImpl` / `clientFactory` precedent.
- **`exactOptionalPropertyTypes`** — `options.discussionsLoader?: ...` follows the "assign-only-when-defined" pattern in the builder.
- **Build-time env behaviour**: when `GITHUB_TOKEN` is unset, the build-time fetch returns `[]`; no discussion items in the digest. Same graceful fallback as Unit 6.5's card badges.
- **Stable `guid`** — `discussion:<discussionId>` uses the GitHub Discussion node ID (a `D_xxxxxx` string) which is stable across renames and edits per GitHub's GraphQL contract. RSS readers de-duplicate on guid.

### Alternatives considered

- **Fetch discussions per-problem (fan-out)** instead of repo-global. Rejected — N+1 problem (1 query per domain × N problems vs. 1 query for the whole repo); GitHub's GraphQL rate budget appreciates the bulk pattern.
- **Skip the regex-parse-the-title trick; pass the slug back from Giscus via a separate metadata channel**. Rejected — Giscus's pathname mapping encodes the slug IN the title by design; the regex IS the contract. A future Giscus version could break this (would surface as zero discussion items in the digest; not catastrophic).
- **Always include manual non-problem discussions** as a separate digest section. Rejected — the digest is **per-domain** (scoped to that domain's problems); manual discussions are out of scope. A separate "site-wide community digest" could land in a future unit.
- **Show discussion items only when count > some-threshold** (e.g., > 5 comments). Rejected — the digest is curator-targeted; signal-vs-noise is acceptable at any count. Visitors filter via the activity-badge surface on cards (Unit 6.5).
- **Hard-code the regex pattern in `build-digest.ts`** vs. exporting it from `lib/discussions/github-graphql.ts`. Going with the latter (export as `TALK_PATHNAME_REGEX`) — keeps the pathname convention in one place; future units that need to parse the title can reuse.

### Tradeoffs / defensible defaults flagged

1. **Repo-global fetch (not per-domain)** — wastes a tiny bit of work parsing items from other domains. Acceptable; 50-item bulk fetch is one call.
2. **`since` parameter aligns with the digest window's cutoff date** — same date semantics as `Date.parse(cutoffDate)`.
3. **Title-regex match is strict** — manual discussions with non-pathname titles are filtered out (correct; only auto-created pathname-mapped discussions belong in a per-domain problem digest).
4. **Default loader is the env-safe wrapper** — never throws; if `GITHUB_TOKEN` is unset, just yields no discussion items.
5. **No retry/backoff** — same as Unit 6.2; one-shot at build time.
6. **The activity-item's `lastActivityAt` is the `date` in the digest** — newest-first sort by `date` means new comments float up alongside new rating actions + new entries.
