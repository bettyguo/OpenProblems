**Framework-only strategy** (Phase-34 gate option (b) verbatim):
"ship Q68 framework as architectural commitment now without external
API." Defines the `ContentModerator` interface + ships `NoopModerator`
default + wires extension points into bio-write + avatar-upload +
rating-challenge-text + subscribe-email surfaces — without committing
to OpenAI moderation API, Perspective API, or a regex/wordlist
provider. The provider commitment is an operational-API-gate parallel
to Q73 / Q75 / Q77: deferred until a real abuse signal arrives.

**Demand-signal-first concern resolution.** Q68 was deferred 12+
phases on the rationale "no users yet → no abuse signals → no demand."
The framework-only-with-no-op design dissolves this concern: the
no-op default imposes **zero API cost, zero latency, zero
false-positive rate**, so shipping the framework now does NOT pre-pay
for unobserved abuse. It pre-positions the extension point so that
when a real signal does arrive (Phase 36+ post-production), the swap-
in is a single env-var change + one new file per provider, not a
multi-surface refactor across bio-write + avatar-upload + rating-
challenge + subscribe.

**Rationale**:

1. **Strongest non-just-surfaced patience signal** at 12+ phases
   carryover. Phase 34 just resolved a 0-phase Q-carryover (Q79); the
   natural counterweight is to ship the longest-pending architectural
   commitment.
2. **Framework-only ADR is a NEW architectural pattern for this
   project.** Every prior ADR pinned a concrete option (ADR-0017
   Vercel Blob; ADR-0021 Resend; ADR-0022 Vercel Cron; ADR-0023
   Option A FK). ADR-0024 framework-only would be the **first ADR in
   project history to commit to an interface without committing to a
   provider in the same ADR**. Establishes the playbook for future
   framework-first decisions (Q66 markdown evolution; Q70 EXIF
   stripping; possibly Q64 privacy opt-out).
3. **Multi-surface coverage from one ADR.** Phase 35 wires extension
   points into FOUR existing surfaces simultaneously: `users.bio`
   write path (ADR-0016 D-B), avatar-upload pipeline (ADR-0019 D-F),
   rating-challenge text submission (Phase 11), and subscribe-email
   submission (Phase 30 ADR-0021). Single architectural unit;
   multi-surface integration unit; hygiene unit; gate. Anticipated
   4-5 units.
4. **No-op default = zero behavioral change for existing tests.**
   `NoopModerator.moderateText(t) → { ok: true }` /
   `NoopModerator.moderateImage(b) → { ok: true }` means all 722
   existing tests pass unchanged. New tests verify (a) the default is

   refusal propagates correctly through each surface.
5. **Operational-API-gate pattern reused.** `MODERATION_PROVIDER` env
   var defaults to `noop`; documented values `openai` /
   `perspective` / `regex-wordlist` carry "Phase 36+ if signal"
   markers per the established Q73 / Q75 / Q77 pattern (operational
   gates curator-track / deploy-gate, not phase-blocking).
6. **Release-readiness audit alignment.** Per the audit shipped this

### D-3. ADR-0024 framework shape anticipated (Unit 35.1)

D-A. **Interface** = `ContentModerator` in `lib/moderation/types.ts`:

```ts
export type ModerationResult =
  | { ok: true }
  | { ok: false; reasons: ReadonlyArray<string>; severity: "block" | "warn" };

export interface ContentModerator {
  moderateText(text: string, ctx: ModerationContext): Promise<ModerationResult>;
  moderateImage(buffer: Buffer, ctx: ModerationContext): Promise<ModerationResult>;
}

export interface ModerationContext {
  surface: "bio" | "avatar" | "rating-challenge" | "subscribe";
  userIdOrEmail: string;
}
```

D-B. **Default provider** = `NoopModerator` in `lib/moderation/noop.ts`:
both methods return `{ ok: true }` synchronously-wrapped-in-promise.
Zero API cost, zero latency, zero false positives. Default when
`process.env.MODERATION_PROVIDER` is unset or `noop`.

D-C. **Factory** = `getModerator(): ContentModerator` in
`lib/moderation/index.ts`. Reads `process.env.MODERATION_PROVIDER`,
dispatches to `NoopModerator` (default), throws clear error on
unknown values. Future providers (`OpenAIModerator` /
`PerspectiveModerator` / `RegexWordlistModerator`) wire in as
additional dispatch arms when the operational-gate signal arrives.
Singleton instance cached per-process.

D-D. **Integration points** (Phase 35 wires all four; each surface
ships a single-line `await getModerator().moderate*(…)` call
followed by refusal-handling):

1. **`users.bio` write path** — `app/api/v1/profile/update/route.ts`
   (or wherever the profile-update server action lives per Phase 15).
   Refuse 422 on `{ ok: false; severity: "block" }`; surface
   moderation reason to the user via the existing error-toast surface.
2. **Avatar upload pipeline** — between `.rotate()` and `.toBuffer()`
   in `lib/images/transcode.ts` per ADR-0019 D-F inheritance contract.
   Refuse 422 on block; same toast surface.
3. **Rating-challenge text submission** — `app/api/v1/rating-
   challenges/route.ts` (Phase 11 surface). Refuse 422 on block;
   surface to the rating-challenge submission form.
4. **Subscribe-email submission** — `app/api/v1/subscribe/route.ts`
   (Phase 30 ADR-0021 surface). The email itself is the moderated
   value (`moderateText(email, { surface: "subscribe", … })`); the
   no-op default makes this a no-cost integration in the canonical
   case. Refuse 422 on block; existing rate-limit-style refusal
   posture per ADR-0021.

D-E. **Test surface** = `lib/moderation/noop.test.ts` (verifies
no-op returns `ok: true` for both text + image methods) +
`lib/moderation/factory.test.ts` (verifies factory dispatches
correctly + caches singleton + throws on unknown provider) +
per-surface integration tests in each of the four wired-surface
test files (each adds 2-3 tests: default-noop-passes-through +
refusal-propagates-correctly using a test-double moderator). Total
anticipated test delta: **+10-14 tests across +4-5 files**.

D-F. **Env var** = `MODERATION_PROVIDER` (12 → 13 env vars).
Documented in `.env.example` with default `noop` + commented future
values `openai` / `perspective` / `regex-wordlist`. Per the
operational-API-gate pattern, provider commitments are Phase-36+
curator-track decisions.

D-G. **Phase 36+ deferrals** (explicit in ADR-0024 D-H):
- Concrete provider implementations (`OpenAIModerator` /
  `PerspectiveModerator` / `RegexWordlistModerator`).
- Persistent moderation-decision audit log (new
  `moderation_decisions` table + migration).
- Per-user appeal mechanism for blocked content.
- Bulk-rescan of historical content under a newly-enabled provider
  (mirrors the Phase-20 backfill pattern flagged in Q68 expansion
  documentation).
- Moderation-policy versioning + curator-facing rule-set editing.
- Per-surface threshold tuning (block-vs-warn severity calibration).

D-H. **What this ADR does NOT decide**: which provider to use; what
the moderation policy text is (verbatim provider docs vs. project-
specific); appeal mechanism; audit-trail retention period; bulk-
rescan cadence. All deferred to Phase 36+ + curator-track.

### Phase-35-blocking decisions deferred to per-unit implementation

- **D-9** (Unit 35.1): exact interface shape — single `ContentModerator`
  with text + image methods vs. separate `TextModerator` +
  `ImageModerator` interfaces; lean = single interface with both
  methods (parallels how `Resend` SDK exposes a single client; simpler
  factory dispatch).
- **D-10** (Unit 35.1): severity field shape — `block | warn` two-state
  vs. numeric score 0-100 vs. provider-specific labels passed through;
  lean = `block | warn` two-state (semantically clear for callers;
  providers map their internal scores to this).
- **D-11** (Unit 35.2): surface-specific refusal UX — generic
  "moderation refused" error vs. per-surface custom messaging; lean =
  generic for Phase 35 (per-surface custom messaging is Phase 36+
  polish; consumer-facing copy is i18n-keyed centrally).
- **D-12** (Unit 35.2): rating-challenge submission surface — moderate
  `rationale:` field only vs. all submitter-text fields; lean =
  `rationale:` only (largest text surface; other fields are
  short-strings less likely to host abuse).
- **D-13** (Unit 35.2): subscribe-email integration — moderate the
  email address (treating it as text-with-domain) vs. skip entirely
  (email validity is already enforced by `validateEmail`); lean =
  skip-entirely (the framework wires the call but the no-op default
  is the right Phase 35 posture; flipping a future regex-wordlist
  provider to "block emails containing slurs" is a one-line change
  in the provider, not a wiring change).
