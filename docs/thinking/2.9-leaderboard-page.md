# 2.9 + 2.10 (combined) — Per-problem leaderboard page + entries scaffold

## THINK

### Inputs

- **§13 Phase 2 deliverables (combined here).**
  - 2.9: "Per-problem leaderboard page with sortable, filterable table; verified-flag rendering."
  - 2.10: "`entries.json` per problem; per-entry verified flag rendered."
- **`LeaderboardEntrySchema`** (`lib/schemas/entry.ts`, Unit 0.5):
  `{ paper_id, benchmark_id, score: number, date, verified, protocol_notes? }`.
  Note `score: z.number()` — not optional, not `TODO(curate)`-able. Real
  numbers required per §15.6.
- **`validate-content`** already handles `content/problems/<slug>/entries.json`
  (Unit 0.7), so the shape layer is ready.
- **Cross-link audit** (Unit 2.11) has the `entries-contributions-agreement`
  check that activates automatically when entries land.

### Constraints

- **§15.6: never invent benchmark numbers.** Real entries content
  requires curated SOTA scores attested to a published source per
  entry. Cannot author at scale without a curator pass; that work
  belongs to PAPER-INGEST-style curation runs.
- **§13 acceptance is met without entries content.** Unit 2.13
  satisfied the §13 acceptance gate (audit green + visual baselines);
  empty entries.json is not a gate criterion.
- **Combined commit.** 2.9 (page) and 2.10 (entries scaffold) are too
  tightly coupled to ship usefully alone — the page needs an entries
  reader. Combining as `2.9` for the route surface gives a single
  commit that:
  1. Replaces the Unit 0.10 stub at `/problems/[slug]/leaderboard`
     with a real page.
  2. Wires a small `lib/content/load-entries.ts` fs helper.
  3. Renders "no entries yet" for every problem at this commit (the
     honest current state).
- **No Velite-collection wiring for entries.** The validate-content
  + cross-link-audit pipelines already read entries.json via fs;
  adding a Velite collection now would be unused infrastructure
  until content lands. Skip; revisit in Phase 3 when entries content
  arrives via curator work or a leaderboard-ingest tool.
- **Generate static params over all 10 problems.** Even though every
  page renders an empty leaderboard at this commit, prerendering
  them surfaces the route in Lighthouse/Playwright and the page
  goes live with content as curators populate `entries.json` per
  problem.

### Page layout

Single page consuming `loadProblem(slug)` + `loadEntriesForProblem(slug)`:

1. **Breadcrumb + header.** `Problems / <problem.title> / Leaderboard`.
   The breadcrumb links the problem name back to the §9 detail page.
2. **Source note.** Plain sentence: every score traces to its paper's
   `evidence:` URL (per §10.2 "every quantity is sourced").
3. **Sortable table.** Columns Paper · Score · Verified · Date ·
   Protocol notes. Sorted by score desc (direction-aware per
   `benchmark.metric_direction`) when the parent problem has
   benchmarks; alphabetical fallback when not.
4. **Empty-state copy** (the Phase-2 norm at this commit). Names
   the next step: curators populate per-problem `entries.json`.
5. **Verified-flag rendering.** `verified: true` → green check pill;
   `verified: false` → muted "unverified" pill. The §13 deliverable
   wording explicitly calls for this.

### Alternatives considered

- **Ship 2.9 (page) and 2.10 (entries) separately.** The 2.0 plan
  ordered them this way; combining is more honest about coupling.
- **Author 1–2 sample `entries.json` files with curated SOTA from
  the public abstracts.** Tempting but risks §15.6 violations — even
  named benchmarks have shifted numbers over time. Cleanest: ship
  the page empty, let curators populate.
- **Read entries via Velite instead of fs.** Validate-content +
  cross-link-audit already read via fs; the page mirrors that to
  avoid a second source of truth. When Velite-entries-as-data lands
  in Phase 3, the page can swap importers in a one-line change.
- **Add a "verified-flag legend" component.** Trivial; inline a
  legend caption above the table.

### Edge cases

- **`entries.json` doesn't exist (current norm).** `loadEntriesForProblem`
  returns `[]`. Page renders the empty-state copy block.
- **`entries.json` is malformed.** validate-content catches it at
  build time; the page reader uses `safeParse` and surfaces a
  Phase-2-friendly error if it slips through.
- **No `paper.contributions[].score`.** Schema is `score?: number`;
  the page doesn't reach into contributions for leaderboard data —
  entries.json is the source of truth for scores. Different surface.

### Tradeoffs / defensible defaults flagged

1. **Page goes live empty.** Acceptable; the §13 gate (Unit 2.13)
   doesn't require entries data. Phase-3 ratings-dynamics units
   populate entries via curator runs.
2. **Sort direction defaults to "higher is better".** Most ML
   benchmarks; specific problems with `lower-is-better` metrics
   (e.g. WeatherBench RMSE) need per-benchmark sort logic when
   content lands. Stub the sort key now; refine on first non-empty
   entries.json.
3. **No tests.** No live data to test against; the fs reader is a
   thin glob. Add a fixture-based test for `loadEntriesForProblem`
   in the same commit alongside the audit fixtures from Unit 2.13.

### What this commit does NOT do

- Author real `entries.json` content. That's PAPER-INGEST-class
  curation per §15.6.
- Wire a Velite `entries` collection. Deferred until content
  warrants it.
- Add a global `/ratings` or `/trending` feed. Those are Phase-3
  units per §12.
