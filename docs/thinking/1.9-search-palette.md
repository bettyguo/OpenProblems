# Unit 1.9 — Search palette (Fuse.js + Cmd/Ctrl+K)

> Stage 1 (THINK) artifact.

## Problem restatement

§13 Phase 1 spec: "Search via Fuse.js — `Cmd/Ctrl+K` palette." A modal overlay launched by the standard shortcut, fuzzy-matching across the project's content with a flat results list and arrow-key navigation. Phase 1 indexes problems only; later units add papers, authors, institutions to the same shape.

## Inputs

- §5.6 — "Phase 1: Fuse.js over a pre-built JSON index. Client-side, ~100 KB gz acceptable for 500 records."
- §10.3 — keyboard navigable, focus rings visible.
- Velite `problems`, `taxonomy`.
- Existing pages and routes for href targets.

## Constraints

- **Fuse.js client-side** with a server-built index (the JSON literal is inlined into the client bundle at build time — well within 100 KB for Phase 1).
- **Cmd-K (mac) + Ctrl-K (Windows/Linux)** open the palette. **Esc** closes. **↑ / ↓** navigate results. **Enter** opens. **Tab** is left alone (palette is not a focus trap competing with the rest of the page).
- **Body scroll-lock** while open. Backdrop click closes.
- **Accessibility.** Trigger button is visible (top-right of root layout) with an `aria-keyshortcuts`. The palette dialog uses `role="dialog"`, `aria-modal="true"`, and focuses the search input on open. Pre-hydration, the trigger renders as a placeholder of equal size (matching the dark-mode toggle pattern from Unit 1.2).
- **Index shape** is a flat array of `{ id, kind, title, subtitle?, tags, href }`. `kind` discriminates `problem` only in Phase 1; later kinds extend the schema without breaking the renderer.
- **No URL routing** for the palette — it's purely transient UI. Phase 2 can add `?q=…` deep linking.

## Alternatives considered

1. **Fuse.js + client-side index** (chosen, per §5.6).
2. Server-route `?q=` page using `searchParams`. Worse UX (full navigation per keystroke); useful as a fallback at Phase 4+.
3. Algolia / Meilisearch. Phase 4+ trigger.

## Edge cases & failure modes

1. **First open is the first time React hydrates this island.** Mount the palette closed; render a static trigger before hydration; mount the actual `<dialog>` lazily so the JS cost is paid only when needed (`next/dynamic` with `ssr: false`).
2. **Empty results.** Show "No matches" copy with a hint to clear / try a domain id.
3. **Very long titles.** Truncate with line-clamp-2.
4. **Mobile.** Cmd-K isn't typing-keyboard-accessible on mobile; the trigger button is the primary affordance there.

## Decision

Files:
1. `docs/thinking/1.9-search-palette.md` — this file.
2. `lib/search/build-index.ts` — `getSearchIndex(): SearchRecord[]`. Pulls every problem from Velite, joins taxonomy titles, emits `kind: "problem"` records.
3. `components/search-trigger/index.tsx` — client component for the button, lazy-loads the palette.
4. `components/search-palette/index.tsx` — client component for the dialog itself (Fuse setup + UI).
5. `app/layout.tsx` — render `<SearchTrigger />` in the body (next to or replacing the home page's local toggle).
6. `CHANGELOG.md` — Unit 1.9 entry.

`<SearchTrigger>` lives in the root layout so the shortcut works on every page. The trigger imports the palette via `next/dynamic({ ssr: false })`.
