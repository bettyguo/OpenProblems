### Why "no top-level Q68 status flip"

Per Phase-15-19 sub-bullet pattern: Q68 was always referred to as
"Q68 candidate" or "Q68 expansion" as a sub-bullet under the
top-level Q that anticipated it. No `## Q68. ...` heading; no
`Status:` line. The Phase-35 resolution edits the **sub-bullet
texts** in three places to add "RESOLVED architecturally Phase 35 ..."
prose, mirroring the way Q70's "Phase-20+ backfill script"
sub-bullet got struck through + RESOLVED-noted in Phase 20.

This is a **first-of-its-kind** documentation pattern for a
candidate Q that ships without ever being promoted to top-level —
prior candidate Qs that resolved (Q66 Phase 17; Q67 Phase 16; Q70
Phase 19) were all promoted to top-level first. Q68 stays sub-
bullet because the framework-only ADR is a category-shift in how
"content moderation" attaches to the project: not a single
ADR-pinned provider commitment (which would have warranted top-
level promotion at provider-selection time) but a multi-surface
framework with deferred provider selection (which makes top-level
promotion meaningless at this point — the provider-selection Q can
be promoted Phase 36+ when the operational gate fires).
