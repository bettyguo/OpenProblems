# 9.5 — Middleware composition + auth-aware SiteHeader UI

Fifth code unit of Phase 9. Wires the Auth.js v5 wrapper from Unit 9.4 into the project's request pipeline: composes `auth()` with the next-intl middleware (Phase 8); makes SiteHeader async + introduces an `AuthControl` server component for the sign-in / sign-out UI; adds `messages.auth.*` keys for EN + FR.

## THINK

### Scope

1. **`middleware.ts`** — wrap the existing `createMiddleware({...})` (next-intl from Unit 8.1 + 8.3) with `auth()` from `lib/auth`. Pattern: `export default auth((req) => intlMiddleware(req))`. Auth.js loads the session (DB-backed via Drizzle adapter); the inner handler delegates to next-intl for locale-prefix routing.
2. **`components/site-header/index.tsx`** — convert from sync to async server component; call `auth()` defensively via a `safeAuth()` wrapper (try/catch returning `null` on any DB failure); pass session to `AuthControl`.
3. **`components/auth-control/index.tsx`** — new server component rendering sign-in / sign-out forms via Auth.js v5's server-action pattern (`<form action={async () => { "use server"; await signIn(...) }}>`). Mirrors `ThemeToggle` / `LocaleToggle` sizing (`h-9` outlined button).
4. **`messages/en.json` + `messages/fr.json`** — `auth.*` namespace: `sign_in` / `sign_out` / `signed_in_fallback`. Three keys per locale.

### Scope NOT in this unit

- **`events.createUser` / `linkAccount` callback** to populate `users.githubLogin` from the GitHub OAuth profile on first sign-in — Unit 9.6 alongside the watchlist write-path (which depends on `githubLogin` for joining to file-system curator-of-record).
- **Watchlist table + write-path** — Unit 9.6.
- **OAuth app smoke-test against real github.com** — Q54 operational unblock.
- **Auth-aware route protection** (e.g., gating `/profile` to signed-in users) — no protected routes yet; future Phase 10+ scope.

### Why `safeAuth()` (try/catch wrapper)

`auth()` calls into the Drizzle adapter which reads from the `sessions` table. Three failure modes need handling:

1. **Local dev, first checkout, no migration run**: `local.db` doesn't exist. `@libsql/client` creates it on first connection, but the `sessions` table is missing until `pnpm db:migrate` runs. `auth()` throws.
2. **CI (test + build)**: no DB at all. `pnpm test` + `pnpm build` would crash on every SiteHeader render at SSG time.
3. **Production with transient DB outage**: rare but possible; the sticky header shouldn't 500 the entire page if auth state is briefly unavailable.

`safeAuth()` catches every failure mode and treats it as "no session" (signed-out branch renders). Production behavior under normal operation: `auth()` succeeds, `safeAuth()` is a no-op wrapper.

Alternative considered: env-gate (`if (process.env.NODE_ENV === "test") skip;`) — rejected as brittle (production might also encounter transient DB failures).

### Why server-action `<form action={...}>`

Auth.js v5 canonical pattern for sign-in / sign-out is:

```tsx
<form action={async () => { "use server"; await signIn("github") }}>
  <button type="submit">Sign in</button>
</form>
```

Server actions run on the server when the form posts; `signIn` / `signOut` are imported from `@/lib/auth` (server-only); the redirect is handled by Auth.js (per ADR-0012 D-D, full-page).

Alternative considered: client-side `<button onClick={() => signIn("github")}>` via `next-auth/react`'s `useSession` + `signIn` hooks — rejected. Requires wrapping the app in `<SessionProvider>` (client context); adds bundle weight; server-action pattern is cleaner + no provider context needed.

### Why `redirectTo: "/"` on signIn + signOut

Auth.js v5's `signIn` / `signOut` accept a `redirectTo` option that controls the post-action destination. Defaults vary by version; explicit value is more predictable.

`redirectTo: "/"` → bare canonical → middleware redirects to `/en/` (or the user's `NEXT_LOCALE`-cookie locale). Clean post-sign-in landing.

Alternative considered: redirect to the originating page via `headers().get("referer")` — rejected as fragile (referer can be missing or cross-origin). Home redirect is the simplest correct choice.

### Composition order (auth-outer vs intl-outer)

Two valid composition orders:

1. **`auth()` outer, intl inner** (chosen):
   ```ts
   export default auth((req) => intlMiddleware(req))
   ```
   - Pros: session loads before intl handles routing; future auth-aware redirects (e.g., `/login` → `/en/login`) get session context for free.
   - Cons: `auth()`'s middleware runs on every request, even ones the intl matcher wouldn't have processed (e.g., before exclusions).

2. **intl outer, auth inner**: technically possible via a custom wrapper but Auth.js v5's API doesn't expose an inner-call shape cleanly; rejected.

Both arrive at the same per-request runtime behavior given the `matcher` config; chose #1 for the future-auth-aware-redirect ergonomics.

### Edge cases & failure modes

1. **`AUTH_SECRET` unset**: Auth.js v5 throws on initialization. Build / dev fails loudly. Documented in `.env.example`. CI / test runs without `AUTH_SECRET` would crash; the `safeAuth()` wrapper doesn't catch initialization errors — they happen at module-import time, before any `auth()` call. Workaround: set `AUTH_SECRET` in CI env via Vercel preview-deploy env config (Q54 operational).
2. **Schema-out-of-sync migration**: if `local.db` exists but is missing tables (e.g., from an aborted `pnpm db:migrate`), `auth()` would throw on the missing-table SQL error. `safeAuth()` handles it.
3. **OAuth callback misconfigured**: if `AUTH_GITHUB_ID` / `AUTH_GITHUB_SECRET` are set but the GitHub OAuth app's callback URI doesn't match `/api/auth/callback/github`, the user gets an OAuth error page. Not catastrophic but confusing; Q54 documents the registration steps.
4. **Multiple tabs / browser sessions**: each tab calls `auth()` independently; DB-backed sessions are read-only per request (read latency ~5ms; Turso edge cache should help). No contention.
5. **`session.user.name` is null** (GitHub user with no public display name): `AuthControl` falls back to `email`, then to the FR/EN `signed_in_fallback` translation. Never renders an empty pill.
