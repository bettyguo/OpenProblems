## Constraints

- **`/methodology/v[N]` URL note.** Â§9 spells it as `/methodology/v[N]` â€” a literal `v` prefix on a parametric segment. Next App Router can't have partial segment prefixes (`v[version]` would be a complete segment named literally `v[version]`, not a prefix). So we accept the URL `/methodology/v1`, `/methodology/v2`, etc. via a `[version]` segment and let the page treat `version` strings like `v1` as the captured token. The IA shown to users is the same; the _implementation_ uses `[version]` as the file name.
- **Async `params`.** Next 15 changed `params` to `Promise<...>`. Every dynamic stub `await`s it.
- **Design-token-consistent stubs.** Use the same `max-w-prose` shell + serif heading + muted-foreground caption as `app/page.tsx`. Stub UI is intentionally repetitive â€” it's the IA proof, not a design exercise.
- **API stubs return 503 with `Retry-After: phase-1`.** Per Â§9 the API is read-only; per the Phase plan, it's wired in Phase 5+. Until then a 503 is honest. JSON body explains the status.
- **No layout duplication.** The root `app/layout.tsx` owns `<html>`/`<body>`. Sub-route layouts are NOT introduced in this unit â€” they would lock in design decisions (sidebar? breadcrumbs?) that belong to Phase 1.
- **No metadata explosion.** Per-route metadata (`<title>`, `<description>`) is wired only on the few routes whose names are stable today: `/about`, `/contributing`, `/methodology`. Dynamic routes can layer this in when content lands.
- **Lint-clean.** Each page is a small RSC; no client hooks; no console. Should pass `pnpm lint` + `pnpm typecheck` + `pnpm build` without changes to config.
- **Build-time complexity bounded.** 22 routes are well within Next 15's static-generation budget. With all-stub pages, the build will produce ~22 static routes; First Load JS won't move.

## Alternatives considered

1. **One stub `page.tsx` per route, sharing inline JSX** (chosen).
2. **One shared `<StubPage />` component in `components/layout/StubPage/index.tsx`** that every route imports. Trade-off: dries up the JSX but adds a component to the layout namespace before Phase 1 actually designs the shell. Rejected â€” premature.
3. **A catch-all `[[...slug]]` route emitting a stub message regardless of path.** Trade-off: one file, all stubs. Rejected â€” it hides the IA in a way that defeats the proof. Locking in `app/<route>/page.tsx` directories is exactly what Â§9 is asking for.
4. **Use route groups (`(marketing)`, `(content)`)** to organize layouts. Tempting for the future but adds layout files now. Defer.

## Edge cases & failure modes

1. **Slug clashes between page and API trees.** `app/api/v1/problems/[slug]/route.ts` and `app/problems/[slug]/page.tsx` cohabit. Next disambiguates by `route.ts` vs `page.tsx` and by the `/api/` prefix. Verified by build.
2. **`[version]` capturing `methodology` itself.** `/methodology` is the index (`app/methodology/page.tsx`); `/methodology/v1` is the dynamic version. Two separate files â€” no ambiguity.
3. **Empty params for an API route.** `app/api/v1/ratings/route.ts` accepts no params; returns a stub JSON.
4. **Stub UI accessibility.** Use `<main>`, `<h1>`, semantic landmarks. Lighthouse a11y on each stub should still be 100. Spot check at Unit 0.12.
5. **Open Graph / metadata.** Each stub page exports a minimal `metadata.title` so the browser tab shows the route name, not just "LLM OpenProblems". Reduces tab confusion in Phase 1 dev mode.
6. **404 behavior on unknown slugs.** Phase 0 stubs accept any slug and return "Phase 1 content pending Â· slug = X". A real 404 (e.g., `app/problems/[slug]/not-found.tsx`) lands in Phase 1 when we have real data to compare against.
7. **API stubs leaking implementation.** Returning `{ "status": "phase-1-pending" }` is fine. Returning an internal class name or an error stack would not be. Keep response shape stable so Phase 5 implementations don't break consumers.

## Open questions surfaced (â†’ OPEN_QUESTIONS.md)

- **Q26. `/methodology/v[N]` URL discrepancy.** Â§9 writes `v[N]` (literal `v` prefix on a parametric segment). Next App Router implements that as a `[version]` segment whose values start with `v`. Same user-facing URL (`/methodology/v1`); flag for transparency.
