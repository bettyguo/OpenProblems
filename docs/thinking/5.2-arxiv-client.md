# 5.2 — `lib/curate/arxiv-client.ts`

## THINK

### Inputs

- **Unit 5.0 D-3**: arXiv API rate limit = 3 req/s with filesystem cache at `.arxiv-cache/` (gitignored). Token bucket: 3 req/s burst, 1 req/s sustained.
- **arXiv API**: Atom feed at `https://export.arxiv.org/api/query?id_list=<id>`. Public; no auth; XML response.
- **arXiv usage policy**: ≤3 req/s ([info.arxiv.org/help/api/index.html](https://info.arxiv.org/help/api/index.html)). The `User-Agent` should identify the project.
- **ADR-0008 D-A**: Anthropic SDK is the only LLM dep; **non-LLM third-party SDKs are not gated** by ADR-0008 (it only forbids competing LLM providers). XML parser is fine.
- **No XML parser** in `package.json` yet.
- **Existing tests** use vitest; mocking patterns established in Phase-1/2 (e.g., `lib/content/*.test.ts`).

### Scope (this unit)

A new `lib/curate/` directory with:

1. **`lib/curate/arxiv-client.ts`** — exports `fetchArxivMetadata(arxivId, options?)` and the `ArxivMetadata` type. Internal: rate limiter (token bucket), HTTP fetch, XML parsing, cache read/write.
2. **`lib/curate/arxiv-client.test.ts`** — vitest tests with mocked `fetch`, mocked filesystem, and unit-tested rate-limiter behaviour.
3. **`.gitignore`** — adds `.arxiv-cache/`.
4. **`package.json`** — adds `fast-xml-parser` as a runtime dep (`devDependency` if it stays out of the Next.js build — verify below).

### Constraints

- **No Anthropic SDK import** in this unit. arXiv is just the metadata fetcher; the LLM-drafted YAML transformation lives in Unit 5.3.
- **`fast-xml-parser` placement**: runtime dep if Next.js bundles it (it doesn't — `lib/curate/*` is only imported from `scripts/*`, which is a `tsx`-driven runtime, not a Next.js build target). **devDependency**. Mirrors how `tsx` itself sits.
- **Filesystem access**: `node:fs/promises` + `node:path`. Scripts run under Node, not the browser; no `next/headers`-style isomorphism.
- **`fetch` is global** in Node 22 (per `engines.node: ">=22 <24"`). No `node-fetch` polyfill needed.
- **`exactOptionalPropertyTypes: true`** — careful with optional fields in `ArxivMetadata`.
- **No partial caching**. If the parsed metadata is missing a required field, **don't** cache it; surface the parse error so the curator sees it.

### Data shape

```ts
export interface ArxivMetadata {
  /** Canonical arXiv ID (e.g., "2310.06770"); version suffix stripped. */
  arxivId: string;
  /** Version tag of the latest revision (e.g., "v3"). */
  version: string;
  title: string;
  /** Full abstract text. */
  abstract: string;
  authors: string[];
  /** Primary arXiv category (e.g., "cs.SE"). */
  primaryCategory: string;
  /** All categories, including the primary one. */
  categories: string[];
  /** ISO date (YYYY-MM-DD) when the v1 was published. */
  publishedDate: string;
  /** ISO date (YYYY-MM-DD) of the latest revision. */
  updatedDate: string;
  /** URL to the abstract page. */
  abstractUrl: string;
  /** URL to the PDF of the latest revision. */
  pdfUrl: string;
}
```

`ArxivMetadata` is JSON-serializable for the on-disk cache.

### API contract

```ts
export interface FetchArxivOptions {
  /** Skip the on-disk cache; force a network fetch. */
  noCache?: boolean;
  /** Override the default cache directory (default: `.arxiv-cache/`). */
  cacheDir?: string;
  /** Override the fetch impl (used by tests). */
  fetchImpl?: typeof fetch;
  /** Override the user-agent (default: "llm-openproblems/0.0 (https://github.com/bettyguo/OpenProblems)"). */
  userAgent?: string;
}

export async function fetchArxivMetadata(
  arxivId: string,
  options?: FetchArxivOptions,
): Promise<ArxivMetadata> { ... }
```

The injected `fetchImpl` is the test seam; production callers omit it and get the global `fetch`. Same shape as how Phase-3 loaders inject content.

### Rate limiter

Token bucket:
- **Capacity**: 3 tokens (burst).
- **Refill rate**: 1 token per 1000 ms (sustained 1 req/s, matching arXiv guidance for batch operations).

Implementation: a module-level singleton. Each `fetchArxivMetadata` call awaits the next available token before issuing the HTTP request. Cache hits **do not consume tokens** (no network round trip).

### Alternatives considered

- **Use `xml2js` instead of `fast-xml-parser`**. Rejected — `xml2js` is older, callback-style by default (promise wrapper needed), heavier bundle. `fast-xml-parser` is well-typed + promise-clean.
- **Regex-parse the Atom XML**. Rejected — fragile; arXiv's Atom feed has nested `<entry>` and `<author>` elements; CDATA in abstracts.
- **arXiv's OAI-PMH endpoint** instead of the Atom API. Rejected — heavier protocol; the Atom API is sufficient for single-ID + small-batch queries Phase-5 needs.
- **Multi-ID batching** in a single API call. Rejected for the cache-aware caller path — cache hits would be wasted by batching. Future optimization in Unit 5.3 if benchmarks reveal it.
- **In-memory rate limiter only** (no persistent state). Accepted — process-lifetime is fine; Phase-5 CLIs are short-lived. The cache provides the persistence story.
- **Reservoir-style ([github.com/SGrondin/bottleneck](https://github.com/SGrondin/bottleneck)) library**. Rejected — extra dep for ~20 lines of token-bucket logic.

### Tradeoffs / defensible defaults flagged

1. **Module-singleton rate limiter.** Trade-off: multiple `arxiv-client.ts` consumers in the same process share one bucket — good for arXiv-side rate compliance, but if a future CLI imports the module from two paths, the limiter still applies (TS resolves to one module instance). No issue today.
2. **Cache key = arxivId** (without version). Trade-off: a curator who specifically wants v1 (not the latest) can't get it cleanly. Mitigation: `noCache: true` forces a re-fetch; future enhancement could parameterize by `id@v1`. Not Phase-5-blocking.
3. **Cache TTL = unlimited.** Trade-off: arXiv metadata is essentially immutable once published; only the `updatedDate` changes on revisions. If a curator needs a refresh, `--no-cache` is the path.
4. **Failure mode for unparseable XML**: throw; don't cache. Curator sees the error. Robustness > graceful degradation here — silent partial caching would corrupt downstream YAML drafts.
5. **`User-Agent` includes project URL** per arXiv's politeness norm. Hardcoded to the canonical GitHub URL.

### Tests

`arxiv-client.test.ts` (≥ 7 tests):

1. **Parse a realistic Atom payload** (canned fixture) into the typed `ArxivMetadata` shape — verifies title / abstract / authors / categories / dates / URLs.
2. **Strip the version suffix** from the canonical `arxivId` (input: `2310.06770v3` → output: `2310.06770`).
3. **Cache hit** — second call with the same id + same cacheDir returns from disk; `fetchImpl` not called.
4. **Cache miss** — first call (or `noCache: true`) calls `fetchImpl`.
5. **Cache write** — first call persists to `<cacheDir>/<id>.json`.
6. **`noCache: true`** bypasses both read + write (i.e., subsequent calls without `noCache` still get the freshly-fetched value? — verify).
7. **Rate-limit smoke** — 5 sequential calls (with cache cleared between each) wait at least the configured refill interval; uses fake timers.
8. **Throws on parse error** — fixture that's malformed XML throws; no cache write.
9. **Throws on HTTP error** — `fetchImpl` that returns 503 throws; no cache write.
