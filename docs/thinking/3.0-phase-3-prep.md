### Order rationale

- **3.1 first.** Multi-action history is the only deliverable that meaningfully exercises ADR-0005's append-only flow. It must ship before vizes 3.6 / 3.8 can render against real multi-action data; everything downstream uses it for storybook fixtures.
- **3.2 right after 3.1.** A single cross-problem loader prevents three pages from each rolling their own. Tests cover sort/filter/pagination.
- **3.3 / 3.4 / 3.5 in a tight cluster.** All three pages read from the same 3.2 loader; shipping them sequentially keeps each commit reviewable but they share the data fixture.
- **3.6 → 3.8 → 3.9.** Vizes individually first, page composition last (3.9 is "compose them together"). Each viz can be Storybook-validated in isolation before page integration.
- **3.7 in parallel with 3.6/3.8** is acceptable — `/trending`'s MoversBoard reads `.velite/ratings.json` directly (no problem-specific dep).
- **3.10 orthogonal.** Recompose UI is client state on the `/problems` index page; doesn't depend on any other Phase 3 work. Could land first if needed, but easier last because it shares the composite-weight formula in §8.3 which 3.11 may rewrite.
- **3.11 deliberately late.** ADR-0006 (Saturation N/A) is a schema-version bump. Units 3.1–3.10 ship under v1.0; 3.11 introduces v1.1 with a migration note. Existing v1.0 actions remain valid (ADR-0005 immutability + §8.1 versioning) — they just can't have `N/A` saturation. This deferral lets 3.1 ship fast without entangling the schema bump.
- **3.13 closes the phase.** Mirrors Unit 2.13 (Phase 2 acceptance gate) — three checks (RSS validates, Lighthouse a11y ≥ 95, visual baselines refresh) plus CHANGELOG roll-up.

### Phase-3-blocking decisions resolved here (with defensible defaults)

These are flagged in the unit description for the implementing session to surface and override if desired.

#### D-1. Number of multi-action problems

Decision: **5 problems get 2 additional actions each (= 3 total per problem); the other 5 keep their single initial action.**

Selected for multi-action treatment (high-traffic + active research areas):
1. `hallucination-reduction`
2. `long-horizon-agent-reliability`
3. `scalable-oversight`
4. `compute-optimal-test-time-reasoning`
5. `mechanistic-interpretability`

Rationale: §13 says "at least 5"; doing all 10 doubles content effort without proportional viz benefit. The 5 picked are the ones whose history is most credibly "active" in the 2025–26 narrative.

#### D-2. Action dating for the simulated revisions

Decision: **Use forward-dated revisions on a quarterly cadence relative to the 2026-05-14 initial.**

| Problem | Initial         | Second          | Third           |
|---------|-----------------|-----------------|-----------------|
| each of the 5 | 2026-05-14 (exists) | 2026-09-01 | 2026-12-15 |

Rationale: §13 says "across past months". Today's harness date is 2026-05-15 — the initial action is 1 day old, leaving zero room to backdate. ADR-0005 forbids editing the initial. Forward-dating relative to today is the only schema-clean option; the file system + RSS feed render the actions in chronological order regardless of where "now" sits.

Slug suffix convention per ADR-0005 (`<ISO-date>-<short-slug>.yaml`):
- second action: `2026-09-01-q3-revision.yaml`
- third action: `2026-12-15-q4-revision.yaml`

The slug part is a brief content tag (e.g., `q3-revision`, `q4-saturation-update`, `q4-watchlist-add`); each is curator-picked when authoring.

#### D-3. Action content for simulated revisions

Decision: **Per-action delta must be modest and signal-driven** — each revision changes 1–2 dimensions by a small amount, with at least one `signals_considered` entry that didn't appear in the prior action. Both unchanged and changed dimensions must be fully snapshotted per §8.5 + ADR-0005.

Per-problem authoring plan (concrete drafts to commit in 3.1; numbers are placeholders for the implementing session):
- `hallucination-reduction`: q3 nudges saturation 35 → 32 ("HaluEval-QA leaderboard refresh shows fresh frontier-model gains; ceiling unchanged"), industry_call stays at 5. q4 bumps confidence on saturation (0.55 → 0.65) after a clearer signal.
- `long-horizon-agent-reliability`: q3 upgrades difficulty B → A on new RE-Bench evidence; q4 nudges urgency (4 → 5) after a frontier-lab evaluation announcement.
- `scalable-oversight`: q3 leaves dimensions but adds confidence; q4 saturation 0 → 5 after first measurable result.
- `compute-optimal-test-time-reasoning`: q3 saturation 20 → 30 (Snell/Kumar 2024 follow-on results); q4 raises industry_call to 5 (frontier-lab spend tracking).
- `mechanistic-interpretability`: q3 stars-only updates; q4 difficulty A → B as SAE-based circuit extraction landscape matures.

Each rationale must cite a primary source per §15.6.

#### D-4. JSON envelope shape (Q25)

Decision: **option (c) hybrid** — `/api/v1/ratings` returns `{ items, page, total }` paginated; `/api/v1/ratings/<id>` (deferred to Phase 4+) would return the flat resource. This is the Q25 OPEN_QUESTIONS lean.

The action `id` for URL/RSS-guid purposes is `<problem_slug>/<filename-without-extension>` (e.g. `hallucination-reduction/2026-05-14-initial`). Stable across renders; encodes both the date + slug.

#### D-5. RSS feed shape

Decision: **one `<item>` per rating action.** Channel-level: title "LLM OpenProblems — Rating actions", link `/ratings`, description from MASTER_PROMPT §3.1 (one sentence on the rating-agency framing). Item-level:
- `<title>`: `"<Problem Title> — <change summary>"` (e.g. `"Hallucination Reduction — Q3 revision (saturation 35 → 32)"`).
- `<link>`: `/problems/<slug>/ratings#<action-id>` (anchor to the action's position on the per-problem ratings page).
- `<pubDate>`: action's `date` field, RFC-822 format at midnight UTC.
- `<guid isPermaLink="false">`: the stable `id` from D-4.
- `<description>`: the rationale of the most-changed dimension (or the first dimension if none changed), prepended by the prior → new vector for that dimension.

W3C validator (https://validator.w3.org/feed/) must pass — checked in Unit 3.13.

#### D-6. Recompose UI semantics

Decision: **client-only state + URL search params** for shareability. The composite formula from §8.3 is recomputed in the client on weight change. URL shape: `/problems?wd=0.25&wv=0.25&wu=0.20&wi=0.15&ws=0.15` (defaults match §8.3). Invalid weight combos (sum ≠ 1 or any < 0) fall back silently to defaults with an inline UI hint.

No persistence to localStorage (Phase 3 simplicity); revisit if user-research signals demand it.

#### D-7. /history page composition

Decision: **single page with three stacked sections** — (1) TimelineRibbon (Phase-3-light — see §11 catalog item 5, minimal version listing publication dates of contributing papers + rating actions; full catalog implementation deferred to Phase 4), (2) SaturationCurve, (3) RatingHistoryStream. No tabs. Each section is its own anchor-target.

Rationale: §11 lists TimelineRibbon as catalog item 5; the section keeps the page coherent even if the full force-graph TimelineRibbon viz is deferred. Tabs add a discovery cost; stacked-with-anchors lets users scan top-to-bottom.

#### D-8. "Last 30 days" window for `/trending` MoversBoard

Decision: **last 90 days, not 30** — relative to the most-recent rating action date across all problems (NOT today's wall-clock). §13 says "last 30 days"; the actual Phase-3 data lands 5–7 months of simulated actions in `2026-05` / `2026-09` / `2026-12`. A 30-day window would render an empty MoversBoard most of the time during development. 90 days widens to a useful default while preserving the "trending" framing.

Override path: query param `?window=30` falls back to 30-day window; default URL stays clean.

### Phase-3-blocking decisions deferred to per-unit implementation

Different colors per dimension (use the `--chart-difficulty` … `--chart-industry-call` tokens already in design tokens from Unit 0.4). Decision deferred: linear time axis or stepped (one column per action)? Default lean: stepped, since rating actions are discrete events not continuous samples.

### Open questions newly surfaced

**Q33. RSS author / managingEditor field.** RSS items conventionally name a person. Use the action's `curator` field directly (`jikun`)? Or aggregate to a project-level address? Lean: `curator` directly for item-level `<dc:creator>`; project-level email for channel `<managingEditor>` once a real email address is decided.

**Q34. /trending watchlist additions.** §13 says MoversBoard tracks "watch-list additions". The `watchlist` boolean is per-action — Phase 3 displays a watchlist add when an action toggles `watchlist: false → true`. No problems in the seed set have `watchlist: true` yet. Decision needed before 3.7 ships: should at least one of the 5 simulated revisions in 3.1 flip a watchlist to demonstrate the rendering? Lean: yes; pick one revision (e.g. `mechanistic-interpretability`'s q4) to add a watchlist signal.

**Q35. "Recompose" UI persistence.** D-6 above defers localStorage. Confirm or override.

**Q36. Composite weights from §8.3 vs Recompose UI** — when user re-weights, do leaderboards on `/domains/[domain]` and `/` also re-sort, or is the recompose scoped to `/problems` only? Lean: `/problems` only for Phase 3; cross-page sync via URL param is a Phase 4 enhancement.
