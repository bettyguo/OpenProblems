### Decisions pinned (per Unit 16.0 D-10 anticipated shape)

**D-A. Storage architecture choice.** Pins Vercel Blob (chosen)
via `@vercel/blob@1.x` SDK; documents URL allowlist + S3/R2
deferral matrix. Storage-key naming
`avatars/<userId>-<timestamp>.<ext>` (Unit 16.3 D-18 lean).

**D-B. Upload pipeline scope.** MIME validation (`image/jpeg`,
`image/png`, `image/webp` only; SVG excluded for XSS); 2 MB size
cap; first-bytes magic-number check (defense-in-depth);
delete-on-replace transactional; clear-by-empty-submit; NO EXIF
stripping (Q70 candidate); NO content moderation (Q68 expansion);
NO cropping UI; NO resizing/transcoding; NO GIF/animated WebP; NO
dimensions check; abandoned-blob cleanup script as Class B
follow-on.

**D-C. Edit surface route shape.** Extend Phase-15's
`/[locale]/profile` inline edit form with new image-upload
section. NOT separate `/profile/edit/avatar`. Continues ADR-0016
D-C "two surfaces per identity" pattern verbatim.

**D-D. Server-action shape.** New sibling `updateProfileImageAction`
distinct from Phase-15's `updateProfileAction` (multipart vs
text-only encoding asymmetry). Mirrors Phase 10/11/12/15
inline server-action precedent.

**D-E. Public consumption fallback chain.** Extends to
`imageOverride → image (GitHub-derived) → fallback initials
placeholder`. Affects `/u/{handle}` (Phase 14) + `/profile`
(Phase 10) + `getPublicProfileByHandle` + `getUserMetadataById`.
SiteHeader + AuthControl pill UNCHANGED Phase 16 (no avatar pill
yet; defer to Phase-14 Class B SiteHeader avatar-dropdown).

**D-F. Validation + sanitization model.** MIME enforcement +
first-bytes magic-number check + 2 MB size cap + 512 char URL
length cap + HTTPS + Vercel Blob URL pattern match.

**D-G. Operational gating.** **Q69 candidate** — `BLOB_READ_WRITE_TOKEN`
provisioning. Parallel to Q54 (GitHub OAuth) + Q55 (Turso DB).
Graceful degradation if unset (upload action returns error;
sign-in / read-side unaffected).

**D-H. Phase 17+ deferrals.** EXIF stripping (Q70 candidate);
content moderation (Q68 expansion); cropping UI; server-side
resizing / transcoding; multiple-avatars / history; image
dimensions check; abandoned-blob cleanup script; GIF / animated
WebP; external URL allowlist composability (Option 2 extension).

### Why an ADR at all?

Three reasons:

1. **First binary storage layer in project history** crosses
   ADR-0004 + ADR-0013 D-F boundaries cleanly (binary storage is
   a SEPARATE primitive from DB + file-system content; this ADR
   defines that boundary explicitly).
2. **First new env-var operational gate since Phase 12**
   (`BLOB_READ_WRITE_TOKEN`); Q69 candidate parallel to Q54 +
   Q55. Pinning the operational shape + graceful-degradation
   contract documents Phase-16 inheritance for Phase-17+ binary-
   asset surfaces.
3. **Q67 promotion was explicit Phase 16+ deferral** (ADR-0016
   D-G): "Image override DEFERRED to Phase 16+. Storage ADR
   needed … ADR-0017 candidate." Phase 16 is the deferral-honor;
   not landing an ADR would be a documentation regression. The
   storage-choice decision is the load-bearing pick Phase 16
   surfaces — Vercel Blob vs S3/R2 vs URL allowlist each have
   different operational, cost, and UX shapes.

Contrast Phase 13: Q58 promoted to resolved WITHOUT new ADR
because the lean was documented across three prior units (11.6 +
12.0 D-10 + 13.0 D-3). Phase 16's image-storage surface spans
broader architectural ground (storage backend + upload pipeline +
validation model + edit surface + operational gating + Phase 17+
deferral matrix); ADR-worthy.

### Options + chosen rationale (mirrors ADR body)

Four options considered:

1. **Vercel Blob with file-upload pipeline** (CHOSEN). First-party
   Vercel integration; no CORS / IAM ceremony; ~$0.02/GB
   negligible at MVP scale; single env var; small dep; natural
   user flow (upload from computer); cleanup-on-replace is one
   SDK call. Closes Q67 with the conventional social-platform UX.
2. **External URL allowlist (no storage)**. REJECTED. Simplest
   architecture; smallest blast radius. But most users have no
   usable image URL handy — power-user-only feature; defeats the
   "I want to use my professional headshot" UX. Phase 17+
   composable extension if power-user demand surfaces.
3. **S3 / R2 (cheapest at scale)**. REJECTED. Cheapest per GB
   but complexity dominates cost savings at MVP scale. Multi-
   env-var operational gate; heaviest dependency (~120 kB);
   IAM/bucket-policy ceremony. Phase 17+ trigger: storage cost
   crosses $10/month or vendor-portability becomes load-bearing.
4. **Hybrid (Blob upload OR URL paste)**. REJECTED. Triples
   implementation surface; pushes Phase 16 scope to 11+ units
   (beyond 9-unit envelope). Phase 17+ if usage signal demands
   both paths.

### Architectural relationships pinned

- **Inherits ADR-0016 D-G** Phase-16+ deferral path verbatim.
- **Inherits ADR-0016 D-A** user-controlled override pattern —
  `imageOverride` (user-controlled) overrides `image`
  (GitHub-derived); new column, not a replacement. Mirrors
  Phase-15's `displayName` / `name` partition.
- **Inherits ADR-0015 D-A** public-data invariant (no NEW
  public-data category; avatar imagery already public at
  github.com).
- **Inherits ADR-0015 D-F** + **ADR-0016 D-C** "two surfaces per
  identity" pattern — `/profile` extends with image-upload section;
  `/u/{login}` extends with image render. No new route.
- **Inherits ADR-0014 D-E** ALTER migration discipline — Phase
  16's `0005_user_image_override` is the THIRD ALTER migration in
  project history; same additive nullable text column pattern.
  Discipline crystallized at third exercise.
- **Preserves ADR-0013 D-F** USER-STATE-only DB — binary storage
  lives in Vercel Blob (a separate primitive); DB stores only the
  URL pointer.
- **Preserves ADR-0008 §3** cost-governance pact (applied by
  analogy to storage cost; MVP-scale cost negligible).
- **Reuses ADR-0012 D-E** `users.githubLogin` joining (excluded
  from editable fields; URL key immutable).
- **Reuses ADR-0011** i18n strategy — new
  `messages.profile_edit.image_*` keys follow sibling-file
  convention; extends Phase-15's `messages.profile_edit.*`
  namespace.
- **Establishes binary-storage pattern** that Phase 17+
  binary-asset surfaces (paper figures? curator-review
  attachments? methodology diagrams?) inherit.
- **Surfaces Q69 candidate** as a NEW operational gate parallel
  to Q54 / Q55.
