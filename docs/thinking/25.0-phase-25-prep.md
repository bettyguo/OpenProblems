### D-3. Item 1 — Methodology-version dynamic read

**Scope** (Phase-22 B.6 item 3 closure):
- New helper `readMethodologyVersion({ contentRoot }): Promise<string>` in `lib/rating-challenges/scaffold.ts`.
- Reads `<contentRoot>/methodology/v*.mdx` (filters out
  locale variants like `v1.fr.mdx`).
- Sorts filenames lexically descending; tries each in order.
- Parses each via simple frontmatter regex (`/^version:\s*["']([^"']+)["']/m`) — full YAML parsing unnecessary for one field.
- Returns first parsed version OR fallback `"1.0.0"` on
  failure / no files.
- `buildRatingActionYaml` accepts new optional `methodologyVersion?: string`
  parameter; defaults to `METHODOLOGY_VERSION` constant when
  not provided. Constant remains `"1.0.0"` as the ultimate
  fallback if dynamic read fails.
- CLI orchestration (`scripts/emit-challenge-action.ts`)
  calls `readMethodologyVersion({ contentRoot: "content" })`
  and passes result to `buildRatingActionYaml`.

**Why this matters**:
- Currently all rating-actions hardcode `methodology_version: "1.0.0"`.
- Phase 26+ if v2 methodology lands (e.g., `content/methodology/v2.mdx`
  with `version: "2.0.0"`), this helper picks it up
  automatically without code change.
- Defensive: helper failure → fallback to hardcoded constant
  (no functional regression).

### D-4. Item 2 — Batch-emit-multiple

**Scope** (Phase-22 B.6 item 4 closure):
- New `--all-under-review` flag on `scripts/emit-challenge-action.ts`.
- When set: skip `<challenge-id>` positional arg; query
  `db.select().from(ratingChallenges).where(eq(status, "under_review"))`;
  process each in series via the existing per-challenge flow.
- Per-row try/catch (one bad challenge doesn't kill the
  batch); summary at end (succeeded / failed counts).
- Exit code: 0 if all succeeded; 1 if any failed.
- Mutually exclusive with `<challenge-id>` positional arg
  (`parseArgs` validates: must provide exactly one of
  `<challenge-id>` OR `--all-under-review`).

**Why this matters**:
- Phase-22 ships single-challenge invocation. Curator with
  ~5 challenges in `under_review` queue would have to run the
  script 5x.
- Batch mode reduces friction without changing per-challenge
  semantics.

### D-5. Item 3 — `--no-prior-fill` opt-out

**Scope** (Phase-23 B.7 closure):
- New `--no-prior-fill` flag on `scripts/emit-challenge-action.ts`.
- When set: skip `readPriorRatingAction` call; pass
  `priorAction: null` to `buildRatingActionYaml` (forces
  Phase-22 placeholder behavior).
- Useful when prior data is suspect / curator wants to re-
  author OTHER 4 dimensions from scratch.
- Compatible with `--all-under-review` (both flags can be
  used together for batch mode without auto-fill).

**Why this matters**:
- Phase 23 ships auto-fill always-on. Curator may need to
  opt out (e.g., methodology changed; prior used different
  scale; signals shifted dramatically).
- Simple flag; preserves Phase-22 fallback path verbatim.

### D-6. Item 4 — `--user-id` flag for `backfill-resize-webp`

**Scope** (Phase-24 B.8 item 1 closure):
- New `--user-id <id>` flag on `scripts/backfill-resize-webp.ts`.
- When set: filter affected rows to just the row where
  `users.id = <id>` (AND existing Vercel Blob URL pattern).
- Mutually compatible with `--dry-run`.

**Why this matters**:
- Phase 24 ships batch backfill (all affected rows).
- Curator surgical scenarios (user reports broken avatar
  after re-upload; one specific row needs re-processing)
  benefit from per-user targeting.
- Defends against accidental full-batch run when only one
  user needs the re-process.
