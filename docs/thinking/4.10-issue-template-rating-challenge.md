# 4.10 — Issue template: rating-challenge

## THINK

### Inputs

- **Unit 4.0 D-8 + Q37**: form-based `.yml`; 3–5 required + 2–4 optional + free-text notes.
- **Units 4.7 / 4.8 / 4.9 (`de9460b` / `1954b1d` / `636da83`)** establish the pattern.
- **`RatingActionSchema`** (`lib/schemas/rating-action.ts`) — 5 dimensions, all required, each snapshotted in full on every action per ADR-0005 immutability. A "challenge" doesn't edit existing actions; it provides input for the curator to draft a **new** rating-action YAML.
- **ADR-0005** enforced by pre-commit hook: no edit / delete / rename on `content/problems/*/ratings/*.yaml`. The dispute resolution path is **always** "write a new YAML", never "amend an old one".
- **MASTER_PROMPT §8 rating methodology**: 5 dimensions = difficulty (S/A/B/C/D/E grade) + saturation (0–100 or qualitative band per §8.2 / ADR-0006) + urgency (★ 0–5) + value (★ 0–5) + industry_call (★ 0–5). All have `confidence ∈ [0, 1]` + `rationale` text.

### Scope (this unit)

`.github/ISSUE_TEMPLATE/rating-challenge.yml` — captures a dispute against an existing rating. The output is curator-side: triage, evaluate evidence, decide whether to draft a new rating-action YAML.

### Constraints

- **Mirror Units 4.7 / 4.8 / 4.9** structural decisions.
- **Use `dropdown` for the dimension** — the set is closed (5 values per §8); freeform invites typos. This is the first issue-template field where a `dropdown` is the right call vs. freeform.
- **Use `dropdown` for direction** — closed set ("up" / "down" / "watchlist"). Helper text explains "watchlist" = add the problem to the MoversBoard watchlist-add signal.
- **No required `confidence` / `rationale` boilerplate** — a challenger isn't drafting the new YAML; they're providing input. Curator interprets.
- **YAML quoting audit** per Unit 4.8 lesson — single-line values with internal `:` get quoted.

### Form-field schema

7 body items: 1 markdown intro + 6 form fields.

| # | Item                                  | Type       | Required | Notes                                                                  |
|---|---------------------------------------|------------|----------|------------------------------------------------------------------------|
| 1 | Intro markdown                        | markdown   | n/a      | Links MASTER_PROMPT.md §8 (rating methodology) + §15.6 + ADR-0005. Notes the curator writes a new rating-action YAML, never amends an old one — what the challenge needs is **evidence**, not a draft. Cross-links the live ratings index `/ratings` for the current rating-action history. |
| 2 | Problem slug                          | input      | yes      | Which problem's rating is being challenged.                            |
| 3 | Dimension                             | dropdown   | yes      | One of: difficulty, saturation, urgency, value, industry_call. Closed set. |
| 4 | Direction                             | dropdown   | yes      | One of: up, down, watchlist. "up" = the dimension should go higher (more difficult / more saturated / more urgent / more valuable / stronger industry call); "down" = opposite; "watchlist" = adding the watchlist signal. |
| 5 | Evidence / rationale                  | textarea   | yes      | The meat of the challenge. What's the new signal? Cite primary sources per §15.6. |
| 6 | Source URL(s)                         | textarea   | no       | Bulleted list of supporting URLs. Helper text reinforces §15.6.        |
| 7 | Additional context                    | textarea   | no       | Anything else — proposed score range, related challenges, watchlist concerns. |

### Alternatives considered

- **Freeform `dimension` input.** Rejected — closed set; typos waste curator triage time. First template where `dropdown` is the right call.
- **Multi-select dimensions.** Considered — a challenge might span 2 dimensions (e.g., "saturation went up AND urgency went up"). Rejected as overhead: 95%+ of real disputes are single-dimension, and a multi-dim challenge can be filed as two issues or surfaced in "Additional context".
- **Score/grade input for the proposed value.** Considered (would speed curator drafting). Rejected as form noise — the challenger may not know the encoding (S/A/B/C/D/E vs. 0–100 vs. 0–5 stars vary per dimension); curator picks the encoding during YAML drafting.
- **Required confidence-level input.** Rejected — challengers aren't curators; asking for confidence is asking them to do the curator's job.

### Tradeoffs / defensible defaults flagged

1. **3 required + 1 textarea required = 4 required fields.** Stays in Q37's 3–5 lean range.
2. **Dropdown for dimension/direction.** First issue-template use of `dropdown` in this project — set explicitly because both are closed enums. Future template revisions can apply the same pattern for other closed sets if they emerge.
3. **No structured "current vs. proposed" pair.** Curator infers the current value from the latest committed rating-action YAML; challenger only needs to convey direction + evidence.
4. **Watchlist as a `direction` option, not a separate `boolean` field.** Keeps the form to a single primary decision tree: "what should change about this rating?" — watchlist toggling is one valid answer.
