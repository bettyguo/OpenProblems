# Unit 1.3 — Methodology page rendering `content/methodology/v1.mdx`

> Stage 1 (THINK) artifact.

## Problem restatement

Three things, tightly coupled:

1. Author the **first** versioned methodology document (`content/methodology/v1.mdx`) covering MASTER_PROMPT §8 — first principles, dimension definitions, composite formula, confidence, rating-actions log, COI policy.
2. Wire **Velite's MDX pipeline** end-to-end: `methodology` collection, KaTeX (math), Shiki (code), MDX → React render at request time / build time.
3. Render at `/methodology` (latest version) and `/methodology/[version]` (specific snapshot per §9 + §8.1 versioning rule).

Closes Q29 (Velite MDX plugin set). Defers Mermaid (not needed for v1).

- §5.4 — KaTeX for math, Shiki for code, Mermaid for diagrams (deferred).

## Constraints

- **v1 must read as workshop-paper-defensible.** Not exhaustive, but every claim cite-able and self-consistent. Composite weights match §8.3 defaults.
- **MDX renders server-side.** No `"use client"` for the methodology page. KaTeX runs at build time and emits HTML+CSS — no client KaTeX runtime.
- **Shiki at build time.** Code blocks compile to HTML at build; zero client JS for highlighting.
- **Versioned by file name.** `v1.mdx`, `v1.1.mdx`, `v2.mdx`. The version is the file slug; URL `/methodology/v1` ↔ `content/methodology/v1.mdx`.
- **`/methodology` redirects mentally to the latest.** Implementation: sort the collection by `version` (semver), render the highest. No HTTP redirect.
- **MDX schema duplication.** Same Velite/Zod-4 wrinkle as Unit 1.1 (Q31). The methodology collection schema lives in `velite.config.ts` using `s`.

## Alternatives considered

1. **Velite `s.mdx()` + remark-math + rehype-katex + rehype-pretty-code** (chosen). All the master prompt's §5.4 pieces, build-time, server-rendered.
2. **`@next/mdx`.** Compiles per route — fine for a one-file methodology, but doesn't generalise to Phase 1.4's problem MDX (background.mdx, definition.mdx, history.mdx). Rejected to keep one MDX pipeline.
3. **Plain TSX page (no MDX).** Fastest to ship but defeats §5.3 (content authored in MDX) and §5.4 (KaTeX). Rejected.
4. **Defer Shiki to Unit 1.4.** Tempting since v1 has minimal code. Rejected — Shiki is small marginal cost in this unit; better to land the full plugin set once.

## Edge cases & failure modes

1. **`new Function(code)` for MDX.** Velite emits compiled MDX as a JS code string. Standard render pattern is `new Function("...code")({ ...jsxRuntime })`. Acceptable in RSC at build time; never expose to user input.
2. **KaTeX font CSS.** rehype-katex injects markup expecting `katex.min.css`. Either import it in `app/layout.tsx` or in the methodology page. Import in layout so any future MDX page works.
3. **Shiki bundle weight.** Shiki loads themes + grammars lazily. Configure to load only the languages we use (yaml, ts, bash, json) and one theme aligned with our two-tone foundation.
4. **Versions ordering.** `v1` vs `v1.1` vs `v10` — string-sort breaks. Parse the leading `v` and compare numerically; in v1 there's only one version, so the bug is dormant.
5. **`/methodology` empty.** If `content/methodology/` has zero files Velite returns `[]` and the page should render a "no methodology yet" message. v1 ships at least one — so practical fallback is "404 with a helpful message" via `notFound()`.
6. **MDX frontmatter shape.** Frontmatter must include `version`, `title`, `summary`, `date`, optionally `supersedes`. Velite's `s.object({...})` validates at build time.
7. **§14.3 numbers-need-sources rule.** v1 has the composite-weight defaults (0.25 / 0.25 / 0.2 / 0.15 / 0.15). Those are project-decided, not externally cited — leave a `<Cite>` placeholder or footnote rationale ("editorial default; revise by ADR").

## Open questions surfaced

## Decision

Files / actions:

1. `docs/thinking/1.3-methodology-page.md` — this file.
2. `pnpm add remark-math rehype-katex rehype-pretty-code shiki katex` (KaTeX CSS comes from the `katex` package).
3. Extend `velite.config.ts`: add a `methodology` collection with `s.mdx({ remarkPlugins: [remarkMath], rehypePlugins: [[rehypeKatex], [rehypePrettyCode, shikiOptions]] })`. Schema includes `version`, `title`, `summary`, `date`, `body` (MDX).
4. `lib/mdx/mdx-content.tsx` — RSC component that takes Velite's compiled `code` string and renders it via `new Function(code)({...runtime})`. Reusable for Unit 1.4's problem MDX.
5. `app/layout.tsx` — import `katex/dist/katex.min.css` so math is styled site-wide.
6. `content/methodology/v1.mdx` — first methodology snapshot covering §8.1–§8.6. ~300 lines. Includes the composite-formula KaTeX block.
7. `app/methodology/page.tsx` — server component; loads `methodology` collection from `#site/content`, sorts by version, renders the latest via `<MDXContent code={latest.body} />`.
8. `app/methodology/[version]/page.tsx` — server component; finds the matching version or `notFound()`.
9. `OPEN_QUESTIONS.md` — close Q29; open Q32.
10. `CHANGELOG.md` — Unit 1.3 entry.
