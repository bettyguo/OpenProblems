# 37.2 — Integration of `MarkdownExtensionRegistry` into all 4 markdown helpers (bio + reviewNotes + rationale + actionRationale) + framework tests + per-surface extension-merge tests

Phase 37 unit per `docs/thinking/37.0-phase-37-prep.md` (HEAD
`55e7ba9` = Unit 37.1 ship). Wires the Phase-37-Unit-37.1
framework into the four existing markdown helpers per ADR-0018
D-G APPEND. Zero behavioral change Day 1 — default empty
extension sets fold into the pipelines as no-ops.

## THINK

### Problem restatement

Unit 37.1 shipped the `MarkdownExtensionRegistry` framework as
five standalone files under `lib/markdown/extensions/`. The
framework is currently disconnected from the four markdown
helpers (`renderBioMarkdown` Phase 17 + `renderReviewNotesMarkdown`
Phase 18 + `renderRationaleMarkdown` Phase 27 +
`renderActionRationaleMarkdown` Phase 29). Unit 37.2 wires the
registry into each helper's `unified` pipeline per ADR-0018
D-G APPEND D-A / D-C / D-D semantics:

- Each helper reads `registry.getExtensions(surface)` once.
- `extensions.schemaOverrides` shallow-merges over the
  surface's base sanitize schema (**override-replace
  semantics** per APPEND-D-C: callers supply complete
  replacement for any field they override; framework does NOT
  deep-merge).
- `extensions.remarkPlugins` fold into the unified pipeline
  AFTER the default `remark-parse` + `remark-gfm` plugins
  (**plugin order: after default plugins** per APPEND-D-D).
- `extensions.rehypePlugins` fold AFTER `rehype-sanitize` +
  `rehypeDemoteHeadings` + `rehypeStripUnsafeHrefs` (same
  after-default ordering) and BEFORE `rehype-stringify`.

Day-1 behavior parity: `DefaultExtensionRegistry` returns `{}`
for all four surfaces; folding empty plugin lists + empty
schema overrides is a structural no-op (an empty
`PluggableList` passed to `.use()` adds no plugins; shallow-
spreading `{}` over the base schema is identity).

### Alternatives considered

**Option 1: Build processors per-call (rejected)** — every
`render*` call constructs a fresh `unified()` chain. **Why
rejected**: trivially correct but loses the module-level
singleton optimization the current code relies on. For
`actionRationale` rendered 5× per rating-action × N actions
per `/problems/[slug]/ratings` page, per-call build adds
measurable overhead at scale.

**Option 2: Build processors at module load (chosen with
caveat)** — call `getExtensionRegistry()` once at module load,
build the four processors with extensions baked in, cache as
module-level constants. **Why chosen with caveat**: preserves
singleton perf; matches the current module shape exactly.
**Caveat**: not test-friendly — once the module-level
constants are built, switching the registry (via the Unit-
37.1 `__resetRegistryForTests` hook) doesn't rebuild the
processors. Solved via Option 2-prime below.

**Option 2-prime: Lazy-built processors with test-reset hook
(chosen)** — convert the four `const xxxProcessor = unified()...`
to lazy getters that build on first call and cache. Add a
`__resetMarkdownCachesForTests()` hook that clears all four
lazy caches; tests call this hook AFTER swapping the registry
to force rebuild. This is the cleanest shape:
- Production: lazy-build-once-then-cache; first-call overhead
  amortized over the process lifetime; module-load-side-
  effect-free (no registry read at import time).
- Tests: per-suite reset clears caches → switching registry
  takes effect on next `render*` call.

**Option 3: Pass extensions explicitly into each render call
(rejected)** — `renderBioMarkdown(text, extensions?)`. **Why
rejected**: leaks framework wiring into every call site;
defeats the "registry is the single source of truth" property
of the framework; harder to test (every call site needs
extension-injection); breaks the existing 4-helper public API.

### Edge cases & failure modes

1. **Empty `PluggableList` passed to `.use([])`**: per unified
   docs, `.use(list)` with an empty array adds no plugins.
   No-op. Verified at integration time.
2. **`schemaOverrides: {}`**: spreading an empty object over
   the base schema yields the base schema verbatim. No-op.
3. **Override of `schemaOverrides.tagNames` replaces the
   entire base allowlist**: per APPEND-D-C override-replace
   semantics. A test extension that overrides `tagNames: ["p",
   "table"]` would lose the rest of the base allowlist; the
   test extension must supply the COMPLETE replacement. Tests
   in this unit verify this semantics explicitly.
4. **Multiple instances of same plugin**: framework does not
   deduplicate; if a caller registers the same plugin twice
   it runs twice. Documented in ADR-0018 D-G APPEND F-list as
   Phase 38+ concern if it surfaces.
5. **Test pollution across suites**: `__resetRegistryForTests`
   + `__resetMarkdownCachesForTests` must both be called in
   `beforeEach`/`afterEach` to guarantee clean state.
   Documented in new tests.
6. **Singleton build cost spike**: lazy-build only runs once
   per process per surface. First page rendering each surface
   pays the build cost (~1ms per `unified()` chain). After
   that the singleton is cached. **No regression vs Phase
   17/18/27/29 baseline** which paid module-load cost up
   front; lazy-build just defers it to first-call.

## DESIGN

### `buildProcessor(surface, baseSchema)` shape

```ts
function buildProcessor(surface: MarkdownSurface, baseSchema: Schema) {
  const extensions = getExtensionRegistry().getExtensions(surface);
  const schema: Schema = { ...baseSchema, ...(extensions.schemaOverrides ?? {}) };
  const remarkExtensions: PluggableList = [...(extensions.remarkPlugins ?? [])];
  const rehypeExtensions: PluggableList = [...(extensions.rehypePlugins ?? [])];
  return unified()
    .use(remarkParse)
    .use(remarkGfm)
    .use(remarkExtensions)                              // APPEND-D-D: after default remark plugins
    .use(remarkRehype, { allowDangerousHtml: false })
    .use(rehypeDemoteHeadings)
    .use(rehypeSanitize, schema)                        // APPEND-D-C: schema override-replace
    .use(rehypeStripUnsafeHrefs)
    .use(rehypeExtensions)                              // APPEND-D-D: after default rehype plugins
    .use(rehypeStringify);
}
```

### Lazy getter shape (per surface)

```ts
let bioProcessor: ReturnType<typeof buildProcessor> | null = null;
function getBioProcessor() {
  if (bioProcessor) return bioProcessor;
  bioProcessor = buildProcessor("bio", bioSchema);
  return bioProcessor;
}
```

Repeated for reviewNotes + rationale + actionRationale.

### `__setRegistryForTests` shape

```ts
let registryOverride: MarkdownExtensionRegistry | null = null;

export function getExtensionRegistry(): MarkdownExtensionRegistry {
  if (registryOverride) return registryOverride;
  if (registryInstance) return registryInstance;
  registryInstance = new DefaultExtensionRegistry();
  return registryInstance;
}

export function __setRegistryForTests(r: MarkdownExtensionRegistry | null): void {
  registryOverride = r;
}

export function __resetRegistryForTests(): void {
  registryInstance = null;
  registryOverride = null;
}
```

`__resetRegistryForTests` now clears BOTH the default
singleton AND any override. Tests call `__setRegistryForTests(r)`
to override, then `__resetRegistryForTests()` in `afterEach`
to restore default.

### Test plugin shape (for integration tests)

```ts
import { visit } from "unist-util-visit";
import type { Plugin } from "unified";
import type { Root } from "hast";

const rehypeUppercaseText: Plugin<[], Root> = () => (tree) => {
  visit(tree, "text", (node) => {
    if (typeof node.value === "string") node.value = node.value.toUpperCase();
  });
};
```

Used in tests to verify the rehype-extension fold-after-
default-plugins path.
