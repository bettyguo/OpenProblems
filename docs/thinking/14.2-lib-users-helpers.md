# 14.2 — `lib/users/` module + `getPublicChallengesByUser` extension + tests

Third Phase-14 unit; first code-shipping unit. Per
[Unit 14.0 D-7](./14.0-phase-14-prep.md), introduces a new `lib/users/`
module mirroring the `lib/watchlist/` + `lib/rating-challenges/` +
`lib/auth/login/` separation conventions. Extends
`lib/rating-challenges/` with `getPublicChallengesByUser` per
[ADR-0015](../adr/0015-per-user-privacy-model.md) D-A.

## THINK

### What this unit ships

Three publicly-visible helper signatures + one extension:

```ts
// lib/users/index.ts

export interface PublicProfile {
  userId: string;
  githubLogin: string;   // canonical case from users row
  name: string | null;
  image: string | null;
  createdAt: Date;
}

export interface ProfileActivity {
  watchedCount: number;
  pendingChallengeCount: number;  // submitted + under_review
  acceptedChallengeCount: number;
}

export async function getPublicProfileByHandle(handle: string): Promise<PublicProfile | null>;
export async function getProfileActivity(userId: string): Promise<ProfileActivity>;
export function getCuratorOfRecordSlugs(handle: string): readonly string[];
```

Plus:

```ts
// lib/rating-challenges/index.ts (extension)

export async function getPublicChallengesByUser(userId: string): Promise<PublicChallengeRow[]>;
```

### Design decisions realized from ADR-0015

- **D-A** (public-profile field partition): `PublicProfile` exposes
  ONLY GitHub-derived fields (no `email`); `ProfileActivity` exposes
  only the three publicly-visible counts (no `rejected` / `withdrawn`
  per Phase-13 Unit 13.0 D-3); `getPublicChallengesByUser` filters to
  `PUBLIC_CHALLENGE_STATUSES` server-side.
- **D-B** (case-insensitive handle lookup): `getPublicProfileByHandle`
  uses `sql\`LOWER(${users.githubLogin}) = LOWER(${normalized})\`` for
  case-insensitive matching; returns the **canonical** case stored in
  the DB row (NOT the URL case). No redirect logic — caller's job is
  to render content with the canonical case in the body while URL bar
  preserves the user-typed case.
- **D-E** (curator-of-record case-SENSITIVE): `getCuratorOfRecordSlugs`
  compares strictly (`===`) against the YAML literal value. Caller
  must pass the **canonical case** from `users.githubLogin`, not the
  raw URL `[handle]` segment.

### Why three helpers, not one

Considered: a single `getPublicUserPage(handle)` returning a
composite shape. **Rejected**:

1. **Independent failure modes** — `getPublicProfileByHandle` may
   return `null` (404); the activity + curator-of-record helpers are
   irrelevant in that case. A composite would force callers to handle
   "profile present but counts failed" coupling.
2. **Parallel-fetch tractability** — the route can `Promise.all` the
   three queries independently once the profile is loaded; coupling
   them in one helper would serialize the COUNT queries inside the
   composite and slow down rendering.
3. **Curator-of-record is pure** (file-system scan; no DB); coupling
   it with DB-bound helpers obscures the cost asymmetry.

### Why `getProfileActivity` returns 3 counts, not 5

Per ADR-0015 D-A + Phase-13 Unit 13.0 D-3, `rejected` + `withdrawn`
counts are SUBMITTER-ONLY. The public `/u/{handle}` surface never
needs them. The signed-in own surface (`/profile`) renders the full
per-status partition via the existing `getUserChallenges(userId)`
helper (Unit 11.1 + Unit 11.4) which returns full rows — counts can
be derived client-side from that array. **No new `getProfileActivityFull`
helper needed** for the signed-in own surface.

If a future Phase needs the full partition publicly (Q62 candidate —
rejection-rationale public visibility), the per-status fields can be
added incrementally without breaking the current `ProfileActivity`
shape (TypeScript additive extension).

### `getPublicChallengesByUser` parallel to `getPublicChallengesByProblem`

The two helpers share identical row shape (`PublicChallengeRow`),
identical status-partition filter (`PUBLIC_CHALLENGE_STATUSES`),
identical sort (`createdAt DESC`), identical limit
(`CHALLENGES_LIMIT = 50`). Only the WHERE clause differs (`userId`
vs `problemSlug`). The two routes that consume them (Phase-13's
per-problem listing + Phase-14's per-user sub-route) can share a
common row-rendering component in Phase 15+ if a third axis surfaces;
for now they stay as separate page-level components per Unit 14.0
D-6 + Phase-13 Unit 13.3 precedent.

### Test surface

13 tests added across the new `lib/users/index.test.ts`:

- **getPublicProfileByHandle** (5 tests): exact-case match, case-insensitive
  match preserving canonical case, no-match returns null, NULL
  `githubLogin` (Phase-9 retrofit edge), empty/whitespace handles
  (defensive — short-circuits before DB).
- **getProfileActivity** (3 tests): parallel COUNT aggregation; never
  exposes rejected/withdrawn (defensive shape assertion); 0-default
  on empty row arrays (defensive).
- **getCuratorOfRecordSlugs** (5 tests): case-sensitive match (lowercase
  curator handle → 2 problems); alternate-case curator (different
  case → different problem set; demonstrates D-E case-sensitivity);
  no-match returns empty array; empty/whitespace handles; single-match.

**Mocking approach**:
- `@/lib/db` mocked at module level: `db.select()` returns a chainable
  stub that lets each test seed its own row set.
- `#site/content` mocked: 4 fake problems with 3 distinct curator
  handles (`bettyguo` x2, `BettyGuo` x1 for case sensitivity demo,
  `jikun` x1). Curator-of-record tests do NOT use the real
  `content/problems/` directory.

**No new test files for the `getPublicChallengesByUser` extension** —
the helper mirrors `getPublicChallengesByProblem` shape exactly; the
existing Phase-13 `public-visibility.test.ts` covers the per-status
filter logic via the `PUBLIC_CHALLENGE_STATUSES` const + predicate
tests. Adding a DB-mocked test for `getPublicChallengesByUser` would
duplicate the Phase-13 partition-correctness checks without adding
coverage; the helper is exercised at request time by Unit 14.4's
per-user sub-route.

### Edge cases handled defensively

- **Empty/whitespace handle** → null OR empty array, no DB hit. Saves
  a round-trip on malformed URLs (e.g., `/u/` resolves to `[handle] =
  ""` in Next.js routing; defensive return prevents a `LOWER(NULL)`
  edge case).
- **NULL `githubLogin` row** → null (Phase-9 retrofit edge per Unit
  9.6's deferred `events.linkAccount` callback on prior sign-ins).
  Even if a row matches (e.g., `LOWER(NULL) = LOWER("")`), the post-
  query check rejects it before returning.
- **Missing count row** → 0 (defensive on Drizzle's `count()`
  helper returning `undefined` in edge cases; never observed in
  practice but cheap to guard).
