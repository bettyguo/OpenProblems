# 6.3 — `app/problems/[slug]/talk/page.tsx` (talk-page route shell + Giscus embed slot)

Second code unit of Phase 6. Lands the talk-page route that Unit 6.4's Giscus embed will populate. Mirrors the existing `app/problems/[slug]/<subroute>/page.tsx` pattern (leaderboard / history / ratings) for layout + SSG conventions.

## THINK

### Inputs

- **[ADR-0010 D-A](../adr/0010-discussions-backend.md)** — Giscus iframe is the comment UI; lazy-loaded; doesn't impact SSG shell render or Lighthouse a11y score.
- **[ADR-0010 D-C](../adr/0010-discussions-backend.md)** — pathname-based mapping: discussion title defaults to the pathname (e.g. `/problems/<slug>/talk`); lazy creation on first comment.
- **[Unit 6.0 D-6](6.0-phase-6-prep.md)** — "talk-page route shape mirrors the existing `[slug]/<sub>` pattern. SSG-rendered shell + client-loaded discussion widget (the widget is the only client-only component on the page; the shell stays server-rendered to keep First Load JS in the 103 kB cohort)."
- **[Q48 (decided-as-lean)](../../OPEN_QUESTIONS.md#q48-talk-page-indexing-posture)** — talk pages should be linked from problem detail + included in sitemap. This unit lands the link from problem detail; sitemap is a future-scope item (no sitemap surface exists at HEAD).
- **`app/problems/[slug]/leaderboard/page.tsx`** — structural precedent (breadcrumb shape; `loadProblem` + `notFound`; `generateStaticParams` from `allProblemSlugs`).
- **`app/problems/[slug]/page.tsx`** — the problem-detail page; gets a one-line "Discuss" link added below the title.

### Scope (this unit)

Two file changes:

1. **`app/problems/[slug]/talk/page.tsx`** (new) — talk-page route shell with:
   - `generateStaticParams` for all problem slugs (10 routes added).
   - `loadProblem(slug)` → 404 if unknown.
   - Breadcrumb: `Problems / <Problem Title> / Discussion`.
   - Heading: "Discussion" (h1 — matches the `<sub>/page.tsx` pattern).
   - `<section id="discussions">` slot with placeholder content. Unit 6.4's `GiscusEmbed` component will render INTO this slot via direct child component placement (when Unit 6.4 lands, this file edits to import + render the component; the slot's structure stays stable).
   - `<noscript>` block linking to `https://github.com/bettyguo/OpenProblems/discussions` with a note describing how to find this problem's discussion thread (per ADR-0010 negative-consequence "JS-disabled visitors see no comments").
   - Back-link to the problem detail page.
   - Citation-style metadata (last-curated date, primary curator) deferred — not load-bearing on a discussion page.

2. **`app/problems/[slug]/page.tsx`** edit — add a one-line "Discuss" link below the curator-stamp line, mirroring the existing "View full leaderboard →" / "View full history →" inline links elsewhere on the page. No badge yet; that's Unit 6.5.

### Scope NOT in this unit

- **Giscus embed component** — Unit 6.4. This unit lands the slot; 6.4 fills it.
- **Activity-count badge on problem cards** — Unit 6.5. Consumes `getDiscussionByPath` from Unit 6.2.
- **Discussion thread integration in digest pipeline** — Unit 6.6.
- **Lighthouse enrolment of one talk page** — Unit 6.7.
- **Real-time activity count in the talk-page header** — could land in 6.5 alongside the card badge, OR be deferred. Default: defer. The talk-page header is informational; the count belongs on the card surface where it informs the click-decision.
- **Sitemap.ts file** — no sitemap surface exists at HEAD. Q48's "sitemap inclusion" promise will land as a separate hygiene unit if/when a sitemap is added. Not blocking.
- **Markdown / MDX content for `/talk` pages** — discussion threads ARE the content; no curator-authored MDX surfacing.

### Constraints

- **§5.5 perf budget** — First Load JS shared chunk = 103 kB held since Phase 1. This unit's page is pure SSG with `loadProblem` only; no new client-side dependencies; should not move the chunk.
- **§14.2 testing** — every page route needs a Playwright smoke (renders, no console errors, key landmarks present). E2E coverage is the test home for this route. No vitest test file needed for the page itself (matches the precedent — `app/problems/[slug]/leaderboard/page.tsx` has no vitest test either).
- **§15.6 primary-source rule** — does NOT apply to community comments (opinion-class, not editorial-claim-class). The shell itself contains no editorial claims requiring sources.
- **`exactOptionalPropertyTypes`** — page props are `{ params: Promise<{ slug: string }> }`; standard pattern.
- **Next.js App Router constraint** (the Phase-5 5.8 caveat): route files restrict exports to a fixed set. This unit's `page.tsx` exports only `default` + `generateStaticParams` — within the allowed set.
- **Path aliases**: `@/*` (project source); use for the loader import.

### Alternatives considered

- **Skip the placeholder slot — just empty `<section>`**. Rejected — a `<section id="discussions">` with explicit "Discussion loading via Giscus" placeholder text at HEAD makes 6.3-only deploys self-explanatory (a visitor reaching the route between 6.3 and 6.4 merges sees the slot label, not a blank section).
- **Land 6.3 + 6.4 in a single commit**. Rejected — splits cleanly per ADR-0009 D-A ("don't refactor surrounding code unless necessary") + Unit 6.0 ordering. The page-shell vs the client-only embed component are two distinct surfaces (server-rendered vs client-rendered); separate commits make review smaller + bisection cleaner.
- **Land 6.3 WITHOUT the problem-detail "Discuss" link** (defer to 6.5). Rejected — the talk route without an inbound link is dead weight at HEAD between 6.3 and 6.5. A one-line link is the right scope-creep level; 6.5 will UPGRADE it to include the count badge.
- **Render `getDiscussionByPath(pathname)` at SSG time** to show "N comments" in the header. Rejected — adds 10 build-time GraphQL calls (one per problem) for cosmetic value. Q47 (repo discussions enablement) means the queries return empty without operator action; the count would always show 0 at HEAD. Defer to 6.5 where it's the load-bearing feature.
- **Use a separate "Discussion" route at top-level (not nested under `[slug]/talk`)**. Rejected — the route mirror with leaderboard/history/ratings is the cleanest place-in-the-IA decision; nesting under `[slug]` reuses `loadProblem` + breadcrumb + 404 conventions for free.

### Tradeoffs / defensible defaults flagged

1. **Placeholder text at HEAD between 6.3 and 6.4**: "Discussion thread loading…" is the placeholder string. After 6.4 merges, the Giscus embed replaces the placeholder. Acceptable; the merge window is short.
2. **No `loading.tsx` for the route**: page is pure SSG, no async data fetch on render; no loading state to surface.
3. **No `error.tsx` for the route**: same reasoning. Errors come from `loadProblem` returning null → `notFound()` → standard 404 UI.
4. **Breadcrumb shape duplication**: same breadcrumb code as leaderboard / history / ratings sub-pages. Per §14 "no premature abstraction" — three near-copies is the minimum-friction shape; abstraction can land when a fifth sub-page joins.
5. **No SEO metadata override**: page inherits root layout title; Unit 6.7 (LHCI) may want to add `export const metadata` later but that's not load-bearing for 6.3.
6. **Page count delta**: 323 → **333** (+10 SSG routes; one per problem). Matches the Unit 6.0 D-6 prediction.
