### Path correction discovery: attributes shape

Looking at Phase-17 `baseSchemaConfig.attributes` in
`sanitize-schema.ts`:

```ts
attributes: {
  a: ["href"],
  input: [["type", "checkbox"], "checked", "disabled"],
  "*": [],
}
```

The `attributes` field is an object of per-tag attribute
allow-lists; entries can be:
- Plain string (attribute name allowed; any value)
- Tuple `[attrName, ...allowedValues]` (attribute name +
  value-restricted to the listed values)

For tables Phase 39:
- `<th>` and `<td>` should allow `align="left|center|right"`
  (GFM column alignment syntax).
- Tuple form: `["align", "left", "center", "right"]`.

**Per APPEND-D-C override-replace semantics, `schemaOverrides.attributes`
must include the FULL base list verbatim PLUS the new
entries**. The framework does NOT deep-merge.

### Schema override-replace + attributes wrinkle

The `attributes` object has multiple keys. Override-replace
at the top level (per APPEND-D-C) means the ENTIRE
`attributes` object is replaced. So
`schemaOverrides.attributes = { th: [...] }` would lose
`a`, `input`, and `*`. **Must include all base keys**:

```ts
attributes: {
  a: ["href"],
  input: [["type", "checkbox"], "checked", "disabled"],
  th: [["align", "left", "center", "right"]],
  td: [["align", "left", "center", "right"]],
  "*": [],
}
```

Documented explicitly in tests + ADR APPEND.

### Alternatives considered

**Option 1: Single `GFM_TABLE_SCHEMA_OVERRIDES` constant
(chosen)** — define the full schema override as a module-level
const; `TablesExtensionRegistry.getExtensions` returns
`{ schemaOverrides: GFM_TABLE_SCHEMA_OVERRIDES }` for enabled
surfaces. Simple; reusable; one source of truth for the
allow-list expansion.

**Option 2: Build override per-call (rejected)** — would
compute the override on each `getExtensions` call. Wasteful;
the override is a static value.

**Option 3: Inline override in registry method body (rejected)**
— harder to test in isolation; harder to document.

**Option 4: Per-surface override variants (rejected)** —
e.g., `TablesExtensionRegistry` enabling tables on
`reviewNotes` with one allow-list + on `bio` with a different
allow-list. Premature Phase 39 (single-surface scope per
Phase-39-prep D-4); Phase 40+ if multi-surface tables ship
with surface-specific schemas.

### Edge cases & failure modes

1. **Table without header row** (GFM requires a header row):
   `remark-gfm` won't parse it as a table. Falls through as
   plain text. No XSS concern.
2. **Malformed table syntax** (unbalanced pipes): `remark-gfm`
   handles parsing gracefully; produces partial tables OR
   plain text. No XSS concern.
3. **Nested HTML inside table cells**: only `tagNames`-allowed
   inside `<td>` / `<th>` (the existing allow-list applies to
   content within cells); rehype-sanitize enforces.
4. **`align` values outside the allowed set**: tuple form
   `["align", "left", "center", "right"]` restricts the
   attribute to those 3 values; any other value is stripped.
   Sanitization audit boundary preserved.
5. **Table on non-enabled surface** (`bio`, `rationale`,
   `actionRationale`): `TablesExtensionRegistry.getExtensions`
   returns `{}` per default-deny; table tags stripped per
   default schema; **Phase-17/27/29 baseline preserved**.
6. **Combined `wikilinks` + `tables` extensions desired**:
   Phase 39 mutually exclusive env-var dispatch — operator
   must pick ONE. Multi-value composition deferred Phase 40+
   per Phase-38-prep D-11.
