### Decisions pinned (per Unit 19.0 D-8 anticipated shape)

**D-A. Library choice.** `sharp@0.34+` — Node.js image processing
standard; native libvips bindings (10x faster than alternatives);
EXIF-stripping-default on `.toBuffer()`; server-only (preserves
103 kB First Load JS invariant); **may already be transitive dep
via `next/image`** (TBD verify Unit 19.2).

**D-B. EXIF allow-list.** Strip everything by default; explicit
`.withMetadata({...})` opts in to preserving. Phase-19 conservative
= strip all (GPS / camera serial / datetime / software / author /
copyright / comment) except color profile (ICC; via
`.withMetadata({ icc: true })` if observed needed Phase 20+) +
dimensions + encoding. **Phase 19 ships with no `.withMetadata()`
call** — strip everything.

**D-C. Pipeline placement.** `lib/storage/putAvatar` extension;
server-side; before Vercel Blob upload. `sharp(buffer).rotate().toBuffer()`
between `Buffer.from(await file.arrayBuffer())` and the Vercel
Blob `put()` call. Encapsulation in storage layer; future Phase
20+ transforms compose inside the same integration point.

**D-D. Auto-rotation preservation.** `sharp(buffer).rotate()` (no
args) BEFORE `.toBuffer()`. Reads EXIF Orientation tag (1-8),
applies rotation/flip to pixel data, resets Orientation to 1.
**iOS portrait shots render correctly across all clients
including those that ignore EXIF Orientation.**

**D-E. Backwards compatibility.** Phase 19 only strips on NEW
uploads. Existing Phase 16-18 avatars NOT retroactively
processed. Phase-20+ backfill script candidate (`scripts/backfill-exif-strip.ts`;
manual curator invocation; CLI dry-run flag; ~1-2 unit scope).

**D-F. Phase 20+ inheritance contract.** Phase-19 `putAvatar`
extension establishes the **transcoding pipeline integration
point**. Phase 20+ surfaces (Q68 expansion content moderation;
cropping UI; resizing; WebP/AVIF transcoding) insert their
transform between `.rotate()` and `.toBuffer()`. Transforms
compose; multiple Phase 20+ transforms can stack.

### Why an ADR at all?

Two reasons:

1. **First server-side image processing surface + first explicit
   privacy-by-default surface** in project history. Pinning the
   library choice + EXIF allow-list + pipeline placement +
   auto-rotation preservation + backwards-compat decision +
   Phase-20+ inheritance contract in ONE document prevents
   downstream drift when Phase 20+ image-processing surfaces land
   (Q68 expansion / cropping / resizing).
2. **Q70 promotion was explicit Phase 17+ deferral** (ADR-0017
   D-B + D-H): *"EXIF stripping — privacy concern (GPS metadata,
   camera serials). Defer until first user privacy report."*
   Phase 19 is the deferral-honor; **the privacy gap is closed
   pre-emptively** (single user privacy report would surface this
   immediately; conservative posture).

Contrast Phase 18: Q71 promoted to resolved WITHOUT new ADR
because the inheritance contract (ADR-0018 D-G) pre-documented
the pattern Phase-17. Phase 19's image-transcoding pipeline spans
new architectural ground (library choice + EXIF strip semantics
+ auto-rotation handling + Phase-20+ inheritance shape);
ADR-worthy.

### Options + chosen rationale (mirrors ADR body)

Five options considered:

1. **`sharp` server-side transcoding (CHOSEN)** — de-facto Node.js
   standard; native libvips 10x speedup; EXIF-strip-by-default;
   server-only; likely transitive via `next/image`; establishes
   Phase 20+ inheritance.
2. **`exifreader` / `exiftool`** — REJECTED. Read-only or
   external-binary; doesn't re-encode; net more work; `exiftool`
   bundled binary exceeds Vercel deploy size.
3. **`piexifjs` browser-side** — REJECTED. Adds ~10 kB client
   bundle; regresses 103 kB First Load JS invariant; trust
   boundary moves to client.
4. **Vercel Image Optimization API** — REJECTED. Read-side
   optimization only (image SERVE, not UPLOAD); doesn't apply to
   Phase 19's upload-path goal.
5. **Cloudinary / Imgix external service** — REJECTED. Per-image
   cost + operational gate (API key + billing) + vendor lock-in;
   over-engineered for Phase 19 scope.
