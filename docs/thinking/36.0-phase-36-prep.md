**Pivot rationale (from Phase-35-gate default (a) ADR-0025 to (b)
Q64)**: Phase-35 acceptance gate explicitly enumerated three Phase-
36 defaults: (a) ADR-0025 concrete content-moderation provider; (b)
Q64 per-user privacy opt-out; (c) pause for review. Default
recommendation was (a). User signaled "Continue" without thread
override — interpreted as "proceed with most defensible default."

On reconsideration at Phase-36 entry, **(a) ADR-0025 concrete
provider is harder to ship autonomously**:

- **OpenAI Moderation**: needs `OPENAI_API_KEY` operational gate;
  curator-track unblock.
- **Perspective API**: needs Google API key; curator-track unblock.
- **Regex-wordlist**: needs curator-authored wordlist; project
  shouldn't bake hardcoded slur/profanity content into the codebase
  (cultural sensitivity + legal/reputational concerns; curator
  policy decision).
- **Custom provider**: needs project-specific moderation logic
  authored by curator.

All four concrete-provider variants require either curator content
authorship OR external-API operational unblock — neither is
unblockable autonomously. The framework Phase 35 shipped is correct
and complete; the provider commitment naturally belongs to a curator-
driven phase or an operational-gate phase.

**(b) Q64 is more autonomously-tractable**:

- Self-contained code unit (schema + UI + filter; no external deps).
- Adds `users.profilePublic` boolean default `true` column +
  migration.
- Opt-out toggle UI on `/[locale]/profile` next to existing
  displayName + bio + imageOverride controls.
- `/[locale]/u/{handle}` returns 404 (or renders a `private-profile`
  notice) when target user has opted out.
- `getPublicProfileByHandle` filters opted-out rows (returns null
  so caller's `notFound()` fires).
- 4-8 i18n keys per locale.
- ADR-0015 D-A APPEND (extending the public-data invariant with
  the opt-out exception; opt-out is the user's right under the
  invariant rather than a violation of it).
- ~3 units of code + 1 docs hygiene + 1 gate = ~5 units.

**Rationale**:

1. **Longest open architectural Q at 15+ phase carryover** —
   strongest patience signal post-Q68 resolution. Phase 34 + 35 just
   resolved Q79 (0-phase) + Q68 expansion (12+ phase); Q64 at 15+
   phases is the natural next-longest closure target.
2. **Self-contained autonomous-shipping shape** — schema + UI +
   filter; no curator content authorship dependency; no operational
   API gate. Matches Phase 9-34 autonomous-shipping rhythm verbatim.
3. **Concrete user-visible privacy value** — opt-out is a real
   feature for users who want their participation private (curators
   submitting rating challenges they don't want their employer
   linking to; junior researchers who want to keep their watchlist
   private until they're ready to surface).
4. **Exercises ADR-0015 D-A privacy-model invariant explicitly** —
   D-A specified "watchlist counts + pending/accepted challenge
   counts are public"; opt-out is the user's right to opt out of
   that public surface entirely. Extending D-A with an opt-out
   exception is architecturally clean.
5. **Sets the public-data-exception pattern** for future surfaces
   that may want similar opt-outs (per-problem watchlist visibility
   Phase 36+; subscription visibility Phase 36+; rating-challenge
   submission visibility Phase 36+). The `users.profilePublic`
   boolean is the simplest possible shape; future per-surface
   opt-outs can extend with finer-grained columns OR a separate
   `user_privacy_preferences` table.
6. **No new ADR strictly required**. The opt-out is an extension of
   ADR-0015 D-A's public-data invariant (extension, not contradiction);
   ADR-0015 D-A APPEND-not-EDIT documents the realization. Saves
   ADR-0025 candidate slot for the eventual concrete content-
   moderation provider (Phase 37+ once curator authors moderation
   policy OR an external API key lands).
7. **§5.7 trigger (b) fires** — first schema migration since Phase
   33 `0008_subscriber_user_id`. Adds `users.profilePublic` boolean
   column with `default true` (existing rows backfill to
   `profilePublic = true` = public-by-default = matches Phase 9-35
   behavior verbatim).
8. **Tight scope (~3-4 code units)** matches Phase-32 and Phase-34
   no-new-ADR phase shape — schema unit + UI unit + hygiene + gate.
   Possibly 5 units if i18n + tests warrant a separate unit.

### D-3. Schema design anticipated (Unit 36.1)

D-A. Column = `users.profilePublic: boolean default true`. SQLite
boolean stored as integer 0/1; drizzle-orm `integer({ mode:
"boolean" })` generates `INTEGER NOT NULL DEFAULT 1` per the
established Phase-9 + Phase-15 pattern.

D-B. Default value = `true` (public-by-default; matches Phase 9-35
behavior verbatim; backfill preserves existing-user behavior). Users
must explicitly opt out via toggle on `/profile`; no "ask for
consent on first sign-in" Phase 36 (Phase 37+ if explicit-consent
UX warranted per ADR-0015 D-A privacy-model lens).

D-C. Migration = `0009_user_profile_public.sql` via `pnpm db:generate
--name user_profile_public`. One ALTER TABLE adding the column with
`NOT NULL DEFAULT 1` clause. Drizzle-kit's ALTER TABLE emission for
SQLite usually requires no manual edit for default-value columns
(unlike Phase-33's manual `ON DELETE CASCADE` add for FK). Verify
generated SQL at unit time.

D-D. `getPublicProfileByHandle` filter = adds `AND profilePublic =
1` to the existing `LOWER(githubLogin) = LOWER(?)` WHERE clause.
Opted-out users return null from this helper; caller's `notFound()`
fires; `/u/{handle}` returns 404. Mirrors Phase-9 retrofit-edge
filtering of NULL `githubLogin` rows.

D-E. UI = opt-out toggle on `/[locale]/profile` between
imageOverride section + Watching section. Single labeled checkbox
form with server action calling new `setProfilePublic(userId,
isPublic): Promise<null>` helper in `lib/users/`. Toggle label = "Make
my profile public" (checked = public; unchecked = opted-out =
private). i18n keys: `profile_edit.privacy_heading` +
`profile_edit.privacy_label` + `profile_edit.privacy_description` +
`profile_edit.privacy_public_status` + `profile_edit.privacy_private_status`.

D-F. Anti-scope (Phase 37+):

- **Granular opt-outs**: per-problem watchlist visibility; per-
  challenge submission visibility; subscription visibility. Phase
  37+ if signal — `users.profilePublic` is the coarsest possible
  shape; finer-grained columns OR a separate `user_privacy_preferences`
  table is Phase-37+.
- **Friend-only profiles** (vs strictly public/private). Phase 37+
  if signal — requires `user_relationships` table or similar.
- **Explicit-consent UX on first sign-in** (vs public-by-default).
  Phase 37+ if regulator/auditor signal — matches GDPR consent
  framework but adds friction.
- **Opt-out audit trail** (who opted out when; for support /
  enforcement). Phase 37+ if operational signal.
- **Bulk opt-out via CSV upload** (for users migrating from another
  platform). Phase 37+; out of MVP scope.

D-G. ADR-0015 D-A APPEND anticipated Unit 36.3 hygiene per the
APPEND-not-EDIT convention. Notes that Phase-36 opt-out extends
D-A's public-data invariant with the user's right to opt out of
the public surface entirely (NOT a contradiction of D-A; the user
controls whether the public-data invariant applies to their account).

D-H. **Private-profile UX on `/u/{handle}`**: pure 404 vs "this
profile is private" friendly notice page. **Lean = pure 404** to
minimize information leakage (404 doesn't reveal whether the handle
exists + is private vs doesn't exist; "private" notice reveals the
former). Mirrors GitHub's `/orgs/private-org` 404 behavior; mirrors
Twitter's deleted-account 404. Phase 37+ revisable if signal warrants
the friendlier notice.
