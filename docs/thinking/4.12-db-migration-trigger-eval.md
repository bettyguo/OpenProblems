# 4.12 — DB-migration trigger evaluation

> **Standalone evaluation note** anchoring the `MASTER_PROMPT.md` §12 DB-migration trigger check.
> Originally referenced in Unit 4.0's THINK doc; this unit promotes it to a top-level doc so
> Phase-5+ curators have a canonical re-check artifact rather than digging it out of CHANGELOG.

## Context

`MASTER_PROMPT.md` §12 (Phase 4 deliverables):

> "(Conditional) DB migration: if `gzip(content-snapshot) > 5MB` **OR** auth is needed for submissions, migrate structured data to Postgres/Drizzle. **Otherwise skip and revisit at Phase 5.**"

Two triggers, OR-joined:

1. **Size trigger.** `gzip(.velite snapshot) > 5 MB` (5,242,880 bytes).
2. **Auth trigger.** Authenticated user submissions required.

If neither fires, DB migration is deferred. Per §5.7, the trigger check is meant to be cheap + reproducible: run `tar -czf -` against the `.velite/` directory and compare the byte count.

## Measurements at Unit 4.12

HEAD = `25801f4` (post-Unit-4.11, just before this commit).

### Size trigger

```
$ tar -czf /tmp/snapshot.tar.gz -C .velite .
$ ls -l /tmp/snapshot.tar.gz
... 69182 bytes
```

| Quantity                        | Value                         |
|---------------------------------|-------------------------------|
| `.velite/` total (uncompressed) | ~465 KB (10 JSON files)       |
| `tar -czf` gzipped              | **69,182 bytes (~67.6 KB)**   |
| §12 trigger threshold           | 5,242,880 bytes (5 MB)        |
| **% of trigger**                | **~1.32%**                    |
| Headroom before trigger fires   | ~76× (current usage)          |

Content breakdown (per Unit 4.0 measurement; little movement since):
- `problemPages.json` — ~300 KB uncompressed (10 problem pages × multiple sub-pages MDX-compiled).
- `methodology.json` — ~40 KB (the §8 methodology MDX).
- `ratings.json` — ~54 KB (20 rating actions × 5 dimensions × rationale text).
- `papers.json` — ~27 KB (30 papers × metadata).
- `authors.json` — ~12 KB (126 authors).
- `problems.json` — ~19 KB (10 problems × benchmarks + tags + editorial).
- Other (`taxonomy.json`, `institutions.json`, `index.d.ts`, `index.js`) — < 10 KB combined.

The dominant byte budget is **`problemPages.json` (~65% of uncompressed)** — driven by MDX prose volume per problem page, not by structured metadata. The rating-action audit log (`ratings.json`) is the second-largest at ~12% and grows linearly with curator activity. Both compress well (the gzipped snapshot is ~15% of uncompressed).

**Size trigger: NOT FIRED. Massive headroom.**

### Auth trigger

§12's auth trigger is qualitative ("auth is needed for submissions"). Phase 4's submission workflow is:

- **Issue templates** (Units 4.7–4.10): 4 form-based templates on GitHub. GitHub handles auth — the contributor's GitHub identity is the audit trail; the encyclopedia itself stores no user accounts.
- **PR review**: curator reviews issues, drafts YAML, opens PR. Same GitHub-mediated auth flow.

Per §5.8 ("Auth deferred to Phase 4"):

> "Clerk or NextAuth.js v5 when needed. GitHub OAuth strongly preferred for community signal."

§5.8's "when needed" language defers to actual user-facing-submission infrastructure, not to issue templates. The §13 Phase-4 deliverable list does not include a write-path UI; that's Phase 5+ scope.

**Auth trigger: NOT FIRED. Phase 4's workflow is GitHub-mediated; no first-party auth.**

## Decision

**DB migration: deferred to Phase 5.**

Both §12 triggers are negative by large margins. `MASTER_PROMPT.md` §12 explicitly contemplates this case ("Otherwise skip and revisit at Phase 5.") — no override required.

### Re-evaluation triggers (when to re-check)

Re-run the size measurement and re-evaluate the auth picture when **any** of the following fires:

1. **Content scale 3×** — `pnpm validate-content` reports `> 600` files (currently 203), or the gzipped snapshot exceeds **1 MB**. The 1 MB internal alarm gives ~5× headroom before the 5 MB hard trigger; an early signal lets the migration plan land before the trigger.
2. **A v1 write-path lands.** The first Phase-5 deliverable that lets a non-curator user create a leaderboard entry, paper, or rating action **through the site** (not via PR / issue) flips the auth trigger.
3. **Phase 5 kickoff (regardless).** Per §12, Phase 5 must explicitly re-evaluate. The Phase-5 prep doc (`docs/thinking/5.0-phase-5-prep.md`, future) should include a fresh measurement.
4. **Rating-action volume reaches 200**. The ratings audit-log grows linearly with curator activity (20 actions today; gzipped contribution ~5 KB per ~10 actions). At 200 actions the contribution is ~100 KB — still well below 5 MB but worth a re-check given the linear vs. step-function difference vs. MDX prose.

### Forward-looking signals to watch

- **Storage-shape stress, not just size.** A trigger we don't have today: per-page query-performance complaints. The `.velite` snapshot is shipped as a static JSON; SSG pages read sliced subsets at build time. If a Phase-5 viz needs cross-cutting queries (e.g., "all rating actions across all problems where saturation > 50 in the last 6 months"), JSON-on-disk starts feeling cramped well before 5 MB. Migration may be motivated by query shape, not just byte count.
- **Real-time data freshness.** Static snapshots are baked at build time. If Phase 5+ adds a daily ingest cadence, every ingest is a re-deploy. A DB lets the cadence decouple from deploy cycles.
- **Multi-curator coordination.** ADR-0005 immutability + git PR review already handles concurrent curators well. If concurrency grows to 5+ active curators per week, optimistic-concurrency primitives a DB provides become relevant.

Each of these is a Phase-5 (or later) consideration. None fires today.

## Methodology version

This evaluation note doesn't modify any schema. It's a docs-only artifact. No methodology-version bump applies.

## Consequences

- **Positive.** Avoids the operational complexity a DB migration would introduce (Drizzle setup, connection pooling, migration scripts, secret management, separate CI test infra). The current file-first stack is shipping at 198 SSG routes / 103 KB First Load JS with zero DB-side state.
- **Positive.** Preserves the §4.5 "composable content via PRs" promise. As long as content is MDX + YAML + JSON on disk, every change is auditable in `git log`.
- **Negative.** Punts the migration to Phase 5; no rehearsal under current load. If Phase 5+ rolls a write-path, the migration could land under pressure rather than in calm conditions. Mitigation: ADR-0004 (file-first; no DB through Phase 3) is the durable record of the choice + its known trade-offs.
