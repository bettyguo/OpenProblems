# 4.8 — Issue template: new-paper

## THINK

### Inputs

- **Unit 4.0 D-8**: GitHub form-based `.yml` per template; 4–6 required fields + `description` blocks linking workflow docs.
- **Unit 4.0 Q37 lean**: minimum-viable schema; new-paper specifically needs arXiv ID + title + primary problem slug + lead-author ORCID per the lean.
- **Unit 4.7 (`de9460b`)** establishes the template convention: form-based `.yml`, `[<Type>]` title prefix, `<type>` + `needs-triage` labels, intro markdown block linking MASTER_PROMPT.md + relevant runbook + live taxonomy/index, freeform inputs over enumerated dropdowns when the underlying set evolves.
- **`PaperSchema`** (`lib/schemas/paper.ts`) required fields: `id`, `title`, `authors[]`, `institutions[]`, `year`, `tldr` (1–400 chars), `contributions[]` (each with `problem_slug` + `evidence` URL).
- **`docs/PAPER_INGEST_RUNBOOK.md`** is the curator-side workflow this template feeds into. The template doesn't replicate the runbook; it points to it.

### Scope (this unit)

`.github/ISSUE_TEMPLATE/new-paper.yml` — a form that captures enough signal for a curator to draft the full paper YAML. Final author/institution backfill, TL;DR drafting, and contribution-array authoring happen during the runbook pass.

### Constraints

- **Mirror Unit 4.7's structural decisions**: form-based YAML, auto-prefix title `[New paper] `, labels `new-paper` + `needs-triage`, freeform inputs where the underlying set (problems, benchmarks) evolves.
- **No required ORCID for lead author.** Unit 4.0 Q37's lean named it as required but author resolution is curator-side (per `lib/schemas/author.ts`'s ORCID regex + the parallel-session's Unit 2.5b backfill precedent). A contributor proposing a paper rarely has the ORCID handy. Demote to optional.
- **No required `tldr`**. 1–400-char abstract distillations are editorial work; the curator authors from the source. Demote to optional with `placeholder` showing the expected shape.
- **No required `authors[]` / `institutions[]`**. These are post-acceptance editorial backfill driven by the runbook + ROR / ORCID lookups (Units 2.5b / 2.6c / 2.6d precedents). The submission form gathers the paper, not the full author graph.
- **GitHub URLs**: pin to `github.com/bettyguo/OpenProblems` per `git remote -v` (same as Unit 4.7).

### Form-field schema

7 body items: 1 markdown intro + 6 form fields.

| # | Item                                  | Type       | Required | Notes                                                                  |
|---|---------------------------------------|------------|----------|------------------------------------------------------------------------|
| 1 | Intro markdown                        | markdown   | n/a      | Links to MASTER_PROMPT.md (§7 paper-object framing + §15.6 primary source), `docs/PAPER_INGEST_RUNBOOK.md`, `content/problems/` for problem-slug lookup. |
| 2 | arXiv ID or DOI                       | input      | yes      | The canonical identifier. Schema field: `Paper.id` (typically the arXiv ID, e.g. `2310.06770`). |
| 3 | Paper title                           | input      | yes      | Schema field: `Paper.title`.                                            |
| 4 | Primary contributing problem slug     | input      | yes      | Existing problem in `content/problems/<slug>/`. Helper text links the directory. Drives `Paper.contributions[0].problem_slug`. |
| 5 | Evidence URL                          | input      | yes      | Per §15.6 primary-source rule. Typically the arXiv abstract URL or paper PDF. Drives `Paper.contributions[0].evidence`. |
| 6 | Benchmark + score (if any)            | textarea   | no       | Free-text "benchmark-name: score (metric)" format. Helper text mentions the existing benchmarks under `content/problems/<slug>/problem.yaml`. |
| 7 | Additional context                    | textarea   | no       | TL;DR draft, related problems, authors / institutions if known, anything that speeds the runbook. |

### Alternatives considered

- **Require lead-author ORCID** (per Q37 lean). Rejected — see Constraints; demote to optional under "Additional context".
- **Require TL;DR**. Rejected — curator authoring during runbook pass is the precedent.
- **`dropdown` over the live problem slugs**. Considered. The set is more stable than the taxonomy (10 problems, slow churn) but still evolves; freeform with the directory link is more robust. Re-evaluate when problem count crosses ~50.
- **Multiple `contributions[]` entries upfront** (one form field per contribution). Rejected — most submissions name one primary problem. Curator multiplies contributions during YAML drafting. Single primary-problem-slug field keeps the form short.

### Tradeoffs / defensible defaults flagged

1. **Single primary-problem-slug field, not an array.** A paper may legitimately contribute to multiple problems; the form captures the headline one and the curator fans out during YAML drafting.
2. **Evidence URL required.** §15.6 is the project's anchor rule (no second-hand summaries). Requiring it at the form layer trains submitters to find the primary source upfront.
3. **No `venue` / `github` / `doi` fields.** All optional in the schema; curator backfills via the runbook.
4. **No `year` field.** Curator parses from arXiv metadata during ingest.
5. **Benchmark + score as freeform textarea, not structured input**: the schema has 4 optional fields (`benchmark_id`, `score`, `metric`, `rank_at_publication`); 4 inputs in a form is overhead. Freeform with helper text is faster for contributors.
