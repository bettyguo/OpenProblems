## THINK

### Problem restatement

Implement the `rehypeResolveWikilinks` rehype plugin per
Phase-38-prep D-3 D-A semantics + the `WikilinkExtensionRegistry`
class per D-3 D-D. The plugin walks the post-sanitize HAST tree,
finds text-node substrings matching `[[slug]]` pattern (kebab-
case `[a-z0-9-]+`), and replaces each match with an `<a
href="/problems/{slug}">{slug}</a>` element node. The
registry's `getExtensions(surface)` returns `{ rehypePlugins:
[rehypeResolveWikilinks] }` for enabled surfaces, `{}`
otherwise.

- HEAD `0c46b9a` (Unit 38.0 ship; Phase 38 prep doc).
- `lib/markdown/extensions/types.ts` — `MarkdownExtensionRegistry`
  + `MarkdownExtensionSet` + `MarkdownSurface` interfaces from
  Phase 37 Unit 37.1.
- `lib/markdown/index.ts` — `buildProcessor` pipeline (Phase 37
  Unit 37.2) that folds `extensions.rehypePlugins` AFTER the
  default `rehype-sanitize` + `rehypeStripUnsafeHrefs` steps per
  ADR-0018 D-G APPEND APPEND-D-D plugin-order-after-default
  discipline.
- `rehypeStripUnsafeHrefs` in `lib/markdown/index.ts` (Phase 17
  Unit 17.2) — strips hrefs that don't match
  `ABSOLUTE_HTTPS_OR_MAILTO`. **Wikilink hrefs (`/problems/{slug}`)
  do NOT match this regex** — but the wikilink plugin folds
  AFTER `rehypeStripUnsafeHrefs` so the strip step never sees
  the wikilink-emitted hrefs.

### Path correction discovery: rehype-sanitize boundary

**`rehype-sanitize` runs BEFORE the wikilink plugin in the
pipeline** (per `buildProcessor`'s ordering). So the wikilink
plugin emits `<a>` elements + `href` attributes that have NOT
been validated by `rehype-sanitize`. The security boundary
therefore relies on the plugin's regex `[a-z0-9-]+`:

- Slug pattern is restrictive: kebab-case only; no special
  chars; no path traversal; no scheme injection.
- Href format is fixed: `/problems/${slug}` — no user-
  controllable URL component beyond the slug itself.
- Slug-as-text is also safe (same character set as href).

**Documented in ADR-0018 D-G APPEND Phase-38 EXTENDED block**
as the wikilink plugin's explicit XSS-safety contract: the
regex IS the validation. Anything not matching `[a-z0-9-]+`
falls through as literal text.

### Alternatives considered

**Option 1: `rehypeResolveWikilinks` walks text nodes,
splice-replaces with mixed text + element nodes (chosen)** —
HAST text node containing `before [[slug]] after` gets
splice-replaced with three sibling nodes: text(`before `) +
element(`<a>slug</a>`) + text(` after`). Standard rehype-
plugin idiom; matches `unist-util-visit` API for in-place
child mutation.

**Option 2: `remarkResolveWikilinks` on the MDAST side
(rejected)** — would run earlier in the pipeline. Tempting
because MDAST has cleaner link-node primitives. **Why
rejected**: would fold via the `extensions.remarkPlugins` slot
which folds BEFORE `rehype-sanitize`; the wikilink-generated
hrefs would be subjected to `rehype-sanitize`'s
`protocols.href` check (rejects relative URLs) AND
`rehypeStripUnsafeHrefs` (rejects relative URLs). Wikilink
href `/problems/slug` is relative → would be stripped. Would
require `schemaOverrides` adding relative-URL acceptance,
which contradicts ADR-0018 D-D blanket relative-URL deny.
Cleaner architecturally: emit wikilinks AFTER both strip
steps via the `rehypePlugins` slot per Phase-37 framework's
APPEND-D-D plugin-order discipline.

**Option 3: Plugin parameterized with URL builder function
(rejected)** — `rehypeResolveWikilinks({ buildHref: (slug) =>
'/problems/' + slug })`. **Why rejected**: Phase 38 single-
purpose extension only resolves `[[problem-slug]]`. Future
Phase 39+ cross-entity wikilinks (`[[paper-id]]`,
`[[author-slug]]`, etc.) would need parametrization — but
that's Phase 39+ scope; over-engineering for Phase 38.

### Edge cases & failure modes

1. **Slug not matching `[a-z0-9-]+`**: e.g., `[[Slug]]` with
   uppercase or `[[my slug]]` with space → regex doesn't
   match → falls through as literal text. **No XSS risk** —
   plugin emits nothing for non-matching input.
2. **Adjacent wikilinks** `[[a]][[b]]` → two separate `<a>`
   elements with no text between. Tested.
3. **Wikilink at start/end of text node** `[[scalable-oversight]] is`
   or `is [[scalable-oversight]]` → leading/trailing text node
   handled correctly via slice + new text nodes.
4. **Multiple wikilinks in one text node** `see [[a]] and [[b]]
   for context` → 5 nodes synthesized: text + `<a>` + text +
   `<a>` + text.
5. **Wikilink inside an existing element** (e.g., bold
   `**[[slug]]**` → MDAST `<strong>[[slug]]</strong>`) →
   wikilink plugin walks `<strong>`'s text child via `visit`,
   splice-replaces text child with mixed nodes; `<strong>`
   now has `<a>slug</a>` as child. Tested.
6. **Wikilink with empty inner** `[[]]` → regex requires
   `[a-z0-9-]+` (at least one char) → doesn't match → falls
   through as literal `[[]]`.
7. **Unresolved slug** `[[nonexistent-slug]]` → still rendered
   as `<a href="/problems/nonexistent-slug">` pointing at a
   404. Phase-38-prep D-9 lean: no validation (matches existing
   markdown link behavior). Phase 39+ may add build-time
   validation.
8. **Visit-during-mutation infinite-loop risk**: the plugin
   splices NEW children into the parent. If `visit` re-visits
   the synthesized text/element children, the new text nodes
   don't contain `[[`, so no infinite loop. The element node's
   inner text is just the slug (kebab-case) which doesn't
   contain `[[`. Confirmed by returning `index + newNodes.length`
   from the visitor to hint `visit` past the inserted nodes.
