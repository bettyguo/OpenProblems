## Class A — 4 carried operational items (unchanged from Unit 11.5)

Phase 12 did not surface any new operational gates per se. The new env
var `LOP_CURATOR_LOGINS` is Phase-12-internal (curator allowlist; ADR-
0014 D-B); it does not require Vercel-side coordination beyond setting
the value once the curator-of-record decides who they trust.

1. **Q54 GitHub OAuth app registration** (operational; carried since
   Phase 9). Phase-12's curator dashboard inherits the same auth-side
   sign-in surface — until Q54 lands, sign-in into `/curator/challenges`
   surfaces `/api/auth/error?error=Configuration`. Curator must register
   under `bettyguo` GitHub org + add `AUTH_GITHUB_ID` /
   `AUTH_GITHUB_SECRET` / `AUTH_SECRET` to Vercel env.
2. **Q55 Turso production DB provisioning** (operational; carried since
   Phase 9). Phase-12's new `0003_rating_challenge_review` migration
   inherits — production DB needs the migration applied. `turso db
   create` + `turso db tokens create` + add `TURSO_DATABASE_URL` /
   `TURSO_AUTH_TOKEN` to Vercel env + run `pnpm db:migrate` against
   production once first.
3. **CI dummy `AUTH_SECRET`** (Phase-9 Class A item 3; carried). Auth.js
   v5 throws at module-load if unset; `safeAuth()` doesn't catch
   initialization errors. Production-mode CI builds will fail without
   env override.
4. **`pnpm db:migrate` doc for new contributors** in `/contributing`
   v1.2 (Phase-9 Class A item 4; carried). Now **4 migrations** to run
   on first setup (`0000_initial_auth` + `0001_watchlist` +
   `0002_rating_challenges` + `0003_rating_challenge_review`).

**Newly surfaced (but Phase-12-internal, NOT Class A operational)**:
- `LOP_CURATOR_LOGINS` env var (ADR-0014 D-B). When unset → empty
  allowlist → ALL signed-in users are non-curators → curator dashboard
  + review API return 403 for everyone. Acceptable failure mode (locks
  the dashboard down by default; explicit opt-in via env value). Treat
  as a Phase-12-internal gate rather than promoting to operational
  Class A.

## Class B — 14 Phase-12-specific follow-ons

Architectural follow-ons surfaced by Phase 12's surface. Each is
overridable as a Phase-13+ thread.

1. **CLI helper `pnpm emit-challenge-action <id>`** — automates ADR-0014
   D-D step 1-3 (reads challenge from DB; scaffolds the rating-action
   YAML at `content/problems/<slug>/ratings/<date>-<slug>.yaml` per the
   methodology v1.0 format; curator commits manually). Phase 13+. Mirrors
   Phase-5's `extract-leaderboard` shape. **Potential Q59 candidate** if
   manual emission proves cumbersome at Phase-12 acceptance smoke.
2. **Web-side automated emission** via GitHub Contents API + GitHub App
   installation. Phase 14+. Needs follow-on ADR for bot-identity vs
   curator-identity semantics (today the rating-action YAML's `curator`
   field MUST match the git commit author; ADR-0012 D-E + §13).
3. **Multi-stage curator approval** (editor-1 reviews → editor-2 accepts).
   Phase 14+ alongside editorial-board ([Q7](../../OPEN_QUESTIONS.md#q7-editorial-governance))
   resolution. ADR-0014 Option 4 rejected for Phase 12 (curator pool solo;
   doubles state-machine complexity).
4. **Per-problem ACLs** — curator-X can only review domain-Y challenges.
   Phase 13+ if curator workload signals demand. ADR-0014 D-B explicitly
   deferred per-problem ACLs.
5. **`curatorRoles` DB table + admin UI** — replaces env-var allowlist
   with DB-backed roles. Phase 14+ alongside editorial-board. ADR-0014
   D-B explicitly deferred.
6. **Full §8.6 24-month collaborator COI check** — joins DB
   `users.githubLogin` ↔ file-system `paper.authors[].slug` ↔
   `paper.year`. Phase 13+ when usage justifies the multi-hop traversal
   cost. ADR-0014 D-C explicitly deferred. Phase 12 ships hard-block
   self-review + soft-warn primary-curator; `reviewNotes` serves as
   human-audit trail for §8.6 attestation until automated.
7. **Email notifications when challenge is reviewed** — couples to
   subscriber-list (Phase-5 D-4 punt). Phase 13+ alongside `lib/email/`.
   ADR-0014 D-? rationale: submitter awareness via profile-page status
   updates (pull-based) is acceptable MVP.
8. **Per-challenge detail page at `/[locale]/profile/challenges/[id]`**
   — submitter-side detail view (mirrors curator-side detail view at
   `/[locale]/curator/challenges/[id]`). Phase 13+. Carried from
   Phase-11 Class B item 6 + Phase-12 Class B per UX-papercut anticipation
   (profile-page truncates rationale at 200 chars; submitter has no full-
   rationale view).
9. **Profile-page filter on challenge status** — show only `accepted` /
   only `rejected` / only `submitted` / only `under_review`. Phase 13+
   when volume justifies. Carried from Phase-11 Class B item 7.
10. **Curator dashboard URL search-param sort** — allow curator to
    re-sort by status / problemSlug / dimension / submittedAt /
    reviewedAt via URL `?sort=...&order=...` search params. Phase 13+.
    Default `createdAt ASC` (fairness queue per ADR-0014 D-F) sufficient
    for Phase-12 volume.
11. **Q58 public visibility surface** — counter on problem detail page
    + per-problem listing + public-profile-of-challenges. Phase 13+
    explicit per ADR-0014 scope cap. Couples to Phase-10 Class B item 1
    (public profile route at `/[locale]/u/[handle]`).
12. **Public profile route at `/[locale]/u/[handle]/challenges`** —
    couples to Q58 (challenge visibility) + Phase-10 Class B item 1.
    Phase 13+.
13. **Middleware-based auth-route protection** (Phase-9 Class B item 12)
    — threshold = 3+ protected page routes. **Phase 12 brings count to
    2** (profile from Phase 10 + curator dashboard from Phase 12; the
    `/curator/challenges/[id]` detail is a sub-route of the dashboard,
    not a separate authorization tier — counts as one). Lift fires
    Phase 13+ if a third protected page route lands (public-profile-of-
    challenges admin surface OR per-challenge submitter-side detail).
14. **Rate-limiting on POST `/api/v1/rating-challenges/[id]/review`** —

    bounded. Phase 13+ when broader unauthenticated surfaces land OR
