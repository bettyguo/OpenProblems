# Unit 0.1 — Initial ADRs (0001–0005)

> Stage 1 (THINK) artifact for Unit 0.1.

## Problem restatement

Land the five architectural decisions that `MASTER_PROMPT.md` §13 lists as Phase 0 deliverables, in a format (MADR) that future ADRs can extend without churn. These five lock in the choices that materially constrain every later unit:

- 0001 — Application framework (Next.js App Router).
- 0002 — MDX pipeline (Velite vs `next-mdx-remote`).
- 0003 — Schema source of truth (Zod).
- 0004 — Storage model for Phase 1–3 (files, no DB).
- 0005 — Rating-action immutability (audit-log discipline).

## Inputs

- `MASTER_PROMPT.md` §5 (tech stack rationale), §6 (repo structure rules), §7 (data model), §8 (rating methodology), §13 (Phase 0 ADR list).
- [MADR 3.0 template](https://adr.github.io/madr/) — chosen format.

## Constraints

- ADRs must be defensible at a workshop-paper level (per §8 ethos for the methodology page; same bar applies to architecture).
- Each ADR lists ≥ 2 alternatives with trade-offs (per §15.1 THINK rule, generalized to ADRs).
- ADRs are accepted on commit; superseding requires a new ADR, never an in-place edit (mirrors §6 rating-action immutability rule applied to architecture decisions).
- Numbering is monotonic and never reused.

## Alternatives considered (format / process, not content)

1. **MADR 3.0 short form (chosen).** Context → Decision Drivers → Considered Options → Decision Outcome → Consequences → Pros/Cons. Lightweight enough to write in 30–60 min per ADR; rich enough for a workshop reviewer.
2. **Nygard's original ADR template (rejected).** Simpler (Status / Context / Decision / Consequences). Misses the Decision-Drivers and explicit Pros-and-Cons sections; we want those for academic-credibility reasons.
3. **Y-Statements (rejected).** Single-sentence form ("In the context of X, facing Y, we decided for Z, to achieve W, accepting V"). Too compressed to capture trade-offs the master prompt cares about.

## Edge cases & failure modes

1. **Decider field unknown.** No GitHub handles are confirmed yet (`OPEN_QUESTIONS.md` Q7 covers editorial governance). Use `Author (primary curator, TBD GitHub handle — see Q7)` as a placeholder. ADRs are still valid; the role is what matters.
2. **Date drift.** All five ADRs are dated `2026-05-14` (today, matching the example rating-action filename in §6). If sign-off slips to a later date, ADRs are _accepted on commit date_, not authored date — leave the authored date in the file and rely on git history for the accepted date.
3. **Premature acceptance.** If the human disagrees with an ADR's chosen option, the ADR is marked `Status: rejected` and a new ADR with the next number proposes the alternative. We do _not_ edit a rejected ADR's chosen option in place.
4. **Velite maturity.** ADR-0002 picks Velite (Zod-validated, type-safe, fast), but Velite is a younger project than `next-mdx-remote`. If Velite proves unmaintained or blocks an upgrade later, the escape hatch is a new ADR (0002 stays as the historical record).
5. **Zod version churn.** Zod 4 arrived in 2025 with breaking changes from Zod 3. ADR-0003 must pin to Zod 4 explicitly so a future bump back to 3 (or forward to 5) requires a superseding ADR.

## Decision

Land seven artifacts:

1. `docs/adr/README.md` — ADR convention + MADR template pointer + numbering rule.
2. `docs/adr/0001-nextjs-app-router.md`
3. `docs/adr/0002-velite-for-mdx.md`
4. `docs/adr/0003-zod-as-source-of-truth.md`
5. `docs/adr/0004-file-first-no-db.md`
6. `docs/adr/0005-rating-action-immutability.md`
7. `docs/thinking/0.1-adrs.md` — this file.

Each ADR contains its own context / drivers / options / consequences. The per-ADR THINK content lives inline (Context + Considered Options + Pros/Cons sections), so no separate per-ADR THINK doc is needed.

DESIGN stage: the MADR template _is_ the design. No further design artifact.
