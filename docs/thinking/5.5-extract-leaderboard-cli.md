# 5.5 — `scripts/extract-leaderboard.ts` (CLI extracting leaderboard entries from a paper PDF)

## THINK

### Inputs

- **Unit 5.0 D-5 / D-6 / D-7**: CLI invocation `pnpm extract-leaderboard`; unified-diff drafts in `drafts/`; no auto-merge.
- **ADR-0008 D-B**: model default for `extract-leaderboard` = **Opus 4.7** (multi-table PDF parsing benefits from frontier capability).
- **Unit 5.3 (`25fd29e`)**: `scripts/ingest-arxiv.ts` establishes the CLI shape — positional input + `--model` / `--dry-run` / `--verbose` / `--no-cache` / `--out` / `--help`; `lib/curate/anthropic.ts` wrapper; `buildDraftFilenames` / `buildUnifiedDiff` helpers.
- **Unit 5.4 (`da50dbf`)**: `lib/curate/pdf-text.ts::extractPdfText(arxivId, options?)` returns extracted text + page count.
- **Unit 4.9 Q-aware design**: the LLM does NOT propose `verified: true`. Verification is curator-side. The script defaults `verified: false` on every new entry.
- **`LeaderboardEntrySchema`** (`lib/schemas/entry.ts`): required `paper_id`, `benchmark_id`, `score: number`, `date: YYYY-MM-DD`, `verified: boolean`. Optional `protocol_notes: string`.
- **`benchmark_id`** must match an entry in the parent problem's `problem.yaml::benchmarks[]`. The prompt enumerates those ids.
- **`content/problems/<slug>/entries.json`** is a JSON array (`[]`-rooted), one row per entry. Existing-vs-new merge logic lives in the script.

### Scope (this unit)

5 new files:

1. **`lib/curate/entry-draft.ts`** — pure helpers: `buildSystemPrompt(benchmarks)`, `buildUserPrompt(paperId, pdfText)`, `parseEntryArrayResponse(text)` (strips JSON fence, validates rows), `mergeEntries(existing, proposed, paperId)` (appends new with `verified: false`), `buildEntriesDiff(targetPath, existingBody, mergedBody)` (uses `diff` npm pkg).
2. **`lib/curate/entry-draft.test.ts`** — vitest tests for the pure functions.
3. **`scripts/extract-leaderboard.ts`** — CLI entry. Orchestrates: PDF text fetch (Unit 5.4) → problem.yaml benchmark loader → prompt build → LLM call (or dry-run) → response parse → merge → diff → write to `drafts/`.
4. **`package.json`** — new script `"extract-leaderboard": "tsx scripts/extract-leaderboard.ts"`.
5. **`drafts/` dir**: gitignored, already in place.

Plus new dep: **`diff@9`** (devDep). Generates `git apply`-compatible patches when the target file already exists (modify case). New-file diffs reuse `buildUnifiedDiff` from Unit 5.3's `paper-draft.ts`.

### Constraints

- **`--problem <slug>` is REQUIRED** (positional + required flag). The script needs to know which problem's benchmarks to feed the LLM and which `entries.json` to target.
- **`<arxiv-id>` REQUIRED** — the paper to extract from. The paper must already be ingested (`content/papers/<id>.yaml` exists). If not, the script aborts with "ingest the paper first via `pnpm ingest-arxiv <id>`".
- **The LLM does NOT propose `verified: true`**. Every new entry's `verified` is forced to `false` by the merge layer; the curator flips on review.
- **Multi-entry output**: a single paper often reports scores on multiple benchmarks. The LLM returns an array. Each row must satisfy the schema or it's rejected.
- **Benchmark-id guardrail (Unit 5.0 D-11)**: when the LLM proposes a `benchmark_id` not in the parent problem's declared benchmarks, the merged entry retains the proposed id but the prompt warned the model; the curator catches the mismatch at `pnpm validate-content` time.
- **Model default = Opus 4.7** per ADR-0008 D-B. `--model` overrides.

### CLI surface

```
pnpm extract-leaderboard <arxiv-id> --problem <slug> [options]

Options:
  --problem <slug>         REQUIRED. Existing problem to attach entries to.
  --model <id>             Anthropic model (default: claude-opus-4-7)
  --dry-run                Don't call the API; emit placeholder diff
  --verbose                Print prompt + estimated cost before the call
  --no-cache               Bypass .pdf-cache/ on PDF fetch
  --out <dir>              Override the drafts directory (default: drafts/)
  --help                   Show this help
```

Examples:
```
pnpm extract-leaderboard 2310.06770 --problem long-horizon-agent-reliability
pnpm extract-leaderboard 2310.06770 --problem long-horizon-agent-reliability --dry-run
pnpm extract-leaderboard 2310.06770 --problem hallucination-reduction --model claude-sonnet-4-6
```

### Prompt design

**System block** (cached):
- Excerpt of MASTER_PROMPT.md §9 (leaderboard framing) + §15.6 (primary-source rule).
- `LeaderboardEntrySchema` description (TS-style).
- The parent problem's declared `benchmarks[]` (id + name + metric + metric_direction).
- The paper's `id` (so the LLM knows what to put in `paper_id`).
- Output format spec: JSON array of `{ benchmark_id, score, date, protocol_notes? }` (paper_id implicit; verified is curator-side).
- Editorial rules: §15.6 no fabrication. If a score isn't explicitly stated, OMIT the row.

**User block**:
- The paper's full extracted PDF text.

**Response**: JSON array (parsed via `JSON.parse` after fence-stripping).

### Diff generation cases

1. **`entries.json` doesn't exist** (create): reuse `buildUnifiedDiff(targetPath, mergedBody)` from `paper-draft.ts`. The merged body is `[ <new entries...> ]`.
2. **`entries.json` exists with N existing entries** (append): use `diff@9::createPatch(targetPath, existingBody, mergedBody, "", "")`. The merged body is `[ ...existing, ...new ]`.

Both cases produce a `git apply`-compatible unified diff. The curator runs `git apply drafts/<file>.diff` to materialize.

### Alternatives considered

- **Single mega-CLI** combining ingest-arxiv + extract-leaderboard. Rejected per Unit 5.3 — separation makes each script's prompt + audit shape simpler.
- **Tool-use mode** (Anthropic's structured output) instead of free-form JSON parsing. Considered; rejected for Unit 5.5 — adds tool-schema authoring overhead; free-form JSON parsing is sufficient. Revisit if Unit 5.13 acceptance gate reveals frequent JSON-parse errors.
- **Apply directly** instead of diff. Rejected per §13 no-auto-merge.
- **Run extract-leaderboard automatically after ingest-arxiv**. Rejected — separate runs let the curator iterate on the paper draft before committing to the leaderboard extraction (which may need different review attention).

### Tradeoffs / defensible defaults flagged

1. **Opus 4.7 default cost**: ~$0.50–$0.75 per 50k-token PDF (per ADR-0008's note). Curator runs this 1-2× per paper, not per-row. Acceptable.
2. **Full PDF in the user prompt.** ~30k–50k tokens per academic paper. Prompt caching on the system block helps; the user block (the actual paper) is uncached because every paper is different.
3. **`verified: false` default**: forces curator review. Trade-off: every script-output entry needs at least one curator flip before it's "verified". Acceptable per Unit 4.9 design.
4. **Benchmark-id guardrail is post-hoc**. The LLM may propose a benchmark_id not declared on the problem; `pnpm validate-content` catches it on the eventual PR. Not a script-time enforcement.
5. **`diff@9`** introduces a new devDep. Mature, widely-used, ~10 KB. Acceptable.
