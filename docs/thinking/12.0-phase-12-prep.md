### D-3. Review-column migration shape (`0003_rating_challenge_review`)

The migration ALTERs `ratingChallenge` to add curator-review columns.
Per ADR-0013 D-B (migration immutability; mirrors ADR-0005's rating-
action-immutability ethos): NEVER edit `0002_rating_challenges`; the
review columns land as ADDITIVE schema deltas in a NEW migration file.

```sql
ALTER TABLE `ratingChallenge` ADD `reviewedAt` integer;
ALTER TABLE `ratingChallenge` ADD `reviewerId` text REFERENCES `user`(`id`) ON UPDATE no action ON DELETE set null;
ALTER TABLE `ratingChallenge` ADD `reviewNotes` text;
ALTER TABLE `ratingChallenge` ADD `acceptedActionId` text;
```

**Schema additions** (per ADR-0014 D-? to be pinned in Unit 12.1):

- **`reviewedAt`** integer timestamp_ms; nullable. NULL means
  `status ∈ {submitted, under_review, withdrawn}`; NON-NULL means
  the curator landed a `accepted`/`rejected` decision at that
  timestamp.
- **`reviewerId`** text FK to `user.id` with `ON DELETE SET NULL`
  (NOT cascade — if the curator's account is deleted, the review
  decision history should survive; the pointer becomes orphan but
  the row stays). Differs intentionally from `userId`'s cascade FK
  (deleting the SUBMITTER cascades the challenge; deleting the
  REVIEWER preserves it).
- **`reviewNotes`** text; nullable. Curator's free-text rationale
  for the decision (acceptance summary; rejection reason). App-level
  validation: max 4000 chars (longer than submitter's `rationale`
  to allow more detailed curator-side commentary).
- **`acceptedActionId`** text; nullable. Pointer to the rating-action
  YAML file emitted on acceptance. Stored as filename (e.g.,
  `2026-06-01-rationale-update.yaml`) for app-level join to
  `content/problems/<slug>/ratings/<filename>`. NULL when status ≠
  `accepted`; NON-NULL after the curator has committed the rating-
  action YAML out-of-band and attached the filename.

**Decisions NOT in this migration** (per ADR-0005 immutability +
explicit-evolution ethos):

- **No `withdrawnAt` column**. The status transition to `withdrawn`
  is recorded by `status = "withdrawn"` alone; the existing
  `createdAt` covers the submission timestamp. If audit-trail
  requirements emerge in Phase 13+, a `0004_*` migration adds
  `withdrawnAt` + `withdrawnBy` columns.
- **No COI-disclosure columns**. COI enforcement is server-side
  (Phase 12 lean: refuse review if curator = submitter); the
  decision itself is logged in `reviewNotes`. A future ADR may
  formalize COI as a structured field.
- **No `rejectionReason` enum**. Free-text `reviewNotes` covers
  both acceptance + rejection contexts; an enum would presuppose
  taxonomy. Phase 13+ if curator workload signals demand.

### D-4. Status transitions + state machine

Phase 12 lifts the `status` column from a single-value default to a
full state machine:

```
                  submitted
                 /         \
                ▼           ▼
        under_review    withdrawn (submitter only)
         /    |    \
        ▼     ▼     ▼
   accepted rejected (curator only)
```

Allowed transitions (server-side enforced; app-level state machine):

| From          | To            | Actor       | Trigger                                                |
|---------------|---------------|-------------|--------------------------------------------------------|
| `submitted`   | `under_review`| curator     | "Start review" button on curator dashboard             |
| `submitted`   | `accepted`    | curator     | "Accept" button (skip `under_review` for fast lane)    |
| `submitted`   | `rejected`    | curator     | "Reject" button (skip `under_review` for fast lane)    |
| `submitted`   | `withdrawn`   | submitter   | "Withdraw" button on profile-page row                  |
| `under_review`| `accepted`    | curator     | "Accept" button on detail view                         |
| `under_review`| `rejected`    | curator     | "Reject" button on detail view                         |
| `under_review`| `withdrawn`   | submitter   | "Withdraw" button on profile-page row (allowed mid-review) |
| `accepted`    | —             | (terminal)  |                                                        |
| `rejected`    | —             | (terminal)  |                                                        |
| `withdrawn`   | —             | (terminal)  |                                                        |

Implementation lives in `lib/rating-challenges/`'s new
`reviewChallenge(challengeId, reviewerId, decision, notes)` helper
(curator-side) + existing `withdrawChallenge(challengeId, userId)`
helper (submitter-side). Both helpers refuse INSERT-time if the
current `status` doesn't allow the transition.

**Decisions NOT in this state machine** (deferred):

- **No `accepted → re-opened` reversal**. Once accepted, the
  decision is final (mirrors ADR-0005's rating-action immutability;
  emitted YAML files are themselves immutable; reversing acceptance
  would orphan the YAML). Phase 13+ would need a new ADR to design
  a "supersede" surface.
- **No multi-stage approval** (e.g., editor-1 approves → editor-2
  reviews). Single-curator decision; matches Phase-9 single-tier
  authorization. Editorial board promotion is a Phase 14+ scope (Q7
  open).

### D-5. Curator authorization model (env-var allowlist)

Phase 12 lean per Unit 11.6 Q57 lean: env var `LOP_CURATOR_LOGINS`
(CSV of GitHub logins) gates curator routes.

```bash
LOP_CURATOR_LOGINS=bettyguo,jikun,otherperson
```

Implementation: `lib/auth/curator.ts` exports `isCurator(login):
boolean` reading the env var. Curator dashboard route + review API
both call `isCurator(session.user.githubLogin)` server-side before
rendering / accepting actions; reject with 403 otherwise.

**Why env-var, not DB column**:

- **Bootstrap-friendly**: no DB-side seeding required; new curator
  added by editing Vercel env. Mirrors Q47 Discussions operational
  gate pattern.
- **Auditable**: env-var changes flow through deploy logs.
- **Reversibility**: trivial to revoke (remove from CSV; restart
  deploy).
- **Avoids premature DB schema**: a `curatorRoles` table would
  require migrations + an admin UI to manage; both premature in
  Phase 12. Per ADR-0005's explicit-evolution ethos: defer the
  table until usage justifies.

**Phase 12+ deferred**:

- Per-problem ACLs (curator-X-can-review-domain-Y; curator-Z-can-
  review-everything). Phase 13+ if curator workload signals demand.
- Curator-roles table. Phase 14+ alongside editorial board.

### D-6. Curator-admin route shape

Curator dashboard at `/[locale]/curator/challenges` (third protected
page route; auth + curator-authz gated). Two sub-routes:

- **`/[locale]/curator/challenges`** — list view. Table of all
  challenges (any status), sortable by status / createdAt / problem
  slug. Default filter: `status ∈ {submitted, under_review}`
  ("pending review queue"). Empty state when queue is empty.
- **`/[locale]/curator/challenges/[id]`** — detail view. Single
  challenge with: submitter handle + submitted-date + problem
  context + dimension + proposedValue + rationale (full text, not
  truncated) + review form (textarea for notes + accept/reject
  buttons + optional "start review" button for `submitted` → `under_review`).

**Why two pages**, not single inline expansion: rationale text may be
~2000 chars; a list view truncates for scannability; the detail view
gives the curator full context. Matches §15.5 reviewer-mode mindset
("could a future curator misuse the surface" — yes, if they accept
based on a truncated rationale).

**Auth check**:

- Page-local check via `auth()` in the server component, mirroring
  Phase-10 `/[locale]/profile/page.tsx` pattern.
- `if (!session?.user?.githubLogin || !isCurator(session.user.githubLogin)) redirect('/')`.
- Mirror for API route(s) — `POST /api/v1/rating-challenges/[id]/review` (curator only;
  401 if no session; 403 if signed-in non-curator).

### D-7. COI policy enforcement (§8.6)

§8.6: "A curator must not rate a problem where they (or their direct
collaborators within the last 24 months) hold a current leaderboard
top-5 entry."

**Phase 12 simplification** (per Unit 11.6 Q57 lean "inherit verbatim";
refined for MVP):

- **Curator ≠ submitter**. A curator cannot review their own
  challenges. Strict refuse server-side. Surfaces "You cannot
  review your own challenge" disclaimer on the detail page; review
  buttons disabled.
- **Curator ≠ problem's `primary_curator`**. Soft check (warn, don't
  block). Reasoning: the problem's primary curator authored the
  current rating; reviewing a challenge to their own rating creates
  a self-review surface that §8.6 spirit forbids. Phase 12 displays
  a disclaimer ("You authored this problem's current rating — review
  carefully"); does NOT block. Phase 13+ may promote to a hard
  block based on usage observations.
- **24-month collaborator check** (§8.6 full): **DEFERRED to Phase 13+**.
  Requires joining DB `users` to file-system `paper.authors` to
  `paper.year` — a multi-hop traversal that's expensive to implement
  before usage justifies. The Phase-12 simplifications above cover
  the most common COI risk (self-review).

**Implementation**: `lib/rating-challenges/coi.ts` exports
`getCoIStatus(reviewerLogin, problemSlug, submitterLogin):
{ blocked: boolean; warning?: string }`. Curator dashboard renders
the warning UI; API route refuses if `blocked = true`.

### D-8. Acceptance → rating-action YAML emission

Phase 12 ships **manual emission**. Reasoning (per Unit 11.6 Q57 lean
+ Vercel read-only filesystem at runtime):

1. Curator clicks "Accept" on the detail view.
2. API route sets `status = "accepted"`, `reviewedAt = now`,
   `reviewerId = curator.id`, `reviewNotes = ...`, `acceptedActionId
   = null`.
3. UI shows a banner: "Challenge accepted. Now create the rating-
   action YAML at `content/problems/<slug>/ratings/<date>-<slug>.yaml`
   per [methodology](/methodology) §rating-action-format, commit it
   via PR, and **return here to attach the filename** in the
   `acceptedActionId` field."
4. Curator commits the YAML out-of-band (separate PR following the
   ADR-0005-enforced editorial workflow).
5. Curator returns to detail view; "Attach action YAML" form lets
   them set `acceptedActionId = "<filename>"`. Server-side validates
   the file exists at `content/problems/<slug>/ratings/<filename>`
   before persisting.

**Phase 12+ deferred** (per Unit 11.6 Q57 + Phase-5 D-4 punt):

- CLI helper `pnpm emit-challenge-action <id>` that reads DB +
  scaffolds the YAML locally for the curator to commit. Phase 13+.
- Web-side automation via GitHub Contents API (server-action commits
  the YAML via installation token). Phase 14+; needs follow-on ADR
  for the GitHub App installation requirement + bot identity for
  the commit.

  authz model; COI policy; emission semantics; ALTER discipline)
  need to be pinned BEFORE code lands. Mirrors Phase-9 precedent
  (ADR-0012 + ADR-0013 in Units 9.1 + 9.2, both before any auth /
  DB code).

  project; preferred dependency for every subsequent unit.

### Phase-12-blocking decisions deferred to per-unit implementation

- **D-13. ADR-0014 D-F decision on route depth** (Unit 12.1).
  Lean: two-page surface (list at `/[locale]/curator/challenges`;
  detail at `/[locale]/curator/challenges/[id]`). Single-page
  inline expansion rejected: rationale text too long for
  scannability.
- **D-14. Migration SQL ALTER vs. recreate** (Unit 12.2).
  Lean: ALTER TABLE 4 columns; SQLite supports `ALTER TABLE ADD
  COLUMN` natively (column adds; not column drops or renames).
  Drizzle-kit generates the ALTER statements automatically from
  the schema edit.
- **D-15. COI check return shape** (Unit 12.3). Lean:
  `{ blocked: boolean; warning?: string; reason: enum }`. Reason
  enum: `"self-review" | "primary-curator" | "collaborator-recent"
  | null`. Enables UI surface to discriminate by reason for
  curator-facing messaging.
- **D-16. Curator dashboard sort default** (Unit 12.4). Lean:
  default sort `createdAt ASC` (oldest pending first; "fairness
  queue"). Curator can re-sort via column header click — but
  client-side sort needs JS; for Phase 12, default is enough +
  Phase 13+ adds re-sort via URL search params (server-side).
- **D-17. Server-action vs API for review action** (Unit 12.4).
  Lean: API route `POST /api/v1/rating-challenges/[id]/review`
  for testability (mirrors Phase-9 watchlist + Phase-11 challenge
  submission); server-action wrapper on dashboard for
  zero-client-JS UX. Both paths converge on the same
  `reviewChallenge` helper.
- **D-18. Profile-page reviewNotes preview length** (Unit 12.5).
  Lean: 200-char truncation (matches Unit 11.4's rationale
  truncation precedent).
