# 7.3 — `app/[locale]/` layout + about-page wiring proof (deliberate scope cut)

Second code unit of Phase 7. Originally scoped (per Unit 7.0 §F) as "App Router locale segment: `app/[locale]/` restructure" — i.e., move every existing page under `app/[locale]/`, update params signatures, and add middleware with `localePrefix: "always"`. **Scope cut to a minimum-viable wiring proof**: ship the `app/[locale]/layout.tsx` + a single `app/[locale]/about/page.tsx` pilot that proves the next-intl rendering pipeline works end-to-end (server-side `getTranslations`, FR locale renders at `/fr/about`, NextIntlClientProvider passes through to client components). Bulk page migration + middleware deferred to a follow-on unit (provisionally 7.3a).

## THINK

### Why scope cut

The original 7.3 scope was a ~22-file structural change that would:
- Move every `app/<route>/page.tsx` under `app/[locale]/`.
- Update each page's `params` signature to include `locale: string`.
- Add `setRequestLocale(locale)` to each page.
- Update every `generateStaticParams` to expand the cartesian product (locale × dynamic params).
- Update every link-generator in `lib/`, `components/`, `app/api/` to include the locale prefix OR rely on middleware redirects.
- Update `lighthouserc.json` URLs to `/en/` prefix.
- Update `TALK_PATHNAME_REGEX` in `lib/discussions/github-graphql.ts` to handle locale-prefixed paths.
- Add `middleware.ts` with `localePrefix: "always"` per ADR-0011 D-B.

That's a 300+ line diff with high blast-radius risk (a single missed `setRequestLocale` call would crash an entire route at build-time). Per the unit-rhythm "ship small, ship often" principle and the fact that this session is already at 17 commits across 3 phases (5.13b → 7.2), the original 7.3 is too big for one safe commit at this point in the session.

**Scope cut**:
- This unit ships the wiring PROOF: `[locale]` layout + one pilot page + extended seed messages.
- A follow-on unit (provisionally 7.3a) does the bulk page migration + middleware atomically.
- Unit 7.4's FR pilot intent (per Unit 7.0 §F) is partially absorbed into this unit (the about-page IS rendered bilingually here as the wiring proof); 7.4 can shift to translating `/methodology` as the original plan intended.

This deviation is recorded transparently per Unit 7.0 D-3 carry-over.

### Inputs

- **[ADR-0011 D-A](../adr/0011-i18n-strategy.md)** — runtime = next-intl@^3.x.
- **[ADR-0011 D-D](../adr/0011-i18n-strategy.md)** — `fr → en` fallback with switch hint (relevant when 7.3a migrates real content; not exercised by the about-page pilot which has FR content from the seed).
- **[Unit 7.2](7.2-lib-i18n-runtime.md)** — `lib/i18n/routing.ts` + `lib/i18n/request.ts` + `messages/{en,fr}.json` + plugin already wired into `next.config.ts`. This unit consumes them.
- **`app/about/page.tsx`** — existing stub. Stays untouched. The new `app/[locale]/about/page.tsx` coexists; Next.js resolves `/about` to the static segment (existing stub) and `/<locale>/about` to the new dynamic-segment file.
- **`messages/en.json` + `messages/fr.json`** — get extended with an `about.{title, description}` namespace.

### Scope (this unit)

Three file additions + one edit:

1. **`app/[locale]/layout.tsx`** (new) — locale-aware layout that:
   - `generateStaticParams()` returns `locales.map(l => ({ locale: l }))`.
   - Validates the requested locale via `isLocale()`; calls `notFound()` for unrecognized locales (e.g., `/xx/about` 404s).
   - Calls `setRequestLocale(locale)` to enable SSG with the next-intl per-request context.
   - Loads messages via `getMessages()` from `next-intl/server` (uses the lib/i18n/request.ts loader transparently).
   - Wraps `{children}` in `<NextIntlClientProvider locale={locale} messages={messages}>` so client components downstream can use `useTranslations`.
2. **`app/[locale]/about/page.tsx`** (new) — about-page pilot that:
   - Awaits `params` to get `locale`.
   - Calls `setRequestLocale(locale)`.
   - Uses `getTranslations("about")` (server-side) to fetch the namespace.
   - Renders title + description in the requested locale. Visually mirrors the existing stub's `<StubPage>` shape but with translated content.
3. **`messages/en.json`** (edit) — add `about.title = "About"` + `about.description = "Project description, governance, and contact. Phase 1."` (matches the existing stub copy verbatim).
4. **`messages/fr.json`** (edit) — add the FR translations: `about.title = "À propos"` + `about.description = "Description du projet, gouvernance et contact. Phase 1."`.

### Scope NOT in this unit

- **Middleware** — deferred. Without middleware, bare URLs do NOT redirect to /en/<path>; both /about (existing stub) and /en/about (new pilot) coexist as separate routes. Acceptable for the wiring proof.
- **Bulk page migration** — every other `app/<route>/page.tsx` stays at its bare path; no `/en/<route>` exists for them yet. Deferred to a follow-on (provisionally 7.3a).
- **Link-generator updates** in `lib/`, `components/`, `app/api/` — bare paths still work without redirects; no updates needed at this scope.
- **`lighthouserc.json` updates** — URLs still bare; no LHCI change needed.
- **`TALK_PATHNAME_REGEX` update** — pathname-based Giscus mapping unaffected; `/problems/x/talk` still works at its bare path.
- **`app/[locale]/page.tsx`** (the home page under [locale]) — not added. `/en/` and `/fr/` would 404 (only `/<locale>/about` resolves). Acceptable; future units add additional pages.

### Constraints

- **Coexistence**: `app/about/page.tsx` (existing static segment) wins for `/about`; `app/[locale]/about/page.tsx` (new dynamic segment) wins for `/<locale>/about`. Next.js routes static segments preferentially over dynamic catch-alls, so there's no conflict.
- **§5.5 perf budget** — First Load JS shared chunk = 103 kB. The `NextIntlClientProvider` adds ~5-8 kB to the FR + EN about routes (the provider serializes messages + locale to the client bundle). Shared chunk should stay 103 kB since the provider is only on `app/[locale]/` routes; the existing pages (which dominate the route count) aren't affected.
- **`exactOptionalPropertyTypes`** — params resolves to `{ locale: string }`; isLocale narrows to `Locale`; setRequestLocale takes Locale; standard typed flow.
- **`getMessages()` from `next-intl/server`** — uses the lib/i18n/request.ts loader; no second message-load implementation needed.
- **API routes** unaffected — `app/api/...` is a separate tree; `[locale]` doesn't match `/api/<anything>`.

### Alternatives considered

- **Skip the pilot page entirely; just add the layout + middleware** — Rejected. Layout without a page under it doesn't validate end-to-end; the wiring proof needs a page to render against.
- **Use `useTranslations` (client) instead of `getTranslations` (server)** — Both work. Going with `getTranslations` (server) because the about page is purely informational; no client-side interactivity needed; smaller bundle.
- **Add app/[locale]/page.tsx (home)** — Rejected. /en/ and /fr/ rendering pilots can wait until the home page actually has translatable content (currently it's a complex composition of the recently-rated panel + problem list, none of which is internationalized yet).
- **Translate the existing /methodology page directly (the Unit 7.0 §F D-5 pilot target)** — Rejected for THIS unit. Methodology is MDX-driven; bilingual MDX needs the sibling-file pattern from Unit 7.5 (Velite collection extensions). Out of order. The about-page pilot is JSON-message-based which is what the runtime ships today.

### Tradeoffs / defensible defaults flagged

1. **Bare /about + /en/about + /fr/about all coexist**. Means visitors see THREE different about pages depending on URL. That's a documented intermediate state during the bulk-migration. Acceptable for the wiring-proof scope.
2. **No middleware** means bare URLs don't redirect; the "every URL has a locale prefix" goal (ADR-0011 D-B) is unmet at HEAD. Strict ADR adherence resumes when middleware lands in 7.3a.
3. **Unit 7.0 §F deviation** — about-page pilot replaces the methodology-pilot intent partly; Unit 7.4 shifts to methodology translation (now sibling-file pattern, which IS the more complete content-pluralism validation).
4. **2 new SSG routes**: /en/about + /fr/about. Page count: 333 → 335. Modest.
