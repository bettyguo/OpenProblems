## THINK

### Class A — In-flight Phase-9 items (4)

These are operational unblocks / configuration tasks that the curator
(or curator-track collaborator) must complete OUTSIDE the code surface
for Phase 9's deliverables to be fully exercisable. They are NOT bugs;
they are deliberate scope splits where Phase 9 shipped the
architectural surface and a separate (often single-step) operational
action turns the feature on.

1. **Q54 — GitHub OAuth app registration** (`open · operational`). Phase 9
   ships [`lib/auth/index.ts`](../../lib/auth/index.ts) configured for
   GitHub OAuth (per [ADR-0012](../adr/0012-auth-provider.md) D-B) but the
   OAuth app itself isn't registered. Steps for the curator:
   1. Register an app under the `bettyguo` GitHub org with
      `Homepage = https://llm-openproblems.org` (Q2 placeholder; might
      shift on DNS landing) + `Callback = .../api/auth/callback/github`.
   2. Register a second app for `http://localhost:3000` (dev).
   3. Add `AUTH_GITHUB_ID`, `AUTH_GITHUB_SECRET`, `AUTH_SECRET` (latter
      via `openssl rand -base64 32` or `npx auth secret`) to Vercel env
      (production + preview scopes).
   Until completed, sign-in clicks surface Auth.js's
   `/api/auth/error?error=Configuration` page. The architectural surface
   is complete; the operational gate is a once-off curator action.

2. **Q55 — Turso production DB provisioning** (`open · operational`).
   Phase 9 ships [`lib/db/index.ts`](../../lib/db/index.ts) with env-fallback
   from `TURSO_DATABASE_URL` → `file:./local.db`. Steps for the curator:
   1. Provision a Turso database via `turso db create llm-openproblems`
      (or via the dashboard).
   2. Generate an auth token: `turso db tokens create llm-openproblems`.
   3. Add `TURSO_DATABASE_URL` + `TURSO_AUTH_TOKEN` to Vercel env
      (production + preview scopes).
   4. Run `pnpm db:migrate` once against the new DB to apply
      `0000_initial_auth` + `0001_watchlist`.
   Until completed, Vercel-deployed instances attempt `file:./local.db`
   which **won't work on Vercel's read-only filesystem** — every
   request to a route that touches `auth()` returns a 500 unless
   `safeAuth()`-style wrappers catch (most routes will catch SSG-side
   but the watchlist API route would return 500). The architectural
   surface is complete; the operational gate is the provisioning +
   env-vars step.

3. **CI dummy `AUTH_SECRET` for build smoke**. Auth.js v5 throws at
   `NextAuth({...})` initialization (module-load time) if `AUTH_SECRET`
   is unset. `safeAuth()` does NOT catch this — the throw happens
   BEFORE any `auth()` call, during module evaluation. Today
   `pnpm build` works locally because `local.db` exists AND Auth.js v5
   tolerates missing `AUTH_SECRET` in development mode. CI under
   `NODE_ENV=production` will fail at the `lib/auth/index.ts` import
   chain. **Unblock**: inject a CI-only `AUTH_SECRET` env var (any
   32-byte base64 value; needn't match production since CI doesn't
   sign anything observable). Defer follow-on per "tactical hygiene
   not yet triggered" criterion (CI is green today on PR builds; risk
   surfaces only when `NODE_ENV=production` build runs without env
   override).

4. **`pnpm db:migrate` doc for new contributors**. ADR-0013 D-C says
   "new contributors run `pnpm db:migrate` on first setup" but no
   document currently tells them. Curator-track addition to
   `/contributing` v1.2 (or its successor) — same surface that Q51 +
   FR backfill will eventually touch. Defer to first contributor
   onboarding feedback or to a `/contributing` v1.x update unit.

### Class B — Phase-9-specific follow-ons (12)

Items that survive Phase 9 because they're either scope-creep (out of
the Phase-9 "auth foundation + ONE write-path" framing per Unit 9.0
D-5), waiting on an architectural unblock that wasn't granted this
phase, or tactical hygiene whose threshold hasn't been crossed.

1. **Profile page** (`/[locale]/profile` or `/[locale]/u/[handle]`)
   showing a user's watched problems + sign-out + (eventually) per-user
   preferences. Phase 10+ scope per Unit 9.0 D-5 explicit deferral.
   Watchlist data is queryable; the UI surface isn't built.

2. **Rating-challenge submission write-path** (§8.6 + Phase-3 deferred
   per Unit 9.0 D-5). Second write-path beyond watchlist; lives under
   `/api/v1/rating-challenges/[slug]` + a form on the problem detail
   page. Deferred because Phase 9 capped at ONE write-path to keep
   scope tractable; rating-challenge UX needs draft + curator review
   pipeline (more architectural surface than watchlist toggle).

3. **Email notifications when a watched problem gets a rating
   action**. Couples to the deferred email subscriber-list thread
   (Phase-5 D-4 punt). Requires email provider choice + transactional
   send infrastructure. Phase 10+ candidate; orthogonal to the
   watchlist storage that landed this phase.

4. **Watchlist count display on `/problems` index**. Surface
   "★ N watching" next to each problem in the index table. Trivial
   SQL (`SELECT problem_slug, COUNT(*) FROM watchlist GROUP BY 1`)
   but needs a UI design pass + signed-in / signed-out behavior
   (do we show the per-user "I'm watching this" badge or the global
   count?). Defer until the watchlist sees adoption traffic.

5. **Bulk-import / bulk-clear watchlist UI**. Power-user surface;
   premature without observed need. Deferred until a curator reports
   the use-case.

6. **Orphan-row cleanup script** per ADR-0013 D-F: `DELETE FROM
   watchlist WHERE problem_slug NOT IN (SELECT slug FROM <content>)`.
   Cron / pre-deploy hook to garbage-collect rows pointing at deleted
   `content/problems/<slug>/problem.yaml` files. Phase-9 explicitly
   tolerates orphans; cleanup is a future curator-track or
   ops-script unit.

7. **Rate-limiting on watchlist POST/DELETE** (`429 Too Many
   Requests` on excessive flips). Phase-9 surface is per-user only
   (auth-gated; per-user damage is bounded to their own watchlist), so
   abuse risk is low; ship rate-limiting when broader unauthenticated
   surfaces land (subscribe-form, public submission, etc.).

8. **Multi-provider OAuth expansion** (Google / GitLab / email-link).
   Per ADR-0012 D-B **forbidden in Phase 9**. Phase 10+ unblock via a
   follow-on ADR or explicit Q-promotion. The provider config slot is
   ready; only the addition is gated.

9. **`events.createUser` vs `events.linkAccount` choice**. Unit 9.6
   picked `linkAccount` for `users.githubLogin` population because
   `createUser` doesn't expose `profile.login`. When multi-provider
   expansion lands (Class B item 8), email-link sign-in WILL fire
   `createUser` without an OAuth profile; the `githubLogin` column
   stays null for those users (acceptable; column already nullable +
   unique-on-not-null). Document the design intent here in case a
   future curator wonders why both events exist in the codebase
   conceptually.

10. **First LHCI run validating auth-aware SiteHeader on `/en/`**.
    Unit 9.5 added `<AuthControl>` to SiteHeader; LHCI's existing 19
    URLs would surface any new a11y / performance regressions, but no
    LHCI run has executed against the post-9.5 surface yet (CI
    workflow runs on push; happens on the first PR with Phase-9
    commits). URL count holds at 19 (no enrolment changes). Survives
    until first preview-deploy PR.

11. **OAuth callback URL stability** (depends on Q2 DNS). If
    `llm-openproblems.org` is replaced by a different domain, the
    OAuth app's registered callback URI needs updating. Coupling not
    architectural — just a one-line change in the OAuth-app dashboard.

12. **Auth-aware route protection** (gating specific routes to
    signed-in users via `auth()` checks in `middleware.ts` or page-level
    early-return). Phase 9 has no protected routes (watchlist API
    self-protects via 401; the toggle UI degrades gracefully when
    signed out). Phase 10+ scope when surfaces like `/profile` or
    `/admin` need it.
