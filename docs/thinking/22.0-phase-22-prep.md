### D-3. CLI invocation + output shape

Mirrors ADR-0014 D-D pre-scoped contract + Phase-5 `extract-leaderboard`
drafts-first precedent:

```
pnpm emit-challenge-action <challenge-id> [--dry-run]
```

- **`<challenge-id>`** — required first positional arg. Matches a
  `ratingChallenge.id` UUID.
- **`--dry-run`** — prints the scaffolded YAML to stdout instead
  of writing to the drafts/ directory. Curator-inspection pass.
- **`--help` / `-h`** — print usage + exit-code matrix.

Output path: **`drafts/ratings/<challenge-id>.yaml`** (gitignored
per Phase-0 `.gitignore` `drafts/` pattern). Curator reviews +
edits + moves the file to
`content/problems/<problemSlug>/ratings/<date>-<slug>.yaml` +
commits manually. **Preserves ADR-0014 D-D "manual commit step"
requirement and ADR-0005 pre-commit-hook immutability protection**
(both fire only on actual `content/` commit — the gitignored
draft is invisible to both).

### D-4. Scaffolded YAML shape (5-step pipeline)

| Step | Action | Source |
|---|---|---|
| 1 | Read `ratingChallenge` row | DB `SELECT * FROM ratingChallenge WHERE id = ?` |
| 2 | Refuse if status terminal | `accepted` / `rejected` / `withdrawn` → exit 1 with "challenge already terminal; scaffold not applicable" |
| 3 | Refuse if reviewerId NULL | Challenge has not been claimed for review; emit-challenge-action runs AFTER curator decides to accept. Exit 1 with "challenge has no reviewer; assign reviewer via UI first" |
| 4 | Read reviewer's `users.githubLogin` | `SELECT githubLogin FROM user WHERE id = ?` — fills `curator` field per ADR-0012 D-E. Refuse if NULL |
| 5 | Generate YAML | RatingActionSchema shape — see breakdown |

**YAML shape** (`RatingActionSchema` from
`lib/schemas/rating-action.ts`):

```yaml
# Auto-scaffolded by `pnpm emit-challenge-action <challenge-id>` on <ISO date>.
# Source: rating-challenge <id> submitted by <userId> for problem <problemSlug>.
# Reviewer: <reviewerId> (githubLogin: <curator>).
#
# REVIEW REQUIRED:
#   - The proposed dimension (<dimension>) is fully scaffolded from challenge data.
#   - The OTHER 4 dimensions are placeholders — copy from the prior rating-action.
#   - `prior_action` is a placeholder — set to the filename of the most recent
#     rating-action for this problem (under content/problems/<slug>/ratings/).
#   - Move this file to content/problems/<slug>/ratings/<YYYY-MM-DD>-<descriptive-slug>.yaml
#     and `git add` + commit. Then return to /curator/challenges/<id> to attach
#     the filename via the acceptedActionId form.
problem_slug: <challenge.problemSlug>
date: <YYYY-MM-DD>
methodology_version: "1.0.0"
curator: <reviewer.githubLogin>
prior_action: "TODO-set-to-prior-action-filename"
dimensions:
  difficulty:        # PROPOSED IFF challenge.dimension === "difficulty"; else PLACEHOLDER
    grade: <parsed proposedValue OR placeholder "S">
    confidence: 0.7
    rationale: |
      <challenge.rationale OR "TODO: copy unchanged from prior action.">
  saturation:        # PROPOSED IFF challenge.dimension === "saturation"; else PLACEHOLDER
    value: <parsed proposedValue OR placeholder 0>
    confidence: 0.7
    rationale: |
      <challenge.rationale OR "TODO: copy unchanged from prior action.">
  urgency:           # similar pattern; stars: 0-5
  value:             # similar pattern; stars: 0-5
  industry_call:     # similar pattern; stars: 0-5
signals_considered:
  - "TODO: list signals (papers, leaderboard movements, industry announcements) considered."
watchlist: false
```

**Phase-22 lean**: pre-fill the 1 PROPOSED dimension fully from
challenge data; leave the OTHER 4 dimensions as placeholder
values with TODO comments. Curator MUST edit the placeholders
before committing (ADR-0005 + Velite + Zod schema validation
together catch placeholder values at commit time via the
`pnpm validate-content` pre-commit gate).

**Phase 23+ enhancement** (deferred): read the most recent prior
rating-action YAML for the same problem (via Velite's
`#site/content` or direct filesystem read), copy its dimensions
into the scaffold, set `prior_action` to its filename
automatically. ~1 additional unit when promoted.

### D-5. Dimension-value parsing

Per `RatingActionSchema` dimension types:

| Dimension | Value type | Parser |
|---|---|---|
| `difficulty` | `grade: "S" \| "A" \| "B" \| "C" \| "D" \| "E"` | string; passthrough |
| `saturation` | `value: 0-100 \| null` + optional `qualitative_band: "low"/"medium"/"high"` | `Number(proposedValue)` with NaN-fallback to placeholder; OR enum check + qualitative_band |
| `urgency` / `value` / `industry_call` | `stars: 0-5 integer` | `parseInt(proposedValue)` with range-check fallback |

`ratingChallenge.proposedValue` is a TEXT column (per Phase-11
Unit 11.0 D-3 + D-6); Phase-22 parsing is best-effort with
placeholder fallback when malformed. Curator catches malformed
values during the manual-edit pass.

### D-6. Pipeline placement

**Decision**: stand-alone CLI script at
`scripts/emit-challenge-action.ts`. NOT a server action / NOT an
API route / NOT a cron job. Manual curator invocation only —
**continues Phase-20 + Phase-21 operational-script-keystone
pattern**.

Why stand-alone CLI:
- ADR-0014 D-D explicitly says "preserves the manual commit
  step" — automation rejected for the YAML scaffolding step;
  drafts-first is the closest helper that respects this.
- Mirror Phase-5 `extract-leaderboard` shape (per ADR-0014 D-D
  wording).
- Drafts/ output is gitignored (Phase-0 contract); ADR-0005
  pre-commit hook fires only on actual `content/` commits.
