### Decision drivers (will be in the ADR proper)

1. **A11y first.** Phase-3 vizes (RatingHistoryStream, MoversBoard, SaturationCurve, RatingRadar) are all SVG-only with `role="img"` + `aria-describedby` → `<desc>`. Lighthouse a11y ≥ 95 is the cross-phase gate.
2. **Bundle weight discipline.** First Load JS shared chunk has held at 103 KB across all of Phase 3. Force-directed graphs introduce the largest dep delta of any Phase-4 unit.
3. **Operator familiarity.** SVG is the working pattern across every existing viz.
4. **Hydration safety.** Server-rendered SVG with deterministic positions → no client-hydration mismatch even when client interactivity is added later.

### Decisions (will be in the ADR proper)

**D-A. Render target = SVG.** Not Canvas, not HTML/CSS.

**D-B. D3 surface = tree-shaken sub-packages.** Specifically:
- `d3-force` for the force simulation.
- `d3-selection` reserved for the future client-drag follow-on (installed Unit 4.1, unused at Unit 4.2).
- `d3-scale` not installed — `Math.sqrt` covers the area-fair node sizing.
- `d3-zoom` not installed — zoom/pan scoped out per Unit 4.0 D-6.
- The umbrella `d3` package is forbidden (~150 KB gz vs. ~15 KB gz tree-shaken; bundle-discipline matters).

**D-C. Deterministic SSR layout.** d3-force normally seeds with `Math.random()`. DomainMap pre-seeds each node's `x` / `y` on a circle around the viewport center, indexed by input-array position. Initial positions + deterministic forces (forceLink + forceManyBody + forceCenter) → identical simulation output across renders.

### Considered options (rejected — will be in the ADR proper)

- **Canvas via `OffscreenCanvas`.** Better raw render speed; trades a11y (no semantic structure for AT) for performance we don't need at ~30 nodes.
- **HTML/CSS with absolute positioning.** Feasible for nodes, but force-graph edges need SVG `<path>` regardless — would end up with two render systems.
- **Umbrella `d3` package.** ~150 KB gz vs. ~15 KB gz tree-shaken; the 10× bundle penalty doesn't buy anything since we use ≤ 5 sub-packages.
- **React-native force-graph wrappers** (`react-force-graph-2d`, `cytoscape-react`, etc.). Most bundle Canvas; conflicts with the SVG-first choice; opinionated about state model.
- **ELK / hierarchical layout libraries.** Different algorithm class; force-layout reveals cluster structure that hierarchical layout doesn't.

### Consequences (will be in the ADR proper)

- **Positive:** Same a11y plumbing pattern as Phase-3 vizes (`role="img"` + `aria-describedby` + `<desc>`); Lighthouse-a11y ≥ 95 gate carries over without bespoke work.
- **Positive:** First Load JS shared chunk **remained 103 KB UNCHANGED** through Unit 4.2 (d3-force used server-side only; static SVG payload). The Unit 4.0 D-2 projection of +20–25 KB gz applies only when client-side filter chips wire into 4.3 / 4.4.
- **Positive:** Deterministic SSR makes hydration-mismatch a non-issue when the future drag follow-on adds `"use client"`.
- **Negative:** Manual force-layout tuning constants — empirically chosen, will need re-tuning if content scales 3× (e.g., Phase-5 ingest).
- **Negative:** `d3-selection` is installed but unused at Unit 4.2; until the drag follow-on lands, it's dead weight in `node_modules` (not in the client bundle — tree-shaking handles that).
