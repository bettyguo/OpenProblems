   Stale `subscriber` rows with `status = "pending_verification"` AND

   cleanup (subscribers who initiate verification but never click).
   This is a **DB-hygiene concern** that fires regardless of subscriber
   volume — even with 5 test subscribers, expired rows pile up. Other
   Phase-32+ threads (Q68 / Q64 / Q76 / Q78 / Markdown-evolution) all
   require **demand-signal-first decisions** the project hasn't
   observed yet.

   entry in `vercel.json` (daily or weekly cleanup schedule TBD); new
   `app/api/v1/cron/cleanup-stale-tokens/route.ts` thin POST handler;

   injectable-deps shape mirroring `lib/cron/digest-send.ts`; same
   `CRON_SECRET` auth via `checkCronAuth` helper Phase 31 extracted
   exactly for this multi-cron reuse case. **The Phase-31 surface was
   designed to be extended**; Phase 32 validates the extension shape.
3. **No new ADR required**. ADR-0022 already pinned Vercel Cron as

### D-3. Cleanup-cron pattern anticipated (no ADR; direct ADR-0022 D-A/D/F instantiation)

D-A. Cleanup logic = `DELETE FROM subscriber WHERE status =
'pending_verification' AND verificationTokenExpiresAt < now()`.
Removes only stale `pending_verification` rows; preserves
`verified` (subscribed users) + `unsubscribed` (audit trail per
ADR-0021 D-E). Side-effect-free aside from the DELETE.
D-B. Schedule = TBD per Unit 32.1 (lean: weekly Tuesday 00:00 UTC
to offset from digest send on Monday — runs after the digest send
loop completed; or daily — bounds maximum stale-row accumulation
window). 24-hour token TTL means a DAILY cron catches expired
tokens within 24 hours of expiry; WEEKLY catches them within 7
days. **Lean: WEEKLY Tuesday 00:00 UTC** (cleanup runs once per
week; bounded accumulation is ~7 days × abandonment-rate;
acceptable for foundation phase; daily can be promoted Phase 33+
if accumulation signal warrants).
D-C. Auth = `CRON_SECRET` bearer-token + `vercel-cron` header (same
as digest-send cron per ADR-0022 D-D; single shared secret).
D-D. Failure handling = single-row DELETE batch; per-row try/catch
not necessary since DELETE is atomic; route returns `{ deletedCount,
durationMs }` for observability.
D-E. Idempotency = inherent — `DELETE WHERE` is naturally idempotent
(re-run on same rows is a no-op).
D-F. Resend dependency = none (no email sends; pure DB cleanup).
