# 2.8 — Papers index `/papers`

## THINK

### Inputs

- **§13 Phase 2 deliverable.** "Paper / Author / Institution detail
  pages" — covered by Unit 2.3 (authors + institutions) and Unit 2.7
  (papers). The Phase 2 index for papers parallels Unit 1.8's
  `/problems` index and is named in the [Unit 2.0 plan](./2.0-phase-2-prep.md):
  *"2.8 Papers index — filterable + sortable list, replaces stub."*
- **30 papers committed across Units 2.4 / 2.5 / 2.6.** Coverage:
  hallucination-reduction (4) · long-horizon-agent-reliability (4) ·
  compute-optimal-test-time-reasoning (2) · mechanistic-interpretability (3) ·
  scalable-oversight (3) · multi-agent-llm-coordination (2) ·
  long-context-rag (2) · operator-learning-foundation-models (5) ·
  genome-foundation-models (2) · benchmark-integrity (3). Densest enough
  for filter UIs to be useful.
- **Pattern reference: [`/problems` index (Unit 1.8)](../../app/problems/page.tsx)
  + [`components/problems-index`](../../components/problems-index/index.tsx).**
  Server pre-computes a flattened `IndexedX[]`; client island owns
  filter + sort UI; navigation links to detail pages. Same shape here.

### Constraints

- **Client filter UI with `"use client"` island.** Server entry is RSC;
  filter / sort state is per-session and lives in client useState. No
  URL-state encoding (deferred to Phase 3, same as `/problems`).
- **No fabricated metadata.** Every facet (`year`, `venue`, problem
  filter) is derived from data on disk; no curated taxonomy of venues.
- **Composite paper rating not yet computed.** `/problems` shows §8.3
  composite; papers have no analogous aggregate. Defer to Phase 3 if
  the project decides papers need a sortable rating.
- **Linkage to detail.** Each row links to `/papers/[id]` (Unit 2.7's
  page). Click on a contribution's `problem_slug` chip filters the
  index to that problem (mirrors Unit 1.8's tag-chip behaviour).

### Indexed paper shape (`IndexedPaper`)

```ts
interface IndexedPaper {
  id: string;
  title: string;
  year: number;
  venue?: string;
  arxivId?: string;
  problemSlugs: string[];          // distinct, from contributions[]
  contributionCount: number;
  authorCount: number;             // resolved + unresolved (string count)
  institutionCount: number;        // same
}
```

Computed once, server-side, by `lib/content/load-papers-index.ts`. No
runtime joins on the client.

### Filter + sort surface

**Filters (3, mutually compose):**

- **Problem slug.** Dropdown of distinct slugs across all papers'
  contributions, sorted alphabetically. Matches via
  `paper.problemSlugs.includes(slug)`.
- **Year.** Dropdown of distinct years descending. Exact match.
- **Venue.** Dropdown of distinct non-empty venues, alphabetical
  (collapses "arXiv 2018" / "arXiv 2019" / … into per-year buckets
  because each appears as a distinct venue string).

**Sort keys (3):**

- **Year** (default, descending).
- **Title** (alphabetical asc).
- **Contributions** (count desc — papers that touch more problems
  rank higher; surfaces the cross-cutting work).

The sort header doubles as a toggle (active sort + arrow direction),
exactly matching `/problems`. "Clear" button resets all filters.

### Row shape

Per row:

- Title (links to `/papers/[id]`).
- Year + venue + arXiv badge (when present) on the meta line.
- Contributions count + author count + institution count chips.
- Problem-slug chips (clickable; toggles the problem filter).

### Alternatives considered

- **Tag-based filter on papers.** Papers don't carry a `tags[]` field
  (only `contributions[]`). The problem-slug filter is the closest
  analogue and is more meaningful — a paper's identity is more about
  which problems it touches than free-form tags.
- **Score-sorting.** No paper-level score is defined in the schema.
  Rejected — adding one without an ADR would violate §15.6's
  no-invent rule.
- **Author / institution filters.** With all 30 papers currently
  carrying empty `authors[]` (see Units 2.4 / 2.5 / 2.6), neither
  filter would surface anything. Defer to the author-backfill unit.

### Edge cases

- **A paper with no resolved problem.** Possible if a paper cites a
  problem slug that doesn't exist on disk. `IndexedPaper.problemSlugs`
  carries the raw slug; row renders the chip with the raw text
  (no link). Cross-link audit (Unit 2.11) surfaces these.
- **Empty filter result.** Render the same italic empty-state line
  that `/problems` uses.
- **`paper.venue` missing.** Don't include in the Venue facet (only
  papers with a venue contribute to the dropdown); render `—` on the
  row's meta line.
- **No client JS budget concern.** With 30 papers the JSON payload is
  small (~3 kB). Bundle target: `/papers` route < 5 kB + 117 kB
  First Load JS, mirroring `/problems`.

### Tradeoffs / defensible defaults flagged

1. **Server-pre-computed `IndexedPaper[]` over a build-time JSON
   index.** Mirrors `/problems` exactly. Fine at 30 papers; revisit
   if Phase 3+ pushes papers past ~500 and the JSON payload bloats.
2. **No URL-state for filters.** Same as `/problems`. URL state is a
   later-phase usability tweak when the project has a stable surface.
3. **No "verified" filter.** `LeaderboardEntry.verified` is the
   relevant flag, but entries (Unit 2.10) haven't shipped yet. When
   they do, this index gets a "show only papers with verified entries"
   toggle.
4. **No new tests.** Pure display + filter UI over already-tested
   data (the loaders have unit tests in Unit 2.2; the index is a thin
   transformation). Visual regression baseline lands in Unit 2.13.
