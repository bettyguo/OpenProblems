# 9.6 — Watchlist write-path (§5.7 trigger (a) FIRES — first first-party write)

Sixth code unit of Phase 9. The **watershed unit** of the phase (and arguably the
project): lands the first write-path against the project's own DB and thereby
fires the §5.7 DB-migration trigger (a) that the project has deferred for 5
phases running (Unit 4.12 → 5.10 → 6.0 → 7.0 → 8.0 → 9.0 D-2 all logged "still
cold; flips on first auth write-path"). Phase-4 "no user accounts" pact is
broken here per the announced consequence — no editorial content moves; only
user-state lands per [ADR-0013](../adr/0013-db-choice.md) D-F.

### Watchlist table schema (Q56 lean realization)

```ts
export const watchlist = sqliteTable(
  "watchlist",
  {
    userId: text("userId")
      .notNull()
      .references(() => users.id, { onDelete: "cascade" }),
    problemSlug: text("problemSlug").notNull(),
    createdAt: integer("createdAt", { mode: "timestamp_ms" })
      .notNull()
      .default(sql`(unixepoch() * 1000)`),
  },
  (table) => ({
    pk: primaryKey({ columns: [table.userId, table.problemSlug] }),
  }),
);
```

Three notes:

- **Composite PK** = `(userId, problemSlug)` (per Q56). Index implicitly
  covers per-user lookups via prefix scan; per-problem reverse lookups would
  need a separate index, which we defer until a use-case lands (none in
  Phase 9).
- **No FK on `problemSlug`** per Q56 lean + ADR-0013 D-F. Plain `text`
  column; the file-system `content/problems/<slug>/problem.yaml` stays the
  source of truth for problem metadata. Orphan rows (slug pointing at a
  deleted problem.yaml) tolerated until a cleanup script lands (deferred
  follow-on). Orphans don't break the UI: the toggle renders only if the
  problem detail page loads, which already calls `loadProblem(slug)`; an
  orphan slug never reaches the toggle.
- **`createdAt`** mirrors the `users.createdAt` convention from Unit 9.3
  (timestamp_ms via `unixepoch() * 1000`). Used for ordering "recently
  watched" lists in Phase 10+; unused in Phase 9 but cheap to include now
  (avoiding a future migration to add it).

### `events.linkAccount` for `githubLogin` population

The cleanest Auth.js v5 path to populate `users.githubLogin` on first sign-in:

```ts
events: {
  async linkAccount({ user, account, profile }) {
    if (account.provider !== "github") return;
    const login = typeof profile?.login === "string" ? profile.login : null;
    if (!login || !user.id) return;
    await db.update(users).set({ githubLogin: login }).where(eq(users.id, user.id));
  },
},
```

The `linkAccount` event fires:

- AFTER the user is created (via the adapter's `createUser`) AND the OAuth
  account is linked (via the adapter's `linkAccount`). The user row exists,
  the FK is intact, the GitHub profile is fully available.
- Once per OAuth-account-link. For users who sign in via GitHub and then
  later add another provider (Phase 10+ scope), the GitHub `linkAccount`
  fires only on the original GitHub link, not on subsequent provider links.

Alternative considered (and rejected): customize the `GitHub` provider's
`profile` callback to return a `{ ..., githubLogin: profile.login }` object,
relying on the Drizzle adapter to persist `githubLogin` on `createUser`.
Rejected because Auth.js v5's `users` table schema is fixed by the adapter
contract; adapter-side persistence of arbitrary extra fields is not part of
the documented v5-beta API and may break with adapter upgrades.

Alternative considered (and rejected): use `events.createUser` instead.
Rejected because `createUser` fires before the account is linked, and the
`profile` argument is NOT in scope (only `user` is). We need the GitHub
profile's `login` field, which only `linkAccount` exposes.

### API route — POST vs DELETE vs toggle

Three viable shapes:

1. **POST adds; DELETE removes (chosen)** — REST-clean; idempotent on both
   methods (POST uses `onConflictDoNothing`; DELETE is a no-op if absent).
   UI uses two distinct server actions; each form posts to its matching
   verb.
2. **POST toggle** — single endpoint; flips state. Simpler UI (one form);
   but non-idempotent (each POST flips). Reload-safe only with explicit
   client logic.
3. **PUT replace** — REST-pure for idempotent state-set; verb semantics
   awkward for boolean state (no body needed).

Picked #1 for REST cleanness + idempotency. UI gets two server actions
(`addToWatchlist`, `removeFromWatchlist`) called from two distinct form
elements; current state drives which form renders.

Test coverage on the route: per-method × per-exit-shape (3 shapes × 2
methods = 6 tests). Mock `lib/auth` for the 401 path; mock `lib/watchlist`
helpers for the 200 path; rely on real `#site/content` for the 404 path
(slug "unknown-problem" definitely isn't in the seed).

### Server-action vs API-route — both, distinct purposes

- **Server actions** drive the UI (form submission → server-side toggle →
  `revalidatePath` → re-render). Internal to the app. No HTTP overhead.
- **API route** is the REST surface for programmatic access (future:
  third-party integrations, mobile apps, scripts). Same underlying helpers
  (`addToWatchlist` / `removeFromWatchlist`) from `lib/watchlist/`; the
  route does the HTTP shaping (verb → helper → JSON response).

Both call the same `lib/watchlist/` helpers so logic stays consistent.

### Edge cases & failure modes

1. **Unknown slug** (POST /api/v1/watchlist/does-not-exist): 404 with
   `{ error: "unknown-problem" }`. Server action also guards: if slug isn't
   in `problems` from `#site/content`, the toggle button doesn't render
   (the page wouldn't have reached this code path — `notFound()` fires
   earlier in `ProblemPage`). The API guard is the belt-and-suspenders.
2. **No session** (POST without auth): 401. Same on DELETE. Server action
   for the signed-out branch never fires (UI renders a sign-in CTA, not a
   form); the API route is the only entrypoint for unauthenticated callers
   and returns 401 cleanly.
3. **Double-watch** (POST when already watched): 200 (idempotent via
   `onConflictDoNothing`); returns `{ watched: true }`. Safe to retry.
4. **Double-unwatch** (DELETE when not watched): 200 (delete is a no-op);
   returns `{ watched: false }`. Safe to retry.
5. **Concurrent toggle from two tabs**: both flip the state independently;
   last-writer-wins. No locking. Acceptable for Phase 9 (per-user surface;
   user-driven not script-driven; no data loss — just UI inconsistency
   between tabs until next render).
6. **`local.db` missing / `sessions` table missing** (CI / fresh clone):
   `auth()` throws; the API route's `auth()` call doesn't have a
   `safeAuth()` wrapper (the route is a code path, not an SSG-time render;
   a throw here surfaces as 500 to the caller, which is correct
   — the system genuinely can't authenticate). SiteHeader's `safeAuth()`
   keeps the SSG build green. Tests mock `lib/auth` to avoid the throw.
7. **CI build with no DB**: SSG of `/[locale]/problems/[slug]` calls
   `<WatchlistToggle>` which itself calls `auth()` (via a `safeAuth()`-
   style wrapper) → returns null on failure → signed-out branch renders →
   no DB read for `isWatched`. SSG-safe.
8. **Slug with special characters** (URL-encoded): Next.js's dynamic route
   already decodes; `decodeURIComponent` is not needed. Slugs are
   kebab-case per `lib/schemas/_primitives.ts`, so no special characters
   in practice.

### Composition with existing surfaces

- **Middleware**: composed from Unit 9.5 (auth-outer + intl-inner). API
  routes under `/api/...` are excluded from the matcher regex
  (`/((?!api|_next|_vercel|.*\\..*).*)`), so the watchlist API route runs
  outside the intl wrapper — clean.
- **`#site/content`**: API route imports `problems` from `#site/content` to
  validate slug. Velite already builds this index; the API route is a
  server-only consumer; bundle impact zero.
- **`lib/auth`**: API route imports `auth` to read the session. The
  `safeAuth()` wrapper in SiteHeader does NOT apply here (different exit
  semantics: SSG → render the signed-out branch; API → return 401).
- **`lib/db`**: API route imports `db` via `lib/watchlist`. Standard.
