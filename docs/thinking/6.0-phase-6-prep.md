### D-1. First-thread recommendation: **GitHub Discussions integration**

**Recommendation**: lead with the GitHub Discussions thread (§13 bullet 1). Surface a `/problems/[slug]/talk` page on each problem detail; embed a discussion widget (Giscus or first-party GraphQL); surface activity counts on problem cards; thread metadata into the digest pipeline.

**Rationale**:

1. **Lowest blast radius.** No first-party auth (Discussions uses GitHub's OAuth by way of the embed). No DB trigger flip (no first-party storage of user identity). No Velite schema changes. Pure additive routes + components.
2. **Closest to the rating-agency framing** (per `project_llm_openproblems.md`). The encyclopedia rates open problems; the rating loop is fed by community discussion. Discussions are the *signal* the rating process consumes; they belong before the rating-challenge auth UX.
3. **Lets the auth thread marinate.** Committing to a provider (Clerk vs NextAuth.js v5 vs GitHub OAuth direct) before knowing the first-party write-path UX requirements is premature. Discussions defer the choice without blocking value.
4. **Independent of bilingual.** A bilingual investigation can run as a parallel-curator session against Phase 6 main without colliding with Discussions code paths (different file trees: `lib/discussions/` vs `lib/i18n/`).
5. **Scope manageable.** Estimated 6-8 implementation units + 2-3 hygiene/gate units = a tight Phase 6 (~10 units), well under the Phase 0-5 cadence of 14.

### D-3. Discussions backend choice (deferred to Unit 6.1 ADR-0010)

Three viable backends for the Discussions integration:

1. **GitHub Discussions GraphQL API direct** — first-party query against repo discussions; deeper integration; needs `GITHUB_TOKEN` for read (rate-limited but acceptable).
2. **Giscus embed** ([giscus.app](https://giscus.app)) — iframe widget backed by GitHub Discussions; no token needed for the embed itself; community-maintained.
3. **Utterances** — backed by GitHub Issues (not Discussions); older, simpler, but issue-based instead of discussion-based.

Lean (subject to ADR-0010 in Unit 6.1): **Giscus for the embed + GraphQL for the read-side metadata** (counts, last-comment-at, surfaced on problem cards + digest). Giscus handles the auth-via-GitHub UX without us building it; GraphQL gives us programmatic access for the metadata side.

Deferred to ADR-0010 (Unit 6.1) for the realized contract.

### D-4. Discussions-per-problem mapping

GitHub Discussions are categorized under a single repo. Two options for problem ↔ discussion mapping:

1. **One discussion per problem** — pre-create discussions; map `/problems/<slug>/talk` to a fixed discussion ID via a `talk_discussion_id` field in `problem.yaml`. Requires admin pre-seeding; clean separation.
2. **Discussion-per-talk-thread, discovered by title** — Giscus's default mapping mode (e.g., map by `og:url` or `pathname`); creates a discussion lazily on first comment.

Lean (Unit 6.1 ADR-0010): **Option 2** (pathname-based mapping; lazy creation). Removes the admin pre-seeding burden; matches Giscus's documented happy path; problems without any talk activity simply have no discussion (no orphan threads).

### Open questions newly surfaced

**Q46. Discussions-backend tradeoff** — Giscus embed vs first-party GraphQL build (server-rendered discussion thread). Lean (Unit 6.1 ADR-0010): Giscus embed for the comment UI; first-party GraphQL for the read-side metadata only. Decoupling the two means the metadata refresh is build-time + cheap; comment-rendering stays on GitHub's infra.

**Q47. GitHub repository discussions enablement** — discussions must be enabled in repo settings for any of this to work. Requires owner action on `github.com/bettyguo/OpenProblems`; out-of-band for this docs unit; check before Unit 6.2 ships.

**Q48. Talk-page indexing posture** — should `/problems/<slug>/talk` pages be in the sitemap / linked from problem detail? Lean: yes, sitemap-included; surfaces with a "Discuss" link from `app/problems/[slug]/page.tsx`. Revisit after talk-page Lighthouse a11y baseline lands (Unit 6.7).

**Q49. Comment moderation routing** — when a Giscus comment is flagged in the embed, where does the curator chain pick it up? Giscus delegates to GitHub Discussions' native moderation; this question is "do we need a first-party moderation queue alongside?" Lean: **no** — defer entirely to GitHub Discussions' moderation; revisit only if curator workload signals demand it.
