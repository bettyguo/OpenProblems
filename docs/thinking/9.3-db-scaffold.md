# 9.3 — DB scaffold: Drizzle + libsql + initial migration

Third code unit of Phase 9 (first that touches `node_modules`). Lands the database layer pinned by [ADR-0013](../adr/0013-db-choice.md): `drizzle-orm` + `@libsql/client` runtime + `drizzle-kit` dev dep; schema with the NextAuth.js v5 canonical tables (`users`, `accounts`, `sessions`, `verification_tokens`) plus the `githubLogin` column from [ADR-0012](../adr/0012-auth-provider.md) D-E; first migration; local-dev `local.db` setup via env-fallback.

## THINK

### Scope

1. **Install** — `drizzle-orm@^0.x` + `@libsql/client@^0.x` (`dependencies`); `drizzle-kit@^0.x` (`devDependencies`). Resolved versions: `drizzle-orm@0.45.2` + `@libsql/client@0.17.3` + `drizzle-kit@0.31.10`.
2. **`lib/db/schema.ts`** — TypeScript-first schema definitions for the four NextAuth canonical tables + `githubLogin` text-unique column on `users` + `createdAt` timestamp default. Type cast `$type<AdapterAccountType>()` on `accounts.type` deferred to Unit 9.4 (when `next-auth` installs) — runtime is a plain `text` column either way.
3. **`lib/db/index.ts`** — Drizzle client export. `createClient` from `@libsql/client` with env-fallback (`TURSO_DATABASE_URL` → `libsql://...turso.io`; otherwise `file:./local.db`). `drizzle()` wrapped with the schema.
4. **`drizzle.config.ts`** — Drizzle-Kit configuration. `dialect: "turso"` (new convention in 0.31.x; supersedes the deprecated `dialect: "sqlite"` + `driver: "turso"` pair); `schema: "./lib/db/schema.ts"`; `out: "./lib/db/migrations"`; `dbCredentials` from env-fallback.
5. **`package.json` scripts** — `db:generate` (drizzle-kit generate), `db:migrate` (drizzle-kit migrate), `db:studio` (drizzle-kit studio for ad-hoc inspection).
6. **`lib/db/migrations/0000_initial_auth.sql`** — generated migration. Note: drizzle-kit names migrations 0-indexed (`0000_...`); ADR-0013 D-E mentioned `0001_initial_auth` — off-by-one in prose. Migration creates the four tables with foreign keys + unique indexes on `user.email` + `user.githubLogin`.
7. **`.gitignore`** — `local.db` + `local.db-*` (SQLite WAL / shm files) + `.env.local` + `.env.*.local`.
8. **`.env.example`** — committed contract for all project env vars (Phase-5 LLM CLIs, Phase-6 Discussions, Phase-8 i18n SITE override, Phase-9 GitHub OAuth + Turso credentials). Documents each var's purpose + when it's required.

### Scope NOT in this unit

- **`next-auth@^5` install** — Unit 9.4. The `AdapterAccountType` type cast on `accounts.type` deferred until then.
- **`lib/auth/index.ts`** — Unit 9.4.
- **Watchlist table** — Unit 9.6 schema + `0001_watchlist` migration per ADR-0013 D-E.
- **Local DB seeding** (running `pnpm db:migrate` against `local.db`) — deferred to first developer setup; documented in CHANGELOG.
- **Production Turso provisioning** — Q55 operational; lands when Vercel deploy ships.
- **DB-trigger (a) flip** — fires at Unit 9.6's watchlist write-path (first write surface). This unit lays the DB foundation but doesn't write to it.

### Constraints

- **ADR-0013 D-A**: only `@libsql/client` as DB driver; other drivers forbidden. **D-B**: only `drizzle-orm` as ORM. **D-C**: local-dev = `file:./local.db`. **D-F**: USER-STATE only; content stays file-first.
- **NextAuth.js v5 Drizzle adapter schema**: the four tables must match the canonical column shapes (verified against [authjs.dev/getting-started/adapters/drizzle](https://authjs.dev/getting-started/adapters/drizzle)).
- **Migration discipline** (ADR-0013 D-B): applied migrations are immutable. The single migration this unit lands becomes immutable on commit; future schema changes ship as NEW migrations.
- **Type-clean without `next-auth`**: avoid importing from `next-auth/adapters` until Unit 9.4. Runtime types resolve via plain `text` columns; the v5 adapter narrows at usage time.

### Alternatives considered

- **Land `next-auth` install in this unit** — rejected. Keeps Unit 9.3 narrowly scoped to the DB foundation; Unit 9.4 owns the auth wrapper. Mirrors ADR-0008 pattern (ADR-0008 in Unit 5.1; `@anthropic-ai/sdk` install in Unit 5.3).
- **`dialect: "sqlite"` + `driver: "turso"` (older convention)** — rejected. Drizzle-Kit 0.31.x rejects this with a Zod union error; the modern `dialect: "turso"` is the only valid value for libsql connections.
- **Inline-define the `AdapterAccountType` literal** — rejected. The type lives in `next-auth/adapters` (Unit 9.4); duplicating here forks the contract.
- **Generate migration without a `--name` flag** — rejected. drizzle-kit defaults to a hash-based name (`0000_<adjective>_<noun>.sql`) which is hard to reference. The named `0000_initial_auth.sql` is explicit.

### Edge cases & failure modes

1. **`TURSO_AUTH_TOKEN` set but `TURSO_DATABASE_URL` is `file:./local.db`**: `@libsql/client` accepts the token but ignores it for `file:` URLs. Harmless. Mitigation: `lib/db/index.ts` conditionally passes `authToken` only when defined.
2. **`local.db` doesn't exist on first dev setup**: `@libsql/client` creates it on first connection. New contributors run `pnpm db:migrate` after clone to apply the migration.
3. **drizzle-kit's generated migration uses 0-indexed `0000_...`**: ADR-0013 D-E text said `0001_initial_auth`. Drizzle-Kit overrides the prose; CHANGELOG documents the correction.
4. **Schema diverges from NextAuth.js v5 canonical**: would cause adapter runtime errors at Unit 9.4 wire-up. Mitigation: schema matches the [authjs.dev/getting-started/adapters/drizzle](https://authjs.dev/getting-started/adapters/drizzle) reference shape verbatim except for the project-specific `githubLogin` + `createdAt` additions on `users`.
5. **`drizzle.config.ts` env-resolution at build time**: drizzle-kit reads config at command-invocation time, not Next.js build time. `process.env.TURSO_DATABASE_URL` is whatever the shell has set; falls through to `file:./local.db` in CI / local dev.
